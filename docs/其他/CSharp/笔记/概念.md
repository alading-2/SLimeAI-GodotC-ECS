# C# 核心概念：类型系统与内存管理

## 1. 核心分类：值类型 vs 引用类型

C# 的数据类型分为两大类，这决定了它们在内存中的存储方式、赋值行为以及性能特征。

| **特性**   | **值类型 (Value Types)**                                      | **引用类型 (Reference Types)**                                                   |
| -------- | ---------------------------------------------------------- | ---------------------------------------------------------------------------- |
| **包含内容** | 变量直接包含**实际数据**。                                            | 变量存储的是指向堆中数据的**内存地址（指针/引用）**。                                                |
| **赋值行为** | **深拷贝**：赋值给新变量时，会逐字节复制完整的数据副本。修改副本不影响原件。                   | **浅拷贝**：赋值时只复制内存地址。两个变量指向同一个堆对象。修改一个会影响另一个。                                  |
| **典型类型** | `int`, `float`, `bool`, `struct`, `enum`, `ValueTuple`     | `class`, `interface`, `string`, `array`, `delegate`, `object`, `record` (默认) |
| **内存位置** | 通常在**栈**上（作为局部变量时）。                                        | 实例总是在**堆**上，栈上只存引用地址。                                                        |
| **默认值**  | `0`, `false`, 或结构体字段全为 0。**不可为 null** (除非是 `Nullable<T>`)。 | `null`。                                                                      |

## 1.5 内存复制机制：浅拷贝 vs 深拷贝

当我们将一个变量赋值给另一个变量（或使用 `with` 表达式）时，数据的复制方式决定了它们是否“藕断丝连”。

| 模式       | **浅拷贝 (Shallow Copy)**                                  | **深拷贝 (Deep Copy)**                                        |
| :------- | :------------------------------------------------------ | :--------------------------------------------------------- |
| **定义**   | 只复制对象最外层的“壳”。对于内部的引用类型字段，只复制**引用地址**。                   | 递归复制对象的所有层级。为内部引用的每个对象都创建全新的副本。                            |
| **形象比喻** | **配了一把新钥匙**：你有两把钥匙，但它们打开的是同一个仓库。                        | **盖了一个新仓库**：你不仅有新钥匙，而且仓库里的东西也全部搬了一份到新仓库里。                  |
| **默认行为** | C# 的赋值操作 `=`、`MemberwiseClone()`、`record` 的 `with` 表达式。 | C# **默认不提供**。通常需要手动实现接口 (`ICloneable`) 或通过序列化（JSON/二进制）实现。 |
| **风险**   | 修改副本内部的**引用对象**（如 `List`），原件也会被修改。                      | 性能开销大，内存占用高。                                               |

**内存图解：**

假设对象 A 内部有一个指向 `List` 的引用。
- **浅拷贝后 (B = A)**：A 和 B 是两个不同的栈变量/堆对象，但它们内部的 `List` 引用指向**堆中同一个列表**。
- **深拷贝后 (B = A.DeepCopy())**：堆中会有**两个**独立的列表，A 指向列表 1，B 指向列表 2。

## 2. 深入理解：可变性 vs 不可变性 (Mutability vs Immutability)

在游戏开发和高并发场景中，理解“对象创建后是否可以被修改”非常关键。

### A. 引用类型的特殊情况：String (不可变)

虽然 `string` 是引用类型（存储在堆上），但它具有**不可变性 (Immutability)**。

- **行为：** 当你修改一个字符串时（如 `s = s + "World"`），C# **不会**在原内存修改，而是：
  1. 在堆上分配一块新内存。
  2. 写入新的组合字符串。
  3. 将 `s` 指向新地址。
  4. 旧的字符串成为垃圾（GC 待回收）。
- **游戏开发警示：** 在 `Update` 循环中频繁进行字符串拼接（`+`）会产生大量垃圾，导致 GC 卡顿。应使用 `StringBuilder`。

### B. 结构体 (Struct) 的陷阱

- **默认是可变的：** 但通常建议将 Struct 设计为不可变。
- **防御性复制 (Defensive Copying)：** 如果你有一个很大的 `struct`，在将其标记为 `readonly` 的字段中访问它，或者在 `in` 参数中访问它，编译器为了保证它不被修改，可能会偷偷对其进行完整复制。
- **最佳实践：** 定义结构体时，尽量使用 `readonly struct`。

```csharp
// 推荐：不可变结构体
public readonly struct Vector2
{
    public double X { get; }
    public double Y { get; } // 只能在构造函数赋值
    public Vector2(double x, double y) => (X, Y) = (x, y);
}
```

> **不可变含义：** `readonly struct` 保证**结构体实例本身不可变**——所有字段必须是 `readonly`，实例创建后字段值无法再次修改；但它**不会递归保证引用类型字段指向的对象不可变**。例如，若字段是 `readonly StringBuilder`，仍可调用 `Append` 修改对象内部状态，只是不能让字段再指向另一个实例。

### C. 记录类型 (Record)

C# 9.0 引入，专为不可变数据模型设计。虽然 `record` 默认是引用类型（class-based），但它提供了值语义的比较和非破坏性修改（`with` 表达式）。

```csharp
public record PlayerState(int Hp, int Mp); // 编译器自动生成不可变属性

var s1 = new PlayerState(100, 50);
// s1.Hp = 90; // 报错！不可修改
var s2 = s1 with { Hp = 90 }; // 创建新对象，复用未修改的属性
```

## 3. 栈 (Stack) 和 堆 (Heap)

理解内存区域有助于理解生命周期和性能。

### 栈 (The Stack) - "办公桌"

- **机制：** 后进先出 (LIFO)。
- **用途：** 存储方法调用的执行帧、局部变量、参数。
- **特点：**
  - 分配/释放极快（移动指针即可）。
  - 生命周期确定（出了作用域立即销毁）。
  - 空间较小（递归过深会导致 StackOverflow）。

### 堆 (The Heap) - "仓库"

- **机制：** 任意顺序分配。
- **用途：** 存储长期存在的对象（引用类型的实例）。
- **特点：**
  - 分配/释放较慢（需要寻找空闲块）。
  - 由 **垃圾回收器 (GC)** 管理。
  - 生命周期不确定（直到 GC 运行才释放）。

一句话总结：  
**栈**保存“值类型”和“引用类型的引用”；**堆**只保存“引用类型的真正对象数据”。

### 关键误区纠正

> 误区： "值类型总是在栈上。"
>
> 真相： 值类型存储在声明它的位置。

1. **局部变量：** `void Func() { int a = 10; }` -> `a` 在 **栈** 上。
2. **类成员：** `class Player { int hp; }` -> `Player`实例在堆上，作为其一部分的 `hp` 也在 **堆** 上。
3. **数组元素：** `int[] nums = new int[5];` -> 数组是引用类型，在堆上，里面的 `int` 也在 **堆** 上。

## 4. 性能杀手：装箱与拆箱 (Boxing & Unboxing)

这是面试和性能优化的重灾区。

- **装箱 (Boxing):** 将 **值类型** 转换为 `object` 或接口类型时，系统会在 **堆** 上分配内存，将值复制进去，并返回引用。
  - _代价：_ 内存分配 + 数据复制 + GC 压力。
- **拆箱 (Unboxing):** 将 `object` 强制转回 **值类型**。
  - _代价：_ 类型检查 + 数据复制。

**代码示例：**

```csharp
int i = 123;
object o = i; // 装箱：在堆上创建对象存123，o指向它 (耗费性能)
int j = (int)o; // 拆箱：把堆上的值拷贝回栈给j

// 危险场景：ArrayList (旧版集合)
ArrayList list = new ArrayList();
list.Add(10); // 发生装箱！因为 ArrayList 存的是 object
list.Add(20); // 再次装箱！

// 解决方案：泛型 List<T>
List<int> listGeneric = new List<int>();
listGeneric.Add(10); // 无装箱，性能高
```

## 5. 参数传递：按值 vs 按引用

C# 默认是**按值传递**，但可以通过关键字改变行为。

| **关键字**    | **作用**                                              | **适用场景**                          |
| ---------- | --------------------------------------------------- | --------------------------------- |
| **无 (默认)** | **值类型**复制副本；**引用类型**复制地址（通过地址能改对象内容，但不能让外部变量指向新对象）。 | 绝大多数情况。                           |
| **ref**    | **双向引用**。传入前必须初始化。方法内修改会影响外部变量本身（包括让外部引用指向新对象）。     | 需要修改外部变量的指向，或交换两个变量。              |
| **out**    | **单向输出**。传入前无需初始化。方法内**必须**赋值。                      | 用于返回多个结果（例如 `TryParse`）。          |
| **in**     | **只读引用**。传递引用但不允许修改。                                | 用于传递大型 `struct` 以避免复制开销，同时保证数据安全。 |

## 6. 综合代码演示

```csharp
public class GameCharacter // 引用类型
{
    public string Name;
}

public struct Point // 值类型
{
    public int X, Y;
}

public void Main()
{
    // --- 引用类型行为 ---
    GameCharacter c1 = new GameCharacter { Name = "Hero" };
    GameCharacter c2 = c1; // 复制引用
    c2.Name = "Villain";
    // c1.Name 现在也是 "Villain"

    // --- 值类型行为 ---
    Point p1 = new Point { X = 10, Y = 10 };
    Point p2 = p1; // 复制完整数据
    p2.X = 20;
    // p1.X 仍然是 10

    // --- 字符串的不可变性 ---
    string s1 = "Hello";
    string s2 = s1;
    s1 = "World"; // 创建新字符串 "World"，s1 指向它
    // s2 仍然指向 "Hello"

    // --- 参数传递 ---
    int number = 5;
    ModifyVal(number); // number 还是 5
    ModifyRef(ref number); // number 变成 10

    // --- 元组 (ValueTuple) ---
    (int x, int y) pos = (100, 200); // 栈上分配
}

void ModifyVal(int x) { x = 10; } // 修改的是副本
void ModifyRef(ref int x) { x = 10; } // 修改的是外部原来的变量
```

### 总结建议 (针对独立开发者)

1. **默认用 `class`**：除非你需要存储大量小型数据（如顶点、坐标、复数）且需要极致的内存密度，才考虑 `struct`。
2. **避免装箱**：尽量使用泛型集合 `List<T>`, `Dictionary<K,V>`，避免使用非泛型的 `ArrayList`。
3. **关注字符串**：UI 更新或日志中尽量减少字符串拼接，使用 `StringBuilder` 或插值字符串（虽然插值最终也是拼接，但在某些情况下编译器有优化）。
4. **元组很好用**：多返回值场景优先用 `ValueTuple` 代替 `out` 参数，代码更优雅且通常在栈上无 GC。
