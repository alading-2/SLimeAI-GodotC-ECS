# C# 异常与错误 (Exceptions & Errors)

## 1. 核心机制：什么是异常？

在 C# 中，**异常 (Exception)** 是一个对象（继承自 `System.Exception`）。当程序运行过程中发生了“不该发生”的事情（比如除以零、文件找不到、网络断开），运行时环境 (CLR) 会创建一个异常对象并把它“抛”出来。

这个过程就像接力跑：

1. **抛出 (Throw)**：当前方法说：“我搞不定了，有个错误！”
2. **堆栈展开 (Stack Unwinding)**：CLR 开始在调用堆栈中往回找，看谁能处理这个错误。
3. **捕获 (Catch)**：如果找到了 `catch` 块，错误被处理，程序继续运行。
4. **崩溃 (Crash)**：如果一直退到 `Main` 函数都没人管，程序就会崩溃并打印堆栈信息。

## 2. 文档模块深度解析

根据微软文档的结构，重点包含以下四个维度：

### A. 基础语法 (Try-Catch-Finally)

这是异常处理的“三板斧”：

```C#
FileStream file = null;
try
{
    // 1. 尝试执行可能出错的代码
    file = File.Open("data.txt", FileMode.Open);
    int b = 0;
    int a = 10 / b; // 这里会抛出 DivideByZeroException
}
catch (DivideByZeroException ex)
{
    // 2. 捕获特定异常 (推荐)
    Console.WriteLine("数学不好？不能除以零！");
}
catch (FileNotFoundException ex)
{
    // 2. 捕获另一种特定异常
    Console.WriteLine("文件丢了！");
}
catch (Exception ex)
{
    // 3. 兜底捕获 (不推荐在底层逻辑使用)
    // 除非你是为了记录日志然后再次抛出
    Console.WriteLine($"发生了未知错误: {ex.Message}");

    // ⚠️ 关键点：保留堆栈信息的写法
    throw;      // ✅ 推荐：保持原始事故现场，继续往上抛
    // throw ex; // ❌ 错误：这会重置堆栈信息，让调试者以为错误是在这里发生的
}
finally
{
    // 4. 善后处理 (无论是否报错，这里一定会执行)
    // 通常用于关闭文件、释放内存、断开数据库连接
    if (file != null) file.Close();
}
```

### B. 语法糖：`using` 语句

在 C# 中，`finally` 最常用的场景就是释放资源。C# 提供了一个极其优雅的替代方案 `using`（自动释放）：

```C#
// 出了这个大括号，file 自动调用 Dispose() 释放资源，相当于自动写了 finally
using (var file = File.Open("data.txt", FileMode.Open))
{
    // 读写操作...
}
```

### C. 异常的属性

捕获到 `ex` 后，有三个属性最重要：

1. **Message**: 错误消息（给“人”看的，比如 "Could not find file"）。
2. **StackTrace**: 堆栈跟踪（给“程序员”看的，指明错误发生在第几行）。
3. **InnerException**: 内部异常（套娃）。当一个异常引发了另一个异常时，原始异常放在这里。

## 3. 深度思考：异常使用的哲学

这是初学者和资深开发者的分水岭：

### 1. 异常 = 性能杀手？

- **是的。** 抛出异常涉及到挂起线程、快照堆栈、回溯调用栈等繁重操作。
- **原则**：**不要用异常来控制业务逻辑。**
  - ❌ 坏味道：`try { int i = int.Parse(str); } catch { return 0; }`
  - ✅ 好味道：`if (int.TryParse(str, out int i)) { ... } else { return 0; }`
  - **解释**：用户输入错误是“意料之中”的事，应该用 `if` 判断；而硬盘突然坏了是“意料之外”的事，才应该用异常。

### 2. 为什么要有自定义异常？

当系统自带的 `ArgumentException` 无法准确描述你的业务错误时（比如“玩家金币不足异常”），你应该创建自定义异常。

```C#
public class InsufficientGoldException : Exception
{
    public InsufficientGoldException(int current, int required)
        : base($"金币不足。持有: {current}, 需要: {required}") { }
}
```

这样上层调用者可以专门 `catch (InsufficientGoldException)` 并弹出充值窗口，而不是弹出一个通用的“系统错误”。

## 4. 横向对比：C# vs TypeScript vs Python

这三种语言在异常处理上逻辑相似，但细节和文化有很大差异。

| **特性**         | **C#**                                      | **TypeScript (JS)**                                        | **Python**                                                                                         |
| ---------------- | ------------------------------------------- | ---------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| **关键字**       | `try` / `catch` / `finally` / `throw`       | `try` / `catch` / `finally` / `throw`                      | `try` / **`except`** / `finally` / **`raise`**                                                     |
| **抛出类型**     | 必须是 `System.Exception` 或其子类 (规范上) | **极其灵活** (可以抛出字符串、数字、对象) `throw "Error"`  | 必须是 `BaseException` 的子类                                                                      |
| **捕获变量类型** | **强类型**。`catch (IOException ex)`        | **Unknown/Any**。需手动判断 `if (e instanceof Error)`      | **强类型**。`except ValueError as e:`                                                              |
| **特有语法**     | `using` 语句 (自动释放资源)                 | 无 (依赖 GC 或手写 try-finally)                            | **`else` 块** (没报错时执行)、**`with` 语句** (上下文管理器，类似 C# using)                        |
| **重抛出**       | `throw;` (保留堆栈)                         | `throw e;` (JS 堆栈处理较特殊)                             | `raise` (不带参数，保留堆栈)                                                                       |
| **文化差异**     | **严谨**。非常依赖特定类型的 catch。        | **混乱**。因为能抛任意东西，很多库只返回 `null` 而不抛错。 | **EAFP** (Easier to Ask for Forgiveness than Permission)。Python 鼓励“先斩后奏”，多用 try-except。 |

### 关键差异总结：

1. TypeScript 的“盲盒”捕获：

   在 TS 中，catch (e) 里的 e 可能是个 Error 对象，也可能是个字符串，甚至是 null。你通常需要写很多防御性代码：

   ```TypeScript
   try { ... }
   catch (e) {
       if (e instanceof Error) console.log(e.message);
       else console.log("Unknown error", e);
   }
   ```

   而在 C# 中，`catch (Exception ex)` 保证了 `ex` 一定有 `Message` 和 `StackTrace`。

2. Python 的 else 块：

   Python 允许 try-except-else。如果 try 里没报错，执行 else。

   C# 没有这个语法，只能把代码写在 try 块的最后，或者 try-catch 之后。

3. 资源管理：

   C# 的 using 和 Python 的 with 是异曲同工之妙，都是为了解决“忘记关闭文件/锁”的问题。TS 目前主要靠手动 finally。

## Godot 开发者特别提示

在 Godot C# 开发中：

1. **不要让异常逃逸到引擎层**：如果你的 C# 脚本抛出异常且没被捕获，Godot 游戏通常不会直接闪退（编辑器内），但会在调试器里打印红色的 StackTrace，并且**当前帧的后续逻辑会中断**。这可能导致游戏状态不一致（比如扣了钱但没发道具）。
2. **信号 (Signal) 中的异常**：如果异常发生在信号的回调函数里，它很难被外部捕获。务必在回调函数内部处理好异常。
3. **调试**：Godot 的 `GD.PushError("Msg")` 是一种“软错误”，它不会中断程序，只是在控制台报错。对于非致命错误，优先用这个，而不是 `throw new Exception()`。
