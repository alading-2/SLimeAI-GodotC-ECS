# 类和对象 (Class & Object) —— 核心机制

在 C# 和 Godot 开发中，**类**是构建世界的蓝图，**对象**是世界里的实体，而**继承与重写**则是让世界丰富多彩的规则。

## 1. 本质区别：图纸 vs 实物

| 概念              | 别名            | 本质                        | 存在位置          | 数量      |
| :---------------- | :-------------- | :-------------------------- | :---------------- | :-------- |
| **类 (Class)**    | 类型 / 模具     | **定义的规则** (属性、方法) | 代码文件 (`.cs`)  | 全局 1 份 |
| **对象 (Object)** | 实例 (Instance) | **具体的数据** (内存中的块) | 运行时内存 (Heap) | 可有 N 份 |

### 代码视角

```csharp
// 1. 定义类 (画图纸)
public class Enemy
{
    public int Hp;
}

// 2. 实例化 (造实物)
// 'new' 是实例化的开关
var slime = new Enemy();
var orc = new Enemy();

// slime 和 orc 是同一个"类"，但它们是两个独立的"对象"。
```

## 2. 继承 (Inheritance)

继承建立了 "Is-A" (是一个) 的关系。

在 Godot 中，所有的脚本本质上都在进行继承：

Slime : Enemy : CharacterBody2D : Node2D : Node : GodotObject

```C#
// 语法：class 子类 : 父类
public partial class Slime : Monster
{
    // Slime 自动拥有 Monster 的所有 Public/Protected 属性和方法
}
```

## 3. 核心机制：Virtual 与 Override (重写)

这是 C# 这种强类型语言最严谨的地方。**子类想要修改父类的行为，必须经过父类同意。**

### 🔑 黄金法则

1. **父类点头 (`virtual`)**：父类必须在方法前加 `virtual`，表示：“这个方法如果你不满意，可以改。”
2. **子类动手 (`override`)**：子类必须在方法前加 `override`，表示：“我要覆盖父类的这个逻辑。”

### ❌ 错误示范

```C#
// 父类
public class Monster {
    public void Attack() { ... } // 没加 virtual，默认是"封死"的
}

// 子类
public class Slime : Monster {
    // ❌ 报错！父类没允许你重写 Attack
    public override void Attack() { ... }
}
```

### ✅ 正确示范 (Godot 场景)

```C#
// 父类：通用怪物
public partial class Monster : CharacterBody2D
{
    // ✅ 允许子类重写受伤逻辑
    public virtual void TakeDamage(int amount)
    {
        GD.Print("怪物发出了通用的惨叫");
    }
}

// 子类：史莱姆
public partial class Slime : Monster
{
    // ✅ 重写：史莱姆受伤有特殊表现
    public override void TakeDamage(int amount)
    {
        // base.TakeDamage(amount); // 可选：如果想保留父类的通用惨叫，就保留这行
        GD.Print("史莱姆分裂成了两个！");
    }
}
```

## 4. Godot 中的终极重写：生命周期

你在 Godot 中写的最多的代码，其实就是在 **Override** 引擎的方法。

Godot 引擎的 C# 绑定层 (`GodotObject`) 已经把 `_Ready`, `_Process` 等核心方法定义为了 `virtual`。

```C#
public partial class Player : Node2D
{
    // ✅ 这就是一个标准的 Override
    // 你正在覆盖 Node 类的 _Ready 方法
    public override void _Ready()
    {
        GD.Print("玩家加载完毕");
    }

    // ✅ 如果你不写 override，编译器会警告你隐藏了父类成员
    // 虽然能跑，但极不推荐，容易产生歧义
    // public void _Ready() { ... }
}
```

## 5. TS/Lua 开发者直觉对比

| **场景**     | **TypeScript / Lua (灵活派)**                | **C# (严谨派)**                                     |
| ------------ | -------------------------------------------- | --------------------------------------------------- |
| **覆盖方法** | **默认允许**。子类直接写个同名函数就覆盖了。 | **默认禁止**。必须显式声明 `virtual` / `override`。 |
| **调用父类** | `super.func()` (TS)                          | `base.Func()`                                       |
| **设计哲学** | 方便优先。假设你知道自己在干嘛。             | 安全优先。防止你"不小心"覆盖了核心逻辑。            |

### 深度思考：为什么 C# 要这么麻烦？

这是一种契约保护。

如果一个第三方库的核心方法没有标记 virtual，说明作者不希望你修改它的内部逻辑，以保证库的稳定性。C# 强迫你显式确认“我是故意要改的”，从而减少了大规模项目中的 Bug。
