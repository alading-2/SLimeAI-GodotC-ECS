# TestSystem 重构方案

## 1. 背景

当前 `TestSystem` 已经具备基础模块化能力，但在真实运行场景下暴露出明显问题：

- 属性模块对高频 `PropertyChanged` 缺少过滤和合帧
- UI 默认以“整页重建”为主，而不是“增量更新”
- 模块生命周期虽然存在，但激活态管理不彻底
- UI 目录拆分粒度没有统一原则，后续继续扩展会越来越散

这些问题在“玩家被默认选中 + 造成多段或多目标伤害”时会被放大，最终表现为面板卡顿、节点 churn、维护成本升高。

本方案目标不是做局部修补，而是把 `TestSystem` 调整为一套可长期演进的运行时测试框架。

---

## 2. 重构目标

## 2.1 核心目标

- 把 `TestSystem` 从“调试面板集合”升级为“通用测试宿主”
- 保持 ECS 的高内聚、低耦合、事件驱动、数据驱动风格
- 让新增测试模块变得低成本、低侵入、可删除
- 把高频 UI 刷新从整页重建改为增量 patch
- 固化模块、共享控件、服务层、宿主层的职责边界

## 2.2 结果目标

重构完成后应达到：

- 高频数据变化时，UI 仍可稳定运行
- 新增模块基本不需要修改宿主主流程
- 共享控件可复用，但不会拆到过碎
- 代码目录和文档目录都能支撑后续继续扩展

---

## 3. 当前问题拆解

## 3.1 一级问题：刷新模型错误

当前属性模块的问题不是“代码有点丑”，而是**刷新策略本身错误**。

### 现状

- 订阅当前实体的全部属性变化
- 任意属性变化都调用 `Refresh()`
- `Refresh()` 清空所有行，再重新创建全部编辑器节点

### 后果

- 高频场景下触发大量 `QueueFree`
- 同帧连续实例化大量 UI 节点
- 容器布局不断重算
- 明明只变了一个值，却把整页重建一遍

### 正确方向

- 区分“结构变化”和“数值变化”
- 结构变化才允许重建
- 数值变化只更新对应行
- 同一帧多次变化合并成一次 patch

## 3.2 一级问题：生命周期管理不彻底

当前模块虽然有：

- `OnSelectedEntityChanged`
- `OnActivated`
- `OnDeactivated`

但还没有形成真正严格的运行规则。

### 现状问题

- 非当前模块仍可能收到实体变更广播
- 整个面板隐藏后模块不一定真正停掉
- 订阅与刷新门禁分散在各模块中，各自判断

### 正确方向

- 宿主统一管理“谁是当前激活模块”
- 宿主统一发出“进入激活 / 进入休眠”信号
- 模块只在激活态下订阅高频事件

## 3.3 二级问题：UI 组织粒度不稳定

目前“一个小 UI 一个场景”的倾向需要收敛。

### 现状问题

- 复合控件和超薄包装控件都被同等对待
- 目录越拆越碎，但语义边界不清晰
- 小场景数量变多后，维护成本高于复用收益

### 正确方向

- 复合条目保留场景化
- 轻薄输入器统一代码生成或统一编辑器工厂
- 模块私有控件放模块目录，共享控件才进入 `Shared`

## 3.4 二级问题：宿主与服务层边界需要再强化

当前 `FeatureDebugService` 和 `AbilityTestService` 的方向是好的，但还可以更清晰：

- UI 只发出操作意图
- 服务层负责调用正式系统
- 模块负责把服务结果映射回界面

这样后续加新模块时不会把业务逻辑再塞回 UI。

## 4.2 角色边界

| 角色 | 负责内容 | 不负责内容 |
|------|----------|------------|
| `TestSystem` | 宿主、模块树、模块切换、实体选择、面板显隐、局部事件广播 | 具体模块渲染逻辑 |
| `TestModuleBase` | 定义标准生命周期和上下文约束 | 具体业务细节 |
| 模块 | 本模块事件订阅、视图构建、局部更新 | 正式系统底层逻辑 |
| `Shared` 控件 | 跨模块复用的复合控件 | 具体模块业务 |
| `Services` | 正式链路适配与数据装配 | 直接渲染 UI |

## 5. 关键重构点

## 5.1 简化刷新与模块路径注册

当前结论是不再保留宿主级刷新调度器。刷新由模块内部直接处理，模块只需要把变化分成两类：

- 结构变化：重建当前模块局部 UI
- 数值或状态变化：patch 受影响控件

### 价值

- 避免简单刷新动作绕过多层协议
- 模块刷新逻辑更容易读懂和调试
- 宿主专注模块切换、选择实体和模块树导航

当前已落地：

- `ITestModule` / `ITestModuleContext` 作为宿主与模块之间的统一协议
- `TestModuleDefinition` 统一描述模块 `Id / ModulePath`
- `TestModulePath` / `TestModuleGroupId` 支撑点分路径分组
- `TestSelectionContext` 只维护选中实体状态
- `TestSystem.Events` 统一广播 TestSystem 局部事件
- `TestModuleContext` 统一注入宿主 / 选择上下文
- `TestModuleBase` 保留简单 `Active / Inactive` 激活态

## 5.1.1 描述驱动的模块注册

当前宿主按场景子节点顺序注册模块，并按 `ModulePath` 自动生成左侧模块树：

- `TestModuleDefinition.Id`
- `TestModuleDefinition.ModulePath`

价值：

- 模块路径天然支持多级分组
- 宿主可以按稳定 Id 切换模块
- 后续做“记住上次打开模块”“模块过滤”时有稳定主键

## 5.2 属性模块改为行缓存模型

建议把属性模块改为：

- 当前分类有一份 `Dictionary<string, AttributeRowPresenter>`
- 每个 `DataKey` 对应一行
- `BuildView()` 时一次性构建当前分类
- `PatchView(dirtyKeys)` 时只更新对应行

### 只在以下情况重建

- 切换分类
- 切换实体
- 当前分类列表结构本身变化

### 其他变化只 patch

- 数值变化
- 布尔切换
- 文本变化
- 临时 modifier 数值变化

## 5.3 技能模块改为“结构重建 + 状态更新”双模式

技能模块不需要追求完全零重建，但要区分：

- 技能增删：允许重建分组结构
- 启用状态变化：优先更新对应条目状态
- 右键菜单、选中态、提示文本：就地 patch

这会比当前“所有变化都整列表刷新”更稳。

## 5.4 强化模块激活态

建议引入明确状态：

- `Initialized`
- `Active`
- `Inactive`
- `Suspended`

### 规则

- 面板隐藏时，当前模块进入 `Suspended`
- 切模块时，旧模块 `Inactive`，新模块 `Active`
- 只有 `Active` 模块可以持有高频订阅

## 5.5 收敛 UI 拆分标准

建议固定标准：

### 保留为独立场景

- 区块
- 卡片
- 复合行
- 共享空状态

### 收敛为代码构建或轻量编辑器

- 单个 `SpinBox`
- 单个 `LineEdit`
- 单个 `CheckButton`
- 纯样式薄包装

这样既保留复用价值，又避免目录过碎。

## 6. 推荐实施阶段

## 阶段一：止血

目标：先解决当前最明显的卡顿与重复刷新。

工作项：

- 属性模块改成脏 key 合并刷新
- 当前分类行缓存，不再每次整页重建
- 宿主隐藏面板时统一停更
- 清理重复 `Refresh()`

交付结果：

- 多目标命中时不会因属性页整页重建明显卡顿
- 当前宿主生命周期更可控

## 阶段二：结构重构

目标：把代码从“能跑”调整为“可长期维护”。

工作项：

- 抽出 `Core / Modules / Shared / Services`
- 模块刷新保持简单直接，结构变化重建，状态变化 patch
- 模块目录按边界收敛
- 合并过碎的小输入器场景

交付结果：

- 新模块接入成本显著下降
- UI 文件组织更稳定

## 阶段三：平台化

目标：让 `TestSystem` 真正成为通用测试平台。

工作项：

- 为新模块提供模板化接入规范
- 支持更多模块类型
- 明确模块能力边界和文档模板
- 建立统一的共享控件和编辑器库

交付结果：

- 后续扩展 Buff / AI / Damage / EventTrace 模块时，不再需要临时拼结构

## 7. 新模块接入模板

新增模块时建议遵循：

### 目录

- `Modules/MyModule/MyTestModule.cs`
- `Modules/MyModule/Views/MyTestModule.tscn`
- `Modules/MyModule/Views/...`

### 最小职责

- 订阅什么
- 构建什么
- 如何 patch
- 如何释放

### 不要做的事

- 直接修改宿主流程
- 把正式业务逻辑写进控件
- 默认整页刷新
- 为一个极薄输入器再新建一堆小场景

---

## 8. 验收标准

重构后的 `TestSystem` 应满足以下标准：

- 属性测试在高频变化时只更新受影响行
- 技能测试在启停变化时不整页刷新
- 面板隐藏后测试模块不再继续高频空转
- 新增模块不需要修改宿主核心代码
- 目录结构能清晰区分模块私有与共享控件
- 文档能够指导后续继续扩展，而不是只描述当前实现

---

## 9. 推荐结论

`TestSystem` 当前不适合继续在原结构上无限加功能。  
正确路线不是继续打补丁，而是先统一：

- 宿主生命周期
- 刷新模型
- UI 拆分原则
- 模块目录边界

这些基础规则一旦定下来，后续扩展才会轻松；如果这些规则不先收住，模块越多，维护成本只会指数式上升。
