# 系统生命周期与项目状态设计

> 文档类型：正式设计文档  
> 适用范围：`System Core`、`AutoLoad` 协同、项目级状态、运行时系统启停  
> 当前状态：第一版已落地

---

## 1. 最终决策

本项目正式采用：

> `方案 C+：统一系统注册表 + 分域项目状态 + 栈式实体状态效果`

它来源于 [系统生命周期三案设计.md](./系统生命周期三案设计.md) 中的方案 C，但这次不再停留在“折中思路”，而是已经落实为明确的运行模型。

核心判断：

- 不继续扩散 `AutoLoad + 静态初始化`。
- 不把项目级状态和实体状态塞进一个大状态机。
- 统一由 `SystemRegistry + SystemManager` 管系统生命周期。
- 统一由 `ProjectStateService` 管项目级运行状态。

---

## 2. 目标

这套设计需要同时满足：

1. 新系统必须能统一注册、创建、启用、禁用。
2. 系统是否运行必须能被项目状态统一裁决。
3. 运行中要能轻松开关系统，而不是只能重启生效。
4. 兼容现有 `AutoLoad` 体系，允许渐进迁移。
5. 利用 Godot 节点生命周期与 `ProcessMode`，但不把“只停 `_Process`”误当成完整禁用。

---

## 3. 生命周期语义

### 3.1 单系统语义

| 术语 | 含义 |
| --- | --- |
| `Add` | 系统实例被创建，并纳入 `SystemManager` 管理 |
| `Remove` | 系统从管理器注销；若是 `Node`，同时离开场景树 |
| `Enable` | 系统被允许参与运行资格判断 |
| `Disable` | 系统被显式关闭，不再运行 |
| `Run` | 同时满足 `Enable == true` 且 `RunCondition == true` |

第一版公开接口重点是：

- `EnsureSystem(...)`
- `EnableSystem(systemId)`
- `DisableSystem(systemId)`
- `Resolve<T>()`

`Remove` 语义在内部已经留出边界，但第一版没有把“批量热卸载系统”做成主能力。

### 3.2 为什么“禁用”不等于“只关 Process”

过去的问题之一是：

- 有的系统停了 `_Process`
- 但全局事件还在收
- 定时器、外部引用、缓存状态仍在继续工作

现在的要求是：

> 禁用必须经过统一入口，并同步处理事件订阅/退订或运行钩子。

对 `Node` 系统，`SystemManager` 会统一设置 `ProcessMode`。  
对实现了 `ISystemRuntime` 的系统，还会统一走：

- `OnSystemRegistered`
- `OnSystemEnabled`
- `OnSystemDisabled`
- `OnProjectStateChanged`
- `OnSystemUnregistered`

## 4. 核心类型

### 4.1 `SystemDescriptor`

文件：

- `Src/ECS/Base/System/Core/Runtime/SystemDescriptor.cs`

职责：

- 描述系统唯一 Id
- 描述运行形态 `SystemKind`
- 描述生命周期域 `SystemLifetime`
- 描述默认启用状态
- 描述依赖系统
- 描述运行条件
- 提供实例工厂 `Factory`

### 4.2 `SystemRegistry`

文件：

- `Src/ECS/Base/System/Core/Runtime/SystemRegistry.cs`

职责：

- 统一保存系统描述符
- 拒绝重复 `SystemId`
- 供 `SystemManager` 启动时批量读取

现在的新系统注册标准是：

- 使用 `[ModuleInitializer]`
- 在静态初始化里调用 `SystemRegistry.Register(...)`

而不是继续走 `AutoLoad.Register(...)`。

### 4.3 `SystemManager`

文件：

- `Src/ECS/Base/System/Core/Godot/SystemManager.cs`

职责：

- 创建各 `SystemLifetime` 对应的宿主节点
- 负责实例创建与依赖展开
- 统一启用/禁用系统
- 监听 `ProjectStateService.StateChanged`
- 根据 `SystemRunCondition` 重新计算系统是否允许运行
- 对 `Node` 系统设置 `ProcessMode`
- 对 `ISystemRuntime` 系统分发生命周期钩子

### 4.4 `ProjectStateService`

文件：

- `Src/ECS/Base/System/Core/Runtime/ProjectStateService.cs`

职责：

- 维护项目级完整快照 `ProjectStateSnapshot`
- 支持状态切换前、中、后事件
- 提供四个状态域的单独切换接口

### 4.5 `SystemRunCondition`

文件：

- `Src/ECS/Base/System/Core/Runtime/SystemRunCondition.cs`

职责：

- 以结构化条件声明“系统允许在哪些项目状态下运行”
- 第一版支持：
  - `AllowedAppPhases`
  - `AllowedSessionPhases`
  - `AllowedOverlayPhases`
  - `BlockedOverlayPhases`
  - `AllowedExecutionPhases`

这意味着系统不再直接散落手写：

```csharp
if (gamePaused || !inGame || overlayOpen)
{
    return;
}
```

而是把运行资格前置到统一调度层。

## 5. 项目级状态模型

### 5.1 四个状态域

#### `AppPhase`

回答应用主流程处在哪一段。

当前枚举：

- `Boot`
- `FrontEnd`
- `InSession`
- `ShuttingDown`

#### `SessionPhase`

回答这一局内的会话流程。

当前枚举：

- `None`
- `Preparing`
- `Playing`
- `Resolving`
- `Ended`

#### `OverlayPhase`

回答是否有覆盖层在接管交互或表现。

当前枚举：

- `None`
- `PauseMenu`
- `ModalUi`
- `Cutscene`
- `DebugOverlay`

#### `ExecutionPhase`

回答当前是否处于真正运行、暂停、单步或阻塞状态。

当前枚举：

- `Running`
- `Paused`
- `Step`
- `Blocked`

### 5.2 为什么要分域

如果只用一个大枚举，状态会很快膨胀成：

- `InGamePausedByMenu`
- `InGamePausedByModal`
- `InGameCutscene`
- `InGameDebugOverlay`

这会把并行维度硬编码进单值状态，扩展性很差。

分域模型的意义是：

- 菜单/局内流程是一维
- 覆盖层是一维
- 执行暂停语义又是一维

系统只关心自己依赖哪几个域，不必理解整个项目所有状态组合。

## 6. 与 Godot 的协作方式

### 6.1 `AutoLoad` 的新定位

文件：

- `Src/Autoload/AutoLoad.cs`

现在的 `AutoLoad` 只承担两件事：

1. 继续兼容已有启动注册。
2. 在启动阶段创建 `SystemManager`，把 `SystemRegistry` 中的新系统接管进来。

它不再是新系统的标准扩展点。

### 6.2 `ProcessMode` 只是一部分

当托管实例是 `Node` 时，`SystemManager` 会按运行资格切：

- `ProcessModeEnum.Inherit`
- `ProcessModeEnum.Disabled`

但这只是节点处理循环的控制，不代表完整语义。

所以每个需要事件订阅/退订的系统，仍然应在 `ISystemRuntime` 钩子里自己完成：

- 开启时订阅
- 关闭时反订阅
- 关闭时清理内部运行态

## 7. 已落地的迁移模式

### 7.1 `InitAction` 型系统迁移

示例：

- `Src/ECS/Base/System/TargetingSystem/TargetingManager.cs`
- `Src/ECS/Base/System/TargetingSystem/TargetingManagerRuntime.cs`

做法：

- 保留原有静态能力与业务逻辑
- 抽出 `EnableRuntime()` / `DisableRuntime()`
- 新增一个桥接 `Node`，实现 `ISystemRuntime`
- 在桥接类的 `[ModuleInitializer]` 中注册 `SystemDescriptor`

这类系统适合先迁移，因为风险低，边界最清晰。

### 7.2 纯静态事件系统迁移

示例：

- `Src/ECS/UI/UI/DamageNumberUI/DamageNumberSystem.cs`
- `Src/ECS/UI/UI/DamageNumberUI/DamageNumberRuntimeBridge.cs`

做法：

- 静态逻辑保留
- 把“何时订阅/何时退订”转移到运行时桥接层
- 让系统启停不再依赖进程启动时机

### 7.3 旧 `AutoLoad + Node` 系统

这类系统第一版没有强制一次性全迁。

迁移原则是：

- 仍由 `AutoLoad` 承载的旧系统先继续跑
- 新系统一律走 `SystemRegistry`
- 老系统后续按模块逐步迁入 `SystemManager`

## 8. 第一版实施规范

### 8.1 新系统接入规范

新系统默认规则：

1. 用 `SystemDescriptor` 注册，不再直接 `AutoLoad.Register(...)`
2. 明确 `SystemId`
3. 明确 `SystemKind`
4. 明确 `SystemLifetime`
5. 明确 `RunCondition`
6. 如果系统涉及事件订阅，必须实现统一启停钩子

### 8.2 生命周期域建议

| 生命周期域 | 典型系统 |
| --- | --- |
| `Persistent` | 全局服务、广域工具、需全程常驻但可按状态禁用的系统 |
| `Gameplay` | 一局内主要业务系统 |
| `Overlay` | 暂停菜单、调试覆盖层、模态 UI 辅助系统 |
| `Debug` | 只在调试环境下需要的系统 |
| `Test` | 运行时测试系统 |

### 8.3 运行条件建议

典型规则示例：

- Gameplay 系统：
  - `AppPhase == InSession`
  - `SessionPhase == Playing`
  - `ExecutionPhase == Running`
  - `OverlayPhase != PauseMenu`
- Overlay 系统：
  - `OverlayPhase == PauseMenu` 或 `DebugOverlay`
- Persistent 基础服务：
  - `SystemRunCondition.Always`

## 10. 运行时启停现在能做到什么

现在已经支持：

- 系统注册后自动接管
- 运行时 `EnableSystem/DisableSystem`
- 状态变更后自动重算系统运行资格
- 旧 `AutoLoad` 与新 `SystemRegistry` 双轨共存

第一版暂未完成：

- 大规模老系统迁移清单
- 统一的系统配置资源化开关
- 大量 gameplay 系统按状态域批量切换
- 更复杂的条件表达式 DSL
