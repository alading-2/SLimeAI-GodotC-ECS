# 系统生命周期三案设计（System Core）

## 1. 背景与目标

当前项目的系统形态并不统一：

- 一部分是 `AutoLoad + Node/.tscn`（如 `SpawnSystem`、`RecoverySystem`、`TestSystem`）。
- 一部分是 `AutoLoad + InitAction`（如 `TargetingManager`）。
- 一部分是静态系统通过 `[ModuleInitializer]` 直接订阅事件（如 `DamageNumberSystem`）。

这会导致“系统是否创建、是否启用、是否运行、是否可在运行时切换”语义分散，难以做统一生命周期治理。

本设计文档给出三个可选方案，统一围绕以下核心诉求：

1. 系统生命周期统一为：`添加 / 删除 / 启用 / 禁用 / 运行`。
2. 支持全局阶段治理：`游戏外阶段` 与 `游戏运行阶段`（后续可扩展）。
3. 优先利用 Godot 原生节点生命周期与 `ProcessMode` 能力。
4. 允许在运行中平滑开关系统，而不是只能重启生效。

---

## 2. 共识术语

### 2.1 生命周期（单系统）

- `添加`：系统实例被创建并挂入场景树。
- `删除`：系统实例从场景树移除并释放。
- `启用`：系统被标记为允许工作（可接收事件/可处理逻辑）。
- `禁用`：系统被标记为暂停工作（拒绝处理逻辑）。
- `运行`：系统满足“已添加 + 已启用 + 当前阶段允许”的条件，实际执行更新或消费事件。

### 2.2 全局阶段（全局）

- `OutGame`：游戏外阶段（启动后、主菜单、配置、调试准备等）。
- `InGame`：游戏运行阶段（一局内战斗、波次、计时、掉落等）。

> 说明：`Paused` 可以作为 `InGame` 子状态，也可以升级为独立阶段；三案会分别给出处理方式。

---

## 3. 方案A：固定 3~4 阶段 + 全节点化（推荐首选）

### 3.1 核心思想

先把所有“有生命周期的系统”统一为 `Node/.tscn`，并引入固定阶段枚举（建议 `Boot/Menu/InGame/Pause` 或 `OutGame/InGame/Pause`）。  
系统声明“在哪些阶段允许运行”，阶段切换由统一核心调度。

### 3.2 架构要点

1. `SystemLifecycleCore`（新核心）
- 维护当前全局阶段。
- 对系统做统一 `Add/Remove/Enable/Disable` 调度。
- 在阶段切换时触发系统 `OnPhaseEnter/OnPhaseExit` 钩子。

2. `ISystemLifecycle`（统一协议）
- 系统最小协议示例：
  - `SystemId`
  - `DefaultEnabled`
  - `SupportedPhases`
  - `OnSystemEnabled/OnSystemDisabled`
  - `OnPhaseEnter/OnPhaseExit`

3. `SystemSwitchConfig`（配置层）
- 放在 `Data/Config`，用于定义系统默认是否启用、允许阶段、是否允许运行时切换。
- 可保留“生成 C# 配置”路径，但建议配置语义对齐生命周期协议，而不是只给一个布尔值。

4. 节点级运行控制
- 启用/禁用优先通过系统统一入口触发。
- 具体到 Godot 节点时，统一控制：
  - `ProcessMode`
  - `SetProcess/SetPhysicsProcess`
  - 事件订阅/反订阅
- 避免“只关 Process 但事件仍在消费”的半禁用状态。

### 3.3 生命周期语义映射

- `添加`：实例化 `.tscn` + `AddChild` + 建立注册表索引。
- `删除`：从注册表移除 + `QueueFree`（统一由核心触发）。
- `启用`：`Enabled = true`，恢复订阅/处理。
- `禁用`：`Enabled = false`，暂停订阅/处理。
- `运行`：`Enabled == true && 当前阶段 in SupportedPhases`。

### 3.4 优点

1. 概念清晰，成本可控，易落地。
2. 与 Godot 节点生态天然一致。
3. 足够覆盖“游戏外/游戏内/暂停”大多数需求。
4. 可逐步迁移，不要求一次重写所有系统。

### 3.5 风险

1. 阶段是固定枚举，后续复杂流程扩展需要改核心。
2. 对“多维并行状态”（如 `InGame + OverlayDebug + Cutscene`）表达能力一般。

### 3.6 适用结论

最适合作为第一阶段统一方案。

## 4. 方案B：完全状态机化（多维状态 + 声明式运行条件）

### 4.1 核心思想

不仅统一系统生命周期，还把“全局阶段”抽象成可扩展状态机框架：支持状态图、转移规则、多维状态、声明式运行条件。  
系统不直接写 `if (InGame)`，而是声明“在什么条件下运行”。

### 4.2 架构要点

1. `AppStateMachine`
- 管理状态节点与合法转移（例如 `Boot -> Menu -> InGame -> Pause -> InGame`）。
- 支持 `OnEnter/OnExit` 回调。

2. `StateDomains`（状态域）
- 支持多维状态并存，例如：
  - 主域：`AppPhase`（Boot/Menu/InGame）
  - 叠加域：`Overlay`（None/PauseMenu/TestPanel）
  - 可选域：`CombatSubPhase`（Prepare/Wave/Reward）

3. `SystemRunCondition`
- 每个系统声明运行条件表达式，例如：
  - `AppPhase == InGame`
  - `AppPhase == InGame && Overlay != PauseMenu`
  - `Overlay == TestPanel`

4. 生命周期融合
- 系统生命周期与状态机统一调度：
  - 状态变更 -> 条件重算 -> 批量启停系统
  - 保证一致性，不依赖各系统手写判断

### 4.3 优点

1. 表达能力最强，适合中长期大型演进。
2. 可以避免状态枚举膨胀。
3. 对复杂玩法流（战斗内子阶段、叠加调试态、过场态）更友好。

### 4.4 风险

1. 架构复杂度明显高于方案A。
2. 首次落地成本高，设计不稳容易过度工程化。
3. 团队必须持续维护状态图与条件规则，否则会形成新复杂度。

### 4.5 适用结论

适合“核心框架成熟后”的二阶段升级，不建议直接作为第一步。

---

## 5. 方案C：分层折中（外显两阶段，内核预留状态机扩展）

### 5.1 核心思想

对外先提供简单模型（`OutGame / InGame` + `Enable/Disable`），保持学习成本低；  
内核接口按状态机能力设计，后续无破坏升级到方案B。

### 5.2 架构要点

1. 外部 API 简化
- 系统声明先只写：
  - `DefaultEnabled`
  - `AllowedMainPhases`（OutGame/InGame）
- 先不暴露复杂条件表达式。

2. 内部结构前置扩展点
- 预留：
  - 可扩展 `PhaseDomain`
  - `RunCondition` 接口
  - `OnEnter/OnExit` 生命周期钩子
- 但第一版不强制全部实现。

3. 渐进式收敛
- 第一阶段：完成“全节点化 + 主阶段开关”。
- 第二阶段：把 `Pause`、`Overlay` 等提升为独立域。

### 5.3 优点

1. 短期可落地，长期可扩展。
2. 用户侧（策划/开发）心智负担小。
3. 可以边迁移边验证，而不是一次性重构。

### 5.4 风险

1. 文档和实现必须明确“哪些是当前生效能力，哪些是预留能力”。
2. 如果后续不推进，可能长期停在“半抽象”状态。

### 5.5 适用结论

适合当前项目：既要快速统一生命周期，又不想堵死未来状态机化路径。

---

## 6. 三案对比（决策视角）

| 维度 | 方案A 固定阶段 | 方案B 完全状态机化 | 方案C 分层折中 |
|---|---|---|---|
| 首次落地成本 | 低~中 | 高 | 中 |
| 认知复杂度 | 低 | 高 | 中 |
| 短期收益 | 高 | 中 | 高 |
| 长期扩展性 | 中 | 高 | 高 |
| 迁移风险 | 低 | 中~高 | 中 |
| 对当前项目匹配 | 高 | 中 | 高 |

---

## 7. 推荐路径（执行顺序）

1. 首先采用方案C（折中）作为实施路径。
2. 第一步按方案A的方式落地：
- 统一节点化系统形态（静态系统逐步迁入 Node）。
- 建立统一生命周期协议与调度核心。
- 先跑通 `OutGame/InGame`（`Pause` 先作为可选扩展）。
3. 第二步按方案B能力增量扩展：
- 引入状态域与运行条件表达。
- 将复杂系统逐步切换到声明式运行条件。

---

## 8. 对当前代码库的直接约束建议

1. 新系统默认必须走 `Node/.tscn + AutoLoad.Register(Scene)`，禁止新增“静态系统直接订阅全局事件”模式。
2. 启用/禁用必须走统一生命周期入口，禁止系统内部私自只停 `_Process`。
3. “默认是否开启”应由系统配置（`Data/Config`）驱动，不再散落常量布尔值。
4. 系统在 `禁用` 状态下不得继续消费核心业务事件。
5. 系统文档必须写清三件事：
- 支持的全局阶段
- 禁用时行为
- 运行时切换是否安全
