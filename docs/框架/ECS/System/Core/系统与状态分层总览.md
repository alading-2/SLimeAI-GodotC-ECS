# 系统与状态分层总览

> 文档类型：概念总览  
> 适用范围：`SystemManager`、项目级状态、实体状态效果、AI 行为树  
> 当前状态：已与目录重组和首批系统迁移同步

---

## 1. 这次重构到底解决什么问题

如果你是第一次进入这块代码，建议阅读起点固定为：

1. 先看本文，建立四层边界
2. 再看 `其他/系统生命周期与项目状态设计.md`，理解正式运行模型
3. 然后打开 `Src/ECS/Base/System/Core/README.md`，看接入步骤和复杂样例
4. 最后回到源码：`SystemManager.cs` -> `SystemRegistry.cs` -> `SystemDescriptor.cs`

这样看比直接进 `SystemManager` 更高效，因为：

- 先分边界，后看实现，不容易把项目状态和实体状态混在一起。
- 先看正式模型，后看源码接入，不容易把“当前实现”误读成“历史试验产物”。

当前项目过去把几类完全不同的问题混在了一起：

- 项目流程、暂停、UI 覆盖层、测试面板，缺少统一的项目级运行状态。
- 实体上的“无敌 / 眩晕 / 禁止攻击 / 禁止移动输入”长期靠零散 `DataKey bool` 表达，无法处理多来源叠加。
- `MoveMode`、`AIEnabled`、`IsMovementLocked` 曾经被拿来间接表示“这个实体现在还能不能行动”，语义混杂。
- `Src/ECS/AI` 里的行为树负责敌人决策，但它不是项目级状态机，也不该接管系统治理。

这次统一后的核心结论是：

> 系统治理、项目级状态、实体状态效果、单体 AI 决策，必须分层处理。

## 2. 新的四层边界

| 层级 | 回答的问题 | 典型代码 | 说明 |
| --- | --- | --- | --- |
| 系统生命周期层 | 某个系统是否已注册、已创建、已启用、当前能否运行 | `SystemRegistry`、`SystemManager` | 统一管理系统实例与启停 |
| 项目级状态层 | 当前处于菜单、局内、暂停、覆盖层还是阻塞态 | `ProjectStateService`、`SystemRunCondition` | 决定哪些系统应该运行 |
| 实体状态效果层 | 某个实体此刻还能不能思考、主动移动、攻击、施法，是否无敌/免控 | `StatusCollection`、`StatusControllerComponent` | 解决多来源叠加与独立结束 |
| 单体 AI 决策层 | 这个敌人下一步怎么决策、索敌、走位、攻击 | `Src/ECS/AI/*` | 只管单实体决策，不管理项目或系统生命周期 |

## 3. 每层负责什么，不负责什么

### 3.1 `SystemManager + SystemRegistry`

负责：

- 用 `SystemDescriptor` 注册系统元数据。
- 按 `SystemLifetime` 创建宿主节点。
- 统一处理 `Add / Enable / Disable / Resolve`。
- 在项目状态变化时，依据 `SystemRunCondition` 重算系统是否允许运行。
- 对 `Node` 系统统一切 `ProcessMode`，对 `ISystemRuntime` 系统统一调用启停钩子。

不负责：

- 决定业务流程本身为什么切到暂停。
- 决定实体此刻是不是无敌或眩晕。

复杂例子：

- `SpawnSystem` 这类 Gameplay 系统，不只是“生成敌人”这么简单。它真实依赖的是项目状态、波次节奏、计时器、目标选择和宿主路径；这些约束应该写进 `SystemDescriptor`，而不是散在系统内部。
- `TestSystem` / `MouseSelectionSystem` 这类 Debug 系统，也不是“随手挂一个面板”那么简单。它们要和正式系统共存，但生命周期域和运行条件不同，所以必须走同一套注册与接管链路。
- `SystemRegistry` 的职责是守住“描述符元数据唯一性”，但不应该在模块初始化阶段到处 `throw` 中断启动。当前约定是记录 `Log.Error` 并保留首个注册项，让问题可见、流程可继续。

### 3.3 `ProjectStateService`

负责：

- 保存项目级四个状态域：
  - `AppPhase`
  - `SessionPhase`
  - `OverlayPhase`
  - `ExecutionPhase`
- 提供统一快照与切换事件。
- 作为系统运行条件的唯一状态源。

不负责：

- 直接管理敌人的行为树。
- 直接表达某个实体是否可攻击。

### 3.4 `StatusControllerComponent + StatusCollection`

负责：

- 管理实体上的状态实例集合。
- 解决“多个来源同时给无敌/沉默/禁攻”时的独立计时与独立结束。
- 把聚合结果同步为统一快照，再写回 `DataKey.Status*`。
- 向旧链路同步兼容布尔值，避免一次性打断全项目。

不负责：

- 处理项目级菜单/暂停。
- 决定系统是否挂载或启停。
- 替代行为树做敌人决策。

### 3.5 `Src/ECS/AI`

负责：

- 单个敌人的感知、决策、行为树 Tick。

不负责：

- 做项目级状态机。
- 直接管理无敌叠加。
- 替代系统生命周期治理。

## 4. 为什么项目级状态不用行为树

行为树适合回答：

- 敌人当前该巡逻、追击还是攻击？
- 满足某个条件时下一步执行什么节点？

项目级状态要回答的是：

- 当前是不是在主菜单？
- 当前这一局是不是正在进行？
- 当前是不是暂停中？
- 当前是不是有覆盖层 UI 占住交互？
- 某类系统现在到底该不该运行？

这两类问题的粒度完全不同。

如果把项目级状态放进行为树：

- 会让行为树承担系统治理职责，边界错误。
- 会让系统是否运行取决于一堆业务节点判断，无法统一启停。
- 会让暂停、Overlay、测试面板这类横切状态难以稳定组合。

所以项目级状态采用**分域状态**，AI 继续只做**单实体决策**。

## 5. 为什么“运动”和“状态”必须分开

这是本次重构最重要的边界之一。

`MoveMode` 回答的是：

- 用哪种策略生成位移意图？
- 是玩家输入、AI 控制、曲线轨迹、回旋镖返程，还是附着移动？

状态系统回答的是：

- 此刻还能不能主动移动？
- 还能不能思考？
- 还能不能攻击和施法？
- 是否无敌、免控、锁定移动？

典型例子：

- 眩晕：通常禁止思考、主动移动、攻击、施法。
- 击退：可能仍然发生位移，但不应该允许主动输入继续写移动意图。
- 剧情接管：可能禁止 AI 和主动行动，但脚本轨迹仍然继续。

所以现在的原则是：

> 主动控制权由状态层裁决，实际轨迹由移动层执行。

## 8. 最终原则

一句话收束这次方案：

> `SystemManager` 作为唯一启动入口，统一管理系统启停；`ProjectStateService` 管项目级运行态；`StatusControllerComponent` 管实体状态效果；`BehaviorTree` 只管敌人决策。
