# Godot 复刻项目 - 框架索引

## 📖 核心架构：Godot 伪 ECS (Pseudo-ECS)

本项目采用 **"Godot 伪 ECS"** 混合架构：利用 Godot 强大的 Scene Tree 处理渲染和物理（Entity），配合组件化（Component）和数据驱动（Data）实现逻辑解耦。

> **架构哲学**：保留 Godot 的易用性，引入 ECS 的解耦和数据驱动优势。

> **2026-03 Data/Config 改造更新**：`UnitConfig` / `PlayerConfig` / `EnemyConfig` / `TargetingIndicatorConfig` / `AbilityConfig` / `ChainAbilityConfig` 已统一为 `[DataKey(nameof(DataKey.*))]` + `DataKey.*.DefaultValue` 直读模式；`DataKey_*` 主域键已改为 `static readonly DataMeta`，并通过隐式转换保持 `Data.Get/Set(DataKey.*)` 兼容。

### 1.1 Entity (实体)

**核心定义**：**"Scene 即 Entity"**。
在本项目中，Entity 不是一个基类，而是一个实现了 `IEntity` 接口的 Godot Scene（.tscn）。

**核心机制**：

- **EntityManager** - 实体的生命周期管家。负责从对象池生成 (`Spawn`)、注册绑定 (`Register`) 和回收销毁 (`Destroy`)。
- **EntityRelationshipManager** - 关系总管。维护 Entity-Component 和 Entity-Entity 的三索引结构，实现 O(1) 复杂查询。
- **VisualRoot 注入时序（2026-03）** - `EntityManager.Spawn` 会在组件注册前按 `VisualScenePath` 注入 `VisualRoot`，供动画、碰撞配置和组件初始化统一复用。
- **运行时视觉覆盖（2026-04）** - `EntitySpawnConfig.VisualSceneOverride` 可直接传入运行时 `PackedScene` 作为视觉场景，优先级高于 `Config.VisualScenePath`；适用于投射物、预览实体等无独立配置资源或需要临时拼装视觉的实体。
- **碰撞形状注入（2026-04）** - 视觉场景可提供纯 `CollisionShape2D` 或 `CollisionPolygon2D` 模板节点；`EntityManager.InjectVisualScene` 运行时会把模板同步到 Entity 根节点的真实物理体。若现有碰撞节点类型不匹配，会删除旧节点并创建对应类型的新节点；若新的视觉场景未提供碰撞模板，则会删除 Entity 根节点已有的 `CollisionShape2D`，避免残留旧碰撞。`collision_layer / collision_mask` 直接在 Entity `.tscn` 根节点设置，无需模板传递。

### 1.2 Component (组件)

**核心定义**：**"逻辑的功能积木"**。
Component 是依附于 Entity 的功能单元，遵循 **"组合优于继承"**。Component 应当是 **"无状态的"**（Stateless），只包含逻辑，状态存储于 Data 容器中。

**核心机制**：
- **纯逻辑设计** - 不持有私有状态（如 HP, Speed），所有数据从 `_entity.Data` 读写。
- **生命周期** - `OnComponentRegistered` (初始化) / `OnComponentUnregistered` (清理)。

### 1.3 Data (数据)

**核心定义**：**"动态状态容器"**。
Data 取代了传统的 C# 属性（Property），是一个挂载在 Entity 上的动态键值对容器。它解决了 ECS 中 "C" (Component Data) 的问题。

**核心价值**：
- **解耦逻辑与状态** - 任何组件都可以读写任何数据，无需互相引用。
- **自动化管理** - 支持从 Resource 批量加载，对象池回收时自动 `Clear`，天然支持 Buff/Debuff 系统。
- **元数据内嵌** - DataKey 为 `static readonly DataMeta`，直接携带类型、默认值、约束等信息。

### 1.4 Event (事件)

**核心定义**：**"解耦通信总线"**。
基于 C# 原生事件封装的高性能、强类型事件系统，取代 Godot Signal 用于核心逻辑通信。

**核心机制**：
- **EventBus** - 支持优先级、一次性订阅 (`Once`) 和重入保护。
- **分层架构** - `GlobalEventBus` (全局跨系统) + `Entity.Events` (实体内部组件通信)。
- **零 GC 设计** - 使用 `readonly record struct` 传递事件数据，避免内存分配。

### 1.5 System (系统)

**核心定义**：**"全局逻辑处理器"**。
处理跨实体或全局性的业务逻辑。

**核心模块**：

- **DamageSystem** - 职责链模式的伤害计算系统，处理 穿透 -> 暴击 -> 减伤 -> 实际伤害 的复杂流程。

### ✅ ObjectPool (对象池)

**定位**：高性能内存管理。
专为高频生灭对象（子弹、敌人、特显）设计，避免 GC 峰值，提升帧率。支持任意 Node 类型。

**关键时序（2026-04 更新）**：
- 对象池实体采用“两阶段激活”流程：`Get(false)` 出池后先保持禁用，不要立刻恢复处理与碰撞。
- 根节点为 `CollisionObject2D` 的池化实体，回池时先停放到远离战场的泊车位，再脱树清理物理世界状态。
- 脱树节点重新挂树后立即同步禁用碰撞，覆盖 `SetDeferred` 尚未生效的窗口。
- `EntityManager.Spawn` 完成数据注入、位置旋转设置、`ForceUpdateTransform()` 与组件注册。
- 最后由 `EntityManager.Spawn` 显式调用对象池激活逻辑，统一执行挂树保障、恢复处理/可见/根碰撞体并触发生命周期；`CharacterBody2D` 再通过 `CallDeferred(MoveAndSlide)` 同步物理代理。
- 对象池效率统计统一使用 `ReuseRate`（`TotalReused / TotalAcquired`）；`TotalCreatedOnAcquire` 仅统计获取时扩容新建，不包含预热创建。
