# Godot AI Game OS 框架彻底迁移方案

> 日期：2026-05-03
> 目标：完全彻底迁移，不考虑旧框架兼容；一切结构、文档、工具、测试、Skill、MCP 接口都服务于 AI 长任务开发游戏。
> 结论：本项目不再以“Godot C# 伪 ECS 框架”为最终形态，而应升级为“Godot AI Game OS”：ECS 只是 Runtime 内核，真正的产品是 AI 可理解、可操作、可验证、可恢复的游戏程序生产系统。

## 0. 最高判断

当前项目已经具备有价值的底层资产：Entity 生命周期、Component 组合、DataNew、EventBus、Feature/Ability/Damage/Movement/AI/Collision、测试场景、DocsAI、Skill 和 Godot 场景测试 Runner。

但现有项目结构仍然是“人类工程项目 + AI 辅助开发”，不是“AI 原生工程项目”。主要问题不是 ECS 能力不足，而是 AI 开发闭环没有成为项目结构的第一公民。

彻底迁移的核心判断：

- 不再围绕 `ECS` 命名和目录组织项目，而是围绕 `Runtime / Capabilities / Protocols / Validation / Observation / Authoring` 组织项目。
- 不再保留旧 API 兼容层；旧目录只作为迁移输入，迁完即删除或归档。
- 不再让 AI 每次在全局代码里搜索和猜测；每个能力模块必须自带契约、入口、测试、日志协议和修改规则。
- 不再把文档当说明书；文档是 AI 的执行上下文、状态记忆、审查协议和验收门禁。
- 不再把 Skill 写成知识堆；Skill 只负责触发、读取入口、执行步骤、验证命令和禁止事项。
- 不再接受“写完代码请用户打开 Godot 看输出”；AI 必须能通过 CLI 自己 build、跑场景、读日志、修复、复验。

## 1. 外部资料核心要点

本节汇总 Web 与 MCP/Context7 检索结论，保留对本项目有直接价值的要点。

### 1.1 OpenAI Codex 长任务工程模式

来源：OpenAI Developers《Run long horizon tasks with Codex》
链接：https://developers.openai.com/blog/run-long-horizon-tasks-with-codex

核心要点：

- 长任务不是“大提示词一次写完”，而是 `Plan -> Edit code -> Run tools -> Observe results -> Repair failures -> Update docs/status -> Repeat` 的闭环。
- 长任务稳定性的关键是持久项目记忆：规格、计划、约束、状态、决策必须写入仓库文件，而不是依赖聊天上下文。
- 每个里程碑必须有验收标准和验证命令；验证失败时先修复，不继续扩范围。
- Codex 更适合运行在有 repo、文件树、diff、命令输出和可更新状态文件的 harness 中，而不是纯聊天窗口中。

对本项目的结论：

- `Plans/` 必须成为长任务 source of truth。
- 每个迁移阶段必须有 `目标 / 输入 / 输出 / 验证 / 阻塞 / 下一步`。
- AI 开发游戏不应按“功能聊天”推进，而应按“阶段计划 + 验证闭环 + 状态文件”推进。

### 1.2 Codex Skills

来源：OpenAI Developers《Agent Skills》
链接：https://developers.openai.com/codex/skills

核心要点：

- Skill 是把任务专用指令、资源和可选脚本打包成可复用工作流。
- Skill 目录至少包含 `SKILL.md`，可选 `scripts/`、`references/`、`assets/`。
- Codex 先只看到 Skill 的名称、描述和路径；需要时再读取完整 `SKILL.md`，这就是渐进披露。
- Skill 应聚焦单一任务，写清输入、输出、步骤和边界；确定性操作优先放脚本。

对本项目的结论：

- Skill 不应该存放大段架构知识；知识进入 `DocsAI/Modules` 或 `Capabilities/*/Contract.md`。
- Skill 应该变成“AI 操作手册”，例如 `develop-capability`、`debug-runtime-test`、`author-game-data`、`research-reference-framework`。
- 每个 Skill 必须指向明确验证命令，不允许只有提示词没有工具。

### 1.3 AGENTS.md

来源：AGENTS.md 官方说明
链接：https://agents.md/

核心要点：

- `AGENTS.md` 是给编码 Agent 的专用 README，用来放构建命令、测试命令、代码规范和项目上下文。
- 它与人类 README 分离，避免把 AI 需要的详细执行规则塞进人类入口。
- 一个 `AGENTS.md` 可以服务多个 Agent 工具，适合项目级协作规范。

对本项目的结论：

- 根 `AGENTS.md` 应升级为 AI Game OS 的最高执行宪法，只保留稳定红线、入口、验证门禁和目录职责。
- 详细模块规则进入 `DocsAI/Modules` 与对应 Capability 契约。
- 未来迁移后根入口应是：`AGENTS.md -> DocsAI/INDEX.md -> Plans/Architecture/Godot_AI_Game_OS_Migration/README.md`。

### 1.4 MCP 控制层

来源：Model Context Protocol 官方 Server Features
链接：https://modelcontextprotocol.io/specification/2025-06-18/server/index

核心要点：

- MCP Server 提供三类核心 primitive：`Prompts`、`Resources`、`Tools`。
- `Prompts` 是用户控制的交互模板。
- `Resources` 是应用控制的上下文数据。
- `Tools` 是模型控制的可执行函数。

对本项目的结论：

- MCP 不应该替代本地仓库 source of truth；本地文件仍是最终真相。
- 适合暴露给 AI 的 Resources：项目索引、Capability manifest、测试矩阵、最新验证结果、Data schema、错误日志摘要。
- 适合暴露给 AI 的 Tools：列出测试场景、运行指定测试、读取日志、校验数据表、生成 Capability 模板、查询资源目录。
- 适合暴露给 AI 的 Prompts：新增能力、迁移能力、Debug 场景失败、生成数据、审查改动。
- MCP 的价值不是“更高级搜索”，而是把项目内部操作变成可发现、可约束、可审计的工具接口。

### 1.5 Godot CLI 与 C# 自动验证

来源：Godot 4.6 官方 Command line tutorial / C# 文档
链接：https://docs.godotengine.org/en/4.6/tutorials/editor/command_line_tutorial.html
链接：https://docs.godotengine.org/en/4.6/tutorials/scripting/c_sharp/index.html

核心要点：

- Godot 官方支持命令行工作流，可指定项目路径、运行游戏、运行指定场景、开启调试、运行脚本。
- `--scene <path>` 可启动项目内指定场景。
- `--headless` 启用无窗口模式，适合服务器和 CI。
- `--build-solutions` 可构建 C# scripting solutions。
- 命令行导出支持 CI，headless 在无 GPU 环境中是必要参数。

对本项目的结论：

- Godot CLI 是 AI 开发闭环的基础设施，不是辅助工具。
- 当前 `.codex/skills/godot-scene-test/scripts/godot-scene-runner.mjs` 方向正确，应升级为 Validation Layer 的正式入口。
- 每个 Capability 必须声明它的最小验证场景和回归场景。
- 日志必须落盘、可 grep、可摘要，不能依赖用户复制 Output。

### 1.6 Bevy ECS

来源：Bevy / docs.rs ECS 文档，Context7 MCP 查询
链接：https://docs.rs/bevy/latest/bevy/ecs/index.html

核心要点：

- `World` 存储 Entities、Components、Resources。
- Entity 是唯一 ID，关联零个或多个 Component。
- System 是普通函数，参数类型声明数据访问；框架据此判断并行安全。
- Schedule 运行一组 Systems，SystemSet 用于控制调度元数据与顺序。
- Resource 是不属于 Entity 的全局唯一数据。
- Bundle 是一组常一起添加的 Component。
- Message 提供系统间通信；Observer 可响应事件触发。

对本项目的结论：

- 当前项目的 `SystemData` 和系统启停应升级为显式 Schedule/Stage/RunCondition 模型。
- Capability 需要类似 Bundle 的安装声明：启用一个能力时需要哪些 Component、DataKey、Event、System、测试和资源。
- `Data` 与 `SystemContext` 应区分 Entity 状态、全局资源、系统运行状态，减少 AI 混写。

### 1.7 Flecs ECS

来源：Flecs Relationships 文档，Context7 MCP 查询
链接：https://www.flecs.dev/flecs/md_docs_2Relationships.html

核心要点：

- Relationship 是二元 pair：`(Relationship, Target)`。
- 同一个 relationship 可以指向多个 target。
- Relationship 可用于层级、库存、交易、归属、父子、继承等多种游戏语义。
- Query 可以按 `(Relationship, Target)` 或 `(Relationship, *)` 查询。
- Flecs 还提供 prefab / inheritance / observer / module 等概念。

对本项目的结论：

- 当前 `EntityRelationshipManager` 方向正确，但应从“关系工具”升级为 Runtime 的第一类 `Relationship Graph`。
- 关系协议需要固定语义：`PARENT`、`OWNER`、`SOURCE`、`TARGET`、`EQUIPPED_BY`、`SPAWNED_BY`、`AFFECTED_BY` 等。
- AI 写玩法时不应手写引用字段，应声明关系 pair，再通过统一查询入口获取对象。

### 1.8 Unity DOTS / Entities

来源：Unity Entities 1.3 官方文档
链接：https://docs.unity3d.com/Packages/com.unity.entities@1.3/manual/index.html
链接：https://docs.unity3d.com/Packages/com.unity.entities@1.3/manual/systems-intro.html

核心要点：

- Entities 是 DOTS 的数据导向 ECS 实现。
- Systems 有不同实现形式，并通过 System groups 管理更新顺序。
- 官方强调组件粒度、系统数据组织、多 World 和性能优化。

对本项目的结论：

- 不需要照搬 DOTS 的复杂度，但要吸收 `System Group / Update Order / Data Granularity / Authoring -> Runtime` 思路。
- Godot 场景和 DataNew 应形成清晰的 Authoring Layer，运行时只吃生成后的标准配置。
- 大迁移后要减少“边运行边猜数据结构”的逻辑，更多在 Authoring/Validation 阶段提前校验。

### 1.9 源码级调研补充

本轮不只查文档，已 shallow clone 并阅读以下仓库源码：

- Bevy：`https://github.com/bevyengine/bevy`，重点阅读 `crates/bevy_ecs/src/world`、`schedule`、`relationship`、`observer`、`message`、`bundle`。
- Flecs：`https://github.com/SanderMertens/flecs`，重点阅读 `docs/Relationships.md`、`src/query`、`src/addons/rest.c`、`examples/*/relationships`。
- Arch：`https://github.com/genaray/Arch`，重点阅读 `src/Arch/Core`、`Buffer`、`Core/Events`、`Arch.Tests`、`Arch.Benchmarks`。
- Godot 4.6.2 本地源码：`/home/slime/Code/Godot/GodotEngine/godot-4.6.2-stable`，重点阅读 `modules/godot_physics_2d`、`scene/2d/physics`、`servers/physics_2d`。

源码调研得到的额外结论：

- Bevy 已把 `relationship` 做成 ECS 内核模块，并通过 hook 维护 source/target 双向一致性；本项目的 Relationship Graph 不应继续只是工具类，应进入 Runtime 一等协议。
- Bevy `schedule` 源码把 schedule、graph、executor、condition 分开；本项目的 SystemManager 迁移时也要拆出 `ScheduleGraph / Executor / RunCondition / SystemGroup`，否则 AI 仍会把“系统开关、执行顺序、运行态门禁”混在一起改。
- Flecs 的 pair 关系不是文档概念，而是 query、REST、stats、observer 都能消费的基础 ID 形态；本项目关系协议必须同时支持查询、dump、日志和测试，不只是存一张关系表。
- Arch 的 C# 源码重点价值不是照搬高性能 archetype/chunk，而是学习它的测试与 benchmark 组织：核心容器、查询、CommandBuffer、事件都必须有独立测试；本项目 Runtime 迁移也要先补测试再扩能力。
- Godot 物理源码确认 `CollisionObject2D` 脚本侧状态、PhysicsServer2D 后端 shape/pair/query 队列并不同步完成；对象池碰撞问题不能只靠业务层猜时序，必须进入 Observation 和 Godot Engine Debug 计划。

### 1.10 Godot 物理与对象池源码结论

本地 Godot 源码阅读确认：

- `modules/godot_physics_2d/godot_physics_server_2d.cpp` 中 `flush_queries()` 在物理 step 后调用各 space 的 query 回调，脚本收到碰撞事件时已经进入后端查询 flush 阶段。
- `godot_collision_object_2d.cpp` 的 shape disable/enable 会影响 broadphase shape 注册和 pair 重建；同帧回池/出池时，如果物理后端仍持有旧 transform 或旧 pair，就可能出现旧位置 ENTER/EXIT。
- `godot_area_2d.cpp` 的 `set_space()` 会清理 `monitored_bodies` / `monitored_areas`，所以脱树能解决很多残留事件，但它是应用层绕开后端状态不可观测问题的手段，不应该成为唯一长期方案。
- `scene/2d/physics/collision_object_2d.cpp` 是脚本节点与 PhysicsServer RID 同步的桥，底层 debug 应优先从这里和 `modules/godot_physics_2d` 加可控日志，而不是在业务组件里堆更多猜测日志。

对本项目的结论：

- 短期继续保留“泊车位 + 脱树 + 同步禁用 + 延迟物理同步”的对象池防线。
- 中期在 Godot fork 增加可开关的 PhysicsServer2D debug trace：RID、ObjectID、shape index、space enter/exit、broadphase add/remove、pair enter/exit、query flush。
- 长期让 GodotSkill 支持读取这类底层 trace，并把 `EntityId / NodePath / RID / ObjectID` 串成一次碰撞链路报告。

### 1.11 数据层外部结论

`.tres` 和纯 C# 初始化器都不是 AI-first 的最终数据形态：

- `.tres` 优点是 Godot 原生、编辑器友好；缺点是文本结构噪音高、UID/Resource 引用复杂、AI 批量修改和校验成本高。
- `DataNew` 纯 C# 优点是可编译、可存 enum、可直接跳转定义；缺点是 AI 修改表数据时容易碰到 C# 语法、对象初始化器、复杂表达式和编译耦合，批量查询、diff、迁移、约束也不如数据库直接。
- AI-first 数据不应存运行时对象、Node、PackedScene 或复杂对象图。数据应以标量为主：`string / int / float / bool / enum文本 / resource path / entity id / relationship id`。
- 更合理的目标是 `SQLite Authoring DB + schema + generator + runtime snapshot`：数据库作为 AI 修改和查询的真相源，生成器输出 C# 快照、JSON 或 Godot 可加载资源，运行时不在热路径上依赖随意 SQL 查询。

## 2. 当前项目问题诊断

### 2.1 架构定位问题

当前项目文档仍以“Godot 伪 ECS”为中心。这个定位对人类解释有帮助，但对 AI 长任务开发不够直接。

AI 需要的不是“这个框架是否正统 ECS”，而是：

- 我现在要做一个能力，应该改哪里？
- 这个能力需要哪些 DataKey / Event / Component / System？
- 怎么启用？
- 怎么验证？
- 失败日志怎么看？
- 修改后需要同步哪些文档和 Skill？
- 上下文清空后如何继续？

因此项目最高抽象应从 `ECS Framework` 迁移到 `AI Game OS`。

### 2.2 目录语义问题

当前 `Src/ECS/Base/System/...`、`Data/DataNew/...`、`DocsAI/Modules/...`、`.codex/skills/...` 已经形成能力雏形，但目录仍按历史演进堆叠。

典型问题：

- Runtime 内核、Gameplay 能力、工具、测试、AI 契约混在 `Src/ECS` 语义下。
- `System` 既表示 ECS System，又表示游戏业务系统，又表示项目管理系统，AI 容易混淆。
- 文档、Skill、测试矩阵已经存在，但和能力模块不是同一结构单元。
- 旧迁移计划、历史文档、归档说明过多，AI 读取后容易被旧上下文污染。

### 2.3 开发闭环问题

已有 Godot 场景测试 Runner，但还没有成为所有模块的强制接口。

必须解决：

- 每个能力是否有固定验证命令。
- 每个能力是否能生成最小测试场景。
- 日志是否能稳定落盘、摘要、关联到能力。
- 长任务失败是否能从 plan 文件恢复。
- AI 是否能通过命令得到“该能力当前状态”。

### 2.4 文档治理问题

已有 Docs / DocsAI / Skill 分层，但彻底迁移后需要更严格：

- `Docs/` 面向人类理解，不作为 AI 执行细节入口。
- `DocsAI/` 面向 AI 任务执行，不写历史长篇思考。
- `Capabilities/*/Contract.md` 面向单能力修改。
- `Plans/*` 面向长任务状态，不写模块知识。
- Skill 只写工作流入口，不携带大段模块知识。

## 3. 迁移总原则

### 3.1 不兼容原则

本次迁移不保留旧兼容。

允许：

- 改目录名。
- 改命名空间。
- 改 System / Component / DataKey / EventType 组织方式。
- 删除旧 Resource `.tres` 运行时入口。
- 删除旧 README、旧迁移记录、过期 Skill 引用。
- 重写测试入口和计划入口。
- 让旧代码在中间阶段无法运行，只要阶段计划和恢复路径清楚。

不允许：

- 为了兼容旧路径保留双入口。
- 为了少改引用继续沿用误导性命名。
- 为了短期省事保留“历史包袱 + 新结构”并行。
- 在迁移中继续新增游戏功能。

### 3.2 AI-first 原则

判断架构优先级的标准：

- AI 能否快速定位？
- AI 能否局部修改？
- AI 能否通过命令验证？
- AI 能否读取日志定位错误？
- AI 能否在上下文清空后恢复？
- AI 能否从成熟项目和游戏资料中提取设计并落到数据表？

如果某个抽象让传统架构更优雅，但提高 AI 理解成本、验证成本或调试成本，则不优先采用。

### 3.3 稳定内核原则

Runtime 内核必须小、稳定、少改。

AI 高频开发应发生在：

- `Capabilities/`
- `Content/`
- `Authoring/`
- `Tests/`
- `Tools/`
- `DocsAI/`
- `.codex/skills/`

AI 低频修改才进入：

- `Runtime/Entity`
- `Runtime/Event`
- `Runtime/Data`
- `Runtime/Schedule`
- `Runtime/Relationship`
- `Runtime/Pool`

### 3.4 模板优先原则

所有新增能力都必须来自模板，不允许自由发挥。

模板必须包括：

- 目录结构。
- Manifest。
- Contract。
- Data schema。
- Event schema。
- 测试场景。
- Debug 文档。
- Skill 入口或 Skill 映射。
- 验证命令。

### 3.5 验证优先原则

任何迁移阶段没有验证命令，就不算完成。

最低验证层级：

- Markdown 链接和入口检查。
- `dotnet build`。
- Capability 相关场景测试。
- 核心回归测试。
- 数据表校验。
- 日志错误摘要检查。

## 4. 目标架构：Godot AI Game OS

目标结构分为 8 层。

### 4.1 Intent Layer

职责：把用户自然语言需求转成结构化 AI 任务。

示例：

```yaml
intent: AddCapability
target: Player
capability: AutoAttack
requires:
  - TargetQuery
  - Cooldown
  - Damage
outputs:
  - DamageRequested
  - AttackVisualRequested
validation:
  - player_attacks_nearest_enemy
  - cooldown_blocks_repeat_cast
```

## 8. Runtime 内核重构目标

### 8.1 Entity Runtime

目标：统一生命周期、注册、销毁、对象池复用、关系绑定、视觉注入。

迁移要求：

- `EntityManager` 拆成清晰子域：Spawn、Register、Destroy、Pool、Relationship、Migration、VisualInjection。
- 对外只保留少量稳定入口。
- 禁止 Capability 直接 `new Entity` 或 `QueueFree`。
- Entity 根节点仍可以利用 Godot Scene，但 Runtime 文档不再把“Scene 即 Entity”作为最高抽象，而是称为 Godot Scene Bridge。

### 8.2 Component Runtime

目标：Component 是能力执行单元，不存业务状态。

迁移要求：

- `IComponent` 保持生命周期接口。
- 基础组件契约进入 Runtime。
- 业务组件迁入对应 Capability。
- Component 私有字段只允许缓存引用和订阅 token，不允许持久业务状态。

### 8.3 Data Runtime

目标：Data 是运行时状态容器；DataNew 是 Authoring 输入，不混淆。

迁移要求：

- `Data` / `DataMeta` / `DataRegistry` 进入 Runtime。
- `DataKey` 定义进入 Authoring，按 Capability 分域生成或注册。
- 禁止字符串字面量读写 Data。
- `CanMigrate`、默认值、分类、约束必须在 Contract 中可见。

### 8.4 Event Runtime

目标：事件是核心通信协议。

迁移要求：

- 保留 Entity local event 与 GlobalEventBus 的分层。
- EventType 按 Capability 分域。
- 每个事件必须有 payload 约定。
- 订阅和取消订阅必须在 Component 生命周期中闭环。

### 8.5 Relationship Runtime

目标：关系图成为一等 Runtime 能力。

迁移要求：

- 使用 pair 语义描述关系：`Source -- Relationship -> Target`。
- 固定核心关系名：`PARENT`、`OWNER`、`SOURCE`、`TARGET`、`SPAWNED_BY`、`EQUIPPED_BY`、`AFFECTED_BY`。
- 所有生成链路统一声明关系，不在业务组件里藏引用字段。
- 提供关系 dump 和关系查询测试。

### 8.6 Schedule Runtime

目标：系统启停、运行条件和执行顺序显式化。

迁移要求：

- 建立 `ScheduleStage`：`Bootstrap`、`PreUpdate`、`Update`、`PostUpdate`、`LateUpdate`、`Debug`。
- 建立 `SystemGroup`：`Core`、`Gameplay`、`PhysicsBridge`、`Presentation`、`Testing`。
- 建立 `RunCondition`：系统启用、游戏状态、暂停状态、测试模式、场景模式。
- `SystemData` 迁移为 Schedule 配置输入，而不是散落的系统开关表。

### 12.3 Data OS 最终形态

数据层目标应从 `Data/ + DataNew/` 升级为 `GameOS/Authoring/DataOS/`。

核心判断：

- 数据真相源使用 SQLite，而不是 `.tres` 或 C# 初始化器。
- 数据只存可序列化值：`string / int / float / bool / enum文本 / path / id / relationship`。
- 复杂运行时对象通过 `id`、`path`、`handler_id`、`relationship` 间接引用，不直接存对象。
- C# 类型安全来自生成物，不来自人手写表数据。
- Godot 编辑器友好来自 DataForge / 自定义 Editor UI，不来自 `.tres` 作为主数据源。

### 12.4 SQLite 表设计原则

AI 修改数据时最怕“看不见约束”。数据库必须把约束写成结构，而不是写在说明里。

原则：

- 每张表必须有稳定主键 `id`，显示名称用 `display_name`，不要把中文名当主键。
- enum 统一存文本，并由 `enum_catalog(enum_type, enum_value)` 或生成器校验。
- 概率仍按项目规则存 `0-100`，用 `CHECK(value >= 0 AND value <= 100)`。
- “无限制”仍存 `-1`，不能用 `0` 混淆。
- 资源路径存 `res://...` 字符串，生成器校验文件存在。
- 多对多、归属、生成来源、技能挂载等语义用关系表表达，不把列表塞进字符串。
- JSON 只允许用于低频、弱结构、暂未固化的实验字段；一旦字段稳定，应迁成正式列或关系表。
- 所有表都要有 `notes` 或 `ai_notes` 可选列，用于 AI 记录设计理由，但运行时不读取。

示例：

```sql
CREATE TABLE ability (
  id TEXT PRIMARY KEY,
  display_name TEXT NOT NULL,
  handler_id TEXT NOT NULL,
  trigger_mode TEXT NOT NULL,
  ability_type TEXT NOT NULL,
  cooldown REAL NOT NULL DEFAULT 0 CHECK (cooldown >= 0),
  cast_range REAL NOT NULL DEFAULT -1 CHECK (cast_range >= -1),
  damage REAL NOT NULL DEFAULT 0 CHECK (damage >= 0),
  damage_chance REAL NOT NULL DEFAULT 100 CHECK (damage_chance >= 0 AND damage_chance <= 100),
  projectile_scene_path TEXT,
  effect_scene_path TEXT,
  ai_notes TEXT
);
```

### 12.5 DataOS 与 Runtime Data 的边界

不要把数据库直接等同于运行时 `Data` 容器。

分层：

- `DataOS SQLite`：AI authoring 真相源，适合查询、批量修改、约束校验、迁移和生成。
- `Generated Snapshot`：构建或验证阶段生成的 C# / JSON 快照，供运行时读取。
- `Runtime Data`：Entity 上的动态状态容器，存当前 HP、速度、状态、临时修正等运行时值。
- `DataKey Registry`：运行时可读写键的协议，由数据库 schema 和 Capability Contract 共同生成或校验。

运行时热路径不直接执行任意 SQL。原因不是 SQLite 不可用，而是游戏帧内逻辑需要确定、少分配、可测试；数据库适合 authoring、验证、启动加载和调试查询。

### 12.6 DataOS 迁移路线

彻底迁移时不需要兼容旧数据入口，但要有清晰迁移顺序：

1. 定义 `schema/core.sql`：enum catalog、resource path、capability、data key、event type、relationship type。
2. 把 `DataNew/System` 迁成第一张数据库样板，因为系统配置最需要查询、排序、运行条件和依赖校验。
3. 迁 `Unit / Ability / Feature`，把 `FeatureHandlerId`、`AbilityTriggerMode`、资源路径都转成显式列。
4. 编写生成器输出当前项目可编译的 `DataSnapshot.generated.cs`。
5. 运行 `validate-data.sh`：schema 检查、enum 检查、路径检查、handler id 检查、重复 id 检查。
6. Runtime 只从生成快照加载，删除旧 `.tres` 和手写 `DataNew` 当前入口。

验收命令目标：

```bash
Tools/ai-game-os/validate-data.sh
Tools/ai-game-os/generate-data-snapshot.sh
dotnet build
Tools/ai-game-os/run-regression.sh data
```

### 12.7 外部资料与源码研究机制

项目自我迭代能力取决于 AI 能否持续吸收外部成熟实现，而不是每次靠记忆或查几页文档。

新增固定机制：

- 任何框架级重构、底层 bug、复杂玩法系统，默认先做 reference research。
- 优先 clone 源码到临时目录，再用 `rg / find / tree / git log` 分析，而不是只读官网文档。
- 研究产物必须写成 `ResearchBrief.md`，包括：问题、候选项目、源码路径、可借鉴点、不能照搬点、落地决策、后续验证。
- 研究结论只能进入计划和协议，不直接把外部框架复杂度搬进 Runtime。
- 形成 `.codex/skills/research-reference-framework`，把搜索、clone、源码阅读、结论落盘变成固定流程。

### 12.8 经验库与 AI 流程复盘

踩坑不能只留在聊天上下文里。

新增固定机制：

- `DocsAI/Experience/`：记录踩坑、症状、根因、错误尝试、最终方案、验证方式。
- `DocsAI/Protocols/AI表现复盘协议.md`：当 AI 找问题找很久、修复反复失败、效果不好时，不只修代码，还要复盘是文档缺口、Skill 缺口、观察工具缺口、测试缺口还是架构边界不清。
- 每次出现高成本 Debug，最终必须回答：下次 AI 如何更早定位？需要新增什么日志、dump、测试、Skill 或文档？

## 16. 高风险点

### 16.1 Godot 场景引用断裂

迁移目录会导致 `.tscn` 脚本路径、资源路径、UID 引用断裂。

应对：

- 分阶段迁移。
- 每阶段运行场景测试。
- 避免手动批量编辑 `.tscn`，优先用 Godot 或可验证脚本处理。
- 迁移前生成资源引用清单。

### 16.2 C# 命名空间大改导致构建雪崩

应对：

- Runtime 先迁，Capability 后迁。
- 每迁一组就 build。
- 不同时改行为和命名空间。

### 16.3 文档和代码再次脱节

应对：

- Contract 与测试命令放进 Capability。
- Skill 只指向 Contract，不复制细节。
- 每阶段强制 `rg` 检查旧路径。

### 16.4 AI 过度抽象

应对：

- Runtime 小而稳定。
- Capability 模板固定。
- 禁止无需求的大重构。
- 每个抽象必须有验证收益。

### 16.5 测试基础设施不足

应对：

- 迁移前先升级 Validation Layer。
- 没有测试的模块不能迁为 stable Capability。
- 先做 Movement/Damage/Ability 样板，再扩散。

### 16.6 数据库引入后失控

风险：

- AI 直接在 SQL 里堆 JSON 字段，数据库退化成大字符串仓库。
- 运行时到处查 SQLite，导致性能和生命周期不可控。
- schema、生成器、C# 快照三者不同步。
- enum 文本和 C# enum 定义漂移。

应对：

- schema 先行，JSON 只能作为实验字段。
- Runtime 只读生成快照，数据库用于 authoring、验证、启动加载和调试。
- 每次 DataOS 改动必须同时跑 `validate-data.sh`、`generate-data-snapshot.sh`、`dotnet build`。
- enum catalog 由生成器和 C# enum 双向校验。

### 16.7 盲目修改 Godot 底层

风险：

- 直接改 PhysicsServer 行为可能破坏 Godot 既有物理语义。
- 自定义引擎分叉可能导致项目难以升级。
- 底层日志过多会明显拖慢测试和污染输出。

应对：

- 第一阶段只加可开关 debug trace，不改行为。
- 所有底层改动必须带最小复现场景和关闭开关。
- GodotSkill 必须能收集、摘要、过滤底层 trace，否则不算闭环。
- 只有当项目层 workaround 明确成本过高且复现稳定时，才进入行为修改。

### 16.8 AI 流程失败未复盘

风险：

- AI 修失败后只继续猜代码，导致同类问题反复发生。
- Skill 和 DocsAI 不吸收失败经验。
- 缺日志、缺 dump、缺测试的问题被误判为“模型不够聪明”。

应对：

- 三轮 Debug 未收敛必须执行 `AI表现复盘协议`。
- 复盘输出必须落到 `DocsAI/Experience/`、相关 Skill 或 Validation 工具。
- 复盘优先找流程缺口：入口不清、观察不足、测试不足、约束缺失、外部资料未研究。

## 17. 第一轮实际执行建议

不要立刻全项目搬目录。第一轮只做“可逆的结构准备 + 一个样板能力迁移”。

建议执行顺序：

1. 新建正式计划目录 `Plans/Architecture/Godot_AI_Game_OS_Migration/`。
2. 建立 `GameOS/README.md`、`GameOS/Capabilities/README.md`、`GameOS/Runtime/README.md` 骨架。
3. 建立 Capability 模板，不迁代码。
4. 把 Movement 作为样板迁移方案写清楚。
5. 包装现有 Godot scene runner 为 `Tools/ai-game-os/run-capability-test.sh`。
6. 更新 `DocsAI/INDEX.md` 指向新协议入口。
7. 跑 `dotnet build` 和 Movement 场景测试。

这一轮目标不是完成迁移，而是验证新结构是否真的让 AI 更容易工作。

## 18. 最终完成定义

迁移完成不是“代码能编译”这么简单，而是满足以下条件：

- AI 新会话只读 `AGENTS.md`、`DocsAI/INDEX.md`、`ProjectState.md`、当前 Plan，就能继续任务。
- 任意 Capability 都能通过 manifest 找到代码、数据、事件、测试、文档和 Skill。
- 任意 Runtime 改动都有门禁和回归命令。
- 任意 Data 改动都有校验命令。
- 任意 Godot 测试失败都有日志摘要和复验命令。
- 旧路径不再作为当前入口出现。
- 人类不再负责复制 Godot Output。
- AI 可以从一句游戏目标生成计划、填数据、接能力、跑测试、修复问题。
