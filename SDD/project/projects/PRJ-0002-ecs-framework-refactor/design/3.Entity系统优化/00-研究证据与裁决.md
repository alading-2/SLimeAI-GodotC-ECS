# 研究证据与裁决

> 更新：2026-05-29
> 范围：`SlimeAI/Src/ECS/Base/Entity/Core/`、`SlimeAI/DocsNew/`、`SlimeAI-AiFirst/`、`Resources/Engine/Docs/FrameworkAnalysis/Reports/`、外部 ECS 官方资料。
> 结论：`EntityManager` 不做兼容 facade，不继续兼任业务入口；Entity / Relationship 按完整重构处理。旧 `EntityRelationshipManager`、`EntityRelationshipType`、`ParentRelationTypes`、业务 partial 和 legacy parent-chain 归因不作为新设计保留。新结构直接以 typed `EntityId`、`LifecycleTree`、typed `DataKey` / `EntityIdList`、capability-owned index 和明确 `DamageAttribution` 为事实源。

> 本文是 `3.Entity系统优化/` 的研究证据与裁决入口；后续执行型 SDD 应以本目录及其子文档为主事实源。

## 0. 一句话裁决

当前框架的问题不是“有 Relationship”。

其他 ECS / 引擎也有实体关系、层级或 parent-child：

- Bevy 有 typed `Relationship`、`ChildOf` / `Children`。
- Flecs 有 first-class relationship / pair，`ChildOf` 是特殊 hierarchy relation。
- Unity Entities 有实体 ID、组件、系统，也有 transform hierarchy 的 `Parent` / `Child`。
- Friflo 有 typed relations / child entities。
- EnTT、Arch、DefaultEcs 更偏最小 registry / storage / query，把业务关系留给用户组件或系统。

问题在于 SlimeAI 当前实现把这些语义混在一个“任意 `string relationType` + 三索引 + `Dictionary<string, object>`”的通用图里。这对人看起来统一，对 AI 和长期维护反而不稳定：同一个 API 既像生命周期树、又像业务归属表、又像组件 owner 表、又像调试图数据库。

推荐裁决：

| 问题 | 裁决 |
| --- | --- |
| `EntityManager` 是否要大改 | 是。不是保留兼容入口后内部委托，而是删除业务兼任和 legacy API，重建为单一 Entity runtime 入口。 |
| Relationship 是否独有 | 不是。其他 ECS 有类似概念，但主流做法是 typed、受约束、语义明确，不是任意字符串图。 |
| 是否继续用 Relationship | 继续保留“生命周期父子关系”，但改名和改形为 `LifecycleTree` / `LifecycleParent`。 |
| `RelationshipType` 字符串怎么办 | 直接从新 runtime 设计中删除，不做新增冻结期、不做长期兼容层；debug 需要的关系另走 observation/fact artifact。 |
| 子弹、武器、玩家、伤害统计怎么建模 | 统计归因不走通用 parent 链猜。用 typed `DamageAttribution` 或明确 DataKey：`SourceEntity`、`OwnerEntity`、`WeaponEntity`、`AbilityEntity`、`DamageCreditEntity`。 |

## 2. 当前代码事实

### 2.1 `EntitySpawnConfig` 已把 lifecycle 和业务关系混在生成参数里

当前 `EntitySpawnConfig` 包含：

- `ParentEntity`
- `AutoAddParentRelation`
- `ParentRelationTypes: string[]?`
- `ParentDestroyPolicy`

这些字段在 `SlimeAI/Src/ECS/Base/Entity/Core/EntityManager.cs:51-61`。语义上它同时表达：

- 生命周期 parent：父销毁时是否递归销毁子实体。
- 归属溯源：投射物 / 技能 / 特效属于谁。
- 业务分类：`ENTITY_TO_PROJECTILE`、`ENTITY_TO_ABILITY`、`ENTITY_TO_EFFECT`。
- 统计推断入口：沿 `PARENT` 查 IUnit / IWeapon。

这导致生成配置不是单一职责。AI 看到 `ParentEntity` 时很难判断它是在表达“销毁树 parent”、“伤害归因 owner”、“技能拥有者”、“场景节点父级”还是“调试关系”。

### 2.2 `EntityManager.Spawn` 责任过宽

当前 `Spawn<T>` 一次性处理：

- 对象池获取或场景实例化：`EntityManager.cs:172-209`
- runtime snapshot / Data 注入：`EntityManager.cs:214-219`、`273-319`
- VisualScene 注入：`EntityManager.cs:221-226`、`389-437`
- Transform 设置和 `ForceUpdateTransform`：`EntityManager.cs:228-240`
- Relationship 绑定：`EntityManager.cs:242-245`
- Register + RegisterComponents：`EntityManager.cs:247-252`
- 对象池 Activate 和物理同步：`EntityManager.cs:254-266`
- `EntitySpawned` 全局事件：`EntityManager.cs:268`

这是典型“流程正确但边界过粗”。它不是马上不能用，但它已经不适合作为 AI-first 的主要维护单元。任何一点失败语义都会牵动多个系统：

- Data apply 失败会调用 `Destroy(entity)`。
- Relationship 绑定失败只 warn，Spawn 仍继续返回 entity。
- VisualScene 缺失只 debug 跳过。
- Component 注册顺序依赖关系先建立。
- 对象池 Activate 必须晚于 Transform 和 Register。

这些都是行为契约，不应只隐藏在一个大函数里。

### 2.3 `EntityRelationshipManager` 是通用三索引字符串图

`EntityRelationshipManager.RelationshipRecord` 当前字段是：

- `string ParentEntityId`
- `string ChildEntityId`
- `string RelationType`
- `Dictionary<string, object> Data`
- `int Priority`

见 `EntityRelationshipManager.cs:62-71`。

关系主存储和三索引：

- `_relationships`
- `_parentIndex`
- `_childIndex`
- `_typeIndex`

见 `EntityRelationshipManager.cs:73-85`。

`AddRelationship` 接口接受 `string parentId`、`string childId`、`string relationType`、`Dictionary<string, object>? data`、`RelationshipConstraint`，见 `EntityRelationshipManager.cs:109-115`。这说明核心 API 对 AI 和编译器都没有语义保护：

- entity id 是 raw string。
- relation type 是 raw string。
- relation data 是任意字典。
- constraint 是调用时一次性传入，不绑定 relation type 定义。

外部框架里也有 relationship，但核心关系通常是 typed component / typed pair / typed struct / typed component storage，而不是让 gameplay 传任意字符串。

### 2.4 `RelationshipType` 常量正在变成业务枚举仓库

当前 `EntityRelationshipType` 包含：

- `ENTITY_TO_COMPONENT`
- `PARENT`
- `ENTITY_TO_ABILITY`
- `ENTITY_TO_ITEM`
- `ENTITY_TO_EFFECT`
- `ENTITY_TO_PROJECTILE`
- `ABILITY_TO_EFFECT`
- `ITEM_TO_ABILITY`
- `ENTITY_TO_UI`

见 `EntityRelationshipType.cs:14-49`。

这些类型不是一个语义层级：

- `PARENT` 是生命周期树。
- `ENTITY_TO_COMPONENT` 是 Godot Node 组合 / owner 关系。
- `ENTITY_TO_PROJECTILE`、`ENTITY_TO_EFFECT` 是生成追踪和清理。
- `ENTITY_TO_ABILITY`、`ENTITY_TO_ITEM` 是玩家/单位库存或能力列表。
- `ENTITY_TO_UI` 更像 GodotBridge/UI adapter。

放在一个字符串类型表里，短期很方便，长期会让 AI 以为“新增一种业务引用 = 新增 relationType”。这正是冗余代码扩张的来源。

### 2.5 生命周期策略藏在 `RelationshipRecord.Data`

`EntityRelationshipLifecycle` 用私有字符串 `parent_destroy_policy` 存父销毁策略，见 `EntityRelationshipLifecycle.cs:12`。写入时把 `ParentDestroyPolicy` 转成 int 放进字典，见 `EntityRelationshipLifecycle.cs:31-39`；读取时从字典解析，失败回退 `DestroyRecursively`，见 `EntityRelationshipLifecycle.cs:48-75`。

这说明当前 `PARENT` 已经不是普通关系，它需要 typed metadata 和明确销毁策略。但实现仍然借用了通用 `Dictionary<string, object>`。对 AI-first 来说，这种“核心生命周期协议藏在字典 key 里”的形态应当淘汰。

### 2.6 `EntityRelationshipTraversal` 把统计归因绑定到 `PARENT` 链

`ResolveEntityId` 优先读 `DataKey.Id`，否则回退 Godot `GetInstanceId()`，见 `EntityRelationshipTraversal.cs:14-35`。而当前 `ApplySpawnData` 写的是 `GeneratedDataKey.Id`，见 `EntityManager.cs:318`。

这里有一个高风险点：需要确认 `DataKey.Id` 和 `GeneratedDataKey.Id` 是否稳定映射到同一个 DataKey。如果不是，Relationship 表和 Entity Data 中的 ID 语义会漂移。

`FindAncestorOfType<T>` 和 `GetAncestorChain` 都沿 `PARENT` 向上查，见 `EntityRelationshipTraversal.cs:90-139`、`148-192`。这直接影响业务：

- `StatisticsProcessor` 沿攻击链统计 IUnit / IWeapon，见 `StatisticsProcessor.cs:31-42`。
- `MovementCollisionPolicy` 沿 `PARENT` 找最近 IUnit 做阵营判断，见 `MovementCollisionPolicy.cs:124-137`。
- `CritProcessor`、`LifestealProcessor` 等也通过 ancestor 追溯攻击者。

这可以工作，但它把“生命周期 parent”偷换成“业务归因 parent”。只要某个对象因为清理策略需要挂到另一个 parent，统计归因就可能变。

### 2.7 业务系统已经依赖通用 Relationship

当前关键业务调用：

- 投射物生成：`ProjectileTool.Spawn` 设置 `ParentEntity = owner`，并写 `ParentRelationTypes = [ENTITY_TO_PROJECTILE]`，见 `ProjectileTool.cs:29-39`。
- 特效生成：`EffectTool` 对 host 或 owner 调 `BindParentRelationships(... ENTITY_TO_EFFECT)`，见 `EffectTool.cs:111-130`。
- 技能添加：`EntityManager_Ability` 用 `Spawn<AbilityEntity>` 生成技能并绑定 `ENTITY_TO_ABILITY`，见 `EntityManager_Ability.cs:96-107`。
- 技能查询和移除：按 `ENTITY_TO_ABILITY` 查子实体或移除关系，见 `EntityManager_Ability.cs:207-243`。
- 特效清理：按 `ENTITY_TO_EFFECT` 查询 host 下的特效，见 `EffectTool.cs:186-205`。

这说明 Relationship 不是“只给人看”。它已经是 gameplay hot path 和清理路径的一部分。也因此不能直接删除，必须分阶段迁移。

### 2.8 `EntityManager_Ability` 是边界污染证据

`EntityManager_Ability.cs` 位于 AbilitySystem，但声明为 `public static partial class EntityManager`。这意味着 `EntityManager` partial 已经承载了 Ability lifecycle 的业务方法：添加技能、移除技能、查技能。

这不是一个单纯命名问题。它说明 `EntityManager` 已经从“实体注册和生命周期入口”扩张成“业务能力管理器集合”。AI-first 的重构应该优先阻止这种 partial 扩张继续发生。

### 2.9 Component 注册也依赖 Relationship

`RegisterComponents` 用两条规则识别 component：

- 实现 `IComponent`
- 类型名以 `Component` 结尾

见 `EntityManager_Component.cs:121-130`、`166-180`。

注册时会建立 `ENTITY_TO_COMPONENT` 关系，并要求在 `OnComponentRegistered` 前建立，见 `EntityManager_Component.cs:188-205`。注销时也通过关系表找 component，见 `EntityManager_Component.cs:230-258`。

这类关系更像 GodotBridge / NodeRegistry 的内部 owner index，不应和 gameplay 的 projectile/ability/effect relation 共用一个公共 RelationshipType 表。

## 3. 外部框架对照

### 3.1 Bevy：relationship 是 typed source-of-truth，不是字符串图

Bevy 的 relationship 机制把 source 侧关系 component 作为事实源，target 侧 collection 作为反向缓存。`ChildOf` / `Children` 是 hierarchy 的典型实现。父实体 despawn 时递归处理 children 的能力只绑定在配置了 linked spawn / recursive descendant despawn 的关系上。

对 SlimeAI 的启发：

- `PARENT` 应该成为 typed lifecycle relation。
- 反向 children 列表可以是缓存，但不能让调用方随意改。
- 递归销毁只适合 lifecycle relation，不适合 `Target`、`Source`、`AffectedBy` 这类业务引用。
- 自定义业务关系即使存在，也要 typed，不应裸字符串。

不采纳：

- Rust derive / 宏式生成。
- 把 Bevy 的 ECS storage 直接搬进 Godot C# 旧框架。

### 3.2 Flecs：有 relationship，但 `ChildOf` 是特殊关系

Flecs 的 relationship / pair 很强，但它也通过 trait 区分语义。例如 `ChildOf` / hierarchy 与普通业务关系不同；它带有 exclusive、acyclic、traversable、delete policy 等行为。Flecs inventory 示例可以用 typed exclusive relation 表达“物品属于容器”，但这不等价于生命周期 parent。

对 SlimeAI 的启发：

- 如果一个关系会影响销毁、遍历、事件传播或查询，它就必须是明确的内核关系，不应只是字符串。
- `LifecycleParent` 可以借鉴 `ChildOf` 的约束：单 parent、防环、可遍历、明确父销毁策略。
- 业务引用要和 lifecycle 分开：`Owner`、`Source`、`Target`、`Container` 不默认递归销毁。

不采纳：

- 不让 AI 自由创建 `(Relation, Target)` pair 作为 gameplay authoring。
- 不把所有事实查询做成通用 relationship DSL。

### 3.3 Unity Entities：EntityManager 是结构操作入口，不是业务归属图

Unity Entities 把 Entity 作为轻量身份，Component 存数据，System 处理数据。Hierarchy / transform 是专项 transform 结构，而不是所有业务关系的统一载体。

对 SlimeAI 的启发：

- Entity 身份应 typed 且稳定。
- `EntityManager` 可以保留为创建、销毁、注册和结构变更入口。
- 业务统计归因应在数据组件 / typed data 中表达，不应通过一个通用 parent 链推断。

不采纳：

- 不把 SlimeAI 的 Godot Node 全部迁成纯 DOTS archetype。
- 不复制 Unity 的 authoring / baking 工作流。

### 3.4 Friflo：typed relations 是能力，但 SlimeAI 暂不需要完整泛化

Friflo 支持 relations、child entities 和多种 typed 关系能力。它证明 C# ECS 可以做 typed relation，但也说明 relation 系统一旦泛化，会带来 API 和验证复杂度。

对 SlimeAI 的启发：

- typed relation 可以作为长期能力参考。
- 当前更适合先做 `LifecycleTree + typed business reference`，而不是一步到位实现通用 typed relation 框架。

### 3.5 EnTT / Arch / DefaultEcs：小 registry 更适合 AI-first

这些框架更强调 registry、storage、query、command buffer、system iteration。它们通常不会把“技能属于谁、投射物来源是谁、UI 绑定哪个 entity”做进一个公共 RelationshipManager，而是由组件、系统或用户层结构表达。

对 SlimeAI 的启发：

- `EntityManager` 应小，不应知道 Ability、Projectile、Effect、UI。
- 查询能力要通过 service / tool 暴露，而不是公开一个全能力关系图。
- 如果需要 index，也应由 capability owner 维护。

### 3.6 Component Pattern：反对巨型对象

Game Programming Patterns 的 Component 模式强调把不同领域拆成组件，避免一个大对象同时处理输入、AI、物理、渲染、声音等职责。

对 SlimeAI 的启发很直接：当前 `EntityManager` 正在变成 ECS 版巨型对象。它应退回“身份和生命周期基础设施”，把玩法和表现桥接交给 owner service。

## 4. AiFirst 参考实现的可采纳思想

`SlimeAI-AiFirst` 已经走过一次 Relationship 收缩，虽然当前主线不应整体迁移到新 GameOS，但其中 Entity / Relationship 设计非常适合反向采纳。

关键设计：

- `EntityId` 是 `readonly record struct EntityId(string Value)`，不提供 implicit string 转换，见 `SlimeAI-AiFirst/GameOS/Runtime/Entity/EntityId.cs:5-54`。
- `EntityIdList` 是 typed 不可变多引用值，`Add / Remove` 返回新值，见 `EntityIdList.cs:7-148`。
- `EntityManager` 在 AiFirst 中是 static facade，见 `EntityManager.cs:9-87`；本轮旧 ECS 裁决不采纳“保留兼容 facade”，只采纳“小入口 + 委托到明确 Entity runtime”的边界思想。
- `LifecycleTree` 是唯一公开 Relationship 入口，并明确业务多引用必须走 typed DataKey，不通过 tree，见 `LifecycleTree.cs:6-18`。
- `LifecycleLink` 是 typed immutable record，生命周期策略是字段，不是字典，见 `LifecycleLink.cs:15-32`。
- `RuntimeOwnedReferenceRegistry` 负责 owner cleanup hook，见 `RuntimeOwnedReferenceRegistry.cs:6-114`。
- `OwnedReferenceDescriptor` 明确 child-to-owner DataKey 和 owner-list DataKey，见 `OwnedReferenceDescriptor.cs:5-16`。
- `EntityRegistry.DestroyImmediate` 的顺序是销毁 lifecycle children、detach lifecycle、owner cleanup、移除 entity、reset data、发 destroyed 事件，见 `EntityRegistry.cs:98-118`。

对应契约已经写入 `SlimeAI-AiFirst/DocsAI/GameOS/Contracts.md:210-227`：

- `EntitySpawnConfig` 只保留 `EntityId / DataCatalog / ParentEntityId / ParentDestroyPolicy`。
- 不再支持 `AutoAddParentRelation` 和 `ParentRelationTypes`。
- 业务 entity 引用必须是 `DataKey<EntityId?>` 或 `DataKey<EntityIdList>`。
- `LifecycleTree` 是唯一公开 Relationship 入口。
- Observation 不输出 `relationType`，业务引用走 typed reference dump。

这组思想和本轮外部框架结论一致，应作为旧 ECS Entity/Relationship 重构的目标形状。

## 5. Relationship 是否继续保留

### 5.1 保留的部分：生命周期树

必须保留：

- parent -> child lifecycle link
- 单 parent 约束
- 防 self parent
- 防 cycle
- parent destroy policy：`DestroyRecursively` / `Detach`
- children snapshot 查询
- parent chain 查询
- attach / detach 事件
- 销毁时稳定处理 owned children

建议命名：

- `LifecycleTree`
- `LifecycleLink`
- `LifecycleParent`
- `ParentDestroyPolicy`

避免继续叫“通用 Relationship”，因为它会诱导 AI 把任何业务引用都塞进去。

### 5.2 迁出的部分：业务引用

应迁出通用 Relationship：

| 当前 relationType | 推荐归属 |
| --- | --- |
| `ENTITY_TO_PROJECTILE` | `ProjectileDataKeys.SourceEntity` + owner 的 `SpawnedProjectileIds: EntityIdList`，由 Projectile capability 注册 cleanup hook。 |
| `ENTITY_TO_EFFECT` | `EffectDataKeys.SourceEntity` + owner 的 `SpawnedEffectIds: EntityIdList`，由 Effect capability 注册 cleanup hook。 |
| `ENTITY_TO_ABILITY` | `AbilityDataKeys.OwnerEntity` + owner 的 `OwnedAbilityIds: EntityIdList`，由 Ability capability 管理。 |
| `ENTITY_TO_ITEM` | Inventory / Equipment capability 的 typed slot 或 `OwnedItemIds`，不进入 Entity core。 |
| `ENTITY_TO_UI` | GodotBridge / UI registry，例如 `UiBindingRegistry(entityId -> control)`，不进入 gameplay lifecycle。 |
| `ENTITY_TO_COMPONENT` | ComponentRegistrar / GodotNodeRegistry 内部 owner index，不作为 public relationship type。 |
| `ABILITY_TO_EFFECT`、`ITEM_TO_ABILITY` | 由 Ability / Feature / Item capability 的 typed DataKey 或 service index 表达。 |

### 5.3 降级的部分：Debug / AI 可视化

如果 AI 或调试确实需要“看见关系图”，不要把它做成 gameplay API。推荐输出低频 observation：

- `LifecycleTree dump`
- `TypedReference dump`
- `CapabilityIndex dump`
- `GodotNodeRegistry dump`
- `EntityFact audit`

`EntityFact` 可以包含字符串字段，因为它是 observation artifact，不参与运行时核心语义。AI-first 不是让 runtime 更像图数据库，而是让 runtime 的事实可导出、可验证。

## 6. 子弹、武器、玩家、伤害统计如何建模

用户提出的例子：

```text
子弹由武器发出
武器属于玩家
子弹造成的伤害加到玩家造成伤害里面
```

当前做法大致是：

```text
Projectile/Effect/Ability -> PARENT ancestor chain -> 找 IWeapon / IUnit -> 累计统计
```

这有两个问题：

1. 生命周期 parent 被用作统计归因。
2. 统计逻辑依赖运行时对象层级形状，而不是依赖伤害事件本身携带的事实。

推荐改为 typed attribution：

```text
DamageAttribution
  SourceEntityId       // 直接造成伤害的实体，例如 projectile/effect/weapon hitbox
  CasterEntityId       // 释放者，通常是 player/enemy unit
  WeaponEntityId       // 可选，武器实体
  AbilityEntityId      // 可选，技能实体
  DamageCreditEntityId // 主计分实体，通常是玩家或召唤物 owner
```

或更贴合旧 DataOS 的 DataKey：

```text
Projectile.SourceEntityId
Projectile.CasterEntityId
Projectile.WeaponEntityId
Projectile.AbilityEntityId

DamageInfo.Attribution
DamageInfo.DamageCreditEntityId
```

统计时：

- `StatisticsProcessor` 先读 `DamageInfo.Attribution`。
- 玩家总伤害写 `DamageCreditEntityId`。
- 武器统计写 `WeaponEntityId`。
- 技能统计写 `AbilityEntityId`。
- 找不到 attribution 时视为数据错误或测试失败，不回退 legacy `PARENT` ancestor chain。

生命周期时：

- Projectile 可以挂到 caster 或 scene root 下，按清理需求决定。
- Weapon 属于 Player 的 inventory，不必等同 lifecycle child。
- 如果 player 销毁要销毁 weapon / ability / projectile，走各自 capability 的 cleanup hook 或 lifecycle link。

这样建模更 AI-first，因为 AI 不需要猜“这个 parent 是统计 parent 还是销毁 parent”。每个字段名称都直接表达用途。

## 7. `EntityManager` 重构目标形状

### 7.1 不保留兼容 facade

本轮裁决改为完整重构：不把旧 `EntityManager.Spawn / Destroy / Register / GetAll` 包成兼容 facade，也不允许 `EntityManager` 继续兼任 Ability、Migration、Relationship、Component owner 和业务查询入口。

新的 Entity runtime 只保留一个清晰边界：

```text
EntityManager
  Create / Destroy / Get / Snapshot
  只处理 EntityId、注册表、生命周期结构变更和事件发布
```

旧 API 不做长期 adapter。执行型 SDD 可以在同一 PR 内批量改调用点，但完成状态必须是：

- 没有 `ParentRelationTypes`。
- 没有 `EntityRelationshipType` gameplay 调用。
- 没有 `EntityRelationshipManager` runtime hot path。
- 没有 `EntityManager_Ability` 这类业务 partial。
- 没有 `EntityRelationshipTraversal.GetAncestorChain` 作为统计主路径。

### 7.2 拆分建议

| 新边界 | 职责 | 从当前哪里迁出 |
| --- | --- | --- |
| `EntitySpawnPipeline` | 新 Entity 创建流程事实源，替代旧 `EntityManager.Spawn` 大函数 | `EntityManager.Spawn` |
| `EntityNodeFactory` | 场景实例化、对象池取出、添加到场景树 | `Spawn` 的创建路径 |
| `EntityDataInitializer` | Runtime snapshot apply、`GeneratedDataKey.Id` 写入、ID 一致性校验 | `ApplySpawnData` |
| `EntityVisualInjector` | VisualScene 加载和碰撞模板同步 | `InjectVisualScene` |
| `EntityTransformInitializer` | Position / Rotation / ForceUpdateTransform / 对象池物理同步前置 | `Spawn` transform 段 |
| `EntityRegistry` / `NodeLifecycleRegistry` | typed `EntityId` 注册、反查、快照查询 | `NodeLifecycleManager` + `EntityManager.Register/Get` |
| `ComponentRegistrar` | component 扫描、注册、注销、owner index；不通过 public Relationship 表 | `EntityManager_Component.cs` |
| `LifecycleTree` | lifecycle parent attach/detach/query/destroy policy；唯一保留的关系内核 | `EntityManager_Relationship.cs` + `EntityRelationshipLifecycle.cs` |
| `OwnedReferenceRegistry` | 业务 owner-list cleanup hook | 新增 capability owner hook |

| `EntityObservationDumper` | 输出 lifecycle / typed reference / registry fact | 新增 observation/debug |

### 7.3 旧 partial 的处理

应停止新增 `EntityManager_*.cs` 业务 partial。已有 partial 迁移顺序：

1. `EntityManager_Ability.cs` 迁到 AbilitySystem owner service，例如 `AbilityInventoryService` / `AbilityService`。
2. `EntityManager_Migration.cs` 迁到 `EntityMigrationService`。
3. `EntityManager_Component.cs` 改为 `ComponentRegistrar`，删除对 public Relationship 的依赖。
4. `EntityManager_Relationship.cs` 替换为 `LifecycleTree`，不保留 relationType adapter。

## 8. 完整重构执行路线

### P0：建立 hard cutover 基线

目标：明确本次不是兼容迁移，而是替换旧 Entity / Relationship runtime。

- 当前行为只作为理解输入和验收对照，不作为新 API 兼容目标。
- 新执行 SDD 必须列出要删除的旧 API：`EntityRelationshipManager`、`EntityRelationshipType`、`ParentRelationTypes`、`BindParentRelationships(params string[])`、业务 `EntityManager_*.cs` partial。
- 先写新行为验收：typed `EntityId`、`LifecycleTree`、typed business reference、explicit `DamageAttribution`、component owner index、Godot scene smoke。
- 旧测试若依赖 relationType，应改写为新行为测试，而不是加 adapter 让旧测试继续过。

### P1：typed `EntityId` 作为唯一身份

目标：先解决 raw string 和 ID 双语义。

- 引入 typed `EntityId`，不做“至少 wrapper”这种折中。
- 确认 `DataKey.Id` 与 `GeneratedDataKey.Id` 是否同一语义。
- 删除 `ResolveEntityId` 的静默 InstanceId fallback；所有 runtime entity 必须注册后拥有 typed `EntityId`。
- 新 API 禁止 `string entityId`，旧调用点在同一切片内改完。
- Observation 中可以序列化为 string，但 runtime 内存 API 使用 typed value。

### P2：用 `LifecycleTree` 替换旧 `PARENT`

目标：不并行、不双写，直接把 lifecycle parent 迁出通用 Relationship。

- 新增 `LifecycleLink(parentId, childId, destroyPolicy, priority)` typed record。
- 新增 `Attach(parent, child, policy)`、`Detach(child)`、`GetParent(child)`、`GetChildren(parent)`、`DetachAll(entity)`。
- 保证单 parent、防环、防 self、空 ID 拒绝。
- `EntityManager.Destroy` 只看 `LifecycleTree` 做递归销毁。
- 删除 `PARENT` relationType 写入和读取，不做 `PARENT` 双写。

### P3：删除业务 relationType，改 typed DataKey / service index

目标：把 gameplay 从通用关系图迁出。

- Projectile：`SourceEntityId`、`AbilityEntityId`、`TargetEntityId`、`SpawnedProjectileIds`。
- Effect：`SourceEntityId`、`AbilityEntityId`、`TargetEntityId`、`SpawnedEffectIds`。
- Ability：`OwnerEntityId`、`OwnedAbilityIds`。
- Item / Weapon：Inventory / Equipment owner service。
- UI：UI binding registry。
- Component：ComponentRegistrar 内部 owner index。
- Damage：`DamageAttribution` 替代 ancestor chain 主路径。

### P4：重建 `EntityManager` 边界

目标：删除大函数和业务 partial，不做 facade 兼容。

- `Spawn` 变成新的 pipeline 编排，旧 signature 不作为保留目标。
- 每个阶段有明确输入、输出和失败策略。
- 注册、Data 初始化、Visual 注入、Transform、Component、Lifecycle、事件广播分别测试。
- `EntityManager_Ability` 从 Entity core 删除，公开入口归 Ability owner service。
- `EntityManager_Migration` 从 Entity core 删除，若需要保留，转为独立一次性 migration tool。

### P5：删除 `EntityRelationshipManager`

目标：让 runtime 没有通用字符串关系 hot path。

- 删除 `EntityRelationshipManager.cs`。
- 删除 `EntityRelationshipType.cs`。
- 删除 `EntityRelationshipTraversal.cs` 的 gameplay 入口；如需 parent chain，改为 `LifecycleTree` typed 查询。
- 删除 `EntityRelationshipLifecycle.cs`，其能力合并为 typed `LifecycleLink` / `LifecycleTree`。
- Debug 另建 observation，不复用旧 Relationship runtime。

## 9. 验证策略

### 9.1 必测行为

Lifecycle：

- attach / detach 成功事件。
- self parent 拒绝。
- cycle 拒绝。
- child 已有 parent 拒绝或显式 reparent 规则。
- parent destroy policy = recursive 时递归销毁。
- parent destroy policy = detach 时 child 存活并断链。
- destroy 中遍历 snapshot，不因关系变更破坏枚举。

EntityId：

- Spawn 后 Data ID、registry ID、lifecycle ID 一致。
- 手动 Register 必须显式提供 typed `EntityId`，不允许 legacy node fallback。
- `EntityId.Empty` 不能作为 parent/child。

业务引用：

- Projectile spawn 写 `SourceEntityId` 和 source 的 `SpawnedProjectileIds`。
- Effect spawn 写 `SourceEntityId` 和 source 的 `SpawnedEffectIds`。
- Ability add 写 `OwnerEntityId` 和 owner 的 `OwnedAbilityIds`。
- Destroy child 后 owner list 自动清理。
- Destroy owner 后 lifecycle children 按策略处理，业务弱引用不误销毁。

Damage attribution：

- projectile 造成伤害时 player / weapon / ability 的统计分别写入。
- lifecycle parent 改变不影响 damage credit。
- 缺失 attribution 时失败，不允许 parent-chain fallback。

Component / Godot：

- component 注册顺序仍满足 `OnComponentRegistered` 前 owner 可查询。
- 对象池复用仍先 transform，再 component，再 activate。
- UI / adapter 绑定不进入 lifecycle tree。

| 方案 | 描述 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| A. 保留现有通用 Relationship，只加类型常量 | 最小改动 | 改动少 | 不能解决 AI-first 和字符串图问题 | 拒绝 |

### Recommendation

选择 C，但按 hard cutover 执行：

```text
LifecycleTree = 唯一 runtime relationship core
Business reference = typed EntityId / EntityIdList DataKey
Debug graph = observation artifact
EntityManager = 新 Entity runtime 入口 + spawn pipeline
Legacy EntityRelationshipManager = 删除
```

原因：

- 保留了其他 ECS 中已经证明有价值的 hierarchy / relationship。
- 删除了 SlimeAI 当前最不 AI-first 的裸字符串通用关系图。
- 不做兼容层，避免旧 API 长期存在并继续误导 AI。
- 和 `SlimeAI-AiFirst` 已验证方向一致。

### Must Confirm

后续进入执行 SDD 前必须确认：

- `DataKey.Id` 与 `GeneratedDataKey.Id` 是否等价。如果不是，先做 ID 统一。
- `PARENT` 当前是否只允许一个 parent。若历史数据有多 parent，需要迁移策略。
- 技能、武器、投射物、特效是否都应该随 owner 销毁，还是部分应 detach。
- Damage 统计最终要同时记 player、weapon、ability，还是只记 player + 可选来源。
- `EntityManager_Ability` 删除后由哪个 Ability owner service 暴露新公开 API。

### Should Confirm

建议确认：

- 是否需要保留一个 debug-only `EntityFact`。
- UI 绑定是否归 GodotBridge 还是 UI module。
- Component owner index 是否继续支持反查 entity，还是改为 component 持 owner typed id。
- 是否先在 `SlimeAI/Src/ECS/Base/Entity/**` 旁文档和 Entity owner skill 增加禁止新增业务 relationType 的规则。

### Defaults I Will Use

如果后续没有额外输入，默认采用：

- 每个 entity 至多一个 lifecycle parent。
- `ParentDestroyPolicy.DestroyRecursively` 是默认策略。
- 业务引用不参与递归销毁。
- Damage attribution 是统计唯一事实，缺失即失败。

不建议：

- 继续新增 `EntityRelationshipType` 字符串常量。
- 把 `RelationshipRecord.Data` 扩成更多业务字段。

- 让 `EntityManager` 继续新增 `EntityManager_Projectile`、`EntityManager_Item`、`EntityManager_UI` 这类业务 partial。

## 11. Research Adoption 记录

### Evidence

- 当前代码直接显示 `EntityManager.Spawn` 承担对象池、场景实例、Data、Visual、Transform、Relationship、Component、事件广播。
- 当前代码直接显示 Relationship 主存储为 raw string id / raw string relationType / dictionary data。
- 当前业务代码直接依赖 `ENTITY_TO_PROJECTILE`、`ENTITY_TO_EFFECT`、`ENTITY_TO_ABILITY` 和 `PARENT` ancestor chain。
- `SlimeAI-AiFirst` 直接提供 typed `EntityId`、`LifecycleTree`、`EntityIdList`、`RuntimeOwnedReferenceRegistry` 的可参考实现。
- Bevy、Flecs、Unity、Friflo、EnTT 官方资料和本地报告都支持：关系/层级存在，但应 typed、受约束、语义分层。

### Inference

- 当前 Relationship 的冗余来自语义过宽，而不是来自“关系”概念本身。
- `EntityManager` 的大改应直接重建边界，删除业务 partial 和 legacy API，而不是 facade 化旧入口。
- Damage attribution 作为数据事实比 ancestor traversal 更稳定，更适合 AI 理解和测试。
- Debug 关系图应从 runtime core 下沉为 observation artifact。

### Unknown

- 当前 `GeneratedDataKey.Id` 与 `DataKey.Id` 的最终映射还需要代码级确认。
- 旧场景或测试中是否存在非 Spawn 注册的 entity，并依赖 InstanceId fallback；若存在，需要改成显式 typed `EntityId`。
- 是否存在某些业务关系确实需要高性能反向索引；如果有，应由 capability owner 建 index，而不是恢复通用 Relationship。

### Adopt Now

- 删除裸字符串 relationType runtime API。
- 设计目标定为 `LifecycleTree + typed business references`。
- `EntityManager` 重建为单一 Entity runtime 入口，不兼任业务 manager。
- 统计归因设计改为 explicit attribution。

### Adopt Later

- Typed relation 框架。
- Deferred structural command buffer 覆盖 spawn/destroy/attach/detach。
- Entity observation graph / fact audit。
- Capability-owned high-performance indexes。

### Reject

- 通用字符串 Relationship DSL。
- 运行时 gameplay hot path 依赖 `Dictionary<string, object>` relation data。
- 把所有业务引用都做成 lifecycle parent。
- 复制 Bevy / Flecs / Unity 的完整 ECS runtime。

### Copied Code Or Assets

无。本文只采纳设计思想和边界判断，没有复制外部代码或资产。
