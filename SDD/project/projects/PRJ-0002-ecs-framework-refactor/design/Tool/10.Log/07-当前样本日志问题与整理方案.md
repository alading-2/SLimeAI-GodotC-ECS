# 当前样本日志问题与整理方案

> 更新：2026-06-10
> 状态：current design note
> 样本：`.ai-temp/log-runs/20260610-013907/raw/scene-log.jsonl`

## 1. 真实问题

本轮样本证明：当前方向不是“JSON 错了”，而是 **JSONL 还没有被整理成 AI 可分析的 artifact**。

`scene-log.jsonl` 已经有 `runElapsedMs / frame / physicsFrame / severity / owner / context / operation / phase / fields` 这些 envelope 字段，这比旧 `GD.PrintRich` 文本强很多。但 AI-first 的目标不是“把每行打印变成 JSON”，而是让 AI 可以稳定回答：

- 这次 run 是否真的通过，依据是 artifact、structured validation，还是 fallback？
- 问题发生在哪个 phase、owner、operation、entity 或 flow？
- 哪些日志是重复噪声，应该合并、采样或只进 JSONL？
- 哪些字段缺失导致无法判断？
- 下一轮应该打开哪个 owner、关闭哪个 context、补哪个 `Log.md`？

如果这些答案仍要 AI 从 4914 行 raw JSONL 里临场归纳，那么只是把“全量 stdout 难读”换成“全量 JSONL 难读”。

## 2. 样本证据

本轮检查范围：

```bash
wc -l .ai-temp/log-runs/20260610-013907/raw/scene-log.jsonl
jq -r '.severity' .ai-temp/log-runs/20260610-013907/raw/scene-log.jsonl
jq -r '.owner' .ai-temp/log-runs/20260610-013907/raw/scene-log.jsonl
Workspace/Tools/logctl/logctl analyze --run-dir .ai-temp/log-runs/20260610-013907 --out .ai-temp/log-runs/20260610-013907/analysis
Workspace/Tools/logctl/logctl suggest --run-dir .ai-temp/log-runs/20260610-013907 --dry-run
```

关键事实：

| 观察 | 证据 | 判断 |
| --- | --- | --- |
| raw 体量已经超出 AI 直接阅读范围 | `scene-log.jsonl` 约 4914 行；`logctl analyze` 解析出 4915 条，最后一行是 invalid JSONL diagnostic | raw 只能作为证据源，不能作为默认提示词输入。 |
| severity 几乎不能降噪 | `Info` 约 4163 条，`Debug` 约 665 条，`Warn` 约 86 条 | 只调 level 会隐藏信息，不能解决重复 owner / operation。 |
| 噪声高度集中 | `TargetSelector/TargetQueryEngine/TargetQueryEntities` 约 3041 条；`ObjectPoolAcquire` 约 271 条；`ObjectPoolRelease` 约 179 条 | 应优先按 owner/operation 做预算、summary 和 query，而不是全局关日志。 |
| fields 仍大量为空 | 约 1109 条 `fields:{}` | JSON envelope 有了，但业务字段不足，AI 仍要读 message。 |
| operation 经常只是 context 复制 | 约 1109 条 `operation == context` | 这不是稳定业务操作名，后续应补 owner `Log.md` 和调用点契约。 |
| 当前 analyzer 产物偏机器 JSON | 有 `by-owner/*.jsonl`、`by-phase/*.jsonl`、`noise/summary.json`、`missing-fields/missing-fields.json`，但没有 `summary.md`、`noise/top-contexts.md`、`missing-fields/index.md` | 产物已经开始拆分，但还不是 AI 默认可读入口。 |
| `ai-context.md` 太薄 | 只列 runDir、resultSource、status、entries、owners 和两个 query 示例 | 它没有回答 top noise、missing fields、flow digest、owner 文档和下一轮建议。 |
| flow 识别过宽 | `flows/flows.json` 当前包含几乎所有有 `operation` 的 entry | flow 应只来自 `channel=Flow`、`entryType` 或明确 OperationTrace 标记，不能把普通 operation 全算 flow。 |
| 样本没有 Validation 事实 | `validationEntries=0`、`artifacts=0`，gate 结果来自 `structured-log` | “passed” 只能说明没有 structured failure，不等价于验证通过。 |

## 3. 为什么会感觉更麻烦

用户的感觉成立。当前 JSONL 让日志变得“机器可解析”，但同时暴露了新问题：

1. **行数没有减少**：4914 行 raw JSONL 比纯文本更长，直接读更累。
2. **字段重复但语义不够**：每行都有 envelope，但 `fields:{}` 和 `operation == context` 让 AI 仍然要读自然语言 message。
3. **过程没有真正聚合**：`ObjectPoolRelease completed` 连续刷几十行，仍然需要 analyzer 合并成一个 flow/noise 摘要。
4. **分析入口缺失**：`ai-context.md` 没有给 AI 足够的第一屏结论，AI 只能回到 raw。
5. **测试事实不足**：没有 Validation artifact 时，AI 只能说“没有看到失败”，不能说“行为符合预期”。

结论：**JSON 是基础格式，不是最终可读结果。AI-first Log 必须把 raw JSONL 再整理成可消费的分析目录。**

## 4. 解决方向

### 4.1 `logctl analyze` 必须生成 AI 入口文件

目标目录必须至少包含：

```text
analysis/
  summary.md
  ai-context.md
  gate-report.json
  raw/
  by-owner/
  by-phase/
  flows/
    index.md
    <Owner>/<Operation>.md
  failures/
    index.md
  noise/
    top-contexts.md
    suppressed.md
  missing-fields/
    index.md
```

`summary.md` 面向人和 AI 的第一眼总览，必须回答：

- run status 和 resultSource。
- entries / validationEntries / artifacts / structuredFailures。
- top owners / top phases / top noisy operations。
- 是否存在 invalid JSONL、missing required fields、unknown owner/phase。
- 这次 run 应该先读哪些 owner 文档和哪些 query。

`ai-context.md` 必须比 `summary.md` 更面向下一步分析，包含：

- failure focus。
- top noise with recommended action。
- flow digest。
- missing field digest。
- owner `Log.md` 链接。
- 需要补日志、调 profile、修测试还是修代码的初步分类。

### 4.2 flow 不能等于 operation

当前 `flows/flows.json` 把普通 `operation` 也纳入 flow，会制造假 flow。正确规则：

- `channel=Flow` 可以进入 flow。
- 或 `entryType=flow_summary|flow_step|flow_suppressed_summary` 可以进入 flow。
- 或 `fields.flowId / correlationId + outcome + durationMs` 满足完整 OperationTrace 契约时进入 flow。
- 普通 runtime log 的 `operation` 只能进入 `by-owner` / `by-phase` / `noise`，不能自动进入 `flows`。

### 4.3 noisy context 要给可执行建议

样本中的优先 owner：

| owner/context/operation | 样本问题 | 建议 |
| --- | --- | --- |
| `TargetSelector/TargetQueryEngine/TargetQueryEntities` | 约 3041 条，远超其他日志 | 默认 stdout 只输出异常和 summary；JSONL 采样；TargetQuery flow summary 记录候选数、过滤数、结果数、reasonCode。 |
| `ObjectPool/ObjectPool/ObjectPoolAcquire` | 约 271 条 | `PoolGetReleaseActivate` flow 聚合，按 poolName 预算；连续成功路径默认 summary-only。 |
| `ObjectPool/ObjectPool/ObjectPoolRelease` | 约 179 条，尾部出现同一帧大量 release | 合并成 release batch summary，保留 active/idle min/max/count。 |
| `Damage/DamageService/DamageProcess` | 约 111 条 | DamageProcess 保留完整 processor chain，但 stdout 只显示失败或异常 summary。 |
| `Runtime/HealthBarUI/HealthBarUI` | 同一 enemy created/bind/success 三条重复 83 次 | 合并为 UI bind flow 或只在失败时输出，成功路径计数即可。 |

### 4.4 missing-fields 要升级为 owner 任务

`missing-fields` 不能只列 JSON 数组。它要按 owner 输出：

- 哪些字段缺失。
- 哪些 message 只能靠自然语言判断。
- 对应 owner 应补的 `Log.md` 字段契约。
- 这属于 `Log gap`、`Profile issue` 还是 `Code issue`。

样本里 `fields:{}` 和 `operation == context` 很多，这不一定是代码 bug，但它是 **Log gap**。

### 4.5 Validation artifact 是通过判断的必要条件

样本 `resultSource=structured-log` 且 `validationEntries=0`、`artifacts=0`。这类 run 只能表达：

```text
没有发现 structured error/fail。
```

不能表达：

```text
测试行为已经通过验收。
```

因此 analyzer 的 gate report 应区分：

| 状态 | 含义 |
| --- | --- |
| `passed` | artifact 或 Validation 明确通过。 |
| `no-failure-observed` | 有 structured log，但没有 Validation artifact，只能说明没看到失败。 |
| `stdout-pattern-fallback` | 只能靠 legacy stdout pattern 判断，低可信。 |
