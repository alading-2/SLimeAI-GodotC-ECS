# 为什么需要第二部分：一次落地失败的复盘

> 更新：2026-06-11
> 状态：retrospective
> 作用：回答用户的追问——"明明做了 deepthink，需求也写清楚了，为什么重构 Log 时功能没实现？是 deepthink 问题，还是设计文档思路有问题？"

## 1. 用户原始问题（原文保留）

> 我记得打印信息总共才几千行，但是现在整理后既然有几万行，不是变得更加复杂了吗？这样根本不是 aifirst，这样对 ai 分析真的有好处吗？

> 而且我不是说不要现实时间的时间戳吗，真正需要的是游戏运行时间，但是也不需要每次打印都带上这些信息啊……如果只是一条信息带一大串标记，这样给 ai 看是没有任何意义的。不过如果你能够整理好，将打印信息的语义提炼出来，给 ai 分析，这对 ai debug 肯定有很大帮助，但是你现在整理之后没有提炼，而是这条信息原封不动就分发到对应的分类当中，这样根本不算整理。

> 为什么会出现这种问题，是设计文档的思路有问题？还是交流有障碍，为什么我的想法没有实现，功能没有实现，改来改去。

> 你要反省为什么会有这个问题，明明已经有 deepthink 了，为什么还会出现功能没实现的情况，文档应该写得挺清楚的……你去检查是 deepthink 问题，还是文档的思路有问题。

## 2. 先承认事实：用户判断全部成立

本轮用实测数据复核了三个抱怨，全部属实：

| 用户抱怨 | 实测证据 | 结论 |
| --- | --- | --- |
| 几千行整理后变几万行 | raw `scene-log.jsonl` = 4914 行；`analysis-next/` 总输出 = **113864 行**（放大约 23 倍） | 成立 |
| 每条日志带一大串标记、还带现实时间 | 每条 raw 行第一个字段就是 `timestampUtc`（墙钟时间）；envelope 元字段无条件写满每行 | 成立 |
| 整理只是原样分发到分类，没有提炼 | `flows/flows.json` 96924 行（占总量 85%，把 3730 条记录逐条美化展开）；`by-owner/*` 合计 4915 行、`by-phase/*` 合计 4915 行，都是 raw 的逐行复制 | 成立 |

真正做了"提炼"的只有 6 个 markdown digest，合计 **281 行**。也就是说：整理产物里 99.7% 是复制和展开，0.3% 才是提炼。用户的感受——"这根本不算整理"——是对实现的准确描述，不是错觉。

## 3. 这不是 deepthink 没做，也不是需求没写

容易给自己找的借口是"沟通有障碍"或"deepthink 不够"。但事实相反：

- **deepthink 做了**：`第一部分/README.md` 第 11 节有完整确认包（Goal / Context / Evidence / Options / Recommendation / Must Confirm）。
- **需求写了**：第一部分 README 第 10 节"完成定义"列了 9 条；`07` 文档把 T2.1~T2.7 拆成了带文件名、带函数名、带验收命令的实施清单。
- **方向对了**：JSONL envelope、sink 分离、profile/CLI 控制、Validation artifact——这些架构判断都正确，也都落地了。

所以问题不在"没想"或"没写"，而在 **想清楚的东西和写下来的东西，恰好漏掉了用户最在乎的那一件**。

## 4. 真正的根因：把"目标"翻译成了"可检查的形状"，而不是"可检查的结果"

用户要的核心动词是 **"整理 / 提炼"**。这个词在第一部分的所有文档里，从头到尾 **没有被下过一个可验证的定义**。

文档非常详细地规定了 **整理产物长什么样**（目录结构、有哪些文件、文件里有哪些小节），却几乎没有规定 **整理本身要达到什么结果**。对比一下：

| 文档写得极其具体的部分 | 文档完全没有量化的部分 |
| --- | --- |
| `analysis/` 下必须有 raw/by-owner/by-phase/flows/failures/noise/missing-fields/summary.md/ai-context.md | 整理后总行数应该 **远小于** raw，而不是更大 |
| `ai-context.md` 必须有哪 7 个小节 | "一次技能释放成功还是失败"应该 **一眼看出**，不用读 raw |
| flow 边界规则（什么进 flows） | 同一查询 3000 次应该 **压成一条模板+计数**，而不是 3000 棵树 |
| LogEntry 有哪 18 个字段 | 共享字段（owner/correlationId）不该 **每行重复** |

当验收口径是"目录和文件存在"时，一个理性的执行者会去**把目录建出来、把数据搬进去**——这恰好就是"按维度复制"。它能 100% 通过"文件存在吗"的检查，却 0% 实现"整理"。

这就是 Goodhart 定律的标准形态：**一旦把"形状"当成可度量的目标，"形状"就会取代"结果"被优化。** 文档把"形状"写得越细，执行就越被吸进"把形状做对"，反而离"把信息提炼出来"越远。

`07` 文档其实已经隐约察觉到这点，自己写下过一句："把结构化等同于整理"是根因。但它开出的药方（T2.1~T2.7）仍然是在加"形状"——补 summary.md、补 noise md、补 missing-fields md——而没有把验收口径换成"结果"。所以同一个根因在 analyzer 这一层又复发了一次。

## 5. 三个抱怨其实是同一个病的三种症状

把三个抱怨摆在一起看，会发现它们不是三件事，是一件事：

```text
病根：整理的"单位"错了——以"单条日志(entry)"为单位，而不是以"一次操作(flow)"为单位
  │
  ├─ 症状A 行数暴增：以 entry 为单位，就只能"按维度把 entry 复制几份"，N 个维度放大 N 倍
  ├─ 症状B 字段冗余：以 entry 为单位，每条都得自带全部上下文(owner/correlationId/时间)，没法上提共享
  └─ 症状C 没有提炼：以 entry 为单位，就永远是"分类摆放"，拼不出"一次技能释放到底成没成功"
```

只要整理单位还是"entry"，这三个症状无论怎么打补丁都治不好。换句话说，第一部分的设计在"记录层"（怎么把一条事实写成结构化 JSON）是对的、也完成了；但它没有意识到"整理层"需要一个 **完全不同的数据单位**。

## 6. 第二部分要换的那一个观念

第二部分不推翻第一部分。第一部分负责"把事实记成结构化 entry"，这层保留。第二部分只改 **整理层的单位**：

```text
第一部分（记录层）：LogEntry —— 一条事实，保留
第二部分（整理层）：Flow 树 —— 一次操作的所有 entry，按 correlationId 重建成一棵树
```

- 整理的产出单位从"entry"变成"flow 结论"。一次技能释放 = 一棵树 = 一行结论（成功/失败 + 耗时 + 失败在哪步）。
- 结论的总量与 flow 数量同级（几百），不再与 entry 数量同级（几万）。**行数暴增从根上消失。**
- 共享字段上提到树根，entry 只留每步真正变化的东西。**冗余字段从根上消失。**
- 成功 flow 默认折叠成统计，失败 flow 才展开。第一屏就是"成功 N / 失败 M + 失败清单"。**"一眼看出成败"自然成立。**

详细方案见 `01-语义提炼整理设计.md`。

## 7. 给流程本身的两条修正（避免第三次复发）

这次复盘的价值不只在修 Log，更在修"为什么 deepthink + 设计文档仍然漏了目标"。两条修正：

1. **deepthink / 设计文档必须给核心动词下"可验证的结果定义"，而不是只给"产物形状"。**
   - 反面教材：本轮"整理"只定义了产物目录，没定义"整理后行数应显著下降""失败 flow 应一眼可见"。
   - 正面做法：任何"整理 / 优化 / 简化 / 提炼"类需求，验收必须包含 **结果指标**（如 `输出行数 < raw 行数`、`失败 flow 在第一屏可定位`），而不是 **存在性指标**（如 `summary.md 存在`）。
   - 这条建议同样适用于 SystemAgent 其它 workflow：当用户的动词是一个"结果"，gate 就不能只检查"形状"。

2. **设计文档要砍掉过程脚手架，把"真实问题 + 结果定义"顶到最前。**
   - 用户原话："设计文档虽然有很多个但是我总感觉没有重点，废话多。"这是准确的。第一部分 README 把整个 DeepThink 确认包（Goal/Context/Evidence/Options/Must Confirm/Should Confirm/Defaults）原样留在正文，真正的"完成定义"被压到第 10 节，"整理到底指什么"反而没人正面回答。
   - 正面做法（已写入 `systemagent-design-document` skill 的精神）：设计文档先写**真实问题**，再写**解决思路和可验证结果**；DeepThink 的内部确认字段压缩进附录或迁到 `source-request.md`，不占正文标题序列。

## 8. 一句话结论

> 不是 deepthink 没做，也不是没沟通。是把用户要的"结果（提炼）"翻译成了"形状（目录）"，然后照着形状实现，形状达标了、结果没达到。第二部分的任务，就是把"整理"重新定义成一个 **有结果指标、以 flow 为单位** 的动作。
