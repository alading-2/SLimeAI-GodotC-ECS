# 语义提炼整理设计

> 更新：2026-06-11
> 状态：current design（analyzer 默认入口已落地）
> 前置：先读同目录 `00-为什么需要第二部分.md`
> 一句话：把整理的单位从「一条日志(entry)」换成「一次操作(flow 树)」，让产物以"结论"为主、"原始行"为辅。

## 1. 真实问题（不重复复盘，只锁定本文要解决的）

第一部分把"记录"做好了：每条日志是结构化 `LogEntry`。但"整理"被实现成了"按 owner/phase/flow 把同一批 entry 复制几份 + 把 flow 全量美化展开"，于是 4914 行变 113864 行，而且看不出"一次技能释放成没成功"。

本文要解决的就一件事：**定义"整理 = 什么"，并让它产出的东西，AI 一眼能用、行数显著变少。**

## 2. 解决思路：以 flow 为单位重建，而不是以 entry 为单位分发

### 2.1 核心模型：correlationId = 一棵 flow 树

业界（OpenTelemetry Trace）早把这件事标准化了，本文直接借用，不发明新概念：

| 业界概念 | 本项目已有字段 | 含义 |
| --- | --- | --- |
| trace_id | `correlationId` | 同一次操作的所有日志共享，用来把散点 entry 收拢成一棵树 |
| span / 子 span | `operation` / `phase` / `stepIndex` | 树上的节点；一次操作内的阶段和步骤 |
| span.status | `outcome`(Succeeded/Failed) + `validationStatus` | 树根的 status 就是"这次操作成没成功" |
| span attributes | `fields` | 挂在对应节点上的业务细节 |
| span events | 失败 entry / warn entry | 挂在出错节点上的关键时间点 |

整理的第一步，就是 **按 `correlationId` group**，把属于同一次操作的 entry 收成一棵树。已经有 `correlationId` 的 flow（如 AbilityTryTrigger，样本里每次释放都有独立 correlationId）天然成树；没有 correlationId 的普通 runtime 日志，按 `(owner, context)` + 时间窗口兜底归并，并在 missing-fields 里标记"该 owner 缺 correlationId，无法精确成树"。

### 2.2 每棵树产出一个「结论对象」，而不是展开原始行

这是和第一部分最本质的区别。整理的产出单位是 **flow 结论**，不是 entry：

```jsonc
// 一次技能释放，整理后就是这一个对象（不是 N 条原始日志）
{
  "flowId": "ad3926eb",              // correlationId 短码
  "type": "AbilityCast",             // owner.operation
  "entity": "player_001",
  "outcome": "Failed",               // 一眼看成败
  "durationMs": 18,
  "frameRange": [1263, 1266],        // 游戏帧，不是墙钟
  "steps": 7,
  "failedStep": "TargetQuery",       // 失败在哪一步
  "reasonCode": "NoCandidateInRange",// 为什么失败（稳定码，不是自然语言）
  "keyFields": { "ability": "冲刺", "targetCount": 0 },
  "entryCount": 12,                  // 这棵树压了多少条原始 entry
  "rawRef": "raw/scene-log.jsonl:3297-3309"  // 要细看时去哪查
}
```

- 结论对象总量 ≈ flow 数量（几百级），不再 ≈ entry 数量（几万级）。**这一条直接消灭 23 倍膨胀。**
- 失败时 AI 看 `failedStep + reasonCode` 就能定位，不用读 raw；要细看才按 `rawRef` 回查。

### 2.3 字段上提：共享的进树根，每行只留差异

回应"每条日志带一大串标记字段"。按 OTel 的属性分层原则——**一个字段放在某一层后，不在其它层重复**：

| 层 | 字段 | 出现次数 |
| --- | --- | --- |
| run 级（整个 run 一次） | runId、sceneName、profile、wallClock 起点 | 1 |
| flow 树根（每次操作一次） | owner、context、correlationId、operation、entity、outcome、duration、frameRange | 每 flow 1 次 |
| step entry（每步一行） | phase、stepIndex、本步特有 fields、reasonCode（失败时） | 每步 1 次 |

step entry **不再重复携带** owner/context/correlationId/时间基准——它们继承自树根。要还原完整上下文时，靠树结构关联，不靠每行自带。

## 3. 呈现规则：failure-first，成功折叠、失败展开

借鉴 Honeycomb / Sentry 的"先结论后细节"：

### 3.1 第一屏永远是结论总览

```text
# Run 结论
gate: no-failure-observed  (无 Validation artifact，只能说"没观察到失败")
flows: 312 total → 成功 305 / 失败 7 / 异常 0

## 失败 flow（7）—— 直接展开
1. AbilityCast  entity=player_001  failedStep=TargetQuery  reason=NoCandidateInRange  (×3)
2. DamageProcess entity=enemy_044  failedStep=ApplyHp      reason=TargetAlreadyDead  (×2)
...

## 成功 flow（305）—— 折叠为统计
| type | count | avgDurationMs | 说明 |
| TargetQuery   | 3041 | 0.4 | 高频成功，已聚合，详见 noise |
| ObjectPoolAcquire | 271 | … |
```

- 失败 flow 在第一屏全部列出（每条一行：类型 + 失败步 + 原因 + 出现次数）。
- 成功 flow 默认只进统计表，不展开。**屏幕和上下文预算全留给异常。**

### 3.2 高频重复用模板聚合（Drain 式）

样本里 `TargetQueryEntities` 出现 3041 次、全部成功。这种 **不该是 3041 棵树**，应聚合成一条模板：

```text
template: TargetQuery(owner=TargetSelector op=TargetQueryEntities)
  count: 3041   outcome: 全 Succeeded
  candidateCount: min/avg/max = 0/4.2/19
  returnedCount:  min/avg/max = 0/2.1/8
  warnings: 0    truncated: 0
  异常样本: 无（若有 warn/fail，单独留 1~3 条完整样本）
```

- 同 `(owner, operation, 归一化结果)` 的成功路径，折叠成 `模板 + 计数 + min/avg/max + 异常样本`。
- 簇内任何 `outcome=Failed/Warn` 的样本 **单独保留完整树**，不被多数正常样本淹没。

## 4. 时间字段修正（回应"不要墙钟、不要每行一串标记"）

| 字段 | 第一部分现状（错） | 第二部分目标 |
| --- | --- | --- |
| 墙钟时间 | `timestampUtc` 是每行**第一个字段**、无条件写 | 改名 `wallClockUtc`，**可选**，默认不写每行；只在 run 级记一个起点，跨进程对齐时才按 profile 开 |
| 游戏时间 | runElapsedMs/frame/physicsFrame 已有但被墙钟淹没 | 成为默认时间基准；flow 树根用 `frameRange`，step 用相对偏移 |

console 与 JSONL 都遵循：**游戏运行时间为主，墙钟为可选对齐字段**。

## 5. 整理产物的目标结构

整理产物以"结论"为主，原始行只作可查询证据：

```text
analysis/
  summary.md              # 第一屏结论总览（3.1），AI 默认先读
  ai-context.md           # 面向下一轮 AI 的上下文和 query 建议
  flows/
    flows.jsonl           # 每行一个 flow 结论对象（2.2），紧凑、可查询；不是美化展开的原始 entry
    index.md              # flow outcome 统计和失败/警告 flow 表
  noise/
    templates.jsonl       # 高频成功模板聚合（3.2）
    templates.md
    top-contexts.md
  missing-fields/
    index.md              # 缺 correlationId / 缺 reasonCode / fields 空 的 owner 任务清单
  failures/
    index.md
  raw/
    entries.jsonl         # 唯一全量原始证据，默认不读
```

刻意**取消**了第一部分的 `by-owner/*.jsonl`、`by-phase/*.jsonl`（它们是 raw 的逐行复制）和 96924 行的 `flows.json`（全量美化展开）。owner/phase 维度改成在 `flows.jsonl` 上用 `logctl query` 二次筛选，不再物理复制成文件。复跑同一个 output 目录时也必须清理旧 `by-owner` / `by-phase` / `flows.json`，否则 stale 文件会继续污染默认 AI 入口。

## 6. 验收口径：用结果，不用存在性

这是第二部分和第一部分最重要的差别——**gate 检查"结果"而不是"文件在不在"**：

| # | 验收项 | 度量方式（可机器检查） | 反面（第一部分犯的错） |
| --- | --- | --- | --- |
| G1 | 整理后体量显著下降 | `analysis/` 总行数 < raw 行数（目标 < 1×，理想 ≤ 0.3×） | 只查"summary.md 存在" |
| G2 | 一眼看出成败 | summary.md 第一屏含 `成功 N / 失败 M`，失败 flow 全部列出 | 只查"ai-context.md 存在" |
| G3 | 失败可定位到步 | 每条失败 flow 有 failedStep + reasonCode + rawRef | 只查"flows 目录存在" |
| G4 | 高频被聚合 | 单一 (owner,operation) 成功路径在产物里 ≤ 1 条模板，不是 N 条 | flows.json 全量展开 3730 条 |
| G5 | 无冗余墙钟 | raw/产物里 step 行不以墙钟时间打头；wallClock 仅 run 级 | 每行 timestampUtc 打头 |
| G6 | 不读 raw 能定位 | 失败分类（Code/Test/Log gap/Runner）只凭 summary+flows 可得出 | 必须回 raw 拼流程 |
| G7 | raw 下钻显式 | `query --analysis-dir` 只查语义索引；需要 raw entry 时必须显式 `--file analysis/raw/entries.jsonl` | analysis query 没有语义结果时自动回退 raw |

只要 G1/G4 不达标（行数没降、高频没聚合），整理就判 **不通过**，无论目录多齐全。

## 8. 默认假设与需要确认

默认假设（无异议即采用）：
- 不引入外部日志平台（Serilog/OTel exporter），继续用本地 `logctl` + JSONL。
- flow 结论对象的字段集（2.2）先按样本里的 Ability/Damage/TargetSelector 定，后续按 owner 扩展。
- 取消 by-owner/by-phase 物理文件，改 query；如你仍想要物理分桶文件，告诉我。
