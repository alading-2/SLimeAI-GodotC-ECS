> 更新：2026-06-11
> 状态：current contract

## 用户原始问题在哪里

用户原始追问很长，已保存在 `00-为什么需要第二部分.md`。这里保留最核心的一句：

> 打印信息原本才几千行，整理后变成几万行；如果只是把原始信息分发到分类中，没有语义提炼，这根本不算整理，也不是 AI-first。

## 真实问题

问题不是“日志字段还不够多”，而是第一版把“整理”理解成了“把 raw entry 按 owner / phase / flow 复制到更多文件”。这会让 AI 读到更多重复字段，却仍然看不出一次操作是否成功、失败在哪一步、是否只是高频成功噪声。

AI-first Log 的整理层必须满足三个结果：

1. 默认可读产物比 raw 少，而不是更多。
2. 失败/警告 flow 在第一屏可见，并带 `failedStep / reasonCode / rawRef`。
3. 高频成功路径折叠成模板统计，不能逐条展开。

## 最终整理契约

`logctl analyze` 的默认产物只把 raw 当证据库，AI 默认读结论层：

```text
analysis/
  summary.md
  ai-context.md
  gate-report.json
  flows/
    flows.jsonl        # 每行一个 flow conclusion
    index.md
  noise/
    templates.jsonl    # 高频成功模板
    templates.md
    top-contexts.md
  missing-fields/
    index.md
    missing-fields.json
  failures/
    index.md
    failures.json
  raw/
    entries.jsonl      # 唯一全量原始证据，默认不读
```

默认不再生成：

- `by-owner/*.jsonl`
- `by-phase/*.jsonl`
- pretty `flows/flows.json`

复跑到已有 `analysis/` 目录时，也必须主动清理这些旧产物。仅仅“不再写新文件”不够，因为 stale `by-owner` / `by-phase` / `flows.json` 会继续误导 AI 读取旧入口。

二次筛选规则：

- `logctl query --analysis-dir <analysis>` 默认查语义索引：`flows/flows.jsonl` + `noise/templates.jsonl`。
- 需要原始 entry 时，显式用 `logctl query --file <analysis>/raw/entries.jsonl ...`。
- 如果语义索引为空，`query --analysis-dir` 返回空结果，而不是回退到 raw。
- 这条规则防止 query 又把几千条 raw entry 重新塞给 AI。

| 要求 | 状态 | 证据 |

逐条代码审查结论见 `04-当前实现审查报告.md`。该审查把第二部分 G1~G8 与 SDD-0040 的 T2.6 / final smoke 分开：analyzer 默认入口已完成；Validation artifact adoption 和 Godot scene smoke 仍未完成。

## 当前样本验证结论

对 `.ai-temp/log-runs/20260610-013907` 复跑语义 analyzer 后，核心结果是：

```text
rawLines=4915
defaultReadableLines=303
defaultReadableRatio=0.062
flowConclusions=3730
outputFlowConclusions=5
templatedSuccessFlows=3725
noiseTemplates=6
```

这说明“整理后更复杂”的核心问题已被默认入口解决：AI 默认读 303 行 digest，而不是 4915 行 raw，也不是 113864 行复制/展开产物。

但这个样本仍不能证明所有业务 flow 已完整打 step，因为它是旧样本，`singleEntry=3730`。这应被归类为 `Log gap / stale evidence`，不是 analyzer 失败。

## 仍未完成或后续优化

| 源码调用点语义化 | live stdout 仍可能分离；需要按 owner 迁移 `_log.Info` / 测试打印 / Debug UI 输出到 flow、summary、Validation artifact 或 debug profile | 第三部分 T3 |

## 不再采用的做法

- 不再用“目录存在”证明整理完成。
- 不再默认生成 owner/phase raw 分桶。
- 不再把 pretty JSON 当 AI 默认入口。
- 不再把 raw JSONL 或完整 stdout 直接塞给 AI。
- 不再把现实时间戳作为每条日志的默认首要时间字段。
