## 1. 用户原始问题

用户原始追问已完整保存在 `00-为什么需要第二部分.md` 的“用户原始问题”小节。本文只保留最核心的问题句：

> 打印信息原本才几千行，整理后变成几万行，这不是更复杂了吗？如果只是把原始信息分发到分类中，没有语义提炼，这根本不算整理，也不是 AI-first。

本文回答三个问题：

1. 当前问题是否真实存在。
2. 是 deepthink、设计文档还是实现/验收出了问题。

## 2. 结论先行

问题真实存在，且不是用户误解。

当前实现已经完成了“记录层”：结构化 `LogEntry`、sink、profile、gate 状态语义和第一批 digest 都有价值。但“整理层”仍没有完成：`analysis-next` 总行数是 raw 的 23 倍，`flows/flows.json` 是 entry 级美化展开，`by-owner` / `by-phase` 是物理复制，summary 只暴露 top noise 和缺字段，没有把一次操作提炼为 flow 结论对象。

这次偏差不是 deepthink 完全没做，也不是用户没有说清楚。根因是：文档和验收把“整理”翻译成了“产物形状”，没有把“语义提炼”定义成可机器检查的结果。代码随后按目录形状实现，自然就变成复制、分桶和展开。

下一轮实现不能再从“补几个文件”开始，必须先换验收口径：

- 默认 AI 入口的总行数必须显著小于 raw。
- 成功路径必须聚合，失败路径必须展开。
- 每个 flow 结论必须有 outcome、failedStep/reasonCode 或明确说明“缺字段，无法判断”。
- raw 只作为证据库，不允许再次物理复制成多个默认入口。

## 5. 为什么 deepthink 后仍然偏离

### 5.1 deepthink 的问题

deepthink 做了问题识别，但没有把用户最关键的动词“整理/提炼”改写成硬结果定义。它问了方向，也给了目录设计，却没有把以下问题设为 Must Confirm 或 gate：

- 整理后默认可读产物应该比 raw 少多少。
- “一次技能释放成功/失败”应该以什么对象表达。
- 高频成功日志应该默认折叠到什么粒度。
- 墙钟时间是否允许每行出现，默认是否关闭。

这不是 deepthink 完全失效，而是 deepthink 没把抽象词落到可检查结果。

### 5.2 设计文档的问题

第一部分设计文档把目录、文件、小节写得很细，但没有把“整理成功”定义成结果指标。于是实现者自然会优化“文件齐不齐”，而不是“AI 读起来是否少、准、能定位”。

第二部分 `01` 已经修正了方向，但仍需要补上本文这种逐条审查，否则下一轮仍可能只实现 `flows.jsonl` 文件名，而不实现 flow 结论质量。

### 5.3 实现和验收的问题

实现按旧目录形状推进，验收命令主要检查：

- analyzer 能跑。
- digest 文件存在。
- query 有结果。
- SDD validate 通过。
- build 通过。

这些都不能证明“语义提炼”成立。缺的 gate 是：

- 默认 digest 行数是否下降。
- flow 结论对象是否存在且字段完整。
- 成功路径是否真的聚合。
- 失败路径是否能不用 raw 定位原因。
- 当前样本是否仍被生成成 10 万行以上。

## 9. 外部资料校准

OpenTelemetry 的 trace/log 模型支持本文方向：一次操作应作为 span/trace 级对象承载状态、属性和事件，日志可以通过 trace/span id 关联到该操作。这只证明“以 operation/flow 为整理单位”是成熟建模方式，不表示本项目要引入外部 exporter 或平台依赖。

本项目采用的是本地化版本：

```text
correlationId -> trace/flow id
operation + phase + stepIndex -> span/step
outcome + validationStatus -> status
fields -> attributes
raw entry -> event/evidence
```

因此真正的 AI-first 产物应该是 flow conclusion，而不是 N 份 entry 副本。
