# Log 工具设计（两部分）

> 更新：2026-06-11
> 状态：current entry
> 作用：Log 工具设计入口。本目录分两部分；先读本文件确定该看哪部分。

## 两部分怎么分

| 部分 | 解决的问题 | 状态 | 入口 |
| --- | --- | --- | --- |
| 第一部分-记录管道与现状 | 怎么把一条日志**记录**成结构化事实（LogEntry / sink / profile / CLI / Validation / 现状复查） | 记录层已落地；整理层验收口径有缺陷 | `第一部分-记录管道与现状/README.md` |
| 第二部分-语义提炼整理 | 怎么把记录好的日志**整理提炼**成 AI 一眼能用的结论（flow 树 / 结论对象 / failure-first / 模板聚合） | analyzer 默认入口和 T2.5 hot spot 当前范围已落地；T2.6 Validation artifact 继续跟 SDD | `第二部分-语义提炼整理/README.md` |

## 一句话背景

第一部分做对了"记录"，但"整理"曾被实现成"按维度复制 + 全量展开"，导致 4914 行日志整理后变成 113864 行，且看不出一次操作成没成功。第二部分已把整理的单位从「一条日志」改成「一次操作(flow 结论)」，并把验收从"文件存在"改成"行数下降 + 失败可定位"。

详细复盘见 `第二部分-语义提炼整理/00-为什么需要第二部分.md`。
当前最终契约与完成清单见 `第二部分-语义提炼整理/03-最终设计与完成清单.md`。

## 该读哪部分

- 想了解 Log **怎么记录**、字段契约、sink/profile/CLI、测试统一：读第一部分。
- 想了解为什么"整理"没达预期、整理层**现在按什么契约实现**：读第二部分。
- 要改 `Log.cs` 记录逻辑：以第一部分 `02-目标架构与数据契约` 为准。
- 要改 `logctl analyze/query` 整理逻辑：先读第二部分 `03-最终设计与完成清单`，再用 `04-当前实现审查报告` 的 G1~G8 gate 验收；`02` 只作为修复前偏差复盘。
