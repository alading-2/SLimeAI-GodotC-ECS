# Log 工具设计（三部分）

> 更新：2026-06-11
> 状态：current entry
> 作用：Log 工具设计入口。本目录分三部分；先读本文件确定该看哪部分。

## 三部分怎么分

| 部分 | 解决的问题 | 状态 | 入口 |
| --- | --- | --- | --- |
| 第一部分-记录管道与现状 | 怎么把一条日志**记录**成结构化事实（LogEntry / sink / profile / CLI / Validation / 现状复查） | 记录层已落地；整理层验收口径有缺陷 | `第一部分-记录管道与现状/README.md` |
| 第二部分-语义提炼整理 | 怎么把记录好的日志**离线整理提炼**成 AI 一眼能用的结论（flow 树 / 结论对象 / failure-first / 模板聚合） | analyzer 默认入口已落地；不能代表源码调用点已全部语义化 | `第二部分-语义提炼整理/README.md` |
| 第三部分-源码调用点语义化 | 为什么用户运行游戏时打印仍然分离；后续 `Src/ECS` 调用点应该怎么改成 flow / summary / Validation / debug profile | draft direction，未实现，方向未冻结 | `第三部分-源码调用点语义化/README.md` |

## 一句话背景

第一部分做对了"记录"，但"整理"曾被实现成"按维度复制 + 全量展开"，导致 4914 行日志整理后变成 113864 行，且看不出一次操作成没成功。第二部分已把离线整理的单位从「一条日志」改成「一次操作(flow 结论)」，并把验收从"文件存在"改成"行数下降 + 失败可定位"。

但这仍不等于整个 Log 已完成。用户运行游戏看到的 live 打印仍可能分离，因为 `Src/ECS` 里大量业务、测试、Debug UI 调用点还没有迁移成 flow / summary / Validation artifact / debug profile。第三部分专门记录这个未完成的大任务，当前不得声称“Log 全部改完”。

详细复盘见 `第二部分-语义提炼整理/00-为什么需要第二部分.md`。
当前最终契约与完成清单见 `第二部分-语义提炼整理/03-最终设计与完成清单.md`。

## 该读哪部分

- 想了解 Log **怎么记录**、字段契约、sink/profile/CLI、测试统一：读第一部分。
- 想了解为什么"整理"没达预期、整理层**现在按什么契约实现**：读第二部分。
- 想了解为什么 live 打印仍分离、后续源码调用点怎么迁移：读第三部分。
- 要改 `Log.cs` 记录逻辑：以第一部分 `02-目标架构与数据契约` 为准。
- 要改 `logctl analyze/query` 整理逻辑：先读第二部分 `03-最终设计与完成清单`，再用 `04-当前实现审查报告` 的 G1~G8 gate 验收；`02` 只作为修复前偏差复盘。
- 要改 `Src/ECS` 日志调用点：先读第三部分并冻结 T3.0 的确认项，不要直接全仓机械替换 `_log.Info`。

## 当前不能说什么

- 不能说“Log 已全部改完”。
- 不能用 analyzer 默认入口 DONE 证明 live stdout 已经 AI-first。
- 不能把旧样本压缩比例当成新源码调用点迁移完成证明。
- 不能在没有新 run / Godot runner / 用户实际日志样本的情况下声称场景行为通过。
