# 目标架构与重构路线

> 更新：2026-05-31
> 状态：current design note

## 1. 目标

ObjectPool 的目标不是成为新的生命周期总线，而是成为 AI 可读、可验证的复用工具：

```text
少职责：
  复用、容量、归属、基础 Node 出入池状态

强契约：
  物理根节点默认脱树，非物理节点不脱树

可观察：
  每个池和关键 Node 都能导出状态、计数、策略和最后操作

可验证：
  有 Godot 场景测试证明旧位置回池/立即复用不产生幽灵碰撞
```

## 2. 模块拆分

### 2.1 ObjectPool<T>

保留为 public 主入口：

- `Warmup`
- `Get`
- `Activate`
- `Release`
- `ReleaseAll`
- `Cleanup`
- `Clear`
- `GetStats`

它不直接解释 Godot 物理细节，而是委托给策略类。

### 2.2 PoolNodeLifecycleStrategy

负责 Node 通用生命周期：

- 设置 `InPool` / `ObjectPoolName` meta。
- 处理 `ProcessMode`。
- 处理 `Visible`。
- 获取或创建 parent。
- discard 时选择 `Free` 或 `QueueFree`。

### 2.3 CollisionIsolationStrategy

负责物理根节点隔离：

- `ShouldDetach(Node node)`：默认 `node is CollisionObject2D`。
- `Park(Node2D node)`：移动到池泊车位并 `ForceUpdateTransform`。
- `DisableDeferred(Node node)`：常规递归关闭碰撞。
- `DisableDirectAfterReattach(Node node)`：挂树后同步关闭碰撞窗口。
- `EnableDeferred(Node node)`：激活时递归恢复碰撞。

### 2.4 PoolLifecycleContext

上下文只保存对象池策略需要的事实：

- `PoolName`
- `ParentPath`
- `ParkingPosition`
- `IsDeferredActivation`
- `Frame`
- `Reason`

不保存业务字段，不保存 `EntityId` 以外的关系语义。

## 3. 状态机

建议显式化池化 Node 状态：

```text
Created
  -> PooledInactive
  -> AcquiredUninitialized
  -> AttachedCollisionDisabled
  -> Active
  -> Releasing
  -> PooledInactive

Discarded 是终态。
```

状态含义：

| 状态 | 含义 |
| --- | --- |
| `PooledInactive` | 已回池，业务不可用；物理根节点应已脱树。 |
| `AcquiredUninitialized` | 已从池取出，但 Entity 数据、位置、组件尚未完成。 |
| `AttachedCollisionDisabled` | 已挂回场景树，但碰撞仍关闭。 |
| `Active` | `Activate` 完成，业务生命周期回调已触发。 |
| `Releasing` | 正在归还，生命周期清理中。 |
| `Discarded` | 超容量或清理时销毁。 |

## 4. 与 Entity hard cutover 的连接

Entity 重构后建议让 `EntityNodeFactory` 只调用对象池最小 API：

```text
AcquireFromPool:
  pool.Get(false)

ActivateAfterInitialization:
  pool.Activate(entity)

ReleaseToPool:
  ObjectPoolManager.ReturnToPool(entity)
```

Entity pipeline 负责：

- `EntityId` 分配。
- Data snapshot apply。
- Visual 注入。
- Transform 初始化。
- Component 注册。
- lifecycle event。

ObjectPool 不负责这些步骤。

## 5. 与 Collision 的连接

Collision owner 负责：

- `Area2D` 信号绑定和解绑。
- 组件退场时关闭监控。
- 事件发射前确认实体仍有效。
- 业务层过滤死亡、回池、阵营、owner 和生命周期状态。
- 场景结构门禁：空间链路必须保持 `Node2D` 继承。

ObjectPool 负责提供：

- 是否在池中。
- 是否已脱树。
- 是否已挂树但未激活。
- 最后 release/acquire 位置与帧。

## 6. 迁移步骤

### Step 1：无行为变化拆分

把 `ObjectPool.cs` 内的私有方法移动到同目录内部类：

- `SetCollisionTreeActive`
- `ApplyInactiveState`
- `ApplyActiveState`
- `ReattachToTree`
- `ForceDisableCollisionsDirect`
- `Discard`

第一步不改行为，不改 public API。

### Step 2：补显式状态和观测

新增只读状态快照：

```text
PoolNodeStateSnapshot
ObjectPoolNodeStateRegistry
```

状态记录可以先只对 Node 开启，纯 C# 对象不强制。

### Step 3：补验证场景

新增或扩展 Godot scene test：

- 根节点 `CharacterBody2D` 敌人回池复用。
- 根节点 `Area2D` 投射物回池复用。
- 非碰撞 UI 池不脱树。
- 普通 `Node` 阻断空间链路的负向检测。

### Step 4：更新 DocsAI 与 Skill

改动框架实现后同步：

- `DocsAI/ECS/Tools/ObjectPool/Concept.md`
- `DocsAI/ECS/Tools/ObjectPool/Usage.md`
- `DocsAI/ECS/Collision/Collision概念.md`
- `DocsAI/ECS/Collision/Collision使用说明.md`
- `.ai-config/skills/...` 中 ObjectPool / Collision / Entity 相关 skill
