# 目标架构与重构路线

> 更新：2026-06-03
> 状态：current design note

## 1. 目标

ObjectPool 的目标不是成为新的生命周期总线，而是成为 AI 可读、可验证的复用工具：

```text
少职责：
  复用、容量、归属、基础 Node 出入池状态

强契约：
  物理根节点默认脱树，非物理节点不脱树

可观察：
  每个池和关键 Node 都能导出状态、计数、策略和最后操作

可验证：
  有 Godot 场景测试证明旧位置回池/立即复用不产生幽灵碰撞
```

## 2. 模块拆分

### 2.1 ObjectPool<T>

保留为 public 主入口：

- `Warmup`
- `Get`
- `Activate`
- `Release`
- `ReleaseAll`
- `Cleanup`
- `Clear`
- `GetStats`

它不直接解释 Godot 物理细节，而是委托给策略类。

### 2.2 PoolNodeLifecycleStrategy

负责 Node 通用生命周期：

- 设置 `InPool` / `ObjectPoolName` meta。
- 处理 `ProcessMode`。
- 处理 `Visible`。
- 获取或创建 parent。
- discard 时选择 `Free` 或 `QueueFree`。

### 2.3 CollisionIsolationStrategy

负责物理根节点隔离：

- `ShouldDetach(Node node)`：默认 `node is CollisionObject2D`。
- `Park(Node2D node)`：移动到池泊车位并 `ForceUpdateTransform`。
- `DisableDeferred(Node node)`：常规递归关闭碰撞。
- `DisableDirectAfterReattach(Node node)`：挂树后同步关闭碰撞窗口。
- `EnableDeferred(Node node)`：激活时递归恢复碰撞。
- `ApplyOptionalDisableMode(Node node)`：可选验证 `CollisionObject2D.DisableMode = Remove`，但不作为替代脱树的默认策略。

2026-06-03 源码复核后补充：Godot 4.6.2 的 `DISABLE_MODE_REMOVE` 在 `ProcessMode.Disabled` 时通过 `area_set_space/body_set_space(RID())` 退场，恢复处理时再设置 world space。它和脱树共享一部分底层效果，但不能覆盖 `Get(false)` 到 `Activate()` 的半初始化窗口，也不能替代 Collision owner 的事件过滤。因此该策略保持 `Adopt Later / control check`，不进入默认行为。

### 2.4 PoolLifecycleContext

上下文只保存对象池策略需要的事实：

- `PoolName`
- `ParentPath`
- `ParkingPosition`
- `IsDeferredActivation`
- `Frame`
- `Reason`

不保存业务字段，不保存 `EntityId` 以外的关系语义。

## 3. 状态机

建议显式化池化 Node 状态：

```text
Created
  -> PooledInactive
  -> AcquiredUninitialized
  -> AttachedCollisionDisabled
  -> Active
  -> Releasing
  -> PooledInactive

Discarded 是终态。
```

状态含义：

| 状态 | 含义 |
| --- | --- |
| `PooledInactive` | 已回池，业务不可用；物理根节点应已脱树。 |
| `AcquiredUninitialized` | 已从池取出，但 Entity 数据、位置、组件尚未完成。 |
| `AttachedCollisionDisabled` | 已挂回场景树，但碰撞仍关闭；这是 `Get(false)` 到 `Activate()` 之间的关键安全窗口。 |
| `Active` | `Activate` 完成，业务生命周期回调已触发。 |
| `Releasing` | 正在归还，生命周期清理中。 |
| `Discarded` | 超容量或清理时销毁。 |

## 4. 与 Entity hard cutover 的连接

Entity 重构后建议让 `EntityNodeFactory` 只调用对象池最小 API：

```text
AcquireFromPool:
  pool.Get(false)

ActivateAfterInitialization:
  pool.Activate(entity)

ReleaseToPool:
  ObjectPoolManager.ReturnToPool(entity)
```

Entity pipeline 负责：

- `EntityId` 分配。
- Data snapshot apply。
- Visual 注入。
- Transform 初始化。
- Component 注册。
- lifecycle event。

ObjectPool 不负责这些步骤。

## 5. 与 Collision 的连接

Collision owner 负责：

- `Area2D` 信号绑定和解绑。
- 组件退场时关闭监控。
- 事件发射前确认实体仍有效。
- 业务层过滤死亡、回池、阵营、owner、source、target 和生命周期状态。
- 场景结构门禁：空间链路必须保持 `Node2D` 继承。

ObjectPool 负责提供：

- 是否在池中。
- 是否已脱树。
- 是否已挂树但未激活。
- 最后 release/acquire 位置与帧。

## 6. 迁移步骤

### Step 1：无行为变化拆分

把 `ObjectPool.cs` 内的私有方法移动到同目录内部类：

- `SetCollisionTreeActive`
- `ApplyInactiveState`
- `ApplyActiveState`
- `ReattachToTree`
- `ForceDisableCollisionsDirect`
- `Discard`

第一步不改行为，不改 public API。

### Step 2：补显式状态和观测

新增只读状态快照：

```text
PoolNodeStateSnapshot
ObjectPoolNodeStateRegistry
```

状态记录可以先只对 Node 开启，纯 C# 对象不强制。

建议首批字段：

```text
poolName
nodeName
nodeType
state
insideTree
inPoolMeta
needsDetach
lastAcquireFrame
lastReleaseFrame
lastAcquirePosition
lastReleasePosition
```

字段只用于 Observation / TestSystem，不进入 DataOS stable key。

### Step 3：重构 `Src/ECS/Tools/ObjectPool/Tests`

当前 `Src/ECS/Tools/ObjectPool/Tests` 目录先按职责拆分，不直接把 UI demo 硬改成自动化门禁：

```text
Src/ECS/Tools/ObjectPool/Tests/
  README.md
  ObjectPoolContractRuntimeTest.cs
  ObjectPoolCollisionIsolationValidation.cs
  ObjectPoolCollisionIsolationValidation.tscn
  ObjectPoolVisualTest.cs/.tscn        # legacy/manual demo，后续可改名
  ObjectPoolManagerTest.cs/.tscn       # legacy/manual demo，后续可改名
```

#### Step 3.1：Runtime contract checks

`ObjectPoolContractRuntimeTest` 覆盖纯对象池契约：

| Check | 内容 |
| --- | --- |
| `contract_warmup_stats` | 预热后 `Count / TotalCreated / ActiveCount` 正确。 |
| `contract_get_release_stats` | `Get / Release` 后统计、复用率和生命周期回调正确。 |
| `contract_duplicate_release_guard` | 重复归还被忽略，不让 `ActiveCount` 负数。 |
| `contract_capacity_discard` | 达到 `MaxSize` 后归还进入 discard，`TotalDiscarded` 正确。 |
| `contract_static_return_node` | Node 经 `ObjectPoolName` meta 静态归还到正确池。 |
| `contract_static_return_plain_object` | 纯 C# 对象只有经池 `Get()` 建立 mapping 后才能静态归还。 |
| `contract_active_snapshot_is_copy` | 活跃快照不因遍历期间 release 被破坏。 |
| `contract_manager_pool_isolation` | 测试池名不覆盖真实 `ObjectPoolNames`，退出只清理自身池。 |

测试池命名必须使用 `Test/ObjectPool/<CaseName>`，禁止直接使用 `ProjectilePool`、`EffectPool`、`EnemyPool` 等真实池名。

#### Step 3.2：Godot collision validation

`ObjectPoolCollisionIsolationValidation` 自动执行，不依赖 UI、鼠标、随机位置或人工观察。

| Check | 内容 | 通过标准 |
| --- | --- | --- |
| `collision_area_release_detaches` | 根节点 `Area2D` release | `InPool=true`，节点脱树，旧位置 entered 计数为 0。 |
| `collision_character_release_detaches` | 根节点 `CharacterBody2D` release | 节点脱树，速度清零，下一物理帧没有旧位置触发。 |
| `collision_get_false_attached_disabled` | `Get(false)` 挂回场景树 | 节点挂树但 `monitoring/monitorable=false`，shape disabled，未触发 entered。 |
| `collision_activate_after_transform` | 设置新位置后 `Activate()` | 只在新位置产生期望碰撞。 |
| `collision_immediate_reuse_same_frame` | 同帧或相邻帧 release → get(false) → activate | 没有旧坐标、旧 entity、旧 frame 的 entered。 |
| `collision_non_collision_node_not_detached` | 根节点 `Node2D` / `Control` release | 不脱树，只切 `ProcessMode` / `Visible`。 |
| `collision_artifact_oracle_complete` | artifact 自检 | 五字段非空，`checks[]` 覆盖本表，`failureReasons=[]`。 |
| `collision_shape_disable_recreates_pair` | 对照只切 `CollisionShape2D.disabled` | 记录 broadphase remove/create 对应 enter/exit 序列；该路径不能作为默认隔离成功条件。 |
| `collision_area_pair_monitorable_snapshot` | 对照 area-area pair 的 `monitorable` 快照风险 | 不用属性值推断事件状态，必须以实际 signal/query 序列为准。 |
| `collision_disable_mode_remove_control` | 对照 `ProcessMode.Disabled + DisableMode.Remove` | 若减少旧 pair，也只标为 control/adopt-later；脱树路径仍必须通过。 |
| `collision_detach_clears_space_and_monitoring` | 验证脱树触发 space=null 后清理监控状态 | release 后 `IsInsideTree=false`，旧位置 entered 计数为 0，并记录下一 physics frame。 |

普通 `Node` 阻断空间链路的负向检测属于 Collision scene structure gate；ObjectPool validation 只引用该风险，不继续在对象池里叠补丁。

#### Step 3.4：Manual demo 降级

现有 `ObjectPoolVisualTest` / `ObjectPoolManagerTest` 只保留人工演示用途：

- README 或文件名标记 `legacy/manual demo`。
- demo 池名改成 `Demo/ObjectPool/...`，避免覆盖真实 `ObjectPoolNames`。
- demo 退出时不调用全局 `ObjectPoolManager.DestroyAll()`，除非场景只创建隔离池并明确验证全局清理。
- demo 不产出 PASS/FAIL，不进入 scene-gate。

## 7. 验证门禁

文档阶段：

```bash
python3 Workspace/SDD/sdd.py validate --all
git diff --check
```

代码阶段：

```bash
Tools/run-build.sh
Tools/run-tests.sh
```

Godot 场景阶段：

```bash
cd /home/slime/Code/SlimeAI/Games/BrotatoLike
Tools/run-godot-scene.sh run res://Scenes/Main.tscn --timeout 10 --log-dir .ai-temp/scene-tests/runs
Tools/analyze-godot-scene-logs.sh
```

如果新增专门对象池碰撞验证场景，应把场景路径和 PASS artifact 写入对应 SDD。

完成定义必须同时包含：

- 文档入口能从 `DocsAI/ECS/Tools/ObjectPool/README.md` 找到设计裁决。
- `ObjectPool<T>` public API 未因内部策略拆分改变。
- 物理根节点仍默认脱树，非物理节点不会被误脱树。
- TestSystem 或 Observation 能看到节点级状态，而不只是池级容量。
- Collision owner 文档明确事件过滤职责，不把业务正确性推给 ObjectPool。
