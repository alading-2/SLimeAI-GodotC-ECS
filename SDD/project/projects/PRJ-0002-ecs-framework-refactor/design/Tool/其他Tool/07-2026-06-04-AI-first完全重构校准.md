## 1. Goal

本轮要解决：

- 用 AI-first 角度重新审查 `CommonTool` / `ResourceManagement` / `Math` / `NodeLifecycle` / `ParentManager` / `TargetSelector` 的方向。
- 将用户裁决写成项目级事实：AI 框架服务 AI，目标是让 AI 快速理解功能、边界和验证，不保护旧代码形状。

- 明确 `ParentManager` 是有用功能：统一管理大量 Entity / Tool / UI 节点在 SceneTree 中的挂载路径。
- 给出进入执行型 SDD 前必须确认的功能问题。

非目标：

- 本轮不改 C# runtime 实现。
- 本轮不创建新的执行型 SDD。
- 本轮不处理已独立设计的 Input / ObjectPool / Timer，不处理用户要求跳过的 Log。
- 不把 BrotatoLike 专属路径、玩法或资产变成框架默认。

## 3. Problem Shape

用户最新裁决改变了本设计包的默认策略：

- 旧设计包默认“先兼容、再 hardening”，这不再适合作为执行策略。
- 用户要的是功能，不是旧代码；如果旧 API、旧类名、旧目录结构阻碍 AI 理解和验证，应允许一次性删除并迁移调用点。
- AI-first 的目标不是“人类能用”，而是 AI 可以从少量入口看清：
  - 这个工具解决什么功能。
  - 输入、输出、owner、scope、生命周期和失败原因是什么。
  - 哪些调用是禁止项。
  - 如何验证功能没有退化。

按这个标准，当前剩余 Tools 的真实问题是：

| Tool | 当前功能价值 | AI-first 问题 |
| --- | --- | --- |
| `ParentManager` | 统一 SceneTree 挂载路径，避免大量 Entity/Pool/UI 节点散挂。 | 名称和 API 是自由字符串注册表，缺 manifest、scope、pending/in-tree diagnostics。 |
| `TargetSelector` | Ability / AI / Feature / Movement 共用目标查询。 | 查询失败无结构化原因，候选源写死 NodeLifecycle，随机不可复现，`OriginProvider` 实际未贯通。 |
| `ResourceManagement` | 统一资源加载与 catalog。 | `Contains` fallback 容易静默加载错资源，`LoadPath` 没有 source policy，失败只有 log。 |
| `Math` | 纯几何、曲线、波形和部分公式复用。 | `MyMath` 混合 Damage/Ability/Random owner，概率随机不可复现，业务公式来源不清。 |
| `NodeLifecycle` | 底层 Node 注册表，支持 Entity/UI/Component 注册。 | 功能必要，但容易被误当业务查询事实源；API 以 `GetType().Name` 和 string instance id 为核心，缺 owner metadata，且放在 `Tools/` 语义不清。 |
| `CommonTool` / Common Utilities | 当前只包装 `LoadPackedScene(path, usageName)`；用户确认通用工具区域概念可以保留。 | 当前 `CommonTool` 名称是杂项入口；`LoadPackedScene` 应迁入资源加载，Common Utilities 需独立位置、manifest、禁止项和测试。 |

## 4. Main Risks

| 风险 | 说明 | 处理方式 |
| --- | --- | --- |
| 继续保留兼容 facade | AI 会继续沿旧入口写新代码。 | 执行型 SDD 默认 hard cutover：迁移调用点后删除旧入口。 |
| ParentManager 被降级成边缘工具 | 会失去“统一节点挂载路径”的框架能力。 | 明确升级为 `RuntimeMountRegistry` / `SceneMountRegistry` 语义，列为 P0。 |
| 一次性重构过大但无验证 | 工具横跨 Entity、Ability、AI、ObjectPool、UI、Resource。 | 可以按功能切片执行，但每个切片内部不保留旧 API 兼容链。 |
| TargetSelector 直接上空间索引 | 未证明瓶颈前实现复杂度高，可能引入索引失效 bug。 | 默认先做 candidate source + diagnostics + deterministic RNG；空间索引需用户确认或 profiling 证据。 |
| ResourceManagement 直接 fail-fast | 暴露旧资源 key 漂移，短期可能让场景无法启动。 | 这是期望的重构红灯；执行前准备 ResourceCatalog diagnostics 和修复清单。 |
| 删除 NodeLifecycle 公共查询 | TargetSelector / EntityManager 旧调用点需同 SDD 迁移。 | 先建立 EntityRegistry / candidate source / manager facade，再删除业务可见查询。 |

## 5. Options

### Option A：只修文档，源码继续兼容

优点：短期成本低。

缺点：与用户要求冲突；旧 API 仍会继续误导 AI。

结论：Reject。

### Option B：按功能切片 hard cutover

做法：

- 每个工具按功能契约拆一个执行型 SDD 或一个大 SDD 内的独立任务组。
- 每个切片内部允许改名、删旧 API、迁移调用点、重建测试。
- 不保留“长期兼容 facade”；只有执行中临时过渡代码，切片完成前必须删除。

优点：符合“功能优先、代码可丢弃”；风险可按功能分桶验证。

缺点：需要更严格的 grep gate、Godot 场景验证和 DocsAI/skill 同步。

结论：Recommended。

### Option C：Tools Kernel 一次性整体重写

做法：把剩余 Tools 统一重构成一个 `RuntimeToolKernel`。

优点：入口最统一。

缺点：会把挂载、查询、资源、数学、Node 注册这些不同性质功能强塞到一个抽象里，形成新的大而全工具。

结论：Reject。

## 6. Recommendation

采用 **Option B：按功能切片 hard cutover**。

优先级调整为：

| Priority | Cutover | 目标 |
| --- | --- | --- |
| P0 | `RuntimeMountRegistry` / `SceneMountRegistry` | 替代 `ParentManager` 自由字符串接口，统一 Entity / Pool / UI / Tool mount manifest、scope、pending/in-tree diagnostics。 |
| P0 | `TargetQueryEngine` | 替代静态 `EntityTargetSelector` 查询模型，补 query validation、resolved origin/forward、candidate source、diagnostics、deterministic RNG、safe sorting。 |
| P1 | `ResourceLoading` / `ResourceCatalogDiagnostics` | 迁出 `CommonTool.LoadPackedScene`，`ResourceManagement` 变 strict lookup + source policy + structured result。 |
| P1 | `MathFormula` / `DeterministicRandom` / `Geometry2D` 边界 | 拆 `MyMath` owner，概率与采样可注入 RNG，公式归属和测试表清楚。 |
| P1 | `NodeLifecycleRegistry` | 从 `Tools` 工具语义迁到 Runtime 底层注册表，补 owner metadata、snapshot、invalid cleanup；业务查询迁走后删除 public global scan API。 |
| P1 | `Common Utilities` | 保留通用工具区域，但不保留 `CommonTool.SomeHelper()` 杂物箱；每个 helper 必须有 owner、用途、禁止项和测试。 |

这不是“为了不兼容而改名”。判断标准是：旧名字或旧 API 是否阻碍 AI 判断功能、owner、scope 和验证。如果阻碍，就 hard cutover。

## 7. Must Confirm

进入执行型 SDD 前必须确认这些功能选择：

1. Common Utilities 的最终目录位置：推荐 `Src/ECS/Common/Utilities/` + `DocsAI/ECS/Common/Utilities/`；备选 `Src/ECS/Runtime/Common/` + `DocsAI/ECS/Runtime/Common/`。
2. `NodeLifecycle` 是否从 `Src/ECS/Tools/NodeLifecycle/` 迁到 `Src/ECS/Runtime/NodeLifecycle/`。我的建议是迁走，因为它是 Runtime 基础设施，不是普通工具。
3. `EntityTargetSelector.Query(query)` 是否只允许作为执行期临时桥，切片结束前删除或 internal 化。我的建议是是；DocsAI current 入口只教 `TargetQueryEngine.Query(...)` 和 `TargetQueryResult`。

用户已确认：

- `ParentManager` 默认挂载 scope 采用 `/root/SlimeAIRuntime`。
- `TargetSelector` 可以重构；第一阶段需要先说清楚怎么做。
- 资源加载接受 hard fail-fast：缺精确 key / 缺 source policy 直接报错。

## 8. Should Confirm

这些可用默认值推进，但建议确认：

1. `ParentManager` 是否直接改名为 `RuntimeMountRegistry`，还是源码保留 `ParentManager` 文件名但 DocsAI 中只暴露 RuntimeMountRegistry 语义。我的建议是执行时直接改名并迁移调用点。
2. `CommonTool.LoadPackedScene` 是否在资源加载 cutover 中迁出后删除当前 `CommonTool` 类。我的建议是迁出后删除或 internal 化当前类，但保留受约束 Common Utilities 区域。
3. `MyMath` 是否拆成 `AttributeFormula` / `CooldownFormula` / `DamageFormula` / `ProbabilityTool`。我的建议是拆，且公式 owner 写到 DocsAI/Capabilities。
4. `NodeLifecycleManager.GetAllNodes()` / `GetNodesByInterface<T>()` 是否从业务可见 API 删除。我的建议是删除业务可见入口，只留低层 registry snapshot / diagnostics。

## 9. Defaults I Will Use

如果用户回复“按建议执行”，默认采用：

- 只保功能兼容，不保 API 兼容；旧类名、旧方法、旧文件可删除。
- `ParentManager` 执行时升级并改名为 `RuntimeMountRegistry` 或 `SceneMountRegistry`，以 mount manifest 为事实源。
- 默认 mount scope 先选 `/root/SlimeAIRuntime` 下的框架统一根，具体 Entity / Pool / UI 子路径由 manifest 定义；测试可注入 test root。
- `TargetSelector` 第一阶段不做空间索引，先完成 query contract、candidate source、diagnostics 和 deterministic RNG。
- `ResourceManagement.Load<T>` 删除 contains fallback；`LoadPath` 必须携带 `ResourceLoadSource`。
- `CommonTool.LoadPackedScene` 迁到资源加载契约；当前 `CommonTool` 删除或 internal 化。
- 保留受约束 Common Utilities 区域，但不允许 `CommonTool.SomeHelper()` 杂物箱。
- `MyMath` 拆 owner；概率和随机采样默认要求 seed/RNG 可注入。
- `NodeLifecycle` 迁到 Runtime registry 语义，只保低层注册/diagnostics，不作为新业务查询入口。

## 10. Not Recommended

- 不建议继续写“兼容 facade 长期保留”。
- 不建议把 `ParentManager` 降级成临时工具；它应是 Runtime mount 功能。
- 不建议让 AI 新增 `CommonTool.SomeHelper()`。
- 不建议让 Ability/AI 直接扫 SceneTree group 或 NodeLifecycle 全局节点。
- 不建议复制 Bevy / Flecs / EnTT / Arch 的通用 query DSL、archetype storage、pair graph 或 registry public API。
- 不建议为了重构一次性引入第三方 ECS、数学、资源库。

| Godot SceneTree / Node child 管理 | Evidence：Godot `Node.add_child` 管父子树、child name 唯一、parent 删除会删除 child；`CallDeferred` 用于延迟安全调用。 | Adopt Now | `RuntimeMountRegistry` 必须显式管理 mount scope、pending/in-tree 状态和 parent 生命周期。 |
