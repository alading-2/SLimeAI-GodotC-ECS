## 1. Goal

本轮要解决的问题不是“这些旧类名是否好听”，而是：

```text
AI 看到框架时，能不能快速知道：
1. 节点放哪里。
2. 目标怎么找。
3. 资源加载失败为什么失败。
4. 通用工具能不能放、放哪里、怎么防止变成杂物箱。
5. Node 生命周期注册是否统一，是否会误导业务随便扫全局节点。
```

非目标：

- 不在本文件里写源码。
- 不复制 Bevy / Flecs / EnTT / DefaultEcs 的通用 ECS query。
- 不为了“无兼容”而删除有价值功能。
- 不让旧 API 名字继续误导 AI。

## 3. Problem Shape

### 3.1 ParentManager

你截图里的树已经说明 `ParentManager` 的核心价值是对的：

```text
root
  ECS
    Entity
      Unit
        EnemyPool
      PlayerEntity
        Player
      EnemyEntity
        Enemy
  UI
    UI
      HealthBarUIPool
      DamageNumberUIPool
```

这对人和 AI 都有帮助，因为一眼能看出运行时对象大致归属。

问题不在“功能错了”，而在当前实现还有三个 AI-first 缺口：

- 挂载路径是调用点传入的自由字符串，AI 不知道哪些路径是框架规定，哪些只是临时写的。
- root 第一层使用 deferred add，同一帧节点可能还没真正进 tree，但当前 API 直接返回 Node，AI 看不出 pending 状态。
- 没有统一报告告诉 AI：哪些 mount 已在 tree、哪些还 pending、哪些 parent 已无效。

所以方向不是删除，而是升级。

### 3.2 TargetSelector

当前 `TargetSelector` 的功能价值也成立：AI、技能、移动、Feature 都需要找目标。

真正问题是：它现在像一个“返回 List 的黑盒”。

例如 AI 找不到敌人时，当前只得到空列表。AI 不知道：

- 是没有实体？
- 是超出范围？
- 是阵营过滤掉了？
- 是目标死了？
- 是 query 参数无效？
- 是随机排序不可复现？
- 是候选来源扫错了？

AI-first 的目标不是“找目标代码更酷”，而是每次查询都能回答“为什么”。

### 3.3 ResourceManagement

用户已裁决：资源找不到要立刻报错。

这意味着 `ResourceManagement.Load<T>` 不能继续用 contains fallback 猜资源。比如要找 `EnemyEntity`，如果精确 key 不存在，就不应该猜一个名字里包含 Enemy 的资源继续跑。

AI-first 下，启动失败比加载错资源更好，因为失败原因可以被修正。

### 3.4 CommonTool

用户指出：确实会有一些常用函数不好归类，通用工具概念可以保留。

这个判断合理，但必须区分两件事：

```text
可以保留：受约束的 Common Utilities 区域。
不建议保留：所有东西都能塞进去的 CommonTool.SomeHelper() 杂物箱。
```

当前 `CommonTool.LoadPackedScene(path, usageName)` 实际是资源加载功能，不是通用工具。它应该迁到 `ResourceManagement` / `ResourceLoading`。

但可以新增一个更规范的位置，例如：

```text
Src/ECS/Common/Utilities/
DocsAI/ECS/Common/Utilities/
```

或者如果项目坚持 ECS 分层内：

```text
Src/ECS/Runtime/Common/
DocsAI/ECS/Runtime/Common/
```

这个区域必须有规则：

- 只能放无明确 Capability owner 的横切小工具。
- 每个文件按功能命名，不再叫 `CommonTool`。
- 每个函数写清输入、输出、副作用、是否访问 Godot、是否热路径安全、测试入口。
- 资源加载、Entity 查询、事件发送、Data 写入、对象池、Timer、TargetSelector、Math 公式都不能放这里。

### 3.5 NodeLifecycle

用户说的本意是对的：Node 注册、注销、id 管理应该统一，不应该 Entity 写一套、UI 写一套、Component 再写一套。

所以 `NodeLifecycle` 有必要。

但它现在放在 `Tools/`，并且对外暴露：

```text
GetAllNodes()
GetNodesByInterface<T>()
GetNodesByType<T>()
```

这会让 AI 误以为：任何业务想找东西，都可以直接扫全局 Node。

AI-first 的改法不是删除 NodeLifecycle，而是把它变成更明确的运行时底层注册表：

```text
NodeLifecycleRegistry
  负责：Node 注册、注销、id、owner metadata、invalid cleanup、diagnostics snapshot
  不负责：Entity 查询、UI 查询、目标选择、对象池状态、业务关系
```

也就是说：

- EntityManager 可以用它做底层注册，但 Entity 查询应该走 `EntityRegistry`。
- UIManager 可以用它做底层注册，但 UI 查询应该走 UIManager 自己的 typed API。
- ComponentRegistrar 可以用它做底层注册，但组件归属查询走 ComponentRegistrar。
- TargetSelector 不直接扫 NodeLifecycle，而是通过 candidate source 间接拿候选。

## 4. Main Risks

| 风险 | 通俗说明 | 处理方式 |
| --- | --- | --- |
| `ParentManager` 只改名不改契约 | 路径还是随手字符串，AI 仍然不知道哪些路径合法。 | 必须加 mount manifest、scope、status diagnostics。 |
| `/root/SlimeAIRuntime` 残留节点 | 全局挂载不随场景销毁，如果没有清理规则会越积越多。 | RuntimeMountRegistry 必须提供 cleanup scope 和 diagnostics。 |
| TargetSelector 结果仍只是空列表 | AI 还是不知道为什么没目标。 | 新入口必须返回 `TargetQueryResult`。 |
| 直接上空间索引 | 性能可能更好，但索引失效会产生更难查的 bug。 | 第一阶段先不做空间索引，只留 source 接口。 |
| CommonTool 继续当杂物箱 | AI 会把“不知道放哪”的函数都塞进去。 | 保留 Common Utilities 概念，但禁止无 owner、无文档、无测试的杂项入口。 |
| NodeLifecycle 被业务滥用 | 所有系统都直接扫全局 Node，Entity/UI/Component 边界重新混乱。 | NodeLifecycle 只做底层 registry 和 diagnostics；业务查询走 typed facade。 |

## 5. Options

### Option A：保留当前类名，只补一点注释

优点：改动小。

缺点：AI 仍会复制旧入口，问题没有真正解决。

结论：Reject。

### Option B：按功能重构，但保留有效概念

做法：

- `ParentManager` 功能保留，升级为 runtime mount 功能。
- `TargetSelector` 功能保留，升级为可诊断查询引擎。
- `ResourceManagement` 保留，改严格加载。
- `CommonTool` 当前资源加载方法迁走，但保留受约束的 Common Utilities 区域。
- `NodeLifecycle` 保留为运行时 Node registry，不再作为业务全局查询入口。

优点：符合用户“功能优先，代码可丢弃”的要求，也不误删有价值能力。

缺点：需要同步 DocsAI、skill、调用点和 grep gate。

结论：Recommended。

### Option C：把全部合并成一个 RuntimeToolKernel

优点：入口表面上统一。

缺点：挂载、资源、查询、生命周期、通用工具性质完全不同，强行合并会形成新的大杂物箱。

结论：Reject。

## 6. Recommendation

采用 Option B。

推荐执行顺序：

1. `RuntimeMountRegistry`
   - 保留你截图里的清晰目录效果。
   - 默认根节点是 `/root/SlimeAIRuntime`。
   - 内部仍可有 `ECS/Entity/...`、`UI/...`、`Tool/...` 子路径。
   - 关键不是视觉层级变化，而是路径来源从“调用点随手字符串”变成 manifest。

2. `TargetQueryEngine`
   - 先做清楚查询，不急着做空间索引。
   - 每次找目标都返回结果和原因。
   - Ability / AI / Feature / Movement 不再直接用旧 list-only 查询。

3. `ResourceLoading`
   - 精确 key 找不到就失败。
   - `LoadPath` 必须说明来源，例如 DataSnapshot、GeneratedKey、Debug、Test。
   - `CommonTool.LoadPackedScene` 迁到资源加载 owner。

4. `NodeLifecycleRegistry`
   - 从 `Tools` 的“可随手调用工具”语义，迁到 Runtime 底层 registry 语义。
   - 保留统一注册、注销、id、owner metadata、diagnostics。
   - 删除或 internal 化业务可见的全局扫描 API。

5. `Common Utilities`
   - 保留通用工具放置区，但不保留 `CommonTool` 杂物箱风格。
   - 每个 helper 都要有 owner、用途、禁止项和测试。

## 7. TargetSelector 怎么重构

通俗地说，新的 TargetSelector 会从“给我一个 List”变成“给我一份找目标报告”。

### 7.1 新流程

```text
输入查询条件
  -> 检查条件是否合法
  -> 算出这次查询真正的原点和朝向
  -> 从候选来源拿实体
  -> 按几何范围过滤
  -> 按阵营 / 类型 / 生命周期过滤
  -> 按最近、最远、血量、威胁、随机排序
  -> 截断最大数量
  -> 返回目标 + 诊断报告
```

### 7.2 输入会变清楚

目标输入大概长这样：

```text
TargetQueryRequest
  谁在查：SourceEntity / CenterEntity
  从哪里查：Origin 或 OriginProvider
  朝哪里查：Forward 或 ForwardProvider
  查什么形状：Single / Circle / Ring / Box / Line / Cone / Global
  查多远：Range / InnerRange / Width / Length / Angle
  查哪些阵营：Enemy / Friendly / Neutral / Self
  查哪些类型：Unit / Projectile / Effect / UI 等
  怎么排序：Nearest / HighestThreat / LowestHealth / Random
  最多几个：MaxTargets
  候选从哪来：EntityRegistry / ExplicitCandidates / TestSource / 后续 SpatialIndex
  随机怎么复现：RandomSource / Seed
  调用来源：Ability / AI / Feature / Test
```

### 7.3 输出会说明原因

目标输出不再只是 `List<IEntity>`，而是：

```text
TargetQueryResult
  Targets
  ResolvedOrigin
  ResolvedForward
  CandidateCount
  GeometryHitCount
  FilteredByTeamCount
  FilteredByTypeCount
  FilteredByLifecycleCount
  MissingDataWarnings
  SortApplied
  LimitApplied
  Errors
  Warnings
```

这样 AI 看到空结果时能知道：

```text
候选 20 个
几何命中 5 个
阵营过滤掉 5 个
最后目标 0 个
原因：都不是 Enemy
```

或者：

```text
Errors:
  Circle 查询缺 Range
```

### 7.4 候选来源会收口

当前 `EntityTargetSelector` 是直接扫：

```text
NodeLifecycleManager.GetNodesByInterface<Node2D>().OfType<IEntity>()
```

目标改成：

```text
ITargetCandidateSource
  EntityRegistryCandidateSource
  ExplicitCandidateSource
  TestCandidateSource
  NodeLifecycleCandidateSource 仅迁移期/internal
  SpatialIndexCandidateSource 后续性能证据触发
```

通俗说：找目标的人不关心“实体从哪张表拿来”，只关心“给我候选和报告”。

### 7.5 随机必须可复现

当前随机有两个问题：

- `EntityTargetSelector` 里 `new Random()`，测试不稳定。
- `PositionTargetSelector` 用当前毫秒 seed，重放不稳定。

目标：

- 测试传固定 seed。
- gameplay 由上层传 RNG 或后续统一 RNG service。
- TargetSelector 自己不偷偷用当前时间当 seed。

### 7.6 调用点怎么变

旧调用：

```text
var targets = EntityTargetSelector.Query(query);
if (targets.Count == 0) return Failure;
```

目标调用：

```text
var result = TargetQueryEngine.Query(request);
if (!result.HasTargets)
{
    // AI / Ability 可以记录 result.Diagnostics，而不是猜。
    return Failure;
}
```

`FindEnemyAction` 可以把“找不到敌人”的失败原因写进 AI 诊断。

`AbilityImpactTool` 可以把“技能没有命中目标”的原因写进 Ability 执行报告。

`TargetSelectorTest` 可以断言：

- 有多少候选。
- 哪些被阵营过滤。
- 哪些因为死亡被过滤。
- 固定 seed 下随机顺序一致。

## 8. NodeLifecycle 是否还有必要

结论：有必要，但不应该继续是 `Tools/` 下随手可扫全局节点的工具。

推荐目标：

```text
Src/ECS/Runtime/NodeLifecycle/
  NodeLifecycleRegistry.cs
  NodeLifecycleRegistration.cs
  NodeLifecycleOwnerKind.cs
  NodeLifecycleSnapshot.cs
  NodeLifecycleDiagnostics.cs
```

核心数据：

```text
NodeLifecycleRegistration
  NodeId
  NodeType
  NodePath
  OwnerKind: Entity | Component | UI | System | Tool | Test
  OwnerId
  RegisteredFrame
  IsInsideTree
  IsValid
```

这样 Entity、UI、Component 仍能共用注册表，但 AI 不会误以为 NodeLifecycle 是 gameplay query。

## 9. CommonTool 是否保留

结论：保留“通用工具区域”，不保留“当前 CommonTool 作为杂物箱”的形式。

推荐规则：

```text
允许：
  Common/Utilities/NodePathUtility.cs
  Common/Utilities/NameUtility.cs
  Common/Utilities/Guard.cs
  Common/Utilities/CollectionSnapshot.cs

禁止：
  CommonTool.LoadPackedScene
  CommonTool.FindEnemy
  CommonTool.ApplyDamage
  CommonTool.GetAllEntities
  CommonTool.StartTimer
```

判断标准：

```text
如果函数能归到 Resource / Entity / Data / Event / Timer / TargetSelector / ObjectPool / Math / Capability，就不要放 Common。
只有真的跨 owner、无业务语义、无状态副作用的小工具，才可以放 Common Utilities。
```
