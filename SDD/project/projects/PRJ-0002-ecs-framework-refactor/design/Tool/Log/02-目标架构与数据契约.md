# 目标架构与数据契约

> 更新：2026-05-31
> 状态：current design note

## 1. 目标架构

```text
LogSource
  -> LogEntry (structured envelope)
  -> LogRouter
  -> LogSink(s)
      -> console sink
      -> jsonl sink
      -> memory sink
      -> validation artifact sink
```

## 2. 核心类型

### 2.1 LogEntry

每条日志都是固定 envelope，不允许只有一段文本。

建议字段：

- `Timestamp`
- `Level`
- `Channel`
- `Owner`
- `Context`
- `Operation`
- `Message`
- `Fields`
- `EntityId`
- `CorrelationId`
- `Phase`
- `Source`
- `RepeatCount`
- `SuppressedCount`

### 2.2 LogProfile

`LogProfile` 是默认策略集合：

- 默认等级。
- context / channel / owner 覆盖规则。
- 预算规则。
- sink 开关。
- validation 专用规则。
- test 专用规则。

### 2.3 LogRule

规则按粒度递增：

1. 全局默认。
2. channel 规则。
3. owner 规则。
4. context 规则。
5. entity 规则。
6. 临时 CLI 覆盖。

优先级应该清晰固定，避免调试时互相打架。

## 3. 数据契约

### 3.1 最少字段

AI 能直接分析的最少字段是：

- `level`
- `context`
- `operation`
- `message`
- `fields`

如果一条日志既没有 `operation`，也没有 `fields`，它通常只是人类阅读友好，不够 AI-first。

### 3.2 高价值字段

高价值字段应尽量标准化：

- `entityId`
- `targetId`
- `system`
- `owner`
- `phase`
- `reason`
- `result`
- `expected`
- `actual`
- `durationMs`
- `count`

### 3.3 低价值文本

完全禁止大量依赖纯描述句。例如：

- `"开始执行..."`
- `"完成了"`
- `"测试结束"`

这种话可以保留在人类 console 里，但在结构化数据里必须附带字段。

## 4. 输出形式

### 4.1 Console sink

用于人类实时观察。

要求：

- 颜色可以保留。
- 必须显示 `level / context / operation`。
- 结构化字段应压缩展示。
- 重复日志要合并。

### 4.2 JSONL sink

用于 AI 和后处理。

要求：

- 每行一条 JSON object。
- 包含完整 envelope。
- 可直接被 scene runner、分析脚本和后续工具消费。

### 4.3 Memory sink

用于测试和 validation artifact。

要求：

- 支持读取完整日志序列。
- 支持写入 check / pass / fail。
- 支持最终导出到 artifact。

## 5. 约束

- 不允许把业务语义埋进颜色。
- 不允许只看字符串就判断日志类型。
- 不允许测试结果只存在 console。
- 不允许日志 sink 反向决定日志内容。
