# 字符串键名统一问题分析

> 更新：2026-05-26
> 范围：Data key、Event name、Relationship type、Relationship data key、Resource key、配置字段映射。
> 结论：字符串变量名不统一是旧 ECS 当前最核心的横切问题，应优先建立规则、扫描基线和分域推进策略。

## 1. 问题本质

旧 ECS 大量使用字符串作为运行时主键：

- Data：`Data.Get<T>(string key)`、`Data.Set(string key, value)`
- Event：`EventBus.On<T>(string eventName, ...)`、`Emit(string eventName, ...)`
- Relationship：`AddRelationship(parentId, childId, string relationType)`
- Relationship data：`Dictionary<string, object>`
- Resource：资源名、场景路径、分类 key
- Config mapping：属性名、`DataKeyAttribute.Key`

字符串不是问题本身。问题是**没有统一规则区分哪些字符串是稳定协议，哪些只是显示文本或临时路径**。

## 2. 典型坏味道

### P1: 同一语义写两遍

例如事件：

```text
GameEventType.Unit.Damaged
GameEventType.Unit.DamagedEventData
```

例如 Data：

```text
DataKey.AbilityDamage
new DataMeta { Key = nameof(AbilityDamage), Type = typeof(float) }
```

一旦“名字”和“类型/数据结构”分离，就会出现同步风险。

### P2: 常量树只能防 typo，不能表达语义

`DataKey`、`GameEventType`、`EntityRelationshipType` 都是常量集中化思路。

它们解决了部分裸字符串问题，但没有回答：

- 这个 key 的 owner 是谁。
- 这个 key 的类型是什么。
- 这个 key 是否能迁移、能修改、能序列化。
- 这个 event 是通知还是请求。
- 这个 relation 是否参与生命周期。

### P3: 隐式 string 转换降低类型化收益

`DataMeta` 有 `implicit operator string`，调用方可以方便地把 DataKey/Meta 当字符串传入。

这让短期迁移容易，但长期会导致：

- API 看起来接收强类型，实际仍是 string。
- 新代码不容易分辨推荐写法和兼容写法。
- 扫描裸字符串时会漏掉隐式转换风险。

### P4: 缺少统一扫描规则

现在很难一眼回答：

- 还有多少 `Data.Get<T>("...")`？
- 还有多少 `Events.Emit("...")`？
- 还有多少 `AddRelationship(..., "...")`？
- 哪些字符串是 UI 文案，哪些是协议 key？

没有扫描基线，就不能安全推进优化。

## 3. 统一原则

### R1: 稳定协议 key 必须有 owner

每个长期存在的 key 都应归属某个 owner：

| Key 类型 | Owner 示例 |
| --- | --- |
| DataKey | Ability / Unit / Movement / Feature / Base |
| Event | Data / Ability / Unit / Global / TestSystem |
| RelationshipType | Entity core / Ability / Projectile / UI |
| RelationshipDataKey | PARENT lifecycle |
| ResourceKey | ResourceManagement / 具体资源分类 |

### R2: 新代码禁止裸协议字符串

允许的字符串：

- UI 显示文本。
- 日志消息。
- Godot 节点路径。
- `res://` 资源路径。
- 临时测试说明。

不允许的字符串：

- Data key。
- Event name。
- Relationship type。
- Relationship data key。
- 长期 Resource key。

### R3: 类型信息必须和 key 绑定

最低要求：

- Data key 绑定 `T` 和默认值。
- Event key 绑定 payload 类型。
- Relationship type 绑定约束和生命周期参与规则。
- Relationship data key 绑定值类型和默认策略。

### R4: 兼容路径必须标注生命周期

如果保留 string API，应明确：

- 是永久公共 API，还是迁移兼容。
- 新代码是否允许使用。
- 如何扫描和逐步替换。

## 4. 推荐第一步

第一步不要直接改大量代码，而是建立扫描基线和规则文档：

1. 搜索 Data 裸字符串访问。
2. 搜索 Event 裸字符串和 `GameEventType` 调用。
3. 搜索 Relationship 裸字符串关系类型。
4. 搜索 `Dictionary<string, object>` 中作为协议数据的 key。
5. 分类为：
   - 必须修。
   - 可保留但需封装。
   - UI/日志/路径，不处理。

## 5. 后续优化顺序

建议顺序：

1. DataKey 规则最先做，因为影响最广、已有类型化基础。
2. Event 定义事实源恢复其次做，因为当前 `GameEventType` 定义缺失风险最高。
3. Relationship type 元信息第三做，因为调用面相对可控。
4. Resource key 最后做，因为需要结合 ResourceManagement 和资产生成流程。
