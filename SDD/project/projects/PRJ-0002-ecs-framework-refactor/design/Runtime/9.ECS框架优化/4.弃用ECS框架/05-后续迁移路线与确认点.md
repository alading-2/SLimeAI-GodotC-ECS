> 本文目标：把“弃用 ECS”转成后续可执行顺序，但不直接进入实现。

## 推荐迁移顺序

### Phase 0：冻结概念

目标：先让后续所有设计不再以 ECS 为前提。

动作：

- 确认框架新名：`SlimeAIFramework`。
- 确认核心词：Object / Component / System / Feature / Event / Data。
- 标记旧 `DocsAI/ECS` 和 `Src/ECS` 为历史路径名。
- 停止从旧 Data SDD 路线继续执行。

### Phase 1：新框架总设计

新增设计包，建议路径：

```text
SDD/project/projects/PRJ-0002-ecs-framework-refactor/design/Runtime/10.GodotOOP框架方向/
```

建议文档：

- `01-Object与Component模型.md`
- `03-Feature与System边界.md`
- `Data/README.md`
- `04-事件驱动与生命周期规则.md`
- `05-迁移路线与验证策略.md`

### Phase 2：Data 重写设计

先写问题和进入条件，再设计代码。

必须回答：

- 哪些状态保留在 Component。
- 哪些状态进入 Data。
- DataOS 是否继续存在，存在于 authoring 还是 runtime。
- modifier/computed 是否保留，是否只用于少数字段。
- UI/debug/log/save/load 如何读取状态。

### Phase 3：最小实现切片

不要一开始全仓重命名。先选一个功能闭环验证新模式，例如：

```text
Unit + Health + Damage
```

目标：

- Health 状态在哪里。
- Damage 如何发事件。
- UI 如何订阅初值和变化。
- AI 如何通过文档和日志定位。
- Godot scene 如何验证。

### Phase 4：命名和目录迁移

只有新模式跑通后再考虑：

```text
DocsAI/ECS -> DocsAI/Framework
Src/ECS -> Src/Framework
Entity -> Object
Capability -> Feature
```

目录迁移必须单独 SDD，包含 grep gate、路径引用迁移、skill 更新和构建验证。
