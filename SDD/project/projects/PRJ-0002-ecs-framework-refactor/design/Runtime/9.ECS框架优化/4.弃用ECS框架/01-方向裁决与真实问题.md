# 方向裁决与真实问题

> 原始问题：见 [`source-request.md`](./source-request.md)。  
> 本文目标：把“弃用 ECS”落成 SlimeAI 当前项目级方向裁决。

## 结论

用户判断成立：SlimeAI 当前所谓 ECS 框架确实走偏了。它没有实现真正 ECS 的底层 storage/query/schedule，却承担了 ECS 这个词带来的设计压力，导致 Data 系统被推成框架核心，Component/System 语义也不断在 Godot OOP 与传统 ECS 之间摇摆。

新的方向是：

```text
不用 ECS 作为 SlimeAI 框架身份。

SlimeAIFramework 是 Godot C# OOP gameplay framework。
Godot Node / OOP / scene composition 是底层事实。
Object / Component / System / Feature / Event 是功能解耦手段。
Data 只是受控共享状态和表格驱动协议，不是 ECS storage 核心。
```

## 真实问题是什么

### 1. 目标错位

SlimeAI 的第一目标不是让几十万实体高性能运行，而是快速开发游戏 MVP，并让功能能快速添加、移除、启停。

真正 ECS 的主要收益是：

```text
数据布局
批量迭代
CPU cache
并行调度
结构变化控制
```

这些收益很重要，但不是当前 SlimeAI 的第一瓶颈。当前瓶颈是：

```text
概念不稳
Data 复杂
AI 路由困难
文档事实源摇摆
功能实现变慢
Godot 原生能力没有被充分使用
```

### 2. Godot 已经解决了很多动态组合问题

Godot Node 可以作为对象、功能组件、表现节点、碰撞节点、UI 节点和生命周期承载者。Scene 可以组合和继承，Node 可以运行时添加移除。

这正好符合用户最核心需求：

```text
功能能马上添加和移除。
功能直接写在 Node / Component 上。
不需要每个功能都绕到 ECS System / Data storage。
```

### 3. Data 系统被 ECS 方向拖复杂了

为了让 Data 成为“ECS 核心数据结构”，旧设计反复把这些职责塞进 Data：

- authoring descriptor
- runtime snapshot
- generated key
- typed slot
- modifier
- computed
- policy
- presentation
- AI routing
- diagnostics
- validation report

结果 Data 不是变成了简单的共享状态协议，而是变成了比业务功能还复杂的中心系统。多轮重构仍然绕不开类型、默认值、转换、policy 和 owner 边界，说明方向本身需要重置。

### 4. 解耦不依赖 ECS

高度解耦可以通过这些机制实现：

- Godot Node / scene composition。
- 小 Component + typed options。
- Feature owner / System / Facade。
- Event / message 驱动。
- 明确的接口和依赖方向。
- 运行时启停与生命周期 guard。
- 文档、skill、测试、日志和验证。

ECS 是解耦的一种方式，不是解耦的唯一方式。

## 新目标

SlimeAI 后续目标应写成：

```text
快速开发游戏 MVP。
功能以 Feature / Component 为中心组织。
Object 可按功能组合和裁剪。
System 只在跨对象协调、批处理、查询、资源、生命周期等场景出现。
事件驱动承担跨功能通信。
Data 只保存真正跨功能共享、表格驱动或需要验证追踪的数据。
AI-first 负责入口、文档、验证、日志和流程。
```

## 非目标

- 不实现完整 ECS storage。
- 不实现 archetype / chunk / query world 作为底层。
- 不把 Component 强制改为纯数据。
- 不把 Godot Node 降级为 ECS bridge。
- 不继续让 DataOS 变成所有状态的统一入口。
- 不直接引入 QFramework 或其他框架依赖。

## 对旧方向的覆盖

本裁决覆盖 2026-06-15 的“保留 ECS 心智模型 / Composable Godot C# Runtime”方向。

保留其中正确部分：

- 功能可组合、可裁剪、可启停。
- AI-first 是文档、验证和观察工程层。
- 外部框架只学机制，不复制 API。

废弃其中错误部分：

- 继续称 SlimeAI 为 ECS。
- 把 Entity / Data / System 当成底层框架必备核心。
- 让 Data 成为实现功能解耦的默认主线。

## 后续工作原则

1. 先改概念，不急着改源码。
2. 先冻结 Object / Component / System / Feature / Event / Data 边界。
3. Data 重写前只描述问题，不预设新实现。
4. 命名和目录迁移单独开 SDD，不在本轮混做。
5. 后续任何实现都要有最小可运行验证场景，不再只靠文档推演。
