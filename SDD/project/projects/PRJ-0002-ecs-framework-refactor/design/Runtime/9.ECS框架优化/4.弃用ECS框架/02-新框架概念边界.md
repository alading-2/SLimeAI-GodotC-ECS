# 新框架概念边界

> 本文目标：定义弃用 ECS 后 SlimeAI 应使用的核心概念。  
> 状态：superseded by [`../../10.GodotOOP框架方向/README.md`](../../10.GodotOOP框架方向/README.md)；本文保留为 2026-06-16 早期方向草案。

## 一句话结论

新框架不再以 ECS 的 Entity / Data / System 为核心。2026-06-16 后续用户裁决已将正式框架名定为 `SlimeAIFramework`，并保留 `Component`、`System`、`Data` 名字；本文件中 `SharedState` 应理解为 “Data 的受控共享状态职责”，不是改名要求。

```text
Object     表示一个游戏对象或运行时对象。
Component  表示挂载在 Object 上的功能单元。
Feature    表示一组可组合、可裁剪的玩法能力。
System     表示跨对象协调、批处理、查询或全局能力。
Event      表示跨功能通信。
State      表示对象或组件自己的状态。
Data       表示确实需要跨功能共享、表格驱动和验证追踪的状态协议。
```

## Object

Object 替代旧 Entity 概念。

Object 可以是：

- Godot scene root。
- Godot Node。
- 纯 C# runtime object。
- 对象池中的实例。
- UI 或测试对象。

Object 需要保留：

- 稳定 runtime id。
- 生命周期：create / ready / active / disabled / destroyed。
- 组件列表或功能组合。
- 调试名称、来源和 owner。

Object 不再意味着 ECS entity。它不是纯 ID + component storage 的实体，而是 Godot/OOP 世界中的对象入口。

## Component

Component 是功能单元，不再强制是纯数据。

推荐规则：

- 可以是 Godot Node 脚本，也可以是纯 C# component。
- 可以持有内部状态。
- 只暴露小接口和事件，不直接依赖其他组件实现。
- 默认可添加、移除、启停。
- 需要跨功能共享、DataOS 初始化或验证追踪的状态才写入 Data。

示例：

```text
MovementComponent
  内部状态：当前输入方向、临时加速度、上次移动结果
  Data 字段：MoveSpeed、IsMovementDisabled

HealthComponent
  内部状态：受击闪烁、死亡动画状态
  Data 字段：CurrentHp、MaxHp、IsDead
```

## Feature

Feature 是玩法能力 owner。旧 Capability 可以迁移为 Feature。

Feature 管：

- 组件组合。
- 服务注册。
- 事件定义。
- Data 进入条件。
- 验证场景和日志。
- AI 文档入口。

Feature 的价值是让功能能被选择、裁剪、启停，而不是把所有逻辑集中成大 System。

示例：

```text
Feature: Damage
  Components: HurtboxComponent, DamageReceiverComponent
  Services: DamageService
  Events: DamageRequested, DamageApplied
  Data: CurrentHp, DamageReduction
```

## System

System 名字保留，但语义从传统 ECS System 改为 SlimeAIFramework runtime system。

System 只在这些情况需要：

- 跨多个 Object 协调。
- 需要全局查询。
- 需要批处理。
- 管理资源、对象池、计时器、输入、日志。
- 需要一个明确 API 避免组件互相调用。

System 不应该变成万能 manager。能留在 Component 内部的逻辑，不要强行上提。

## Event

Event 是主要解耦通信方式。

保留事件驱动，但需要收窄规则：

- 同一 Object 内组件通信优先用 typed local event 或 owner API。
- 跨 Feature 通信用 typed event/message。
- Godot Signal 主要用于 Godot UI、动画、编辑器和表现层桥接。
- 事件 payload 必须强类型。
- 事件订阅必须绑定生命周期。

事件不是数据存储，也不是替代所有直接方法调用的银弹。局部、同步、明确 owner 的调用可以保留。

## State 与 Data

这是 Data 重写的核心边界。

### State

State 是对象或组件自己的状态，可以直接放在 Component / Object / System 中。

适合 State：

- 单组件内部缓存。
- 临时计算结果。
- UI 动画状态。
- 输入中间状态。
- 一帧内状态。
- 只被一个 Feature 使用的字段。

### Data

Data 是多个 Feature 共享，且需要 DataOS、AI/validator/log/test 追踪的状态。

适合 Data：

- HP、MoveSpeed、Attack、Level 这类多个功能会读取/修改的数值。
- 需要从 DataOS / 表格初始化的运行时字段。
- 需要进入 save/load、debug panel、AI 分析、测试断言的字段。
- 需要跨组件观察变化的状态。

### 关键裁决

```text
默认状态在 Component 内。
证明需要共享、表格驱动或验证追踪后，才进入 Data。
```

这和旧 Data 方向相反。旧方向是“默认进 Data，特殊情况留组件”；新方向是“默认留组件，确实需要共享才进 Data”。

## AI-first 在新框架中的位置

AI-first 不是 runtime 架构，而是工程使用层：

- 文档入口少。
- owner 边界清楚。
- 代码命名稳定。
- 事件和状态强类型。
- 日志可分析。
- 验证命令可重复。
- 设计和 SDD 可恢复。

OOP 也可以 AI-first。关键不是 ECS，而是事实源和验证闭环。

## 初始命名建议

短期：

- 文档正文使用新概念。
- 源码路径暂不动。
- `Entity` 出现时说明历史含义接近 `Object`。
- `Data` 出现时说明新方向为受控共享状态和表格驱动协议。
- `System` 出现时说明新方向为 runtime system / coordinator，不是传统 ECS System。

中期：

```text
Entity -> Object
Capability -> Feature
Runtime -> GameRuntime / FrameworkRuntime
```

是否改名必须单独评估成本，不能在 Data 未定前机械替换。
