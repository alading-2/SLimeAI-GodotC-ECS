# 外部方案证据与采纳边界

## 目的

用户要求广泛搜索 Data 方案。本文只记录会影响 SlimeAIFramework Data 裁决的外部证据，不把外部框架当成依赖或实现模板。

## 证据摘要

| 来源 | 事实 | 对 SlimeAI 的结论 |
| --- | --- | --- |
| Godot Nodes and Scenes | Godot 用 Node tree 组织 scene；scene 可以保存并实例化，组合节点形成复杂对象。 | `Object + Component` 应顺着 Godot scene/node composition，不应再伪装成 archetype ECS。 |
| Godot C# exports | C# `[Export]` 字段/属性会随 scene/resource 保存，并可在 inspector 编辑。 | Component 内部字段可以存在；若只是 Godot editor 配置，不必强制进入 DataOS/Data。 |
| Godot Resources | Node 负责功能，Resource 是数据容器；Resource 可序列化、可有属性和信号。 | DataOS/Data 与 Godot Resource 都是数据入口候选，但 SlimeAI 需要数据库、validator、typed DataKey 和 AI 可读 descriptor，所以保留 DataOS 主链路。 |
| Unity Entities Archetypes | Unity ECS 用 archetype 组织 entity，同 archetype 的 component 数据存入 16KiB chunk，chunk 内按 component array 紧密排列。 | 这是“真正 ECS storage”的证据；SlimeAI 不做这套结构，因此弃用 ECS 身份是正确方向。 |
| Unity Entities Structural Changes | 新建/销毁 entity、增删 component、改 shared component 会搬迁 chunk，并可能产生 sync point。 | SlimeAI 不应把 OOP 组件组合硬套成 ECS 结构变更模型；对象池复用应走 DataBinding 刷新，不做 archetype 搬迁。 |
| Unreal GAS Attributes | Attribute 用于存储、计算、修改 gameplay 浮点值，有 base/current 区分，并放在 AttributeSet 中。 | SlimeAI DataModifier 应收窄为 attribute-like numeric Data 字段，不扩展到所有状态。 |
| Unreal GAS DataTable 初始化 | Attribute 可以用 DataTable 的 AttributeMetaData 初始化，包含 BaseValue、MinValue、MaxValue 等列；但默认 min/max 不自动 clamp。 | SlimeAI 的 descriptor/range/allowed values 必须由自己的 validator 和 Data.Set/ApplyRecord enforcement 实现，不能只把元数据写在表里。 |
| Unreal GAS Gameplay Effects | GameplayEffect 修改 Attribute，支持 instant/duration/infinite、modifier、stacking 和 runtime spec。 | `Feature.Modifiers -> DataModifier source -> DataChanged -> DataBinding` 的方向成立；modifier source 生命周期必须可回滚。 |

## 采纳原则

### Godot

采纳：

- Object / Component 顺着 Node、scene、composition 组织。
- Component 可以保存内部字段。
- 对只服务 editor 或表现层的配置，优先用 exported property、typed options 或 Resource。

不采纳：

- 不把 Godot Resource 替代 DataOS。Resource 缺少当前项目已建立的 SQLite authoring、snapshot generator、DataKey、DataOS validator 和 SDD/AI 文档链路。
- 不用动态 `Object.set/get/call` 做 DataBinding 默认实现。Godot 支持动态属性，但官方也提示 direct reference 更快；SlimeAI 绑定应显式映射到 typed setter。

### Unity Entities

采纳：

- 作为反证使用：真正 ECS 要有 archetype、chunk、query、structural change 和 job/schedule 语义。

不采纳：

- 不在 SlimeAIFramework 重建 archetype/chunk storage。
- 不把 Component 字段全量外置成 Data 数组。
- 不把对象池 acquire/release 做成 ECS structural change 模型。

### Unreal GAS

采纳：

- Attribute 类数值保留 base/current 或 base/effective 的区分。
- GameplayEffect / modifier 的思想映射为 `DataModifier`。
- duration/stack/source cleanup 的生命周期意识映射到 Feature activation 和对象池 release。
- 表格初始化属性值的思路映射到 DataOS records。

不采纳：

- 不照搬 GAS 的 Actor/ASC/AttributeSet 类型体系。
- 不把所有 Data 字段都做成 Attribute。
- 不相信表格里的 min/max 自动生效；SlimeAI 必须在 generator、validator、Data.Set、ApplyRecord 和 modifier pipeline 中明确执行约束。

## 对当前 Data 裁决的影响

最终裁决不变：

```text
Data = 受控共享状态 + 表格驱动 + descriptor 约束 + modifier/computed + 可观察同步
```

更精确地说：

- Component 内部字段可以存在。
- DataOS 到 Data 的表格链路必须保留。
- Data 到 Component 不能靠反射猜测，必须通过显式 DataBinding。
- 对象池复用必须刷新 Data、清理 modifier source、重置 Component local state、重新绑定镜像字段。
- Description、range、allowed values、modifier policy 只属于进入 Data 的字段。
- DataModifier 保留，但只作用于 attribute-like numeric Data 字段。
