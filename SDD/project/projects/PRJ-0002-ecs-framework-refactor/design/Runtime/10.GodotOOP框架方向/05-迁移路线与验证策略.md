目标：先冻结 Data 字段路由、authority、DataBinding、Command 修改入口和 DataModifier 裁决，再写代码。

动作：

### Phase 2：最小行为切片

推荐第一个切片：

```text
Unit + Health + Damage + Recovery
```

原因：

- `CurrentHp` / `MaxHp` 是典型共享状态。
- Damage 需要 Event。
- Heal / Recovery 能验证 Command / owner API 与直接 Data.Set 的边界。
- UI / TargetQuery / Test 都会读生命值。
- Buff / modifier 是否参与生命值或减伤可以被验证。
- 对象池复用时 HP 初始化和 Component 字段同步容易暴露问题。

DoD 草案：

- Unit 从 DataOS record 初始化 Data。
- HealthComponent 从 DataBinding 获得 `CurrentHp` / `MaxHp`。
- DamageSystem / DamageService 通过 typed Command / Request 或 owner API 处理伤害。
- RecoverySystem 通过 Heal Command / owner API 处理恢复，不再随意直接写共享 Data。
- Data changed 同步到 HealthComponent 和 UI/debug/test。
- 对象池复用时 HP、IsDead、modifier source 被正确刷新。
- Godot headless 场景输出 Validation artifact。

## 需要确认

### Must Confirm

暂无阻塞性 Must Confirm。用户已经明确：

- 框架名为 `SlimeAIFramework`。
- `Entity -> Object` 方向可行。
- `Component`、`System` 语义可保留。
- `Data` 名字不改。
- `SDD-0044` 已删除。

### Should Confirm

- **首个实现切片是否用 `Unit + Health + Damage + Recovery`？**
  为什么问：它最容易验证 Data、Event、Component 字段和对象池复用。
  默认值：采用。

- **DataBinding 是否作为显式层，而不是 Component 自己散点读 Data？**
  为什么问：这决定后续代码组织和验证方式。
  默认值：采用显式 DataBinding，并由 Feature owner 声明绑定规则。

- **DataModifier 是否只保留在 attribute-like numeric Data 字段？**
  为什么问：这能保留 Buff/装备/Feature modifier 能力，同时避免 DataModifier 泛化到所有字段。
  默认值：保留但收窄。

- **Command 是否作为共享状态写入入口？**
  为什么问：这能避免外部直接获取 Component 或散点 `Data.Set`。
  默认值：采用 SlimeAI 风格 typed Command / Request + owner handler，不引入 QFramework `AbstractCommand` 对象体系。
