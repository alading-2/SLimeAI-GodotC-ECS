# 问题复盘与重新裁决

## 用户真正想解决的问题

用户原始问题见 [`source-request.md`](./source-request.md)。

本轮不是单纯问“QFramework 好不好”，而是在问：

```text
SlimeAI 现在 Data / Event / Command / System / Feature 一直改，
是不是因为没有一个足够简单的底层框架根？
如果 QFramework 核心很小，我愿意改底层，
直接用它或 hard fork 它，会不会比继续自研更稳？
```

这是一个真实问题。当前 SlimeAI 的文档和代码已经有很多高质量部件，但缺一个“开发者一眼能理解的应用层规则”：

- `SystemManager.Execute` 有命令门禁，但不是所有 owner 写入口都统一进它。
- Data 有 `DataKey<T>`、modifier、computed、DataChanged，但“哪些字段进 Data”仍要靠设计文档解释。
- Feature 有 owner 边界，但功能组合、Profile、Command、Query、Event 的统一语言还不够短。
- Godot Node / Component / System / DataOS / Log / Test 的关系，对人和 AI 都偏重。

QFramework 的价值正在这里：它把规则压到很少，容易记，也容易教 AI。

用户追加判断“SlimeAI 严格说还不是框架，而是一堆底层和功能集合”，大体成立。更精确地说：

```text
SlimeAI 有不少 framework-grade components，
但缺一个 architecture-grade contract。
```

`Data` 和 `Event` 现在承担了过多“框架根”的职责。它们应该是框架根下面的协议，而不是框架根本身。一个框架根至少要能回答：

- 角色有哪些。
- 谁拥有状态。
- 谁能修改状态。
- 谁能查询状态。
- 事件表达事实还是请求。
- 功能如何注册、启停、组合、验证。
- 新功能从哪里开始写，写完如何被框架接管。

QFramework 的强项正是这些问题有短答案。

## 用户思路是否成立

### 成立的部分

1. **QFramework 核心确实小**  
   本地 `Framework/Scripts/QFramework.cs` 约 956 行，单文件 `QFramework.cs` 约 1449 行。核心由 `Architecture<T>`、`IOCContainer`、`ICommand/IQuery`、`TypeEventSystem`、`BindableProperty<T>` 和 `ICanXxx` 接口组成。

2. **Command / Architecture 确实是关键改造点**  
   如果只是学习 QFramework，不需要改它；如果要让它变成 SlimeAI 底层，必须动 `Architecture` 和 `Command`。`Architecture<T>` 的静态单例、手动注册、Type-key IOC、Model/System 初始化顺序，都不等价于 SlimeAI 的 Godot AutoLoad、DataOS config、Feature profile、System diagnostics。`AbstractCommand` 把消息和 handler 合在一个引用对象里，也不适合 SlimeAI 热路径和 AI-first 审计。

3. **QFramework 的少规则能解决 SlimeAI 的一类噪音**  
   “共享数据放 Model，不共享留脚本”“状态变更走 Command”“下层用 Event 通知上层”“接口能力由 ICanXxx 编译期限制”这些规则，比很多长篇架构解释更有执行力。

### 不成立或风险大的部分

1. **“底层东西不多”不等于“能作为 SlimeAI 底层”**  
   QFramework 小，是因为它不处理 SlimeAI 的大部分底层复杂度：Godot scene lifecycle、对象池复用、碰撞、DataOS authoring、per-object state、System run condition、validation artifact、AI 日志分析。把这些加进去后，QFramework 的小核心不会保留原样。

2. **“直接用”会把 SlimeAI 拉回应用层 MVC，而不是 Godot/OOP Feature runtime**  
   QFramework 的 `IController / ISystem / IModel / IUtility` 是 Unity 应用层组织方式。SlimeAI 当前方向是 `Object / Component / System / Feature / Event / Data`，其中 Component 可以持有状态，Data 只承载共享和治理字段。直接套 `Model` 容易让 Data 又变成“所有共享字段大容器”。

3. **改完核心后，长期维护对象其实是 SlimeAI Kernel，不是 QFramework**  
   如果去掉 static singleton、改 Command 为 record/request、改 Event 为 scoped EventBus、改 Model 为 Data/Profile/Component 分区、改 Controller 为 Godot bridge，那么保留下来的主要是名字和思想。此时还说“用 QFramework”，会让后续文档和 AI 路由误判事实源。

## 真实问题

真实问题不是“SlimeAI 需要一个 QFramework 依赖”，而是：

```text
SlimeAI 缺一个足够短、足够硬、可编译期部分约束的应用层 Kernel 规则。
```

它应该回答：

- 哪些状态进入 Data，哪些留在 Component / Profile / System。
- 谁能写状态，写操作怎么命名、怎么追踪、怎么验证。
- 谁能读状态，Query / readonly view / projection 的边界是什么。
- Event 是事实通知，不是命令请求；Command 是意图，不是事实。
- Feature owner 的 `Init` 清单如何像架构图一样可读。
- Godot Node 到 gameplay owner 的边界怎么防止变成万能 controller。

QFramework 可以提供规则样板，但 SlimeAI 必须保留自己的底层事实源。

## 重新裁决

本轮裁决需要比上次更细：

```text
应该深入学习 QFramework。
可以通过“大改 QFramework”做 disposable prototype 和训练。
不建议把 QFramework fork 作为长期 SlimeAI 事实源。
最终目标是 SlimeAI-native Architecture Contract。
```

这不是保守拒绝，也不是否定“魔改”。恰恰相反：如果你缺框架经验，魔改 QFramework 是很好的学习方式，因为它小、清晰、反馈快。问题只在于不要把训练原型误认为最终产品根。

真正可落地的方向是：

```text
SlimeRuntimeKernel
  -> FeatureManifest / SystemManifest / DataOS snapshot / Service registry 的只读总览
  -> CommandDispatcher 承载 typed request + handler/pipeline
  -> QueryFacade 承载 readonly view
  -> EventScope 绑定 Object / Feature / Runtime scope
  -> DataObservable 提供 RegisterWithCurrentValue 语义
```

它吸收 QFramework 的“少规则”，但不复制 QFramework 的“根容器形态”。

新的推荐路线：

```text
Learn:
  完整学习 QFramework 的 Architecture / Rule / Command / Shared Data。

Prototype:
  fork QFramework 核心，删掉 Unity 假设，改成 Godot + SlimeAI 概念。

Extract:
  只把验证后仍清晰的规则写回 SlimeAI Architecture Contract。

Migrate:
  用 Health/Damage/Recovery 之类小切片证明新规则能统领 Data/Event/System/Log/Test。
```

## 默认假设

- 当前仍沿用 `Src/ECS` / `DocsAI/ECS` 历史路径，不在本轮做机械改名。
- 后续若实施，先做 Health/Damage/Recovery 或 Spawn 的最小切片，不全仓重写。
- Command 用 `readonly record struct` 或不可变 request/result，不使用每操作一个 `AbstractCommand` 引用类型对象作为默认形态。
- Architecture 如果要出现，只作为 SlimeAI Kernel facade / manifest，不拥有全部事实源。
- QFramework Toolkits 另行研究，不能和本轮核心架构采纳混在一起。
- “魔改 QFramework”只作为原型/训练，不作为正式依赖引入。
- 正式 SlimeAI 框架文档需要新增 `Architecture Contract` 或同等入口，不能继续只靠 Data/Event 串联。

## 需要确认

### 思路问题

- 你说“用 QFramework”，是想保留 QFramework 的 API 名字，还是只想保留它的规则和代码组织方式？  
  影响：前者会要求兼容 `Architecture<T>` / `IController` 风格；后者可以直接做 SlimeAI-native Kernel。

### 信息缺口

- 你最痛的“改来改去”是 Data 字段归属、Command 写入口、还是整体模块注册/初始化看不清？  
  影响：决定第一刀切 Health/Data，还是切 Kernel manifest / Command dispatcher。

### 决策未定

- 是否接受“QFramework-inspired，但不叫 QFramework，也不引依赖”的方案？  
  默认值：接受。这样后续实现和文档都以 SlimeAI 事实源为准，QFramework 只作为参考资料。

- 是否愿意先做一个一次性 QFramework fork 原型，而不是直接改正式 SlimeAI 源码？  
  默认值：愿意。这样你可以通过改 QFramework 学框架形状，同时避免正式仓继续混乱。
