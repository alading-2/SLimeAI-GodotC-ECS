# 迁移路线与验证门禁

## 路线原则

这不是全仓重写任务。正确顺序是：

```text
先验证规则是否变简单。
再验证一个写入口是否可追踪。
再扩到 DataBinding / FeatureManifest。
最后才考虑目录和命名调整。
```

如果第一刀就改 `Architecture<T>` 或把所有系统迁到新框架，风险会非常高。

## Phase 0：冻结选择

必须先回答：

- 是否接受不引入 QFramework 依赖，只做 SlimeAI-native Kernel。
- 第一刀是解决 Command 写入口，还是解决 Feature / Architecture 清单可读性。
- 是否允许新增 `Runtime/Command` 或 `Runtime/Kernel` 目录作为临时历史路径下的新 owner。

默认建议：

```text
不引依赖。
先切 Command 写入口。
路径暂用 Src/ECS/Runtime/Command 和 DocsAI/ECS/Runtime/Command。
```

## Phase 1：最小 Command Dispatcher

目标：证明 SlimeAI 可以获得 QFramework Command 的好处，而不引入 `AbstractCommand` 对象体系。

建议切片：

```text
Health / Damage / Recovery
```

原因：

- 已有 `DamageProcessRequest -> DamageProcessResult`。
- `DamageService` 已经是 Service Pipeline。
- `RecoverySystem` 已经有 register/unregister request。
- 当前 Data 文档已把 Health/Damage 作为 Data/Command 第一切片。

候选新增：

- `CommandContext`
- `ICommandHandler<TRequest, TResult>`
- `CommandExecutionReport`
- `KernelCommandDispatcher`
- `CommandTrace` 或复用现有 Log trace

不做：

- 不迁移所有 request。
- 不新增通用 string/object command bus。
- 不做 undo/replay。

## Phase 2：DataObservable / RegisterWithCurrentValue

目标：把 QFramework `RegisterWithInitValue` 的价值移到 SlimeAI Data。

候选能力：

```text
Data.SubscribeWithCurrentValue<T>(DataKey<T> key, Action<T> handler)
```

或在 UI binding 层提供等价 wrapper。

验证：

- UI / Debug 绑定注册后立即拿到当前值。
- DataChanged 后再收到增量。
- Node exit / component unregister / pool release 时订阅释放。

## Phase 3：FeatureManifest / KernelManifest

目标：获得 QFramework `Architecture.Init()` 的可读性。

FeatureManifest 应列出：

- owner。
- data authoritative fields。
- projection fields。
- commands / requests。
- queries。
- fact events。
- systems / services。
- DataBinding。
- tests。

这个 manifest 不替代 DataOS 或 SystemManifest，只是 owner 级路由入口。

## Phase 4：能力接口

目标：采纳 QFramework `ICanXxx` 编译期限制。

第一批只做三个上下文：

- `CommandContext`
- `QueryContext`
- `DataBindingContext`

不要一开始就复制 QFramework 的 9 个接口。接口必须服务实际边界。

## 失败条件

出现任一情况应停止扩大范围：

- 新 Command dispatcher 绕过 `SystemManager.Execute`，导致系统运行条件无效。
- 新 Kernel facade 开始保存 gameplay state。
- Data 写入口更多了，而不是更少。
- UI 或 Godot Node 可以直接写 gameplay Data。
- 事件 scope 变模糊，重新出现大量全局 command event。
- 文档必须解释很多例外才能说明新规则。

## 成功标准

最小切片成功应表现为：

- 写 HP / Damage / Heal 的入口更少、更可 grep。
- Command / Event / Query 名称不再混用。
- Data authoritative 与 Component authoritative 字段有 manifest。
- UI 订阅能拿当前值，不手写“先 Get 再 On”。
- `SystemManager.Execute` 的 command gate 仍然有效。
- Log / Test 能看到一次 command 的开始、阻断、完成或失败结果。
