# 方向裁决与概念边界

> 原始问题：见 [`source-request.md`](./source-request.md)。  
> 本文目标：把“弃用 ECS”后的新概念冻结为 `SlimeAIFramework` 可继续实施的方向。

## 真实问题

旧文档已经裁决“弃用 ECS”，但还留下两个会继续误导实现的问题：

1. 把 `Data` 改称 `SharedState`，会让已有 DataOS、DataKey、DataModifier、Data changed event 和技能路由继续发生命名漂移。
2. 把 `System` 改称 `Service`，会让已经完成的 System manifest、preflight、diagnostics、scheduler 等设计重新失去入口。

用户这轮补充是正确的：改方向不等于把所有名词都换掉。真正需要换的是 `ECS` 心智模型和 `Entity` 概念；`Component`、`System`、`Data` 可以保留，但要重新定义语义。

## 新框架名

```text
SlimeAIFramework
```

这个名字避免继续绑定 ECS，也比 `AI-first Godot C# Gameplay Framework` 更适合作为项目和源码命名。`AI-first Godot C# Gameplay Framework` 可以保留为说明语，不作为正式框架名。

## 概念映射

| 旧词 | 新语义 | 是否改名 | 说明 |
| --- | --- | --- | --- |
| ECS Framework | SlimeAIFramework | 改 | 不再以 ECS 为身份 |
| Entity | Object | 改概念 | 表示 Godot/OOP 运行时对象；短期不全仓替换源码路径 |
| Component | Component | 保留 | Godot/OOP 功能组件，可持有内部字段 |
| System | System | 保留 | 跨对象协调、批处理、查询、生命周期、资源等 runtime service |
| Capability | Feature | 倾向改 | Feature 更贴近玩法能力 owner |
| Data | Data | 保留 | 受控共享状态、表格驱动、约束、modifier、computed 和可观察同步 |
| Event | Event | 保留 | typed event/message，Godot Signal 只做表现层或引擎桥接 |
| Runtime | Runtime | 保留 | 对象、组件、系统、事件和 Data 的运行期上下文 |

## Component 与 System 为什么能保留

`Component` 在 Godot 语境中本来就常用于“挂在对象上的功能节点/脚本”。SlimeAI 只需要明确它不是纯数据 ECS component：

- 可以是 Godot Node script 或纯 C# component。
- 可以保存内部字段。
- 只把跨功能共享、需要表格驱动或验证追踪的字段接入 Data。
- 通过 Event、owner API 或 System 与外部协作。

`System` 也可以保留，但语义要从传统 ECS System 改成 SlimeAI runtime system：

- 负责跨多个 Object 的协调。
- 负责批处理、查询、输入、计时器、对象池、资源加载、日志、生命周期。
- 负责 Feature 的全局规则或调度。
- 不负责替代所有 Component 内部逻辑。

## Data 为什么不改名

现有项目已经围绕 `Data` 建立了：

- DataOS SQLite authoring。
- runtime snapshot。
- descriptor / DataDefinition。
- generated `DataKey<T>`。
- typed slot。
- DataModifier。
- computed resolver。
- Data changed event。
- DataOS validator 和 Godot Data 场景测试。

把名字改成 `SharedState` 会让已经稳定的路径、skill、代码和文档继续漂移。正确做法是保留 `Data` 名字，收窄它的职责：

```text
Data 不是 ECS storage。
Data 是 SlimeAIFramework 的受控共享状态和表格驱动协议。
```

## 外部资料采纳边界

本轮外部资料只用于判断方向，不复制 API：

- Godot 官方设计哲学支持 OOP、Scene composition 和 Node hierarchy，说明 SlimeAI 应顺着 Godot 原生结构设计。
- Unity Entities 官方资料说明真正 ECS 会围绕 archetype / chunk 存储 component 数据，反证 SlimeAI 当前不是 ECS storage。
- Unreal Gameplay Ability System 的 Attribute / GameplayEffect / modifier 模型说明“属性 + modifier + 表格默认值”值得保留，但应限制在玩法属性，不应扩成所有状态。
- QFramework 继续只学习强类型 Model、Command、Query、Event 和少规则分层，不接入依赖。

## 推荐方向

后续文档和实现按下面原则推进：

```text
Object 是运行时对象。
Component 是对象上的功能单元。
System 是跨对象/跨功能协调器。
Feature 是玩法能力 owner。
Event 是通信机制。
Data 是需要共享、表格驱动、验证追踪或 modifier/computed 的状态协议。
```

状态默认可以放在 Component。只有证明需要共享、表格初始化、AI/validator/log/test 追踪、持久化、debug panel 或 modifier/computed 时，才进入 Data。
