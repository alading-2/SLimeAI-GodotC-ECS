# 类型转换与生成期检查深化

> 本轮原始问题摘要：用户认为类型转换应该统一，不能 loader 一套、storage 一套、compute 一套；并希望尽量在代码执行前做类型检查，例如数据库或生成 JSON 时发现 enum/default/record 类型问题，减少 Data runtime 类型验证。  
> 本文目标：说明其他引擎的类型转换在哪里发生，并给出 SlimeAI 的统一类型转换边界。

## 结论

其他引擎不是没有类型转换，而是把转换放在 authoring / baking / import / reflection 边界；runtime hot path 使用已经确定的类型。

SlimeAI 应采用同样原则：

```text
DB / CSV / seed / authoring text
  -> validator 严格检查
  -> generator 输出 typed JSON
  -> catalog build 绑定 CLR Type 和 codec
  -> DataSlot<T> typed 存储
  -> runtime Get/Set 不再猜类型
```

当前不应该继续让 `RuntimeDataSnapshotLoader`、`RuntimeDataRecordQuery`、`DataValueConverter`、`DataComputeRegistry`、`generate-data-key-handles.py` 各写一套类型映射。

## 其他框架怎么做

### Unity Entities

Unity 的 authoring component 经过 Baker 转成 entity data。runtime component 是实现 `IComponentData` 的 C# struct，系统按 typed component 访问数据。

类型转换发生在：

```text
MonoBehaviour / authoring / asset
  -> Baker<T>.Bake
  -> IComponentData struct
```

runtime 系统读取时，类型已经是 `T`。

### Bevy

Bevy 组件是 Rust 类型，storage 只决定 Table / SparseSet。反射需要组件实现 `Reflect` 并注册到 type registry。也就是说，动态访问是工具边界，普通 ECS 系统仍通过静态组件类型读写。

类型转换发生在：

```text
asset / scene / reflection input
  -> type registry / component registration
  -> typed component storage
```

### Flecs

Flecs 中 component 有 id、type、size 等 metadata，storage 知道某个 id 对应什么 payload layout。pair / relationship 很动态，但 component payload 仍有明确类型和大小。

类型转换发生在：

```text
component registration
  -> component id / size / hooks
  -> table or sparse component storage
```

### Unreal GAS

GAS 的 Attribute 是 `AttributeSet` 上的显式属性，`GameplayAttribute` 引用这些属性，DataTable / Effect / Modifier 最终作用到明确的 attribute。

类型转换发生在：

```text
DataAsset / DataTable / GameplayEffect
  -> AttributeSet / GameplayAttribute
  -> AbilitySystemComponent runtime
```

它不靠每次读写 `Dictionary<string, object>` 决定类型。

## SlimeAI 的类型问题在哪里

当前类型映射至少散在这些地方：

- `RuntimeDataSnapshotLoader.ParseValueType()`
- `RuntimeDataSnapshotLoader.ConvertDefaultValue()`
- `RuntimeDataRecordQuery.ParseValueType()`
- `DataValueConverter.TryConvert()`
- `DataValueConverter.IsCompatible<T>()`
- `DataValueConverter.ConvertForRead()`
- `DataComputeRegistry.ResolveExpectedOutputType()`
- `generate-data-key-handles.py` 的 `TYPE_MAP`
- `validate-dataos.sh` 的 Python `value_types` / default parse 逻辑

这些重复说明 Data 类型契约没有统一事实源。

## 推荐：建立 DataTypeContract / DataValueCodec

需要一个统一概念，不一定第一轮就写成一个大类，但设计上必须有。

```text
DataTypeContract
  logicalType: float/int/bool/string/enum/object_ref...
  clrTypeHint: Team/Godot.Node2D/Texture2D...
  clrType: System.Type
  codec: DataValueCodec
  isNumeric
  supportsRange
  supportsModifier
  supportsAllowedValues
```

```text
DataValueCodec
  ParseAuthoringText(text) -> typed object
  ParseJsonElement(json) -> typed object
  ValidateAllowedValue(value)
  ToSnapshotJson(value)
  FormatDiagnostic(value)
```

运行期的 `DataSlot<T>` 不应该每次拿 `DataValueType` switch。它应该拿到已经构建好的 typed policy / codec，只在 loader/debug 这种边界使用 codec。

## 是否应该用 `System.Type`

应该用，但不是在数据库里直接存 `System.Type`。

数据库适合保存：

```text
value_type = enum
clr_type_hint = Team
```

runtime catalog build 后再得到：

```text
LogicalType = Enum
ClrType = typeof(Team)
Codec = EnumCodec<Team>
```

原因：

- SQLite / JSON / CSV 不适合直接保存 CLR `Type`。
- `System.Type` 在 C# runtime 有价值，可以做 resolver output 校验、slot 创建、diagnostic。
- `clr_type_hint` 需要由 generator/validator 解析，不应在 runtime 每次猜。

## 类型检查应该前移到哪里

### 1. 数据库 validator

DataOS validator 应尽早检查：

```text
value_type 是否合法
default_value_text 是否能按 value_type 解析
enum default 是否在 enum manifest 中
min/max 是否是数字且 min <= max
range_policy 是否只用于 numeric
modifier_policy=numeric 是否只用于 numeric
compute_id 是否存在
dependencies_json 是否引用已存在 stable_key
record field value_text 是否能按 descriptor type 解析
runtime_type_id / clr_type_hint 是否在需要时填写
```

如果 DB validator 过不了，不生成 snapshot。

### 2. snapshot generator

generator 应输出 typed JSON，而不是把 text 推到 runtime：

```json
{
  "stableKey": "BaseHp",
  "valueType": "float",
  "defaultValue": 10.0
}
```

record field 也应是 typed JSON：

```json
{
  "type": "float",
  "value": 100.0
}
```

非标量可以保留结构化 JSON，例如 `modifier_list`、resource ref、vector2。

### 3. generated handle gate

生成 `DataKey<T>` 后，validator 必须检查：

```text
descriptor.value_type + clr_type_hint == GeneratedDataKey 的 T
```

例如：

```text
Team enum -> DataKey<Team>
TargetNode object_ref + Godot.Node2D -> DataKey<Node2D?>
BaseHp float -> DataKey<float>
Feature.Modifiers modifier_list -> DataKey<FeatureModifierEntryData[]>
```

这个 gate 通过后，业务代码就不需要靠 `DataValueConverter.IsCompatible<T>()` 当主要保护。

### 4. catalog build

catalog build 只做防御性检查和绑定：

```text
descriptor -> DataRuntimeDefinition
defaultValue -> typed default
computeId -> resolver
resolver.OutputClrType == definition.ClrType
policy combination report
```

catalog build 失败时应先生成 `DataCatalogBuildReport` / structured Log，再 fail-fast。

### 5. runtime storage

runtime storage 应只做：

```text
Data.Get<T>(DataKey<T>) -> DataSlot<T>
Data.Set<T>(DataKey<T>, T) -> typed policy -> typed slot
SetUntyped(loader/debug) -> boundary codec -> typed slot
```

业务热路径不再：

```text
object -> DataValueType switch -> Convert.ChangeType -> T
```

## 统一工具怎么避免 Python/C# 双写

完全消除 Python 和 C# 的重复不现实，因为 validator/generator 在 Python/shell，runtime 在 C#。但可以把重复压到一个事实源。

可选方案：

### 方案 A：保留双实现，但加一致性测试

```text
Python validator TYPE_MAP
C# DataTypeContract
snapshot fixture
一致性测试验证同一批 value_type/default/record 结果一致
```

优点：成本最低。  
缺点：仍有两套映射。

### 方案 B：用 JSON type manifest 作为单一事实源

```text
Data/DataOS/Schema/DataValueTypes.json
  logicalType
  clrType
  isNumeric
  supportsRange
  supportsModifier
  defaultParser

Python validator 读它
C# generator 或 runtime 读/生成它
```

优点：事实源统一。  
缺点：需要设计 manifest 和生成 C# 映射。

### 方案 C：C# 生成 validator 辅助代码

用 C# 作为类型映射事实源，再生成 Python/JSON 供 DataOS CLI 使用。

优点：runtime 类型最权威。  
缺点：DataOS 工具链会依赖 C# build，复杂度更高。

推荐：第一轮采用方案 A，第二轮进入方案 B。不要一开始把工具链改成 C# 驱动。

## DataValueType 的新定位

`DataValueType` 可以保留，但它的定位要改：

```text
旧定位：
runtime 到处 switch 的转换中心

新定位：
authoring/snapshot 的逻辑类型枚举
```

runtime catalog build 后应使用：

```text
DataRuntimeType
  LogicalType: DataValueType
  ClrType: Type
  Codec: DataValueCodec
```

这样 `DataValueType` 不再“不靠谱”。不靠谱的是把它放到 runtime 各处重复判断。

## 报错、Log 和 throw 的边界

类型错误分三类：

| 场景 | 处理 |
| --- | --- |
| DB / seed 错 | validator 输出 error，阻止生成 snapshot。 |
| snapshot / catalog 错 | 生成 report + structured Log，然后 throw，阻止启动。 |
| runtime debug/untyped 写错 | 返回 `DataWriteReport`，必要时 Log，不直接吞掉。 |

坏 catalog 和坏 snapshot 仍然应该 throw，因为继续运行会污染全局状态。问题不是 throw 本身，而是 throw 前缺少 AI 可读的 report/log。

## 实施顺序建议

第一阶段：

```text
增强 validate-dataos.sh：
  enum default / allowed values
  policy combination
  record value type

增强 snapshot generator：
  typed defaultValue
  typed record value
  删除 legacy mirror 投影
```

第二阶段：

```text
新增 C# DataTypeContract / DataValueCodec
RuntimeDataSnapshotLoader 只调用 codec
DataComputeRegistry 使用 definition.ClrType，不再自己 switch DataValueType
```

第三阶段：

```text
GeneratedDataKey type gate 强制通过
DataRuntimeStorage typed API 不再依赖 IsCompatible<T> 作为主保护
SetUntyped 只保留 loader/debug
```

第四阶段：

```text
DataValueTypes.json 或等价 manifest 统一 Python/C# 映射
```

这条路线可以最大限度减少 runtime 类型转换，同时不牺牲 DataOS descriptor-first 和 AI-first 路由。
