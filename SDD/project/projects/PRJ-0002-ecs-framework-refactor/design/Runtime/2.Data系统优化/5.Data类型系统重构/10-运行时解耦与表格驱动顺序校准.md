# 运行时解耦与表格驱动顺序校准

> 原始问题：见 [`../6.架构学习/source-request.md`](../6.架构学习/source-request.md) 的“2026-06-15 运行时解耦补充问题”。  
> 上游裁决：[`../6.架构学习/06-运行时解耦第一原则与框架目标.md`](../6.架构学习/06-运行时解耦第一原则与框架目标.md)、[`09-Data系统根本裁决与重构路线.md`](./09-Data系统根本裁决与重构路线.md)。

## 结论

Data 当前问题需要再校准一次：不是 Data 集中、表格驱动、DataOS 这些需求本身错误，而是它们被放到了底层 runtime 之前。

用户说“需要什么功能直接填表格很方便”是对的。这是 SlimeAI 未来应该保留的上层体验。但它不能成为当前 Data runtime 的第一目标。

当前正确顺序是：

```text
先固定底层 runtime 解耦
  Entity / Component / System / Event / Data 的边界简单直接

再固定 Data 进入条件
  只有跨功能共享、需要验证追溯的 runtime state 进入 Data

再做 Data 类型契约
  generated DataKey<T>、typed default、typed slot、typed computed

最后做表格驱动体验
  DataOS / profile / capability manifest / validator / editor
```

反过来就会形成现在的问题：

```text
为了“填表方便”
  -> 把 authoring、presentation、policy、AI routing、runtime storage 都压进 Data
  -> Data 变成最复杂的底层核心
  -> Component / System 解耦目标反而被淹没
```

## Data 不再承担框架第一目标

SlimeAI 第一目标是运行时功能解耦，不是统一数据形态。

Data 的职责应降为：

```text
跨功能共享的 typed runtime state protocol
```

它不应该继续承担：

- 所有配置表。
- 所有展示 metadata。
- 所有“谁能写”的权限规则。
- 所有 Capability 内部 cache / index / 临时状态。
- 所有表格驱动体验。
- 所有 AI 路由上下文。

DataOS 可以保存 authoring 信息，validator 可以检查大量规则，generator 可以产出 manifest；但 runtime `DataDefinition` 和 `DataRuntimeStorage` 必须变小。

## Data 进入条件重新固定

字段进入 Entity.Data 必须至少满足一个条件：

- 两个以上 Capability 需要读写或观察。
- 需要从 runtime snapshot 初始化为 Entity runtime state。
- 需要 modifier、computed、range、allowed values 这类数据形态 contract。
- 需要 generated `DataKey<T>` 作为 AI、测试、日志和 debug 的稳定追溯入口。
- 需要 Data changed event 让 UI、Ability、Feature、Damage 等 owner 订阅。

默认不进 Entity.Data：

- 单 Capability 内部 cache / index。
- 一帧内局部计算值。
- 只服务某个算法的临时集合。
- UI 展示文案、图标、分组、格式。
- 只服务 authoring/editor 的表格字段。
- “谁可以写”这类组织纪律规则。
- 纯配置表对象，除非它会成为 Entity runtime state。

## 表格驱动是后续上层能力

表格驱动的长期价值仍然存在：

- AI 可以在表格里看字段、默认值、依赖和描述。
- 游戏可以通过 profile 选择能力组合。
- DataOS validator 可以提前发现错误。
- 未来可以做“填表组合玩法”的上层工具。

但表格驱动不应直接决定 runtime 结构。它应该通过明确生成边界进入 runtime：

```text
Authoring table
  -> validator
  -> generator
  -> runtime snapshot / generated handle / manifest
  -> runtime Data / System / Capability
```

运行时不回头查询 authoring 表，不把展示字段投影进热路径，不让 runtime `Set` 处理组织纪律。

## 与后续 SDD 的关系

### SDD-A：Data Runtime Simplification

必须先做。

目标：

- `RuntimeDataDescriptorDto` 不再投影 owner / skill / display / description / uiGroup / icon / legacy mirror 字段。
- `DataDefinition` 只保留 runtime 必需字段。
- presentation / authoring / AI routing 信息迁到 sidecar manifest、DocsAI 或 generator report。
- `write_policy` 从 runtime enforcement 核心降级为 DocsAI / validator / code review / grep gate。
- catalog build fatal 前输出 report / structured log。

### SDD-B：Data Type Contract

第二步。

目标：

- 统一 `DataTypeContract` / `DataValueCodec`。
- default typed value 只转换一次。
- record value 只在 loader/debug 边界转换一次。
- slot 类型由 catalog 决定。
- computed resolver output type 在 catalog build 校验。
- `SDD-0044 DataComputeRegistry` 并入这里或作为前置子任务。

### SDD-C：Generated RuntimeId Storage

第三步。

目标：

- `DataKey<T>` 携带 stable key + runtime id。
- hot path 通过 runtime id 查 slot。
- stable key 继续服务 DataOS、日志、debug 和 AI 追溯。

### SDD-D：Capability/Profile Manifest

Data runtime 简化后再推进。

目标：

- 明确每个 Capability 的 DataKeys、Events、Systems、Components、Dependencies、Validation 和 Reject list。
- 明确每个 game profile 启用哪些 capability/system/data/resource。
- 支持启动前裁剪，而不是运行时用万能 Data 判断所有功能是否存在。

### SDD-E：表格驱动组合体验

最后做。

目标：

- 用表格、profile 和 validator 提高玩法组合效率。
- 不把编辑器体验倒灌进 runtime core。

## Feature 复杂度后续记录

当前 Feature 等上层系统也有类似问题：它试图承接太多扩展点，导致概念复杂、调用路径难读。

后续记录：

- 重新评估 Feature 是否只保留最小 modifier/action 能力。
- Ability / Damage / Feature 的状态改变入口尽量用简单 request / service / command。
- 删除或继续压缩 raw object / dynamic action 边界。
- 每个 Capability 的最小可运行闭环应先简单，再组合复杂功能。

这个记录不进入当前 Data SDD-A/B/C，但必须登记为后续上层简化方向。

## 当前执行提醒

后续不要把“Data 很方便”当成继续扩 Data runtime 的理由。正确表述应是：

```text
DataOS 表格体验值得保留。
Data runtime 必须变小。
表格驱动必须建立在清楚的 runtime 解耦之上。
```
