# 需求归纳与真实问题

> 原始问题：见 [`0.Prompt.md`](./0.Prompt.md)。  
> 本文目标：先判断用户提出的问题是否真实存在，再给出 Data 类型系统重构的方向边界。

## 用户真正想解决什么

用户表面上提出了几类具体不满：

- `DataDefinition` 字段太多，比旧 `DataMeta` 更冗余。
- 默认值在 DataOS / 知识库中像是只能存 text，进入 runtime 时类型恢复不清楚。
- `DataValueType` 和当前转换方法不可靠。
- `DataRuntimeStorage` / `DataSlot` 为了 modifier、computed、loader、debug 做了太多类型转换。
- 很多 Data 初始化和验证错误直接 `throw`，缺少 Log / Report。
- 最大疑问是：为了保持 Data 解耦，把所有字段放进字典是否可行。

归纳后的真实需求是：

```text
保留 Data 作为跨 Capability 的统一状态协议，
但把类型契约从运行时转换逻辑前移到 authoring/generator/catalog build，
让 runtime Data 像 typed storage，而不是像弱类型字典。
```

这不是一次“删几个字段”的小优化，而是 Data 核心契约重新分层。

## 问题是否真实存在

真实存在。

### 1. `DataDefinition` 已经变成混合对象

当前 `Src/ECS/Runtime/Data/DataDefinition.cs` 同时包含：

- 运行时读写必须字段：`StableKey`、`ValueType`、`DefaultValue`、`StoragePolicy`、`WritePolicy`、`RangePolicy`、`ModifierPolicy`、`ComputeId`、`Dependencies`。
- DataOS / AI 路由字段：`OwnerDomain`、`OwnerCapability`、`OwnerSkill`。
- 展示字段：`DisplayName`、`Description`、`UiGroup`、`ResetGroup`、`Unit`、`Format`、`IconPath`。
- 迁移和历史字段：`MigrationPolicy`、`AllowedValues`、`RuntimeTypeId`。

这些字段不是都没用，但不应该都进入 `DataDefinition` 这个运行时字段定义对象。

补充审计结论：

- `StableKey`、`ValueType`、`DefaultValue`、`StoragePolicy`、`WritePolicy` 是 runtime catalog 必需字段。
- `RangePolicy/MinValue/MaxValue`、`ModifierPolicy`、`AllowedValues`、`ComputeId/Dependencies/ComputeParams` 是条件必需字段，只应在字段真的使用对应能力时进入 runtime。
- `OwnerDomain/OwnerCapability/OwnerSkill`、`DisplayName/Description/UiGroup/ResetGroup/Unit/Format/IconPath` 更像 authoring、AI 路由或 presentation metadata。
- `RuntimeTypeId` 当前确实有少数用途：`object_ref` 校验和 enum/object_ref generated handle 生成；但普通标量字段不应携带它。

### 2. snapshot descriptor 仍携带旧 mirror 字段

当前 `Data/DataOS/Schema/core.sql` 的 `data_key_descriptor` 同时有新字段和旧字段：

```text
modifier_policy / allowed_values_json / storage_policy / compute_id
category / min_value / max_value / options_json / is_percentage / supports_modifiers / is_computed
```

`generate-runtime-snapshot.sh` 又把这些旧字段继续投影到 `runtime_snapshot.json.descriptors`。字段统计显示：220 个 descriptor 都带 `category`，但 `options` / `allowedValues` / `format` / `unit` / `iconPath` / `resetGroup` 等大多为空。这说明 snapshot 既承担 runtime catalog，又承担展示/历史镜像，边界混杂。

更具体地说，当前 snapshot 里 220 个 descriptor 都携带 `category/ownerCapability/ownerDomain/ownerSkill/displayName/uiGroup/policy`；但 `allowedValues/options/format/iconPath/unit/resetGroup/computeParams` 基本为空。这个现象说明冗余不只是代码观感问题，而是 generator 正在把长期为空或非 runtime 字段复制到每个字段定义里。

### 3. 类型转换没有真正前移

当前主链路大致是：

```text
SQLite default_value_text / record value_text
  -> generate-runtime-snapshot.sh CAST 成 JSON number/bool/string
  -> RuntimeDataDescriptorDto.DefaultValue object?
  -> RuntimeDataSnapshotLoader.ConvertDefaultValue()
  -> DataDefinition.DefaultValue object?
  -> DataSlot<T>.ConvertToSlotValue()
  -> DataValueConverter.ConvertForRead()
  -> DataSlot<T>._defaultValue
```

record 写入也类似：

```text
record field JsonElement/object?
  -> NormalizeRecordValue()
  -> DataValueConverter.TryConvert()
  -> Data.SetUntyped(... object?)
  -> TryApplyWritePoliciesWithReport()
  -> GetOrCreateBoundarySlot()
  -> DataSlot<T>.SetValueFromBoundary()
  -> DataSlot<T>.ConvertToSlotValue()
```

也就是说，类型虽然被描述出来了，但转换发生了多次，且不同边界各有一套判断。

重复判断主要集中在三处：

- 类型映射重复：snapshot loader、record projection、`DataValueConverter`、`DataComputeRegistry`、handle generator 各维护一套 `valueType -> CLR type` 判断。
- record 转换重复：`ApplyRecord()`、`SetUntyped()` policy、slot boundary writer 都会再次转换。
- projection 转换旁路 catalog：`RuntimeDataRecordQuery` 主要依赖 `field.Type` 做读取转换，未完全复用 catalog build 后的 typed definition。

### 4. 其他框架不是没有转换层，而是转换层不在热路径

本地 `Resources/Engine` 和官方资料的共同点：

- Unity Entities：authoring 经 `Baker<T>` 转成 `IComponentData`，runtime 组件是 `struct/unmanaged` 类型。
- Bevy / EnTT / Friflo：组件/存储按类型组织，字典或 type-erased 结构主要是内部 storage 索引。
- Entitas：内部用 component index 存数组，但对外推荐生成 API。
- ET Numeric：可以用 `Dictionary<int, int>` 做数值系统，但 key 是枚举编码，值域受公式系统限制，不是任意 `string -> object`。

推断：SlimeAI 可以保留 `stableKey -> slot` 字典，但类型必须由生成期和 typed handle 固定下来，不能靠 runtime 每次读写再转换。

## 用户思路哪里成立，哪里需要修正

### 成立：Data 可以保持解耦

Data 作为统一状态协议是成立的。它让 Ability、Feature、Damage、AI、UI 不需要直接互相持有组件字段，也适合 AI 通过 DataOS descriptor 和 generated handle 理解字段。

正确的解耦是：

```text
DataOS descriptor 解耦字段定义
Generated DataKey<T> 解耦调用点和裸字符串
DataSlot<T> 解耦 typed value 与跨类型管理
IDataSlot 解耦 loader/debug/diagnostic 边界
Capability service 解耦业务流程和 Data 存储
```

### 需要修正：解耦不等于运行时动态类型

如果 Data 的统一性靠 `Dictionary<string, object>` 和 `DataValueType` switch 维持，Data 会成为全框架最弱的类型边界。

正确目标不是“Data 没有字典”，而是：

```text
字典只做 stableKey -> slot 寻址；
slot 内部保存 T；
业务 API 只允许 DataKey<T>；
loader/debug 才允许 object/text。
```

### 成立：`throw` 不能替代 observation

坏 catalog、坏 snapshot 需要阻断，所以 `throw` 仍必要。但现在很多错误在抛出前缺少结构化报告，AI 只能从异常字符串里猜。

更好的边界是：

```text
Report 负责记录所有可诊断错误
Log / Validation artifact 负责让 AI 可搜索
throw 负责阻断无法继续运行的 fatal 状态
```

### 需要修正：不是所有分类都应该删掉

`OwnerCapability`、`OwnerSkill`、`DisplayName`、`Description` 不是完全无用。它们对 DataOS 裁剪、AI 路由、TestSystem/UI 展示有价值。问题是它们不应该默认进入 runtime `DataDefinition` 热路径对象。

## 当前最大问题

当前 Data 主要问题不是“字典”，而是四层契约没有分清：

| 层 | 应承担 | 当前问题 |
| --- | --- | --- |
| Authoring descriptor | 字段定义、说明、owner、展示、默认值 text、校验输入 | 太多字段直接投影到 runtime snapshot |
| Generated contract | `DataKey<T>`、类型映射、字段 id、可选 accessor | 现在只生成 thin handle，未生成 typed descriptor / field id |
| Runtime catalog | Data 读写、modifier、computed、range、allowed value 必须信息 | 混入 owner/presentation/历史 mirror 字段 |
| Runtime storage | typed default/current/modifier/computed cache | loader/default/debug 多次走 object/valueType 转换 |

如果不先分层，单纯删 `RuntimeTypeId` 或 `OwnerCapability` 会变成局部补丁，类型复杂度仍会留在 `DataValueConverter` 和 `DataRuntimeStorage`。
