# 类型系统与运行时存储重构方案

## 类型问题的根因

当前 Data 的类型信息分散在四个地方：

1. DataOS `data_key_descriptor.value_type` / `runtime_type_id`。
2. `runtime_snapshot.json.descriptors[*].valueType/runtimeTypeId/defaultValue`。
3. `GeneratedDataKey` 的 `DataKey<T>` 泛型参数。
4. `DataRuntimeStorage` 的 `DataValueConverter` 和 `DataSlot<T>`。

这四处都在表达“字段是什么类型”，但权责不同。真正的问题是：runtime 仍需要用 `DataValueType` 和 `object?` 重新判断很多次，说明类型没有在生成期和 catalog build 阶段被充分收敛。

目标应改成：

```text
DataOS / generator 决定字段类型
GeneratedDataKey<T> 固定业务调用类型
Catalog build 验证 descriptor / generated handle / resolver 一致
DataRuntimeStorage 只按已知 T 创建 slot 和执行 typed policy
```

## `DataValueType` 怎么看

用户觉得 `DataValueType` 不靠谱，这个判断需要拆开看。

`DataValueType` 作为 DataOS schema 枚举是有价值的，因为 SQLite / JSON / CSV 不适合保存 `System.Type`：

```text
float / int / bool / string / enum / object_ref / modifier_list
```

但 `DataValueType` 不应长期承担 runtime 转换中心。当前问题不是 enum 本身，而是很多 runtime 逻辑都在 switch 它：

- `RuntimeDataSnapshotLoader.ParseValueType()`。
- `RuntimeDataSnapshotLoader.ConvertDefaultValue()`。
- `DataValueConverter.TryConvert()`。
- `DataValueConverter.IsCompatible<T>()`。
- `DataValueConverter.ConvertForRead()`。
- `DataComputeRegistry.ResolveExpectedOutputType()`。
- `DataRuntimeStorage.ResolveBoundarySlotType()`。

推荐方向：

```text
DataValueType 保留在 authoring/snapshot 边界
DataRuntimeType / ValueClrType 在 catalog build 后固定
typed policy / typed default / typed resolver 进入 runtime
```

换句话说：`DataValueType` 是输入语言，不是运行时到处判断的核心架构。

## 转换链路要前移

### 当前链路的问题

default 和 record 进入 DataSlot 前经历多次 object/text 转换。即使 typed `Data.Set(DataKey<T>, T)` 已经避免了部分装箱，下面这些路径仍然复杂：

- `DataSlot<T>` 构造时从 `definition.DefaultValue object?` 再转一次。
- `SetUntyped` 先 `TryConvert`，再 `SetValueFromBoundary`，slot 内又 `ConvertToSlotValue`。
- `ReplaceSlot<T>` 通过 diagnostic object 迁移旧值。
- `GetOrCreateBoundarySlot()` 可能用实际 object_ref value 类型创建 slot，导致同一个 descriptor 的 slot 类型依赖第一次写入。

### 推荐链路

```text
DataOS validator
  校验 default_value_text、record value_text、allowed_values、range、resolver id、generated handle 类型。

Snapshot generator
  输出 typed JSON value；非标量保留结构化 JSON，不输出额外 mirror 字段。

Catalog build
  将 descriptor 转为 DataRuntimeDefinition / DataRuntimeDefinition<T>。
  default 转成 typed default。
  resolver output type 与 field type 绑定校验。

Runtime loader
  record value 只做一次 boundary parse。
  按 generated/runtime definition 直接调用 typed slot writer 或 typed boundary writer。

Runtime Get/Set
  不再从 object/text/valueType 推断类型。
```

## DataRuntimeStorage 简化方向

### 1. slot 类型由 catalog 决定，不由边界 value 决定

当前 `ResolveBoundarySlotType(definition, value)` 对 `object_ref` 可能返回实际运行时对象类型。这会让同一个 stable key 的 slot 类型依赖第一次写入对象。

建议：

- `ResourceRef` 字段固定 `DataSlot<ResourceRef>`。
- runtime Godot Node 引用字段如果 descriptor 写 `clr_type_hint=Godot.Node2D`，就固定 `DataSlot<Godot.Node2D?>`。
- 如果暂时无法在非泛型边界创建 nullable typed slot，至少固定为 `DataSlot<Node2D>` 或 `DataSlot<object>`，不要按具体子类变化。

### 2. modifier 只对 numeric typed slot 开放

当前 `DataSlot<T>.ApplyModifiers()` 用 `TryGetNumeric` + `ConvertNumericToSlotType` 支持 int/float/double。这个方向可接受，但应把 modifier 能力从通用 slot 中抽出来：

```text
DataSlot<T>
  保存 default/current/changed

NumericDataSlot<TNumber>
  只对 int/float/double 开启 modifier pipeline
```

短期如果不拆类，也要保证：

- `modifier_policy=numeric` 只能绑定 numeric `ValueClrType`。
- 非 numeric 字段不会创建 modifier 列表或执行 modifier 逻辑。
- Feature/Buff 添加 modifier 时不再走 string target，而走 generated `DataKey<TNumber>` 或经过一次 typed resolver。

### 3. policy 泛型化

当前 typed set 已有 `TryApplyTypedWritePoliciesWithReport<T>`，但仍会调用 `ToStableText`、`TryGetNumeric(object)` 这类通用路径。

建议拆出：

```text
IDataValuePolicy<T>
  ValidateWrite(T value, DataWriteSource source)
  ApplyRange(T value, DataWriteSource source)
  ValidateAllowedValue(T value)
```

catalog build 时按字段生成/绑定 policy，runtime 写入不再每次 switch `DataValueType`。

### 4. computed resolver 已经 typed，继续收口 catalog 校验

当前 `IDataComputeResolver<T>` 是正确方向。后续应配合 `DataComputeRegistry` 单例与 catalog build report：

- registry 只保存 `computeId -> resolver`。
- catalog build 检查 resolver output type。
- runtime `GetComputedValue<T>` 只做 typed resolver 调用，不再转 object。

这与 `4.Data验证与Registry简化/01-DataComputeRegistry单例与Catalog验证收敛.md` 方向一致。

## 生成期应新增的契约

### 1. generated handle type gate

`validate-dataos.sh` 已检查 snapshot descriptor 与 generated handle 是否一致，这是好方向。后续要让它成为强门禁：

```text
descriptor.valueType + clr_type_hint
  == GeneratedDataKey.<Identifier> 的 DataKey<T>
```

这个检查通过后，业务代码就不应该再依赖 `DataValueConverter.IsCompatible<T>()` 作为主要保护。

### 2. generated field id / index

当前 `DataRuntimeStorage` 用 `Dictionary<string, IDataSlot>`。这可保留，但长期可以给 generated key 增加稳定 runtime id：

```csharp
public readonly record struct DataKey<T>(string StableKey, int RuntimeId);
```

收益：

- runtime catalog 可用 `RuntimeId -> definition` 数组。
- slot 可用数组或 sparse array 存储，减少 string lookup。
- diagnostics 仍输出 stable key。

这属于 Adopt Later，不建议在第一轮字段瘦身里做。

### 3. generated typed accessor 可选

对高频字段可以生成：

```csharp
public static float GetBaseHp(this Data data) => data.Get(GeneratedDataKey.BaseHp);
public static bool SetBaseHp(this Data data, float value) => data.Set(GeneratedDataKey.BaseHp, value);
```

这不是必须，但可减少 AI 手写泛型和字段名的机会。

## object 边界保留规则

不追求全删 `object`，但必须命名清楚：

| 允许位置 | 规则 |
| --- | --- |
| `RuntimeDataDescriptorDto.DefaultValue` | 只作为 JSON DTO 边界；catalog build 后必须 typed。 |
| `RuntimeDataRecordDto.Field.Value` | 只作为 snapshot record 输入；ApplyRecord 后必须 typed。 |
| `SetUntyped` / `TrySetUntyped` | 只给 loader/debug/TestSystem；业务 grep gate 禁止新增。 |
| `GetDiagnosticSnapshot` / `GetAllValuesForDiagnostics` | 只给 debug dump；名称必须带 Diagnostics。 |
| structured Log / Report | 可以保存 string/object 摘要，不参与 gameplay 读写。 |

禁止位置：

- Capability 业务逻辑新增 `Data.Get<T>(string)`。
- Feature modifier 以任意 string target 直接写 Data。
- computed resolver 返回 `object?`。
- Data changed 业务事件继续使用 untyped payload。

## 其他框架参考如何落到 SlimeAI

### 采纳

- Unity Entities 的 authoring/baking 分层：DataOS/generator 相当于 SlimeAI 的 bake 阶段，runtime 不回读 authoring。
- Entitas 的 generated API 思路：内部可以是索引结构，外部必须是生成的 typed API。
- ET Numeric 的受限数值字典思路：如果未来 HP/Attack/Speed 成为性能热点，可引入 numeric lane，但 key 必须 codegen，不是自由 string。
- EnTT/Friflo 的 typed storage 思路：字典可作为 storage registry，但 payload 应按类型保存。

### 不采纳

- 不复制 Unity DOTS chunk/archetype 存储。
- 不把 AttributeSet 作为第二套状态源引入。
- 不引入通用 ECS query DSL。
- 不把 DataOS descriptor 改回 C# 组件字段定义事实源。

## 建议目标架构

```text
DataAuthoringDescriptor
  stable_key
  value_type
  clr_type_hint
  default_value_text
  owner_capability / owner_skill
  presentation
  policy authoring

RuntimeDataSnapshot
  descriptors: runtime-only subset
  presentationDescriptors: optional sidecar
  records: typed JSON values

DataDefinitionCatalog
  stableKey -> DataRuntimeDefinition
  compute dependency graph
  resolver binding report

DataRuntimeStorage
  stableKey/runtimeId -> DataSlot<T>
  typed Get/Set
  typed policy
  typed computed
  diagnostic boundary only when requested
```

第一阶段可以仍使用 `Dictionary<string, IDataSlot>`；这不是问题。问题是不能再让 value 类型由 runtime 反复猜。
