## Evidence

- QFramework README 说明其架构支持 SOLID、DDD、事件驱动、数据驱动、分层、MVC、CQRS、模块化，并给出四层规则：Controller、System、Model、Utility 和 Command。
- QFramework `QFramework.cs` 中 `Architecture<T>` 注册 System / Model / Utility，并提供 SendCommand / SendQuery / SendEvent。
- QFramework `IOCContainer` 使用 `Dictionary<Type, object>` 保存模块实例；`EasyEvents` 使用 `Dictionary<Type, IEasyEvent>` 保存事件表。
- QFramework `BindableProperty<T>` 内部保存 `T mValue`，变更后触发 typed event。
- FrameworkDesign README 复述 QFramework 四层规则，示例 `PointGame` 用 `Architecture.Init` 注册 system/model/utility，用 `GameModel` 保存 `BindableProperty<int>`，用 `KillEnemyCommand` 修改 model 并发送 event。
- Context7 QFramework 返回 `/liangxiegame/qframework`，说明其核心 API 包含 `IArchitecture`、`Architecture<T>`、`IModel`、`ICommand`、`IQuery`、`TypeEventSystem` 和 `BindableProperty`。
- Unity Entities 文档和 Context7 示例显示 runtime component 使用 `IComponentData` struct；authoring / baking 把编辑器数据转换成 runtime component。
- Unity Entities Context7 文档显示 authoring component 是编辑器/创作侧输入，Baker 在 baking 阶段把它转换成 ECS runtime component；ECS runtime 不使用 authoring component。
- Bevy Context7 文档显示 system 可以通过 `run_if` 条件性执行，app 可以用 `add_plugins(DefaultPlugins)`、diagnostics plugin 和 schedule 系统组合功能。
- Bevy ECS 文档显示 `StorageType` 包含 `Table` 和 `SparseSet`；Table 适合缓存友好迭代，SparseSet 适合频繁增删。
- Flecs Context7 文档显示 system 可从 pipeline registration 中移除，也可以 enable/disable；disabled system 会从 pipeline 执行中排除。
- .NET CA1854 文档指出 `ContainsKey` 后再 indexer 是两次 lookup，推荐 `TryGetValue` 一次 lookup。这证明字典很实用，但 hot path 不能忽视 lookup 成本。
- Friflo 报告显示纯托管 C# ECS 可以有高性能，`CommandBuffer.Playback` 有阶段顺序，并且遍历中结构变更会 hard fail。

## Inference

- QFramework 避免 SlimeAI 当前 Data 类型复杂度，是因为它把状态定义留在 C# 强类型字段，而不是因为它有更强的动态数据系统。
- QFramework 的字典使用证明“字典不是脏结构”；问题在于字典是否承担业务字段类型系统。
- Unity Entities / Bevy / Friflo 证明成熟 runtime 倾向于在 authoring / registration / catalog build 阶段固定类型，运行时不应反复 `object -> T`。
- SlimeAI 可以继续保留 DataOS，但 Data runtime 必须瘦身；DataOS 不应把所有 authoring / presentation / AI routing 字段投影进 runtime `DataDefinition`。
- SlimeAI 可以用 runtime id array 改善 hot lookup，但应排在类型契约收口之后。
- 成熟框架共同支持用户对 System 解耦的拆分：启动前功能组合和运行中系统启停都是合理目标，但运行中结构变化必须有 schedule phase、command buffer、guard 和 diagnostics。

## Unknown

- QFramework Toolkits 是否有更复杂的数据配置方案，本轮未全量展开。
- SlimeAI 当前 Data dictionary lookup 是否已经是真实性能瓶颈，本轮没有 profiler 证据。
- `runtimeId` 如何跨 snapshot / profile 保持稳定，需要后续 Data RuntimeId SDD 设计。
- 如果未来游戏规模扩大，是否需要完整 ECS storage 分支，当前只能作为条件性分支。
- SlimeAI 完整 Capability/Profile manifest schema 尚未冻结；当前只确认方向和边界。

## Adopt Now

| 项 | 决策 | SlimeAI 落点 |
| --- | --- | --- |
| QFramework 少规则分层 | Adopt Now | Data 进入条件、Capability owner 边界、Event 边界 |
| QFramework Command/Query/Event 思路 | Adopt Now | 用于文档表达读写分离，不引入对象层 |
| BindableProperty 初值通知语义 | Adopt Now | UI / Debug / bridge 订阅 Data 时支持当前值快照 |
| QFramework 字典边界 | Adopt Now | 字典做 registry/index，payload 回到 typed contract |
| Unity authoring -> runtime | Adopt Now | DataOS validator/generator 是 SlimeAI Baker |
| Bevy storage lane 思路 | Adopt Now | Data storage 可分 stableKey index / runtimeId array / later numeric lane |
| Friflo CommandBuffer guard | Adopt Now | RuntimeCommandBuffer / EntityManager tick mutation guard 后续 RFC |
| Friflo SystemPerf | Adopt Now | Observation 应输出 ms / alloc / entity count / executions |
| Bevy run condition / Flecs system enable-disable | Adopt Now | System 启停必须是调度层事实，并输出 blocked reason / lifecycle trace |
| 启动前 plugin/profile 组合 | Adopt Now | Capability/Profile manifest、System preset、DataOS snapshot trim 后续作为 runtime 解耦主线 |

## Reject

| 项 | 决策 | 理由 |
| --- | --- | --- |
| 直接采用 QFramework 替换 SlimeAI | Reject | QFramework 是应用层架构，不是 SlimeAI DataOS / ECS runtime |
| 引入 `Architecture<T>` 静态单例 | Reject | 破坏测试隔离和 Godot lifecycle 边界 |
| Godot Node 实现 `IController` 承载玩法 | Reject | SlimeAI Node 是 bridge/view，不是 gameplay controller |
| QFramework `ICommand` 对象层 | Reject | 与现有 Capability service / runtime handler 重叠，增加 GC 和 AI 路由复杂度 |
| `TypeEventSystem.Global` 替代 EventBus | Reject | 不表达 entity/world/broadcast scope |
| `BindableProperty<T>` 替代 Entity.Data | Reject | 缺 DataOS、snapshot、modifier、computed、generated key |
| 直接引入 Friflo / Arch / DefaultEcs | Reject | 会替换 runtime，而不是增强 AI-first 协议 |
| 立刻改完整 archetype/chunk ECS | Reject | 范围过大，缺 profiler 和 GodotBridge 适配证据 |

## 后续 SDD 建议

1. `Data Runtime Simplification Hard Cutover`  
   从 `DataDefinition` / descriptor / presentation / write policy 边界开始，不改 storage 性能结构。
2. `Data Type Contract Hard Cutover`  
   建立 `DataTypeContract` / `DataValueCodec`，并把 SDD-0044 的 `DataComputeRegistry` 单例方向并入或作为前置子任务。
3. `Generated RuntimeId Storage`  
   让 generated `DataKey<T>` 携带 runtime id，storage 优先数组索引。
4. `RuntimeCommandBuffer RFC`  
   参考 Friflo / Arch / DefaultEcs，但只覆盖 SlimeAI 结构变更边界。
5. `Observation SystemPerf Contract`  
   把系统执行时间、alloc、entity count 和 skip/block reason 变成可查询 artifact。
6. `Capability/Profile Manifest`  
   把多游戏功能组合写成启动前 profile、系统 preset、DataOS trim、资源 catalog 和验证矩阵。
