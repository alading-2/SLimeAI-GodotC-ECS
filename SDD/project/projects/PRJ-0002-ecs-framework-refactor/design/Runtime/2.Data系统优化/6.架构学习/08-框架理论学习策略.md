# 框架理论学习策略

> 原始问题：见 [`source-request.md`](./source-request.md) 的“2026-06-15 是否需要查教程学习框架理论”。  
> 本文目标：回答“现在是否应该网上查教程学习框架搭建理论，以及应该查什么、不查什么”。  
> 当前上下文：SlimeAI 已把第一目标校准为 runtime 功能解耦；DataOS 表格驱动和 AI-first 都是上层能力，底层 Runtime 必须简单、直接、可验证。

## 结论

需要补框架理论，但不能把“网上查教程”当主线。

当前正确策略是：

```text
理论学习：20%
源码学习：50%
回到 SlimeAI 对照设计：30%
```

理由很简单：SlimeAI 当前不是缺更多概念，而是缺少数能约束底层 Runtime 的原则。

泛搜“如何搭建游戏框架”“ECS 框架教程”“架构设计教程”风险很大，因为很多教程会把 MVC、MVVM、事件总线、Service、Manager、DI、Module、Plugin 等概念堆在一起。它们可能让框架看起来更完整，但会继续增加底层复杂度。

现在应该只学能直接回答 SlimeAI Runtime 问题的理论。

## 应该学什么

### Game Programming Patterns

优先级最高。

来源：<https://gameprogrammingpatterns.com/contents.html>

它适合当前阶段，因为它不抽象地讲“架构高大上”，而是直接解释游戏里常见机制为什么存在、什么时候会失控。

建议只读这些章节：

- `Component`：理解为什么组合比继承更适合游戏功能拆分。
- `Command`：理解为什么写操作需要明确入口。
- `Event Queue`：理解事件为什么能解耦，也为什么可能造成隐式流程。
- `Game Loop`：理解 Runtime 更新节奏、阶段和副作用边界。
- `Service Locator`：理解全局服务为什么方便，以及为什么需要控制使用边界。

SlimeAI 落点：

```text
Component 解耦不是随便挂删组件。
Command / request 是状态改变入口。
Event 只负责通知，不承包所有依赖。
Game Loop / SchedulePhase 决定结构变更何时生效。
全局服务必须少，并且有 owner 和验证。
```

### .NET Framework Design Guidelines

优先级高。

来源：<https://learn.microsoft.com/en-us/dotnet/standard/design-guidelines/>

SlimeAI 是 C# 框架，底层 API 是否好用、能不能被误用、命名是否清楚，比“看起来高级”重要。

建议只学这些问题：

- public API 怎么命名。
- 什么时候用 interface，什么时候用 abstract class，什么时候只用 concrete type。
- exception / validation / guard 应该放在哪里。
- collection / dictionary / callback API 如何避免误用。
- 泛型 API 怎么保持可读，不做过度泛型炫技。

SlimeAI 落点：

```text
Runtime API 要少。
错误要早暴露。
DataKey<T> / RuntimeCommandBuffer / SystemManager 这类 API 必须容易理解。
不要为了“可扩展”暴露万能 object / string / query DSL。
```

### Unity Entities authoring / baking

优先级中高，但只学概念。

来源：<https://docs.unity3d.com/Packages/com.unity.entities%401.4/manual/ecs-workflow-example-authoring-baking.html>

它最有价值的不是 ECS storage，而是 `authoring -> baking -> runtime` 的边界。

SlimeAI 落点：

```text
DataOS authoring DB
  -> validator
  -> generator
  -> runtime snapshot / generated handle / manifest
  -> Runtime Data / System / Capability
```

这能帮助我们继续坚持：

- Runtime 不查 authoring DB。
- authoring 展示字段不进入 Runtime hot path。
- runtime snapshot 是运行时输入，不是编辑器表格本身。

### Bevy schedule / plugin / run condition

优先级中高，但只学概念。

来源：Bevy ECS / Bevy 文档，建议从本地研究报告和官方文档入口开始：

- 本地报告：`Resources/Engine/Docs/FrameworkAnalysis/Reports/bevy/01-Bevy-源码分析报告.md`
- 官方入口：<https://bevyengine.org/learn/>、<https://docs.rs/bevy_ecs/latest/bevy_ecs/>

Bevy 的源码是 Rust，不适合作为 C# 主读材料；但它的 plugin、schedule、commands、run condition 很适合 SlimeAI。

SlimeAI 落点：

```text
Capability/Profile manifest = plugin/profile 组合边界。
System enable/disable = 调度层事实。
RuntimeCommandBuffer = 延迟结构变更。
SchedulePhase = 命令播放和观察边界。
```

### 企业架构模式只选择性看

Martin Fowler、DDD、Clean Architecture、CQRS 都可以看，但当前不作为主线。

来源：<https://martinfowler.com/eaaCatalog/>

原因是这些理论偏应用层和业务系统，对 SlimeAI 当前底层 Runtime 的帮助有限。它们容易让框架继续增加 Service、Repository、UseCase、Module 等概念。

只允许选择性吸收：

- Layering：分层边界。
- Service Layer：写操作入口。
- Repository：读取/存储边界，但不要照搬到 Runtime hot path。
- CQRS：读写分离思想，而不是引入完整对象层。

## 不应该优先学什么

| 类型 | 暂缓原因 |
| --- | --- |
| “从零搭建 Unity 游戏框架”教程 | 多数偏 MonoBehaviour/MVC/Manager 堆叠，容易污染 SlimeAI Runtime 目标。 |
| “万能 ECS 框架教程” | 容易把 query/world/archetype 当终点，忽略 Godot C#、DataOS 和多游戏组合目标。 |
| “DDD/Clean Architecture 完整课程” | 适合应用层，不适合现在的底层 Runtime 简化。 |
| “大型商业框架设计”泛读 | 概念太多，会诱导继续加 Feature/Module/Manager。 |
| “AI-first 框架教程” | 当前 AI-first 是上层文档和验证问题，不是底层第一目标。 |

判断一个教程是否值得读，只看它能否回答下面六个问题：

```text
Component 怎么解耦？
System 怎么启停？
结构变更怎么延迟？
共享状态什么时候进 Data？
authoring 和 runtime 怎么分开？
API 怎么少而清楚？
```

不能回答这些问题的教程，当前阶段先不读。

## 推荐学习顺序

当前最稳的顺序是：

```text
1. QFramework
   学少规则、分层、Command/Query/Event、共享数据边界。

2. Game Programming Patterns
   只读 Component / Command / Event Queue / Game Loop / Service Locator。

3. Friflo.Engine.ECS
   学 Runtime 边界、CommandBuffer、System 组织、SystemPerf。

4. Arch / DefaultEcs
   学 storage/query 和易用 API。

5. Unity Entities / Bevy 文档
   学 authoring/runtime 分层、plugin/profile、schedule/run condition。

6. 回到 SlimeAI
   先做 Data Runtime Simplification，再做 Type Contract 和 RuntimeCommandBuffer。
```

这条路线避免两个错误：

- 只看理论，越学概念越多。
- 只看源码，看懂实现却不知道为什么这么做。

## 学习产出要求

每读一个理论或框架，只产出一页，不写长篇读书笔记。

固定回答：

```text
1. 它解决什么问题？
2. 它靠哪个最小机制解决？
3. 它把复杂度放在哪里？
4. 它的规则是否少而清楚？
5. SlimeAI Adopt Now / Adopt Later / Reject 是什么？
6. 它是否支持 Runtime 功能可组合、可裁剪、可启停？
```

如果读完不能转成 SlimeAI 的 `Adopt / Reject`，说明这次学习没有形成设计价值。

## 对 SlimeAI 的当前裁决

Adopt Now：

- 补少量框架理论，但只补 Runtime 解耦相关理论。
- `Game Programming Patterns` 作为理论第一入口。
- `.NET Framework Design Guidelines` 作为 C# API 设计入口。
- Unity Entities / Bevy 只学习 authoring/runtime、plugin/schedule/run condition。
- 每次学习必须回写 SlimeAI 的规则表、RFC 或 SDD，不堆读书笔记。

Adopt Later：

- DDD / Clean Architecture / Fowler 企业架构模式。
- Unreal GAS / ET-Framework 这类上层能力体系。
- AI-first 框架资料。

Reject：

- 泛搜教程后按教程重搭框架。
- 为了“理论完整”增加 Runtime 概念。
- 把能靠文档、validator、test、code review 控制的规则塞进 Runtime `Set`。
- 在底层未稳定前继续扩表格驱动和 Feature 上层体验。

一句话：

```text
现在要学理论，但只学能让 SlimeAI Runtime 更简单、更解耦、更可验证的理论。
```
