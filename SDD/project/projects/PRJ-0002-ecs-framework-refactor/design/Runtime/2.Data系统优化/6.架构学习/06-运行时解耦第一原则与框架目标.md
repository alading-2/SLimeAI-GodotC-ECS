# 运行时解耦第一原则与框架目标

> 原始问题：见 [`source-request.md`](./source-request.md) 的“2026-06-15 运行时解耦补充问题”。  
> 本文目标：把 SlimeAI 的第一目标从“AI-first / Data 表格驱动 / 某个成熟框架替代”校准为“底层运行时功能可组合、可裁剪、可启停”。

## 结论

用户判断成立，而且这是比 Data 类型系统更上层的框架目标问题。

SlimeAI 首要目标应该写成：

```text
在同一个 Godot C# 框架内，把大量可复用游戏功能做成可组合能力。
每个游戏通过 profile / submodule 版本 / DataOS preset / manifest 选择需要的能力。
未使用的功能可以在启动前不加载；已加载的系统可以在运行中受控启停。
```

因此，Data、Event、Component、System、DocsAI、Skill、SDD 都不是目的本身。它们服务同一个底层目标：

```text
运行时功能解耦。
```

更准确的框架定位是：

```text
Composable Godot C# Runtime
  + ECS / Component pattern 的成熟心智模型
  + Capability / Profile 的功能组合边界
  + Data / Event / System 的强类型运行时协议
  + AI-first 文档、skill、验证和观察层
```

AI-first 是工程使用层，不是底层架构的第一原因。没有 AI，这套框架也应该能作为多游戏可组合 runtime 使用；有 AI 后，DocsAI / skill / rules / SDD / validation 只是让 AI 更稳定地使用它。

## 是不是没人做过

不是完全没人做，也不是因为这个方向不可能。

成熟框架已经分别证明了这些机制：

| 机制 | 成熟来源 | 说明 |
| --- | --- | --- |
| Component + System 解耦 | ECS / Component pattern | 用组合替代继承，把数据、桥接、行为分到不同 owner。 |
| 启动前功能组合 | Bevy plugin / plugin group、Unreal GameFeature、Unity package / baking | 项目启动前决定启用哪些模块、数据、系统和资源。 |
| 运行中系统启停 | Flecs system enable/disable、Bevy run condition、SlimeAI SystemManager | 系统是否执行由调度条件和启停状态控制，不靠散落的 `if`。 |
| authoring/runtime 分层 | Unity Entities Baker、SlimeAI DataOS | 编辑友好的表/场景不等于运行时结构。 |
| 延迟结构变更 | Bevy Commands、Arch / DefaultEcs / Friflo command buffer | 运行中增删实体、组件、关系要有安全播放阶段。 |
| 观察与诊断 | Unity Entities debugger、Friflo SystemPerf、SlimeAI Log/Validation | 系统是否启用、阻断、耗时、分配和失败原因应可查。 |

SlimeAI 比较少见的地方不是某个单点机制，而是组合方式：

```text
Godot C# runtime
  + 多游戏功能库
  + Capability/Profile 组合
  + DataOS authoring/generator/validator
  + DocsAI/skill/SDD/observation 给 AI 使用
```

所以结论不是“世界上没人做，所以只能闭门造”。更正确的是：

```text
单点机制都能学习成熟框架。
完整形态需要 SlimeAI 自己整合。
```

这也是为什么不建议直接换成 QFramework、Unity Entities、Bevy、Flecs、Friflo、Arch 或 DefaultEcs：它们能证明机制，却不能直接满足 SlimeAI 的 Godot C#、DataOS、AI 文档约束和多游戏功能库目标。

## Component 解耦必须保留

Component 解耦是底层目标的一部分，应保留，但要重新限定含义。

SlimeAI 的 Component 不应该泛化成“所有逻辑都挂一个组件”。当前更稳的定义是：

```text
Component = GodotBridge Component
```

它负责把 Entity、Data、Event、Capability service 和 Godot Node 生命周期连接起来。具体功能仍归 Capability owner。

应该保留的部分：

- Entity 可以通过 C# composition profile 组合不同 Component。
- Component 默认组合不依赖 Inspector / `.tscn` Preset 作为事实源。
- Component 参数用 typed options 注入，注册前完成配置。
- Component 间通信优先 `Entity.Events`、Capability service 或 Runtime command，不直接互调具体组件。
- Component 不使用私有字段承载跨功能共享业务状态。

不应该保留的误用：

- 用 Component 私有字段隐藏多个系统都需要的状态。
- 用 `[Export]` / Inspector override 当框架默认配置。
- 用 `object` / `Dictionary` 传 Component options。
- 任意代码在运行中直接增删 Component 并期望所有系统自动安全。

因此，Component 解耦不是“任何组件随便挂随便删”。正确版本是：

```text
启动前组合由 profile/composer 决定。
运行中结构变化走 RuntimeCommandBuffer 或明确 owner API。
变化后有 validator / diagnostics / observation。
```

## System 解耦必须分两层

用户把 System 解耦拆成“游戏开始前是否使用某个系统”和“运行中打开/关闭某个系统”，这个拆分是对的。

### 启动前组合

这是默认主线，也最稳定。

目标形态：

```text
Game / GenreProfile
  -> enabled capabilities
  -> enabled system ids / tags
  -> DataOS preset / snapshot trim
  -> resource catalog trim
  -> validation scene matrix
```

启动前组合应该回答：

- 这个游戏用了哪些 Capability？
- 每个 Capability 依赖谁？
- 需要加载哪些 System？
- 需要哪些 DataKey / descriptor / record？
- 需要哪些 GodotBridge Component？
- 哪些验证场景必须通过？

当前 SlimeAI 已有部分基础：

- `SystemManager` / `SystemRegistry` 是 System Core 入口。
- `system.config` 和 `system.preset` 已表达 `Required`、`AutoLoad`、`StartEnabled`、`Tags`、`Dependencies`。
- `SystemPreflight` / `SystemDiagnosticsSnapshot` 已经是三源对齐和 diagnostics 的基础。
- `ComponentCompositionProfile` / `ComponentComposer` 已经把默认组件组合转为 C# profile。

缺的是项目级统一目标：这些机制必须服务“多游戏功能拼装”，不能只作为局部优化存在。

### 运行中启停

运行中启停也应该保留，但必须受控。

成熟框架共同证明：运行中系统启停不是问题，问题是结构变化和副作用必须有边界。Flecs 可以 enable/disable system，Bevy 有 `run_if`，SlimeAI 也已有 `SystemManager.EnableSystem/DisableSystem`、`CanExecute`、`BlockedReasonCode`、`SystemLifecycleTrace`。

SlimeAI 应把运行中启停定义为：

```text
系统执行资格 = loaded && enabled && stateAllowed && dependencyReady
```

运行中禁用一个系统时，必须明确：

- `_Process` / `_PhysicsProcess` 停止。
- `OnStopped` 释放或暂停事件订阅、Timer、外部命令入口。
- `SystemManager.Execute` 返回稳定阻断码。
- Diagnostics 能说明是 Disabled、NotLoaded、NotRunning、DependencyMissing 还是 StateMismatch。
- 必要时发出 lifecycle trace。

运行中启停不等于任意时刻直接删除功能。删除实体、增删组件、替换关系、切换 GodotBridge 节点这类结构变化应进入 RuntimeCommandBuffer，按 schedule phase playback。

## Data 的位置要降级

当前 Data 最大问题不是“集中在一起不方便”。相反，用户说得对：集中 Data 很方便，填表就能改功能，这是有价值的。

问题是顺序错了。

```text
底层 runtime 还没简单稳定
  -> 先做了表格驱动和万能 Data
  -> Data runtime 背上 authoring / presentation / policy / debug / AI routing / type conversion
  -> 核心比业务还复杂
```

表格驱动应该是上层能力，不是底层第一原则。它应建立在清楚的 runtime contract 之后：

```text
先有简单直接的 Entity / Component / System / Event / Data 边界
再决定哪些共享 runtime state 进入 Data
再用 DataOS 表格、generator、validator 提高编辑和 AI 追溯效率
最后才做“填表组合功能”的上层体验
```

因此，Data 后续的优先级应保持：

```text
Data Runtime Simplification
  -> Data Type Contract
  -> Generated RuntimeId Storage
  -> Capability/Profile manifest
  -> 表格驱动组合体验
```

不要反过来。

## 上层 Feature 复杂度后续记录

Feature、Ability、Damage、AI 等上层能力以后仍然要做，但底层未稳定前不要继续扩大概念。

后续记录：

- Feature 需要重新审查是否过度抽象。
- Ability / Feature / Damage 的写操作应统一成更简单的 request / command / service 入口。
- Feature action / handler 应减少 object、动态分派和隐式副作用。
- 每个 Capability 应有最小 Contract、DataKeys、Events、System、Component、Tests、Observation，而不是把所有扩展都塞进 Feature。
- 复杂功能应由简单机制组合出来，不通过一个大而全 Feature runtime 直接承包。

这不是本轮实施内容，但必须写入后续路线，避免继续在上层复杂度上加码。

## 推荐的新第一原则

后续所有 PRJ-0002 设计和 SDD 都应先问：

```text
这个改动是否让功能更容易被组合、裁剪、启停和验证？
```

如果答案是否定的，即使它看起来更 AI-first、更表格驱动、更通用，也不应该优先做。

判断表：

| 问题 | 正确倾向 |
| --- | --- |
| 这个状态是否多个 Capability 共享？ | 是才进 Data，否则留 owner 内部。 |
| 这个功能是否可作为独立 Capability 被启用/禁用？ | 是才进入 Capability/Profile manifest。 |
| 运行中启停是否会影响事件、Timer、实体生命周期？ | 是则需要 lifecycle / command buffer / diagnostics。 |
| 这个表格是否只是 authoring 方便？ | 只保留在 DataOS / generator，不投影进 runtime 热路径。 |
| 这个 API 是否让 AI 能绕过 owner？ | 拒绝，例如 global query DSL、裸字符串 event、任意 component surgery。 |

## 采纳与拒绝

Adopt Now：

- 保留 Component + System 解耦。
- 把启动前 profile / capability / system 组合写成框架主线。
- 把运行中 system enable/disable 写成受控 runtime 能力。
- 把 RuntimeCommandBuffer 作为后续结构变化必备 RFC。
- 把 diagnostics / blocked reason / lifecycle trace / SystemPerf 作为解耦可信度的一部分。
- Data 只作为跨功能共享 typed runtime state protocol。

Adopt Later：

- 表格驱动功能拼装体验。
- Capability activation actions。
- 更完整的 profile trim：DataKey、resource、system、component、validation matrix。
- 上层 Feature runtime 简化。

Reject：

- 直接替换为成熟框架。
- 把 AI-first 当成底层架构第一原则。
- 把所有状态和配置塞进 Data。
- 在任意 tick 任意增删 component/system。
- 暴露 global world query DSL。
- 用 `object`、裸字符串和隐式全局副作用伪装成解耦。

## 需要确认

本文不阻塞文档更新；后续进入执行型 SDD 前需要确认：

1. 运行中启停第一阶段是否只覆盖 System `Enable/Disable`，暂不做完整 Capability activation/deactivation？
   - 默认值：先做 System 级启停和 diagnostics，Capability 运行中启停留到 RuntimeCommandBuffer 后。
2. 启动前组合是否先以 `GenreProfile / Capability manifest / System preset` 文档和 validator 表达，不急着做 UI 或表格编辑器？
   - 默认值：先做 manifest + validator，不做编辑器。
3. 表格驱动是否明确排在底层 runtime 简化之后？
   - 默认值：是。DataOS 继续保留，但先瘦身 runtime，再谈上层填表体验。
