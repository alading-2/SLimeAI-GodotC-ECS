# 问题判断与总体裁决

> 原始问题：见 [`source-request.md`](./source-request.md)。  
> 本文目标：回答“我是否应该直接用成熟框架，先学习别人怎么做，后面再自己做”。

## 真实问题

用户的直觉是对的：现在的难点不只是 Data 某个类写复杂了，而是框架能力本身进入了高复杂区。

SlimeAI 当前 Data 已经经历了多轮重构：

```text
DataMeta / DataRegistry
  -> DataOS descriptor-first
  -> runtime snapshot
  -> generated DataKey<T>
  -> DataSlot<T> + IDataSlot
  -> typed computed / modifier
  -> 2026-06-15 根本裁决：功能解耦优先，数据形态统一不是目标
```

这条历史说明：真正的问题不是“AI 没写好代码”，而是缺少成熟框架经验时，容易把一个局部目标上升为全局架构原则。例如把“功能解耦”误实现成“所有数据都进入统一动态 Data 管线”。这会导致 runtime 承担 authoring、presentation、policy、debug、AI routing、type conversion 和 storage 的过多职责。

所以问题真实存在，严重程度也值得处理。后续如果不先建立框架学习路线，继续做 Data / Event / System / Capability 时仍会反复出现同类问题。

2026-06-15 用户补充问题进一步校准了真实目标：SlimeAI 要做的不是“AI 框架”这个新词，而是一个底层可组合 runtime。AI 只是使用这套 runtime 的工程层。真正要先固定的是：

```text
Component 组合
System 启动前裁剪
System 运行中受控启停
Capability / Profile 依赖和验证
结构变化的安全阶段
```

Data、Event、DocsAI、Skill 都应该服务这个目标。Data 表格驱动很有价值，但它是上层便利性，不应继续压倒底层 runtime 简单性。

## 是否直接用成熟框架

不建议直接替换为某个成熟框架。

成熟框架解决的问题不同：

| 框架 | 擅长 | 与 SlimeAI 的冲突 |
| --- | --- | --- |
| QFramework | Unity/Godot 应用层架构、MVC/CQRS、Command/Query/Event、UI 数据绑定 | 不是底层 ECS runtime，不提供 SlimeAI 的 per-entity DataOS / snapshot / modifier / computed 主链路 |
| Unity Entities | 高性能 data-oriented ECS、authoring baking、chunk storage | 绑定 Unity DOTS 生态；复制它等于重建 Entity/Query/Schedule/GodotBridge |
| Bevy ECS | plugin / schedule / table/sparse-set storage / typed system | Rust 生态和宏系统不同；适合学模式，不适合直接移植 API |
| Arch / Friflo / DefaultEcs | C# ECS storage、query、command buffer、perf | 引入后会替换 SlimeAI runtime，而不是增强 AI-first DataOS |
| Unreal GAS | Ability / Effect / Tag / Cue 的大型能力框架 | 概念重，和当前 Godot C# 小 runtime 目标不匹配 |

SlimeAI 的目标更特殊：它不是只给人写游戏，也要让 AI 能稳定路由、验证和复盘。因此直接采用成熟框架反而会把 AI-first 的事实源、owner skill、DataOS、Godot 场景验证、Observation 重新打散。

## 是否应该学习成熟框架

应该，而且要系统学。

推荐把学习分成两条线：

```text
架构纪律线：QFramework / FrameworkDesign
运行时机制线：Unity Entities / Bevy ECS / Friflo / Arch / DefaultEcs / Flecs / EnTT
```

QFramework 的价值不是“它的数据结构替代 SlimeAI Data”，而是它把复杂框架压成几条容易遵守的规则：

- ViewController 接收输入和表现。
- Model 保存共享数据。
- System 处理跨表现层业务逻辑。
- Command 处理写操作。
- Query 处理读操作。
- Event / BindableProperty 负责反向通知。
- Architecture.Init 注册模块，本身就是架构图。

传统 ECS 的价值也不是“SlimeAI 立刻改 chunk storage”，而是它反复证明：

- 类型应在 component / generated contract / catalog build 阶段固定。
- 动态 registry 可以存在，但 payload 不应长期是 `object`。
- authoring 和 runtime 必须分层。
- 结构变更要有 deferred command / playback 阶段。
- perf / diagnostics 应是系统内建能力，不是出事后补 log。
- 启动前模块组合和运行中系统启停都需要显式 manifest / schedule / run condition / blocked reason，不能靠散落的 `if` 或全局副作用。

## 推荐路线

当前推荐路线是“学习后继续自研”，不是“闭门自研”，也不是“直接换框架”。

```text
第一步：QFramework / FrameworkDesign
  目标：建立框架分层、Command/Query/Event、数据放置条件、纸上设计习惯。

第二步：Unity Entities / Bevy ECS
  目标：理解 authoring -> runtime、typed component、storage lane、schedule。

第三步：Friflo / Arch / DefaultEcs
  目标：理解 C# ECS 如何做纯托管性能、query、command buffer、perf observation。

第四步：回到 SlimeAI
  目标：只把机制落到 Runtime 解耦 / DataOS / Data runtime / Capability owner / Observation，不复制 public API。
```

## 默认假设

- 当前 SlimeAI 仍以 Godot C# 为主，不切 Unity。
- 当前 SlimeAI 不引入第三方 ECS runtime 依赖。
- 当前 Data 后续继续 hard cutover，不保长期兼容 adapter。
- QFramework 作为学习对象，不作为代码依赖。
- FrameworkDesign 作为 QFramework 架构课程实践，不作为正式框架源。
- 底层 runtime 先固定 Component/System 解耦、Capability/Profile 组合和 Data 进入条件；表格驱动体验排在后续。

## 必须确认

当前文档阶段没有必须阻塞的问题；默认可以按本文路线继续。后续进入实现型 SDD 前，需要确认：

- 是否先做 `Data Runtime Simplification Hard Cutover`，而不是继续孤立执行 SDD-0044。
- 是否把 `RuntimeCommandBuffer RFC` 和 `Capability/Profile Manifest` 作为 Data runtime 简化之后的 P1，而不是继续把表格驱动塞进 Data runtime。
- 是否为 SlimeAI 增加一个极小“Counter/Combat Slice”学习样例，专门展示 Entity/Data/Event/Capability 的最小闭环。
- 是否允许把成熟框架学习变成固定流程：每个大型 Runtime 改动前必须先写“Adopt / Reject / Unknown”清单。
