### P3：实际效率问题的根因未解决

从 6/9 会话数据看，真正的效率瓶颈不是 git diff/status（只占 5-10%），而是：

| 问题 | 实证 | 根因 |
|---|---|---|
| 验证循环 | 17:45 会话有 20 个循环，`sdd.py validate` 连续跑 7 次 | AI 每改一个文件就跑完整验证，缺乏"批量修改后统一验证"意识 |
| 文件读放大 | Log.cs 被读 22 次，godot-scene-runner.mjs 被读 22 次 | DeepThink/Retrospective/实现之间切换时重新读取事实源 |
| Skill 重复加载 | 同一 SKILL.md 读 3 次 | 每次 context switch 重新加载 |

这些是 AI 行为问题，需要通过规则/skill 引导来解决，不是工具问题。

### P4：SDD 完成后的自动化流程不完整

当前规则说"完成 SDD task 时，更新 tasks.md / progress.md + commit + push 是默认动作"，但：

1. **验证步骤缺失**：commit/push 前是否应该跑 `dotnet build`、`sdd.py validate`？
2. **submodule 更新缺失**：框架仓 push 后，游戏仓的 submodule 指针是否应该自动更新？
3. **ChatHistory digest 缺失**：SDD 完成后是否应该自动生成 session digest？

### 第三批：效率引导规则（P3）

在 `.ai-config/rules/rules.md` 新增 Efficiency 段：

```markdown
## Efficiency

- 同一轮任务中编辑多个文件时，全部编辑完成后再统一验证（build + validate + git status）；不要每改一个文件就跑一次完整验证。
- DeepThink 或 Retrospective 切换时，如果 5 分钟内刚读过同一文件，引用路径即可，不要重新读取全文。
- 已读取的 skill 文件在同一会话中不需要重新加载。
```

### 第四批：SDD 自动化流程补全（P4）

在 `.ai-config/rules/rules.md` 的 SDD 完成流程中补充：

```markdown
- 完成 SDD task 时：
  1. 更新 tasks.md / progress.md
  2. 跑验证（dotnet build + sdd.py validate）
  3. git commit + push
  4. 如涉及框架仓改动，提醒用户更新游戏仓 submodule
```
