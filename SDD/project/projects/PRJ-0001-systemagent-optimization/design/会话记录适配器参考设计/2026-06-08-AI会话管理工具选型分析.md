# AI 会话管理工具选型分析

> 状态：reference-current
> 日期：2026-06-08
> 更新：2026-06-09
> 原位置：`design/优化/2026-06-08-AI会话管理工具选型分析.md`
> 当前定位：会话记录适配器参考设计资料；主裁决见 `../优化/2026-06-08-SystemAgent工作流内化与核心优化裁决.md`。

## Goal

本文件只回答一个较窄的问题：

```text
SlimeAI 做 ChatHistory / session-adapter 时，应该参考或调用哪些现成工具？
```

它不再承担 SystemAgent 总体路线设计。总体路线已经收敛为：

- 不做外层 AI CLI manager。
- 不魔改 Warp。
- 不魔改 `codbash`。
- SystemAgent 自己做薄层整理、命名、索引、摘要和 SDD 复盘字段。
- 外部工具只负责发现、读取、导出或提供设计参考。

## Core Requirements

第一阶段需求：

- Linux 可用。
- 至少支持 Claude Code、Codex、OpenCode 的本地会话发现或导出路径。
- 能按当前 repo、最近会话、指定 session id 增量处理。
- 能帮助解决 session id 难记、原始路径难找、会话难复盘的问题。
- 尽量本地、只读、低侵入。
- 不改变原始 session 文件名。
- 不复制完整原始 JSONL / SQLite / transcript 到仓库。
- 不采集或伪造私有 chain-of-thought。

非目标：

- 不做远程手机控制台。
- 不做多 agent 编排工作台。
- 不把会话记录强制绑定到 git commit。
- 不引入 Langfuse / Phoenix 这类应用级 observability 平台作为第一阶段主线。
- 不一次性处理全部历史会话。

## Evaluation Criteria

| Criterion | Weight | Why |
| --- | ---: | --- |
| Claude / Codex / OpenCode 覆盖 | 5 | 用户明确修正第一阶段必须跨这三类工具。 |
| Linux 实用性 | 5 | 当前用户环境是 Linux。 |
| 只读和低侵入 | 5 | 不能让会话工具接管 git、hook、workflow 或原始 session。 |
| 搜索 / 定位 / handoff | 4 | 直接解决“对话难找、ID 难记”。 |
| 高保真导出能力 | 4 | 复盘需要 tool output、失败、验证证据。 |
| 机器可读输出 | 3 | SlimeAI 需要统一 index 和 digest。 |
| 活跃度 / stars / 安装成熟度 | 2 | 是加分项，但不能压过需求匹配。 |

## Final Ranking

### 1. `codbash` — 跨工具发现入口

链接：<https://github.com/vakovalskii/codbash>

推荐程度：最高，作为第一阶段发现入口。

能解决：

- 跨 Claude Code / Codex / OpenCode 查找本地 sessions。
- 按项目路径列出当前仓最近会话。
- 搜索、查看、handoff、tag、preview、browser。
- OpenCode 数据源覆盖 `~/.local/share/opencode/opencode.db`。
- 对用户来说能快速缓解“session id 不可读、路径难找”的问题。

不能直接承担：

- 不能把 `handoff` 当完整复盘证据，因为 detail / handoff 有消息数量和内容截断。
- 不应启用 delete / launch / resume 自动化作为 SystemAgent 默认能力。
- 不应把 dashboard 当 SlimeAI 事实源。

SlimeAI 采纳方式：

- 作为 `session-adapter` 的本地发现入口。
- 只消费结构化 session 数据或轻量 detail。
- 生成 SlimeAI 自己的 `Workspace/DocsAI/ChatHistory/index.json` 和 Markdown sidecar。
- 不 fork，不改上游源码。

当前状态：

- `SDD-0039` 已采用本地 `Workspace/Resources/tool/codbash` 作为第一版发现入口。

### 2. `codlogs` — Codex 高保真补充

链接：<https://github.com/tobitege/codlogs>

推荐程度：高，但只用于 Codex 专项路径。

能解决：

- 只读扫描 `~/.codex/sessions`。
- 按 repo root 匹配 Codex sessions。
- `--json` 输出机器可读列表。
- Markdown / HTML 导出。
- `--include-tool-results` 包含 tool calls 和 tool outputs。
- 大 session streaming 和 bounded scan。
- sanitize 派生副本，不破坏 Codex resume。

不能直接承担：

- Codex-only，不能作为 Claude Code / OpenCode 总入口。
- 许可证口径曾出现根目录 MIT 与 `package.json` `UNLICENSED` 不一致，正式 vendor 前需复核。

SlimeAI 采纳方式：

- 后续作为 Codex 高保真导出 fallback。
- 特别用于补充 tool output、失败 tool call、token/error summary。
- 不作为第一阶段统一入口。

当前状态：

- `SDD-0039` 已在 README / sidecar 中保留 `codlogs --include-tool-results` 提示路径，但尚未自动化接入。

### 3. `tracebase` — 复盘维度参考

链接：<https://github.com/ssreeni1/tracebase>

推荐程度：参考，不作为第一阶段依赖。

能解决或启发：

- 本地 trace / run inspection。
- failure、resteer、context waste、repeated command、scorecard。
- redacted incident packet。
- raw events 本地存储、dashboard、MCP、hook/watch。

不适合第一阶段的原因：

- 概念和依赖重。
- 会引入新的 store、dashboard、MCP 和 watcher。
- 容易和 SDD / DocsAI / ChatHistory 形成第二套事实源。
- 本地 clone 缺依赖时无法零安装运行 help，说明不是最小可用入口。

SlimeAI 采纳方式：

- 只参考复盘维度：失败、验证、工具重要性、上下文浪费、可分享证据包。
- 不引入 runtime。
- 不接它的 hook/watch 体系。

### 4. `claude-replay` — 人工 HTML replay

链接：<https://github.com/es617/claude-replay>

推荐程度：可选。

能解决：

- 生成 HTML replay。
- 支持 Claude Code / Cursor / Codex / Gemini / OpenCode 等 transcript。
- 支持 secret redaction、tool call 展示、file activity sidebar。

不适合第一阶段主线的原因：

- 更偏人工查看和分享，不是 session index / digest 系统。
- HTML 不适合作 SystemAgent 默认机器事实源。

采纳方式：

- 对重要任务可选生成 replay artifact。
- 不作为 ChatHistory 默认入口。

### 5. `abtop` — 运行态监控

链接：<https://github.com/graykode/abtop>

推荐程度：可选。

能解决：

- 监控 Claude / Codex / OpenCode session 运行状态。
- 看 token、上下文、rate limit、进程和端口。
- `--json` 可输出运行态 snapshot。

不适合第一阶段主线的原因：

- 它不是 transcript 管理和复盘工具。
- 不能替代 search / export / digest。

采纳方式：

- 多开会话时可作为人工状态面板。
- 不接入 SystemAgent 默认流程。

## Explicit Rejections

| Project / Direction | Decision | Reason |
| --- | --- | --- |
| Warp / 自研 terminal workbench | Reject | 已过时；不解决项目事实源和复盘质量。 |
| 外层 AI CLI manager | Reject | 会形成新的控制面和事实源冲突。 |
| `entireio/cli` | Reject first phase | stars 高，但会接 git hooks / checkpoint branch，侵入性高。 |
| `memento` | Reject first phase | 把 transcript 绑到 git notes，不符合当前 SDD-first 记忆模型。 |
| `agent-sessions` | Reject Linux first phase | 功能好但偏 macOS native。 |
| `polpo` | Reject first phase | 移动端远程控制范围过大。 |
| `llm-wiki` | Adopt Later / Reference | 长期知识库方向强，但会和 DocsAI / SDD 重叠。 |
| Langfuse / Phoenix | Reference only | 适合自建 LLM app trace，不适合本地 AI CLI transcript 管理第一阶段。 |

## Current Adopted Stack

```text
跨工具发现入口：codbash
SlimeAI 整理层：Workspace/SystemAgent/Tools/session-adapter/
ChatHistory 输出：Workspace/DocsAI/ChatHistory/
Codex 高保真补充：codlogs（后续）
复盘维度参考：tracebase
人工 replay：claude-replay（可选）
运行态监控：abtop（可选）
```

## Decision After SDD-0039

`SDD-0039 Cross-agent Session Adapter` 已经完成第一层能力，因此本文不再作为待实现选型报告，而是作为参考资料。

已落地：

- `session-adapter list --repo . --limit N`
- `session-adapter index --session <session-id>`
- `session-adapter summarize --session <session-id>`
- ChatHistory sidecar。
- `Workspace/DocsAI/ChatHistory/index.json`。
- Codex visible transcript 月度导出。

仍待后续：

- ChatHistory AI-first digest。
- Codex `codlogs --include-tool-results` 自动化补充。
- Claude Code / OpenCode 高保真导出路径。
- 只读资料 subagent pilot。
- retrospective 可选引用 ChatHistory locator。

## Defaults

后续继续默认：

- 不改上游源码。
- 不改原始 session 文件名。
- 不全量导入历史。
- 不接 hook 自动抓取。
- 不复制完整原始 transcript。
- 不把工具 dashboard 当 SystemAgent 事实源。
- 只把工具输出转成 SlimeAI 自己的 index、digest、evidence 和 SDD progress 引用。
