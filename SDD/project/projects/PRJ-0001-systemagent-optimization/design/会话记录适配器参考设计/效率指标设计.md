# 会话效率指标设计

> 状态：draft
> 日期：2026-06-09
> 任务来源：分析 6/8-6/9 共 11 个 Codex 会话 digest 后，发现 git diff/status 重复执行过多、同文件重复读取、skill 文件重复加载等效率问题。需要在 session_adapter 中新增效率指标，并更新 retrospective actor/skill 使 AI 复盘时能看到效率瓶颈。
> 设计裁决：在现有 digest 层新增 `derived/efficiency.md`；index.json 加效率摘要字段；VALIDATION_RE 扩展覆盖 git status；Retrospective actor 和 skill 加"会话效率"分析维度。

## Goal

本设计要解决的问题：

- session_adapter 当前不识别"验证循环"模式：每次 apply_patch 后的 `git diff --check` + `git status --short` + `sdd.py validate` 被计为独立 validation signal，无法看出"这次 edit 触发了 3 次验证"。
- session_adapter 不跟踪同文件重复读取：AI 在 DeepThink / Retrospective / 实现之间切换时反复读取同一文件，当前 tools.md 只列调用不计重复。
- retrospective 复盘时看不到效率瓶颈：actor 只检查流程缺口和 verdict 一致性，不检查"这次会话是否在重复劳动"。
- 无法从 index.json 快速筛选"低效会话"：没有效率摘要字段，list-digests 无法按效率过滤。

非目标：

- 不修改 AI 的行为规则（CLAUDE.md / AGENTS.md）：效率问题是工具层面的度量缺口，不是规则层面的错误。
- 不在 session_adapter 中调用外部 LLM。
- 不改变现有 digest 文件结构（ai-context.md、summary.md 等保持不变）。
- 不做自动优化建议：只记录指标和模式，由 retrospective 或人工判断是否需要改进。

### 效率问题分类

**P1：验证循环（Verification Loop）**

每次 `apply_patch` 后，AI 执行：
1. `git diff --check`
2. `git status --short`
3. `python3 Workspace/SDD/sdd.py validate ...`
4. `bash Workspace/SystemAgent/Tools/skill-test/lint.sh ...`

有时在修改同一组文件时，步骤 1-4 重复执行 5-8 次。这不是 bug——规则要求"改完总结"——但 AI 缺乏"批量修改后统一验证"的意识。

**P2：文件读放大（File Read Amplification）**

6/9 17:45 会话中，`Src/ECS/Tools/Logger/Log.cs` 被读取 7 次，每次行范围略有不同但有大量重叠。原因：每次 context switch（DeepThink → 实现 → Retrospective）都重新读取事实源。

**P3：Skill 文件重复加载**

同一会话中 `systemagent-deepthink/SKILL.md` 读取 3 次、`sdd-workflow/SKILL.md` 读取 3 次。原因：每次触发 skill 都重新加载。

**P4：git diff/status 不被计为 validation signal**

当前 `VALIDATION_RE` 包含 `git diff --check` 但不包含 `git diff`、`git diff --name-only`、`git diff --stat`、`git status --short`。这些是频繁使用的检查命令，但不被计为 validation signal。

## Design

### 1. 扩展 VALIDATION_RE

当前：

```python
VALIDATION_RE = re.compile(
    r"py_compile|pytest|unittest|dotnet build|validate|validation|git diff --check|"
    r"run-godot-scene|analyze-godot|build-solutions|sdd\.py validate",
    re.IGNORECASE,
)
```

改为：

```python
VALIDATION_RE = re.compile(
    r"py_compile|pytest|unittest|dotnet build|validate|validation|"
    r"git diff|git status|git log|"
    r"run-godot-scene|analyze-godot|build-solutions|sdd\.py validate",
    re.IGNORECASE,
)
```

影响：`git diff`、`git diff --name-only`、`git diff --stat`、`git status --short` 等都会被计为 validation signal。这更准确地反映了"AI 在做检查"的行为。

注意：`git diff` 也会匹配 `git diff --check`，所以原有行为不变。

### 2. 新增效率指标字段

在 `CodexDigestAnalysis` dataclass 中新增：

```python
@dataclass
class CodexDigestAnalysis:
    # ... 现有字段 ...

    # 效率指标
    verification_loop_count: int = 0        # 验证循环次数
    file_read_amplification: dict[str, int] = field(default_factory=dict)  # 文件名 -> 读取次数
    repeated_reads_gt3: list[str] = field(default_factory=list)  # 读取 >3 次的文件列表
    edit_to_validation_ratio: float = 0.0   # apply_patch 次数 / validation 次数
    avg_validation_per_edit: float = 0.0    # 平均每次 edit 触发的验证次数
```

### 3. 验证循环检测逻辑

定义"验证循环"：一个 `apply_patch`（或 code edit signal）之后，连续出现 2+ 个 validation signal 命令，且这些命令之间没有新的 edit。

```python
def detect_verification_loops(tool_calls_detail: list[CodexToolCall]) -> int:
    """检测验证循环次数。"""
    loops = 0
    last_edit_index = -1
    validation_count_since_edit = 0

    for call in tool_calls_detail:
        if call.code_edit_signal:
            # 上一次 edit 后有 2+ 次验证，算一个循环
            if validation_count_since_edit >= 2:
                loops += 1
            last_edit_index = call.event_index
            validation_count_since_edit = 0
        elif call.validation_signal and last_edit_index >= 0:
            validation_count_since_edit += 1

    # 最后一次 edit 的验证
    if validation_count_since_edit >= 2:
        loops += 1

    return loops
```

### 4. 文件读放大检测逻辑

```python
def detect_file_read_amplification(
    tool_calls_detail: list[CodexToolCall],
) -> tuple[dict[str, int], list[str]]:
    """统计每个文件的读取次数，返回读取次数 dict 和 >3 次的文件列表。"""
    read_counts: dict[str, int] = {}

    for call in tool_calls_detail:
        if call.code_edit_signal:
            continue  # 只统计读取，不统计修改
        for path in call.files_read:
            read_counts[path] = read_counts.get(path, 0) + 1

    repeated = [path for path, count in read_counts.items() if count > 3]
    repeated.sort(key=lambda p: read_counts[p], reverse=True)
    return read_counts, repeated
```

### 5. 效率摘要字段（index.json）

在 index.json entry 中新增：

```json
{
  "efficiency": {
    "verification_loops": 5,
    "repeated_file_reads": ["Src/ECS/Tools/Logger/Log.cs", "SDD/.../README.md"],
    "avg_validation_per_edit": 2.3,
    "edit_count": 21,
    "validation_count": 48
  }
}
```

字段说明：

| 字段 | 类型 | 含义 |
|---|---|---|
| `verification_loops` | int | 验证循环次数（edit 后 2+ 次验证） |
| `repeated_file_reads` | string[] | 读取 >3 次的文件路径列表（最多 10 个） |
| `avg_validation_per_edit` | float | 平均每次 edit 触发的验证次数 |
| `edit_count` | int | apply_patch + explicit code edit 总次数 |
| `validation_count` | int | validation signal 总次数（使用扩展后的 VALIDATION_RE） |

## Options

### Option A：只改 VALIDATION_RE，不加效率文件

优点：改动最小。

缺点：retrospective 仍然看不到效率问题；index.json 无法筛选低效会话。

### Option B：加 efficiency.md + index 字段，不改 retrospective

优点：有数据但不强制使用。

缺点：retrospective 复盘时不会主动检查效率。

### Option C（推荐）：完整闭环

优点：session_adapter 生成指标 → efficiency.md 记录 → index.json 可筛选 → retrospective 复盘时检查。

缺点：改动范围较大，需要跨 3 个文件。

## Recommendation

Option C。效率问题已从实证数据确认，需要完整闭环才能真正改善。
