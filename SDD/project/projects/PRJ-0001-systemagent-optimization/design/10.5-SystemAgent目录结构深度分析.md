# SystemAgent 目录结构深度分析与整改方案

> 分析日期：2026-05-25
> 范围：`Workspace/SystemAgent/` 及其关联目录
> 前提：SDD-0006 ~ SDD-0010 已按 design 要求完成整改，验证通过。
> 核心发现：**按 design 要求改的是"入口收敛、分层清晰、规则落盘"，但改完后物理目录仍然过多过散，旧噪音未清。**

---

## 一、现状量化

### 1.1 物理规模

| 指标 | 数值 |
|------|------|
| 一级子目录数 | 13 个 |
| 文件总数（maxdepth=2） | 38 个 |
| 总大小 | 147.8 KB |
| 平均每目录文件数 | 2.9 个 |

### 1.2 各子目录文件数与大小

| 目录 | 文件数 | 总大小 | 问题标记 |
|------|--------|--------|----------|
| `Workflows/` | 7 | ~21 KB | 正常 |
| `Roles/` | 13 | ~21 KB | 文件多但合理 |
| `Protocols/` | 6 | ~31 KB | 正常 |
| `Policies/` | 6 | ~12 KB | 正常 |
| `Catalog/` | 3 | ~25 KB | 机器索引，合理 |
| `Gates/` | 2 | ~14 KB | 偏少 |
| `Capabilities/` | 2 | ~5 KB | **过少**（仅1个落地capability+INDEX） |
| `Skills/` | 2 | ~1.3 KB | **过少且名不副实**（无skill，只有policy说明） |
| `Config/` | 2 | ~26 bytes | **几乎为空**（review-mode.txt + .last-sync） |

| 文件 | 大小 | 职责 |

| `README.md` | 5.4 KB | 事实源根说明、入口链、目录地图、边界摘要 |

根目录承载三种不同类型的内容：README（元说明）、INDEX（路由+索引）、BDDSceneFormat（契约格式）。

## 二、分散问题的四层分析

### 2.1 物理分散：13个一级目录，平铺过浅

**问题**：一级目录数量过多，但每个目录深度只有一层（除 Tools/ 外）。这形成了"宽而浅"的结构，视觉上像摊开的零件，而非组装好的模块。

**证据**：
- Config/ 只有 2 个文件，共 26 字节
- Skills/ 只有 2 个文件，且内容与目录名严重不符
- Capabilities/ 只有 2 个文件，其中 INDEX.md 占了大半
- Tools/ 甚至没有直接文件，只有嵌套的 skill-test/ 和 systemagent-hooks/

**认知负担**：AI 或人类浏览时，面对 13 个并列目录，需要逐一判断每个目录的职责边界。目录名本身（Workflows、Capabilities、Roles、Protocols、Gates、Policies、Catalog、Config、Tools、Skills）是 10 个不同概念，学习成本高。

### 2.2 逻辑分散：概念过度拆分，边界模糊

当前 SystemAgent 正文按 10 个概念类别拆分：

```
Workflow（流程编排）
Capability（可复用能力）
Role（执行视角）
Protocol（长期协议）
Gate（检查点）
Policy（策略边界）
Catalog（机器索引）
Config（运行配置）
Tool（维护工具）
Skill（技能策略）
```

**模糊边界**：

- **Gate vs Workflow**：workflow-catalog.yaml 中每个 workflow 已声明其 gates，Gate 目录的 `ReviewGates.md` 只是共享 gate 库定义。Gate 与 Workflow 是强耦合的，但物理分离。
- **Protocol vs Policy**：Protocol 定义 AI 开发契约（AIFeatureDevelopment、TDD、CapabilityChange），Policy 定义边界规则（Git、AIConfig、Subagent）。两者都是"规则"，但分在不同目录。
- **Catalog vs INDEX**：manifest.yaml、workflow-catalog.yaml、systemagent-catalog.yaml 是机器可读索引；INDEX.md 是人工路由索引；README.md 是目录地图。三者内容有大量重叠。
- **Capability vs Role**：DesignDiscovery 是 capability，但 DesignCritic 是 role。两者都是"设计阶段的能力/视角"，但分类不同。
- **Skills/ 目录**：名为 Skills，实际内容是 "wrapper skill 作者规则"，没有真正的 skill 定义。真正的 wrapper skill 在 `.ai-config/skills/systemagent/` 下。

### 2.3 索引分散：同一信息多处维护

SystemAgent 的索引信息至少分布在 6 处：

| 索引位置 | 维护内容 | 大小 |
|----------|----------|------|
| `README.md` | 目录地图、边界摘要、Memory分层 | 5.4 KB |
| `INDEX.md` | 路由规则、workflow选择表、目录入口表 | 4.8 KB |
| `manifest.yaml` | 机器可读包清单、组件列表、路径映射 | 7.3 KB |
| `workflow-catalog.yaml` | workflow 机器索引、must_read、conditional_read、gates | 8.4 KB |
| `systemagent-catalog.yaml` | skill 机器索引、rubric、category | 9.9 KB |
| 各子目录 `INDEX.md` | 子目录内文件列表和说明 | ~1 KB × 4 |

**同步负担**：新增一个 workflow 需要同时更新：
1. `Workflows/INDEX.md`（新增一行）
2. `Workflows/<NewWorkflow>.md`（新建）
3. `INDEX.md`（可能新增 workflow 描述或目录索引）
4. `README.md`（可能新增目录地图一行）
5. `manifest.yaml`（新增 workflow 组件）
6. `workflow-catalog.yaml`（新增 workflow 条目）

这是 6 处同步，纯人工维护必然漂移。

### 2.4 引用分散：内容与入口物理分离

**wrapper skill 问题**：

| 内容位置 | 入口位置 | 距离 |
|----------|----------|------|
| `Workspace/SystemAgent/Workflows/` 正文 | `.ai-config/skills/systemagent/<name>/SKILL.md` wrapper | 跨目录树 |
| `Workspace/SystemAgent/Roles/` 正文 | `.claude/agents/`、`.codex/agents/` 启动器 | 跨配置层 |
| `Workspace/SystemAgent/` 规则 | `AGENTS.md`、`CLAUDE.md`、`.windsurf/rules/` | 同步副本 |

`.ai-config/skills/systemagent/` 下有 8 个 wrapper skill，每个目录只有 `SKILL.md`，总计 9 KB。这些 wrapper 的核心内容是：
- 触发条件（1-2 句）
- 必读路径列表（8-9 个 SystemAgent 路径）
- 输出要求（1-2 句）
- 禁止事项（4 条）

这种"一个文件一个目录"的结构（8 个目录 × 1 个文件）本身就是物理分散的体现。

## 三、旧目录噪音未清理

`Workspace/DocsAI/` 下仍有以下非当前入口的历史/噪音目录：

| 目录 | 内容 | 状态 |
|------|------|------|
| `ChatHistory/` | 3 个聊天历史归档 | 历史噪音 |
| `Idea/` | 空目录（0 items） | 空壳 |
| `Reviews/` | 7 个历史 review 文档 | 历史分析，非当前入口 |
| `SlimeAINewReorientation/` | 12 个旧规划文档 | 历史参考 |
| `Temp/` | 1 个临时文件 | 临时噪音 |

虽然 `Workspace/DocsAI/INDEX.md` 已声明这些不是当前入口，但物理存在仍会形成以下问题：

1. **AI 扫描干扰**：当 AI 使用 `find` 或 `rg` 搜索 SystemAgent 相关内容时，Reviews/ 和 SlimeAINewReorientation/ 会产生大量命中。
2. **git 信号疲劳**：这些目录如果是未跟踪的，会长期出现在 `git status` 中（如 `Resources/`、`SlimeAIOld/`），削弱边界信号。
3. **新成员困惑**：新 AI 或人类浏览时，看到这些目录名称类似正式文档，难以判断哪些是当前规则。

## 四、与 design 要求的差距

PRJ-0001 的 design 要求（来自 `01-10` 设计文档和 `SystemAgent问题清单.md`）已落实了：

- ✅ 入口收敛（README→INDEX→Workflow）
- ✅ 分层清晰（Route/Workflow/SDD/Capability/Role/Gate/Policy/Tool）
- ✅ 规则落盘（Workflow、Role、Gate、Policy 都写入 SystemAgent 正文）
- ✅ DesignDiscovery / DesignCritic 落地
- ✅ Git/Worktree/Subagent 策略落盘
- ✅ Hook/Gate P0 稳定

**但 design 没有明确要求"物理目录合并"**，只说了"收入口、清分层，不做目录物理合并"。

现在的矛盾是：
- **分层逻辑上已清晰**，但物理上仍是 13 个并列目录
- **旧目录已声明不是入口**，但物理上仍未清理
- **索引多处维护**，但设计文档未要求索引合并

## 五、整改方案

### 5.1 选项A：保守合并（最小改动）

**合并内容过少/名不副实的目录**：

```
Workspace/SystemAgent/
├── README.md
├── INDEX.md
├── BDDSceneFormat.md
├── Workflows/
├── Roles/
├── Protocols/     ← 保持不变
├── Gates/         ← 保持不变
├── Policies/      ← 保持不变
├── Capabilities/  ← 保持不变（等后续capability落地后自然充实）
├── Catalog/       ← 保持不变
├── Tools/         ← 保持不变
└── Config/        ← 与 Skills/ 合并为新目录 Meta/ 或 Internal/
```

**具体操作**：
- `Skills/` → 并入 `Policies/AIConfigBoundary.md` 或新建 `Meta/`（内部工具规则）
- `Config/` → 并入 `Meta/` 或 `Catalog/`
- 删除空的 `Idea/` 目录
- 将 `Reviews/` 迁移到 `Workspace/Archives/` 或 `.gitignore` 排除

**优点**：改动小，不影响核心 workflow/role/policy
**缺点**：没有解决根本的"概念过多"问题

### 5.2 选项B：职责重构（中等改动）

**按 AI 执行路径重组为 4-5 个核心目录**：

```
Workspace/SystemAgent/
├── README.md
├── INDEX.md
├── BDDSceneFormat.md
├── Route/                    ← 原 Workflows/ + 路由相关（INDEX职责上移）
│   ├── Workflows/
│   ├── workflow-catalog.yaml
│   └── route-contract.md     ← 原 INDEX.md 的路由部分
├── Actors/                   ← 原 Roles/ + Capabilities/（执行者=角色+能力）
│   ├── Roles/
│   └── Capabilities/
├── Rules/                    ← 原 Protocols/ + Policies/ + Gates/ + Skills/
│   ├── Protocols/
│   ├── Policies/
│   ├── Gates/
│   └── WrapperPolicy.md      ← 原 Skills/ 内容
├── Index/                    ← 原 Catalog/ + Config/（机器可读索引+配置）
│   ├── manifest.yaml
│   ├── systemagent-catalog.yaml
│   └── review-mode.txt
└── Tools/
```

**优点**：
- 目录从 13 个收敛到 5 个核心目录
- 按 AI 执行路径组织：Route（怎么选）→ Actors（谁来做）→ Rules（什么规则）→ Index（机器索引）→ Tools（怎么验证）
- 概念从 10 个收敛到 5 个

**缺点**：
- 改动量大，需要更新所有 wrapper skill 的引用路径
- 需要更新 manifest.yaml、workflow-catalog.yaml、hook、subagent launcher
- 需要更新 `.ai-config/skills/systemagent/` 的必读路径
- 属于"大重构"，建议用独立 SDD

### 5.3 选项C：索引自动化（不改动物理结构）

**核心思路**：不改物理目录，但用脚本/模板消除人工同步负担。

**具体做法**：
1. **子目录 INDEX.md 自动化**：每个子目录 INDEX.md 由脚本从文件列表自动生成，不作为 hand-written 事实源。
2. **README 目录地图自动化**：manifest.yaml 已包含目录映射，README 的目录地图改为引用 manifest.yaml。
3. **INDEX.md 路由表去重**：INDEX.md 的 workflow 选择表改为引用 workflow-catalog.yaml，不再 hand-written 维护。
4. **wrapper skill 合并**：将 8 个独立的 wrapper skill 合并为 1-2 个通用入口，通过参数区分 workflow。

**优点**：不改物理路径，风险最小
**缺点**：没有解决"目录视觉分散"问题，只是降低了维护成本

### 5.4 选项D：旧目录清理先行（最低风险）

无论选择 A/B/C，都可以先做以下无风险清理：

1. **删除空目录**：`Workspace/DocsAI/Idea/`（空）
2. **归档历史噪音**：
   - `Workspace/DocsAI/ChatHistory/` → 压缩归档或移入 `.git/info/exclude`
   - `Workspace/DocsAI/Reviews/` → 如需要保留则标记为 `archived/`
   - `Workspace/DocsAI/SlimeAINewReorientation/` → 同上
   - `Workspace/DocsAI/Temp/` → 删除内容
3. **确认 Resources/、SlimeAIOld/、Games/BrotatoLikeOld/ 的处理策略**：这些长期未跟踪目录对 git status 和 AI 扫描都形成干扰

## 六、推荐路径

### 第一步（立即做，零风险）：旧目录清理

按选项D清理 DocsAI 历史噪音，明确未跟踪大目录的处理策略。

### 第二步（用独立 SDD）：物理结构重构

如果用户确认"目录必须合并"，建议：

1. **新建 SDD**（如 SDD-0011），专门处理 SystemAgent 物理目录重构
2. **选择方案**：推荐选项B（职责重构），因为当前 13 个目录的"平铺式"结构在长期来看不可维护
3. **设计细化**：明确新目录结构、路径映射、manifest 更新、wrapper skill 路径更新、hook/subagent 配置更新
4. **迁移执行**：分阶段迁移，先做无风险目录合并，后做路径引用更新
5. **验证**：skill lint、SDD validate、旧路径搜索、git diff --check

### 不推荐的做法

- **把 Protocols/ 合并到 Policies/**：虽然两者都是规则，但 Protocol 是 AI 开发契约（AIFeatureDevelopment、TDD），Policy 是行为边界（Git、Subagent）。合并会模糊"契约"与"边界"的区别。
- **把 Gates/ 合并到 Workflow/**：Gate 是跨 workflow 共享的检查点，合并后会失去共享性。
- **删除 Catalog/ 并入 README/**：Catalog 是机器可读索引，README 是人类可读说明。合并会牺牲机器消费能力。

## 七、关键矛盾

本分析揭示了一个深层矛盾：

> **"分层清晰"（逻辑正确）与"目录精简"（物理正确）之间的张力。**

当前 design 已经完成了逻辑分层：
- Route 层：INDEX.md
- Workflow 层：Workflows/*.md
- Capability/Role 层：Capabilities/、Roles/
- Rule 层：Protocols/、Policies/、Gates/
- Index 层：Catalog/、Config/
- Tool 层：Tools/

逻辑上这是清晰的。但如果逻辑清晰需要 13 个一级目录来实现，那说明**概念粒度可能过细**，或者**目录作为组织单元的粒度与概念粒度不匹配**。

建议的解决方向不是"放弃分层"，而是"让物理结构更好地反映逻辑分组"——例如把紧密相关的概念放入同一父目录下（如选项B的 Actors/、Rules/），而不是让每个概念独占一个一级目录。

---

## 八、结论

1. **当前 SystemAgent 已完成 design 的逻辑要求，但物理目录仍然过散。**
2. **分散问题有四层**：物理（13目录/38文件）、逻辑（10概念/边界模糊）、索引（6处维护）、引用（内容与入口分离）。
3. **旧目录噪音未清**：DocsAI 下的 ChatHistory/、Reviews/、SlimeAINewReorientation/、Temp/ 仍物理存在。
4. **整改需要独立 SDD**：这不是单个 task 能完成的，需要路径映射设计、迁移计划和分阶段执行。
5. **推荐方案**：先清理旧目录（选项D），再启动目录重构 SDD（选项B）。
