name: "OPSX: Apply"
description: 执行 OpenSpec 变更中的任务。当用户要求开始实现、继续实现或推进任务时使用。
---

执行 OpenSpec 变更中的实现任务。

**输入**：可指定变更名称。若未提供，从对话上下文中推断；若仍然模糊，必须让用户从可用变更中选择。

**步骤**

1. **选定变更**

   若已提供名称则直接使用，否则：
   - 从对话上下文中推断用户是否提到过某个变更
   - 若只有一个活跃变更，自动选中
   - 若存在多个候选，运行 `openspec list --json` 获取列表，并使用 **AskUserQuestion 工具** 让用户选择

   始终告知：「正在使用变更：<name>」，并说明如何切换（例如 `/opsx:apply <other>`）。

2. **检查状态以理解工作流模式**

   ```bash
   openspec status --change "<name>" --json
   ```

   解析 JSON，关注：
   - `schemaName`：当前工作流模式（如 `"spec-driven"`）
   - 哪个 artifact 包含任务列表（spec-driven 模式通常为 `"tasks"`，其他模式以状态输出为准）

3. **获取 apply 指令**

   ```bash
   openspec instructions apply --change "<name>" --json
   ```

   返回内容：
   - `contextFiles`：artifact ID 到具体文件路径的映射（依模式而异，可能是 proposal/specs/design/tasks，也可能是 spec/tests/implementation/docs）
   - 进度统计（总数、已完成、剩余）
   - 任务列表及状态
   - 基于当前状态的动态指令

   **状态处理：**
   - `state: "blocked"`（缺失 artifact）：提示用户，建议先使用 openspec-continue-change 补全
   - `state: "all_done"`：恭喜完成，建议归档
   - 其他情况：继续执行

4. **读取上下文文件**

   从 apply 指令输出中的 `contextFiles` 读取每一个文件路径。
   文件内容取决于工作流模式：
   - **spec-driven**：proposal、specs、design、tasks
   - 其他模式：以 CLI 输出的 `contextFiles` 为准

5. **展示当前进度**

   输出：
   - 当前工作流模式
   - 进度：「N/M 任务已完成」
   - 剩余任务概览
   - CLI 返回的动态指令

6. **执行任务（循环直至完成或被阻塞）**

   对每个待办任务：
   - 明示正在处理哪个任务
   - 执行所需的代码修改
   - 保持修改最小化、聚焦当前任务
   - 完成后立即在 tasks 文件中标记：`- [ ]` → `- [x]`
   - 继续下一项

   **暂停条件：**
   - 任务描述不清晰 → 请求澄清
   - 实现过程中发现设计问题 → 建议更新 artifact
   - 遇到错误或阻塞 → 上报并等待指导
   - 用户主动中断

7. **完成或暂停时展示状态**

   输出：
   - 本次会话完成的任务
   - 总进度：「N/M 任务已完成」
   - 若全部完成：建议归档
   - 若暂停：说明原因并等待指导

**实现过程中的输出格式**

```
## 正在执行： <change-name>（模式：<schema-name>）

执行任务 3/7：<task description>
[...实现进行中...]
✓ 任务完成

执行任务 4/7：<task description>
[...实现进行中...]
✓ 任务完成
```

**全部完成时的输出格式**

```
## 实现完成

**变更：** <change-name>
**模式：** <schema-name>
**进度：** 7/7 任务已完成 ✓

### 本次会话完成
- [x] 任务 1
- [x] 任务 2
...

所有任务均已完成！可以归档该变更。
```

**暂停时的输出格式（遇到问题）**

```
## 实现暂停

**变更：** <change-name>
**模式：** <schema-name>
**进度：** 4/7 任务已完成

### 遇到的问题
<问题描述>

**选项：**
1. <选项 1>
2. <选项 2>
3. 其他方案

你希望如何处理？
```

**约束**

- 持续推进任务，直到全部完成或被阻塞
- 开始前务必读取 apply 指令输出中的上下文文件
- 任务描述模糊时先暂停确认，不要盲猜
- 实现暴露设计问题时，暂停并建议更新 artifact
- 每次修改保持最小范围，紧扣当前任务
- 完成一项任务后立即更新对应复选框
- 遇到错误、阻塞或需求不清时暂停，绝不臆测
- 使用 CLI 输出的 `contextFiles`，不臆断具体文件名

**流式工作流集成**

本 Skill 支持「变更上可任意操作」的模型：

- **随时可触发**：不必等所有 artifact 就绪（只要已有任务即可），也支持在部分实现后穿插其他操作
- **允许 artifact 更新**：实现中发现设计问题时，建议更新 artifact —— 不被阶段锁定，灵活推进
