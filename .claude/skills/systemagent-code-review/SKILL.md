# systemagent-code-review

## 定位

这是只读代码审查 skill。第一优先级是判断**功能有没有按用户需求、FeatureSpec、SDD 设计或 BDD 场景实现**；第二优先级才是检查必要代码质量边界。

测试结果、ValidationSession artifact、构建输出可以作为 evidenceRef，但不主导判定。缺测试通常记录为证据缺口或交给 `RV-TEST-COVERAGE`；只有测试失败暴露实现错误，或缺少可复验证据导致无法判断功能时，才影响本 skill 的功能判定。

## 触发条件

- SDD task 实现完成后、声称完成前。
- 用户要求"Code Review"、"代码审查"、"审查实现"、"检查代码质量"、"检查实现是否符合需求"、"总结为什么功能没实现"。
- Retrospective 发现实现偏差需要独立审查。

## 必读

- SDD design 文档（需求 / 完成定义 / 数据契约 / 目标架构）
- 设计旁 `.FeatureSpec.md`（如存在）
- 用户当前明确要求或问题描述
- 当前实现代码（git diff 或指定文件）
- 相关 owner skill / DocsAI（如涉及特定模块）
- 可用的测试、构建、ValidationSession artifact 或日志证据（如存在）

## 流程

### Phase 1: 提取审查目标

从用户请求、FeatureSpec、SDD design、BDD、tasks 或代码变更说明中提取可审查目标。每项必须是具体、可验证的陈述，不能是模糊的"能"字句。

提取规则：

- 用户明确点名的功能或问题 → 独立审查目标
- FeatureSpec 的每个 Feature item / Review Checklist → 独立审查目标
- 完成定义（Definition of Done）的每条 → 独立需求项
- BDD Scenario 的关键 Given/When/Then → 独立行为目标
- 目标架构中的关键契约（字段、接口、行为）→ 独立需求项
- "不采用"列表中的每条 → 独立需求项（应验证确实没做）
- 明确的裁决（如"默认关闭 GodotEditorSink"）→ 独立需求项

输出格式：

```markdown
| # | 来源 | 审查目标 | 主要证据 |
|---|------|--------|----------|
| 1 | 完成定义.1 | 业务日志可通过结构化 envelope 输出 | 实现代码 + artifact |
| 2 | FeatureSpec.FS-2 | Data 修改后能触发目标事件 | 实现代码 + Runtime test |
```

### Phase 2: 逐条审查

对每条审查目标，执行：

1. **定位实现代码**：找到对应的生产代码文件和函数；不要只看测试文件。
2. **判断状态**：
   - `DONE` — 功能实现完整，代码质量和架构边界没有阻塞问题。
   - `PARTIAL` — 实现存在但不完整、质量不足或证据不足。
   - `MISSING` — 完全没有实现。
   - `WRONG` — 实现了但方向错误或违反需求。
3. **检查必要质量维度**：
   - 功能/设计实现度：行为是否真的符合用户需求、FeatureSpec、SDD、BDD 或完成定义。
   - ECS 架构红线：Entity 生命周期、DataKey、EventBus、ResourceManagement、TimerManager、TargetSelector、DamageService、对象池等硬约束是否被破坏。
   - 修改范围和依赖边界：是否越界改无关模块、引入不必要依赖、跨 git 边界混改。
   - 可维护性底线：命名是否误导、重复是否明显、函数是否过长或嵌套过深；只抓明显影响维护的问题。
   - 热路径性能和日志噪音：`_Process`、高频循环、调度器、查询器中是否新增 `new`、LINQ、过量日志或不必要分配。
   - 失败语义和错误处理：失败是否有明确返回、异常、日志或 artifact；是否吞错、伪造成功、用默认值掩盖错误。
4. **收集证据**：引用具体文件:行号、关键代码、命令输出或 artifact 内容。
5. **分析根因**（仅对 PARTIAL/MISSING/WRONG）：
   - 需求理解偏差：AI 误解了需求的含义。
   - 验收口径过低：验证命令只检查存在性，不检查质量。
   - 概念边界模糊：如 flow 和 operation 没有明确区分。
   - 技术约束阻塞：如 Godot API 限制。
   - 遗漏：需求存在但执行时被跳过。
   - 质量底线缺失：功能出现了，但实现违反架构红线、维护性或性能边界。

### Phase 3: 输出报告

```markdown
# 代码审查报告

## 概览
- 审查目标总数：N
- DONE：X | PARTIAL：Y | MISSING：Z | WRONG：W
- 主要风险：功能缺口 / 架构红线 / 质量问题 / 证据缺口

## 逐条审查

### DONE
| # | 需求项 | 证据 |
|---|--------|------|
| 1 | ... | file:line, 命令输出 |

### PARTIAL
| # | 需求项 | 当前状态 | 差距 | 根因 | 修复建议 |
|---|--------|----------|------|------|----------|
| 3 | ... | 已有 X，缺 Y | ... | 验收口径过低 | 补 Z |

### MISSING
| # | 需求项 | 根因 | 修复建议 |
|---|--------|------|----------|
| 5 | ... | 遗漏 | ... |

### WRONG
| # | 需求项 | 实际实现 | 期望行为 | 根因 | 修复建议 |
|---|--------|----------|----------|------|----------|
| 7 | ... | ... | ... | 概念边界模糊 | ... |

## 质量问题
| 严重度 | 维度 | 位置 | 问题 | 建议 |
|---|---|---|---|---|
| high | ECS 架构红线 | file:line | ... | ... |

## 证据缺口
- 缺少或未运行的测试 / artifact：...
- 这是否影响功能判定：是 / 否，原因：...

## 根因分析

### 根因 1: [名称]
- 影响的需求项：#3, #5, #7
- 具体表现：...
- 为什么发生：...
- 建议修复方式：...

## 修复优先级

1. [最高影响] ...
2. ...
```

## 禁止

- 不写实现代码；只审查和报告。
- 不接受"应该会过"作为 DONE 证据；必须引用具体代码或命令输出。
- 不把编译通过（`dotnet build`）当作需求完成的证据。
- 不把测试通过直接当作代码质量合格；必须读实现代码。
- 不把缺测试直接等同于功能未实现；缺测试先记录为证据缺口或交给 `RV-TEST-COVERAGE`。
- 不把目录/文件存在当作内容合格的证据。
- 不自由发挥 verdict 文本；状态只允许 DONE / PARTIAL / MISSING / WRONG。
- 不直接修改 `.ai-config/sync-targets.json` 定义的 skill 同步副本作为源。

## 与现有 skill 的关系

- 本 skill 在 `test-designer`（设计验收标准）和 `verifier`（确认证据完整性）之间，负责**需求/设计 vs 实现代码 + 必要质量边界**的逐条对比。
- `reviewer` 检查流程合规（gate、边界、禁改副本）；本 skill 检查需求合规（实现是否符合设计）。
- `RV-TEST-COVERAGE` 检查 TDD 和测试覆盖；本 skill 可引用测试 evidenceRef，但不替代测试 gate。
- `retrospective` 检查流程缺口；本 skill 检查实现缺口。
- 审查报告的 MISSING/WRONG 项可直接转为 SDD `tasks.md` 的 follow-up 任务。
