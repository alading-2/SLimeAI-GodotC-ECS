# CSharp 与 Godot 数学工具选型分析

## 1. 要回答的三个问题

这次分析要回答的是：

1. C# / .NET 有没有现成数学库？
2. Godot 有没有现成的曲线、补间、路径工具？
3. 你现在大量逐帧写运动策略，算不算重复造轮子？

先给结论：

- 有现成基础库，但大多停留在“向量 / 矩阵 / 数值接口 / 通用数值算法”层。
- Godot 有路径和补间工具，但不是所有玩法轨迹都适合直接替代 ECS 运动策略。
- 你现在不是单纯重复造轮子，更准确地说是“高层游戏轨迹语义仍需自研，低层基础能力应尽量复用”。

---

## 2. .NET 自带了什么

### 2.1 `System.Numerics` 提供的是基础数值与向量能力

官方文档明确表明，.NET 自带：

- `Vector2`
- `Vector3`
- `Vector4`
- `Quaternion`
- `Matrix3x2`
- `Matrix4x4`
- 以及泛型数学接口

这意味着：

- 基础向量运算不是没有轮子。
- 通用数值接口也不是没有轮子。
- 但“游戏轨迹工具箱”不是 BCL 的目标。

对项目的意义：

- 如果只是做纯算法层，并且希望降低对 Godot 类型的耦合，可以在部分底层工具考虑 `System.Numerics.Vector2`。
- 但当前项目深度依赖 Godot 节点、`Transform2D`、`Mathf`、`Vector2`，全面切到 `System.Numerics` 并不划算。

### 2.2 .NET 泛型数学解决的是“数值泛型复用”，不是“轨迹表达”

官方 `generic math` 文档说明：

- `.NET 7+` 提供 `INumber<T>`、`IFloatingPointIeee754<T>` 等接口。
- 适合写泛型数学算法。

这对项目的实际价值主要在：

- 写通用插值工具时，可以少写重复重载。
- 写数值工具库时，接口表达更现代。

但它不能直接替代：

- 曲线采样。
- 弧长映射。
- 抛物线 / 椭圆 / 圆弧轨迹。
- 技能几何范围判定。

也就是说，泛型数学是“底层抽象能力增强”，不是高层游戏路径库。

### 2.3 .NET 标准库没有高层现成“轨迹系统”

这点非常关键。

.NET 自带的是：

- 数值类型。
- 向量矩阵。
- 泛型数值接口。
- 一些底层优化。

但默认并不提供：

- 面向游戏设计的抛物线工具。
- 通用 boomerang 回旋轨迹。
- 扇形 / 胶囊 / 环形技能范围工具。
- 匀速 Bézier 路径行走框架。

所以如果你问“我是不是在重复造轮子”：

- 如果你在手写 `Vector2.Lerp`、基础点积叉积，那有一部分确实重复。
- 如果你在写“技能轨迹语义工具”，那基本不算重复。

---

## 3. Math.NET Numerics 这类库能不能直接用

### 3.1 它能做什么

Math.NET Numerics 官方资料显示，它提供：

- 插值。
- 样条。
- 积分。
- 回归。
- 优化。
- 线性代数。
- 概率统计。

这类库的优势在于：

- 学术和工程数值算法很全。
- 插值和数值分析能力成熟。
- 对“编辑器工具”和“离线预处理”很有价值。

### 3.2 它不太适合直接顶替项目里的实时运动层

原因有四个。

#### 第一，它的目标不是 Godot 实时游戏运动

它偏向数值计算库，而不是“游戏运行时轨迹语义库”。

#### 第二，它的抽象过于泛化

你项目真正关心的是：

- 按弧长匀速。
- 2D 朝向切线。
- 轨迹参数对设计师是否好理解。
- 是否能和 ECS `MovementParams` 对上。

这些都不是 Math.NET 的核心目标。

#### 第三，它会引入额外依赖和心智负担

对当前项目来说，想要的是：

- 小而稳。
- 热路径可控。
- 无多余抽象。

而不是引入一整套数值计算框架来解决 2D 技能轨迹。

#### 第四，很多需求你们自己实现反而更轻

例如：

- 半椭圆采样。
- 抛物线顶点控制。
- 简单弧长 LUT。
- 扇形与胶囊范围判定。

这些自己写出来反而更清晰，也更适合当前项目性能边界。

### 3.3 适合把它放在哪

如果未来确实需要：

- 曲线拟合。
- 配置生成。
- 编辑器预烘焙。
- 离线分析。
- 高阶样条实验。

Math.NET 可以考虑用于：

- 编辑器工具链。
- Debug / Test / Analysis。
- 数据生成脚本。

不建议优先用于：

- 战斗运行时热路径。
- 每帧核心移动策略。

## 4. Godot 自带了什么

### 4.1 `Curve2D` / `Path2D` / `PathFollow2D`

Godot 官方文档给出的能力非常明确：

- `Curve2D` 用来描述 2D Bézier 曲线。
- 它会缓存 baked points，加速后续采样。
- `sample_baked(offset)` 是按“沿曲线距离”取样。
- `sample_baked_with_rotation(offset)` 还能直接给出旋转方向。
- `Path2D` 持有一条 `Curve2D`。
- `PathFollow2D` 可以按 `progress` 或 `progress_ratio` 让子节点沿路径前进，并可自动朝向路径方向。

这说明 Godot 对“作者化路径”是有成熟支持的。

适合场景：

- 设计器预设路径。
- 固定路线特效。
- 非碰撞核心物体。
- 展示型、表演型运动。

### 4.2 `Tween`

Godot 的 `Tween` 不是只会补位置。

官方文档明确说明：

- `Tween` 适合“你事先不知道最终值，但想做数值插值”的情况。
- 它比 `AnimationPlayer` 更轻。
- 可以 fire-and-forget。
- 支持 `tween_method()`，也就是把一个值从 `from` 插到 `to`，每一帧回调给你。

这意味着它可以做两类事情：

1. 直接补 Node 属性  
   比如位置、旋转、透明度。
2. 驱动自定义参数  
   比如把 `progress` 从 0 补到 1，然后在回调里调用你自己的曲线采样函数。

因此，Godot 并不是没有“弧线工具”，而是它提供的是：

- 路径系统。
- 通用插值系统。
- 而不是专门的“boomerang / parabola / cone-sample gameplay toolkit”。

### 4.3 Tween 不能自动替代你的 ECS 移动系统

这是最容易误判的点。

虽然 `Tween` 很强，但它不是你当前项目运动系统的直接替代品。

原因：

#### 第一，`Tween` 本质是插值器，不是运动策略框架

你当前运动系统包含：

- 模式切换。
- 与 `MovementParams` 对齐。
- 与 `Velocity`、`FacingDirection` 对齐。
- 与 `CharacterBody2D` / `Node2D` 统一调度。
- 到达判定。
- 阶段切换。
- 返程跟随。

这些都不是 `Tween` 自动替你完成的。

#### 第二，`Tween` 更适合“结果驱动”，你们现在是“策略驱动”

例如：

- “把位置从 A 拉到 B”很适合 Tween。
- “先去程半椭圆，停顿，返程跟随移动目标，再命中结束”不适合只用 Tween 直接接管。

#### 第三，碰撞和战斗语义仍需要你自己控制

如果运动与：

- 命中检测
- 实时目标丢失
- 移动锁定
- 多阶段状态机
- Data 写入

强耦合，那么继续保留手写策略调度更稳。

## 5. 你的“每帧更新很多逻辑”到底有没有问题

### 5.1 每帧更新不是问题，边界不清才是问题

游戏运动系统本来就是高频更新逻辑。

问题不在于“是不是每帧更新”，而在于：

- 每帧里是不是混进了太多可复用公式。
- 每个策略是不是都在重复造相似轮子。
- 业务层是不是直接手写底层数学。

当前项目的真实问题是：

- 有些通用曲线数学已经抽好了。
- 有些还内嵌在策略里，导致后续新增曲线会继续复制。

### 5.2 应当减少的不是“逐帧”，而是“逐策略重复”

例如：

- `SineWave` 的数学已经抽出去，复用良好。
- `BezierCurve` 的数学也抽出去，复用良好。
- `Boomerang` 的半椭圆数学还没抽出去。
- `GeometryCalculator` 虽是纯几何，但还挂在 `TargetSelector` 下。

所以真正优化方向是：

- 保留逐帧调度。
- 抽离共享曲线公式。
- 抽离共享几何算法。

而不是试图把整个运动系统都交给 Tween。

## 6. 给这个项目的选型结论

### 6.1 应继续使用 Godot 原生 `Vector2` 作为主运行时向量类型

原因：

- 与 Node2D、Transform2D、Curve2D、Path2D、Tween 完全一致。
- 少做类型转换。
- 当前项目绝大多数代码已经围绕 Godot API 构建。

### 6.2 不建议为了“更通用”而全面迁移到 `System.Numerics.Vector2`

除非你要做：

- 引擎无关核心库。
- 跨运行时测试库。
- 脱离 Godot 的数值层。

否则整体收益不高，还会增加转换成本。

### 6.3 运行时核心轨迹层继续自研，但尽量小型化

建议自研保留：

- `WaveMath`
- `BezierCurve`
- `EllipseArc2D`
- `Parabola2D`
- `CircularArc2D`
- `ShapeContains2D`
- `ShapeSampling2D`
- `Distance2D`

这些都属于“轻量、明确、面向游戏语义”的工具。

### 6.4 Godot 工具适合作为补充，不适合作为唯一方案

推荐使用：

- `Curve2D` / `Path2D` / `PathFollow2D`
  用于作者化路径、展示型路径、编辑器预览。
- `Tween`
  用于简单参数插值、阶段过渡、UI / 特效 / 非核心运动。

不推荐直接替代：

- 复杂战斗投射物策略。
- 需要碰撞与命中对齐的物体。
- 需要实时改终点和阶段机的移动。

### 6.5 Math.NET 放在“工具链”比放在“热路径”更合适

推荐用途：

- 编辑器插件。
- 曲线拟合实验。
- 离线生成控制点。
- 数据分析和测试。

不推荐优先用途：

- 战斗中每帧驱动子弹。
- 高频策略核心更新。

## 7. 建议的项目落地策略

### 方案 A：运行时核心层

由项目自研，保持轻量纯函数：

- `Src/ECS/Tools/Math/Curves`
- `Src/ECS/Tools/Math/Geometry`

目标：

- 无额外依赖。
- 热路径可控。
- 和 `MovementParams` / `TargetSelector` 清晰分层。

### 方案 B：Godot 作者化层

使用：

- `Curve2D`
- `Path2D`
- `PathFollow2D`
- `Tween`

目标：

- 做设计器友好的曲线资产。
- 做路径预览。
- 做非关键碰撞对象的表演运动。

### 方案 C：离线分析层

按需引入 Math.NET Numerics。

目标：

- 编辑器工具。
- 配置生成。
- 曲线拟合和测试验证。

## 8. 最终结论

### 8.1 你没有严重重复造轮子

更准确地说：

- 低层向量与数值接口，.NET 和 Godot 已经有轮子。
- 高层游戏轨迹语义，标准库和引擎并不会替你完整做好。

所以你当前的自研重点方向是合理的。

### 8.2 你真正需要优化的是“自研层级”

正确做法不是：

- 停止手写轨迹工具。

而是：

- 停止在每个策略里重复写底层公式。
- 把能复用的曲线与几何能力沉淀到独立数学层。
- 在适合的场景下借用 Godot 的 `Curve2D`、`PathFollow2D`、`Tween`。

### 8.3 一句话判断

> 你不是不该自己写，而是不该每次都从策略里重新写一遍。

对于这个项目，最优解不是“完全依赖现成库”，而是：

- Godot 负责引擎级路径与补间。
- .NET 负责基础数值能力。
- 项目自研负责战斗语义明确、热路径可控的曲线与几何工具层。
