# 摄像机激活与移动测试陷阱

在 Godot 开发中，经常会遇到“代码逻辑正确但视觉效果异常”的情况，其中最经典的坑之一就是 **Camera2D 的激活机制与移动错觉**。

## 1. 现象描述
*   **现象 A**：按下移动键，控制台打印位置在变化，但屏幕上的玩家看起来纹丝不动。
*   **现象 B**：在 `Player.tscn` 里测试不走，但把 `Camera2D` 移出 `Player` 或者调整场景树顺序后又“正常”了。

## 2. 核心坑点：Camera2D 激活规则

Godot 的 `Camera2D` 遵循 **“唯一激活制”**：
1.  **Enabled 属性**：只有勾选了 `Enabled` 的相机才会工作。
2.  **优先级（排位赛）**：如果场景中同时存在多个 `Enabled` 的相机，Godot 默认会激活 **场景树中位置最靠下（最后进入场景）** 的那一个。

> **陷阱场景**：你的 `Player.tscn` 内部自带一个相机，而你的 `Test.tscn` 根节点下又手动加了一个相机。此时，这两个相机在争夺控制权，最终生效的往往不是你预期的那个。

## 3. 视觉假象：为什么“不动”？

当 `Camera2D` 作为 `Player` 的 **子节点** 时：
*   **逻辑**：相机像挂件一样死死锁定在玩家身上。
*   **假象**：如果背景是纯黑的（没有任何参照物），玩家在相机视野中的相对位置始终是中心。此时即使玩家在全球坐标系中跑了一万米，由于屏幕上只有玩家，看起来就像在原地踏步。

## 4. 验证与测试技巧

当你怀疑移动失效时，请尝试以下方法验证，而不是盲目改代码：

### A. 增加参照物（最直观）
在场景中放置一个不随玩家移动的物体（如 `ColorRect` 或 `Sprite2D`），或者直接加一个背景图。只要看到背景在后退，就说明玩家在前进。

### B. 使用“远程 (Remote)”检视器
1.  游戏运行时，回到编辑器。
2.  在“场景”面板左上方切换到 **“远程 (Remote)”** 模式。
3.  选中玩家节点，观察右侧属性面板中的 `Position` 或 `GlobalPosition`。如果数值在变，逻辑就是对的。

### C. 临时开启 Top Level
在 `Player` 下随便建一个显示节点（如 `ColorRect`），在属性中勾选 **Top Level**。
*   开启后，该节点会脱离父节点坐标系，固定在世界坐标。
*   如果玩家走开了，你会看到方块被“甩”在后面。

### D. 调试菜单
勾选编辑器顶部的 **调试 (Debug) -> 显示碰撞形状 (Visible Collision Shapes)**。运行后能看到蓝色的碰撞体，有助于判断空间位置。

## 5. 最佳实践建议
1.  **相机原子化**：决定好相机是属于“玩家”还是属于“关卡”。通常俯视角射击游戏（如土豆兄弟）建议相机跟随玩家，但务必在测试场景中放入背景。
2.  **唯一相机**：避免在多个层级嵌套 `Camera2D`。如果需要切换视角，请通过代码手动调用 `camera.MakeCurrent()`。
3.  **坐标日志**：在 `_Process` 中打印 `GlobalPosition` 而不是 `Position`，以排除父节点位移带来的干扰。
