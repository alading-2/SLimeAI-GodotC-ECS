# C# 模式匹配 (Pattern Matching) 全解

#CSharp #Syntax #Refactoring #Godot

## 1. 核心概念：什么是模式匹配？

在传统的编程中，我们通常关注对象的**类型**（Type）或**值**（Value）。 **模式匹配**则是一种更高级的机制，它允许我们同时检查数据的 **“形状” (Shape)** 和 **“内容”**，并从中提取信息。

简单来说，它把以下三步合而为一：

1. **检查：** 你是某种类型吗？（或者你符合某个条件吗？）
2. **解构：** 如果是，把里面的数据拿出来。
3. **绑定：** 把拿出来的数据赋值给新变量，供我马上使用。

> [!TIP] 深度思考：声明式 vs 命令式
>
> - **传统写法 (命令式)：** "先判断 `obj` 是不是 `null`，如果不为空，再判断是不是 `int`，如果是，强转成 `int`，再比较是否大于 10..."
> - **模式匹配 (声明式)：** "如果 `obj` 是一个 `大于 10 的整数`，执行 X。" —— 代码更接近人类的自然语言逻辑。

## 2. 第一战场：`is` 运算符的进化

在 C# 7.0 之前，`is` 只能用来检查类型。现在，它是模式匹配的入口。

### 2.1 声明模式 (Declaration Pattern)

这是 Godot 开发中最常用的模式。 **作用：** 判断类型 + 赋值变量。

```C#
// ❌ 老派写法 (啰嗦，需强转)
void OnBodyEntered(Node body)
{
    var enemy = body as Enemy;
    if (enemy != null)
    {
        enemy.TakeDamage();
    }
}

// ✅ 现代模式匹配 (推荐)
void OnBodyEntered(Node body)
{
    // 语义：如果 body 是 Enemy 类型，就把它赋值给变量 e，并进入 block
    if (body is Enemy e)
    {
        e.TakeDamage(); // 变量 e 在这里可以直接用，且类型安全
    }
}
```

### 2.2 常量模式 (Constant Pattern)

用于检查 `null` 或特定值。

```C#
if (input is not null) { ... } // 这里的 not 也是模式匹配的一部分
if (damage is 0) { return; }   // 比 damage == 0 读起来更有“匹配”的感觉
```

## 3. 第二战场：`switch` 表达式 (Switch Expression)

这是 C# 8.0+ 的重头戏。它把 `switch` 从一个“控制流语句”变成了“表达式”（可以返回值的算式）。

**特点：**

- 使用 `=>` 而不是 `case ... :`。
- 是一个表达式，结果必须被赋值给变量或返回。
- **穷尽性检查 (Exhaustiveness)**：编译器会强迫你覆盖所有可能的情况，否则报警。

### 3.1 基础语法对比

**传统 `switch` (语句):**

```C#
string typeName;
switch (state)
{
    case GameState.Idle:
        typeName = "待机";
        break;
    case GameState.Run:
        typeName = "奔跑";
        break;
    default:
        typeName = "未知";
        break;
}
```

**现代 `switch` (表达式):**

```C#
// 极其简洁，类似函数式编程
string typeName = state switch
{
    GameState.Idle => "待机",
    GameState.Run  => "奔跑",
    _              => "未知" // _ 是弃元，相当于 default
};
```

## 4. 进阶模式库 (The Pattern Zoo)

模式匹配的真正威力在于组合使用以下模式。

### 4.1 关系模式 (Relational Patterns)

使用 `>`、`<`、`>=`、`<=`。

```C#
string GetDescription(int health) => health switch
{
    <= 0  => "死亡",
    < 50  => "危险",
    >= 50 => "健康" // 编译器能看懂这个逻辑链
};
```

### 4.2 逻辑模式 (Logical Patterns)

使用 `and`、`or`、`not` 连接词。这让代码极其易读。

```C#
// 比如在 Godot 的 _Input 中判断按键
if (event is InputEventKey { Keycode: Key.Space } and { Pressed: true })
{
    Jump();
}

// 检查某个值是否在区间内
if (score is >= 0 and <= 100) { ... }
```

### 4.3 属性模式 (Property Pattern) ✨

**这对 Godot 非常重要！** 可以直接深入对象内部检查属性，而不需要提取变量。

```C#
// 假设有一个物品类 Item { string Name; int Rarity; }

// "如果是稀有度为 5 的 'Excalibur'"
if (item is { Name: "Excalibur", Rarity: 5 })
{
    PlayLegendarySound();
}

// 结合 switch 表达式
int goldValue = item switch
{
    { Name: "Coin" } => 1,           // 名字叫 Coin 的
    { Rarity: > 4 }  => 1000,        // 稀有度大于 4 的任何物品
    { Name: var n }  => n.Length * 10, // 将 Name 捕获为变量 n 并在右侧使用
    null             => 0,           // 处理 null
    _                => 5            // 其他杂物
};
```

### 4.4 元组模式 (Tuple Pattern)

用于同时检查多个变量的状态。非常适合**状态机 (State Machine)** 逻辑。 _(参考您之前提供的 ReducedAnd 例子)_

```C#
// 剪刀石头布逻辑 (Rock, Paper, Scissors)
string Battle(string p1, string p2) => (p1, p2) switch
{
    ("Rock", "Scissors") => "P1 Wins",
    ("Scissors", "Paper") => "P1 Wins",
    ("Paper", "Rock")     => "P1 Wins",
    (_, _) when p1 == p2  => "Draw", // 使用 when 关键字添加额外条件
    (_, _)                => "P2 Wins"
};
```

#### 解释

`string Battle(string p1, string p2) => ...`
这里 `Battle` **不是一个变量**，它就是一个正儿八经的 **方法 (Method)**（也就是函数）。

这里的 `=>` 并不是“箭头函数”（Lambda），在 C# 方法定义这里，它的学名叫做 **“表达式体定义” (Expression-bodied definition)**。

它的意思是：**“这个方法只有一句话，我懒得写 `{ return ...; }` 了，直接用 `=>` 后面算出来的结果当作返回值吧。”**

##### 代码等价翻译

为了让你彻底看清，我们把它还原成“老式写法”：

**你的写法（现代 C#）：**

```C#
// 看起来像赋值，其实是“直接返回”
string Battle(string p1, string p2) => (p1, p2) switch { ... };
```

**等价的老式写法（传统 C#）：**

```C#
string Battle(string p1, string p2)
{
    // 1. 运行 switch 表达式
    string result = (p1, p2) switch
    {
        ("Rock", "Scissors") => "P1 Wins",
        // ...
        _ => "P2 Wins"
    };

    // 2. 返回结果
    return result;
}
```

### 4.5 列表模式 (List Pattern) (C# 11+)

用于匹配数组或列表的结构。

```C#
int[] numbers = { 1, 2, 3, 4 };

// 只有 C# 11 以上支持
if (numbers is [1, 2, ..])
{
    Console.WriteLine("以 1, 2 开头");
}
```

## 5. Godot 实战场景

### 1. 补充：输入事件 (InputEvent) 的终极清理

Godot 的 `_Input` 或 `_UnhandledInput` 通常充满了 `if-else` 和类型转换。使用模式匹配可以把它变成极其整洁的路由表。

**建议添加到笔记的代码：**

```C#
// 1. override: 重写 Godot 父类 Node 的虚方法
// 2. @event: 
//    - "event" 是 C# 的保留关键字（用于定义事件）。
//    - 为了用它做变量名（配合 Godot 的命名习惯），必须加 "@" 进行转义。
//    - 编译器看到 "@event" 会把它当作普通的变量名 "event" 处理。
public override void _UnhandledInput(InputEvent @event)
{
    // 使用 switch 表达式 (Switch Expression)
    // 这不是传统的 switch 语句，它是一个能"返回结果"的表达式。
    // 逻辑：把 @event 扔进去匹配，匹配成功后返回对应的 string 赋值给 action。
    var action = @event switch
    {
        // 模式 A：键盘事件 (类型匹配)
        // { ... } : 属性模式 (Property Pattern)。只有当大括号里的属性都匹配时才成立。
        // Keycode: Key.Escape -> 必须是 ESC 键
        // Pressed: true       -> 必须是"按下"瞬间 (排除松开按键的事件)
        InputEventKey { Keycode: Key.Escape, Pressed: true } => "PauseGame",

        // 模式 B：鼠标按键事件 (类型匹配)
        // 细节：必须把"双击"放在"单击"前面！
        // 因为 switch 是从上到下匹配的，如果不先检查双击，双击事件可能会先被下面的单击逻辑捕获。
        InputEventMouseButton { ButtonIndex: MouseButton.Left, DoubleClick: true } => "SelectAll",

        // 模式 C：鼠标按键事件 (普通单击)
        // Pressed: true -> 排除鼠标抬起的事件
        InputEventMouseButton { ButtonIndex: MouseButton.Left, Pressed: true } => "Attack",

        // 模式 D：弃元 (Discard Pattern)
        // "_" 代表"所有其他情况" (相当于 default)。
        // switch 表达式必须穷尽所有可能性，所以必须写这个兜底，否则编译报错。
        _ => null
    };

    // 只有当 action 不为 null (即匹配到了上面定义的有效操作) 时才打印
    if (action != null) 
    {
        GD.Print($"执行操作: {action}");
    }
}
```

- **价值点**：展示了如何用属性模式 `{ Keycode: ... }` 替代繁琐的 `e.Keycode == ...`。

### 2. 补充：物理碰撞的高级筛选 (when 关键字)

你笔记里已经有了基本的碰撞检测，但游戏开发中往往需要结合 **“向量数学”**。比如：只有**从上方**踩到怪物才算攻击，碰到侧面算受伤。

**建议添加到笔记的代码：**

```C#
public void OnBodyEntered(Node body)
{
    // 假设这是在一个 Area2D (比如马里奥的脚底判定框)
    // 我们可以结合 `when` 关键字引入复杂的逻辑判断
    var result = body switch
    {
        // 情况 A: 这是一个怪物，且它是活的 (属性判断)
        Monster m when !m.IsDead => "Hit Monster",

        // 情况 B: 这是一个弹簧床，且我们是向下落的 (结合自身状态)
        SpringBoard s when Velocity.Y > 0 => "Bounce",

        // 情况 C: 这是一个墙壁 (普通类型匹配)
        TileMap => "Hit Wall",

        _ => "Nothing"
    };
}
```

- **价值点**：强调 `when` 关键字在游戏物理逻辑中的核心作用（结合速度、方向判断）。

### 3. 补充：列表模式解析命令/对话 (C# 11+)

如果你的游戏有“作弊码”或者简单的“对话指令”系统，列表模式是神技。

**建议添加到笔记的代码：**

```C#
// 假设输入是一个字符串数组：var args = command.Split(' ');
// 比如输入: ["give", "gold", "50"]
void ProcessCommand(string[] args)
{
    var feedback = args switch
    {
        // 1. 匹配 ["quit"]
        ["quit"] => "Game Quit",

        // 2. 匹配 ["teleport", x, y] -> 捕获 x, y
        ["teleport", var x, var y] => $"TP to ({x}, {y})",

        // 3. 匹配 ["give", "item_id", 数量] -> 忽略 id，捕获数量
        ["give", _, var amount] => $"Spawned {amount} items",

        // 4. 匹配以 "say" 开头的任意长度数组 -> 列表切片模式
        ["say", .. var words] => $"Chat: {string.Join(" ", words)}",

        _ => "Unknown Command"
    };

    GD.Print(feedback);
}
```

- **价值点**：展示了 C# 11 的列表模式如何处理变长参数，这对解析游戏内控制台指令非常有用。

### 4. 处理不同类型的碰撞体

假设你的主角有个 `OnAreaEntered` 方法，你会碰到各种东西。

```C#
public void OnAreaEntered(Area2D area)
{
    // 直接根据 area 的属性或类型做分发
    var effect = area switch
    {
        // 1. 如果是陷阱，且伤害大于 100 (致命陷阱)
        Trap t and { Damage: > 100 } => "Instant Death",

        // 2. 普通陷阱 (把 Trap t 捕获出来以便后续使用 t.Damage)
        Trap t => $"Take Damage: {t.Damage}",

        // 3. 如果是金币 (属性模式)
        { Name: "Coin" } => "Get Money",

        // 4. 其他任何 Area2D，但不是 null
        not null => "Touch something",

        // 5. 理论上 Godot 信号不会传 null，但 switch 表达式要求穷尽
        _ => "Nothing"
    };

    GD.Print(effect);
}
```

## 6. 深度思考：使用建议与陷阱

1. **性能 (Performance)：** 模式匹配非常快。编译器通常会将复杂的 `switch` 表达式优化为跳转表 (Jump Tables) 或高效的二叉树查找。不要担心性能问题，它通常比手写的 `if-else` 链还要快。
2. **顺序很重要 (Order Matters)：** `switch` 表达式是**从上到下**匹配的。必须把**最具体**的条件放在上面，**最通用**的条件（如 `_`）放在最后。如果顺序反了，编译器会报错 "Unreachable code"（代码不可达）。
3. **空值处理 (Null Handling)：**

   - `x is Type y`：如果 `x` 是 `null`，结果为 `false`，非常安全。
   - `switch` 表达式：如果你没有处理 `null` 情况（且没有用 `_` 兜底），传入 `null` 时会在运行时抛出异常。**建议养成最后加 `_ => ...` 的习惯。**

4. **心智模型转换：** 作为从 TS 转过来的开发者，你可以把 C# 的属性模式 `{ Prop: Val }` 类比为 TS 的**对象解构 + 判等**，但 C# 是在运行时检查，且支持递归模式（模式里套模式）。

# 📝 总结

C# 现代模式匹配旨在让你写出 **“这就是我想要的数据样子”** 的代码，而不是 **“如何一步步检查数据”** 的代码。在 Godot 开发中，善用 `is` 进行节点类型判断，善用 `switch` 表达式处理状态机，能让你的代码量减少 30% 以上，且逻辑更加清晰。
