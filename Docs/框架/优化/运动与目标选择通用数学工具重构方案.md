# 运动与目标选择通用数学工具重构方案

## 1. 背景

当前项目已经存在三类和“数学”相关但职责不同的能力：

1. 轨迹曲线数学  
   典型代表：`WaveMath`、`BezierCurve`、`BoomerangStrategy` 内部的半椭圆采样逻辑。
2. 空间几何数学  
   典型代表：`GeometryCalculator` 中的圆、环、盒、线段胶囊、扇形判定与随机采样。
3. 业务调度层  
   典型代表：`MovementParams`、`IMovementStrategy`、`TargetSelectorQuery`、`PositionTargetSelector`。

问题不在于“是否要做通用数学工具”，而在于当前边界还不够清晰：

- `BezierCurve`、`WaveMath` 已经是通用数学层。
- `BoomerangStrategy` 里还有一段明显可复用的半椭圆轨迹数学，尚未抽离。
- `GeometryCalculator` 本质上是纯几何算法层，但目前挂在 `TargetSelector` 目录下，容易被误解为选择器私有实现。

结论：

- 应该继续抽通用数学工具。
- 但不能把所有“和数学有关”的代码塞进一个总类或一个超大目录。
- 应按“曲线数学”和“几何数学”分别沉淀，再由运动系统和目标选择系统调用。

---

## 2. 现状判断

### 2.1 已经是通用层的部分

#### `Src/ECS/Tools/Math/WaveMath.cs`

已经具备典型通用工具特征：

- 无 Entity / Data / Event 依赖。
- 只关心输入输出，不关心业务生命周期。
- 同时提供采样值、导数、速度方向等高复用能力。

这类文件应该继续保持“静态纯函数 + 明确语义”的风格。

#### `Src/ECS/Tools/Math/Bezier/BezierCurve.cs`

这是当前项目最成熟的通用数学层之一：

- 任意阶 Bézier 求值。
- 导数、切线、曲率。
- 曲线分裂。
- 长度近似。
- 弧长 LUT 重参数化。

这类代码已经不是“策略辅助方法”，而是“可被多个系统共享的底层曲线工具”。

### 2.2 应继续抽出的部分

#### `Src/ECS/Base/System/Movement/Strategies/Projectile/BoomerangStrategy.cs`

以下能力本质上不是 Boomerang 私有业务，而是通用二维弧线轨迹数学：

- 基于起点和终点构建局部曲线坐标系。
- 基于半弦长与弧高采样半椭圆轨迹。
- 采样曲线切线方向。
- 估算半椭圆弧长用于等速推进。

真正属于 Boomerang 业务层的只有：

- 去程 / 返程阶段切换。
- 暂停时间。
- 返程跟随宿主。
- 返程速度倍率。
- 命中阶段终点后的完成逻辑。

因此，半椭圆采样必须从策略内部抽离。

### 2.3 适合升级为通用几何层的部分

#### `Src/ECS/Tools/TargetSelector/GeometryCalculator.cs`

该文件已经明显是“纯几何层”：

- 点是否在圆 / 环 / 盒 / 胶囊 / 扇形内。
- 几何体内均匀随机采样。
- 点到线段距离。
- AABB / hollow box 采样。

这类算法可以被：

- TargetSelector 复用。
- 技能预览范围复用。
- AI 生成巡逻点复用。
- 运动系统做路径几何判断时复用。

但 `TargetSelectorQuery`、`GeometryType`、`PositionTargetSelector` 仍然是领域层，不应整体下沉到通用数学目录。

## 3. 推荐的分层方式

推荐不要继续扩大 `MyMath` 一类的混合工具类，而是拆为如下结构：

```text
Src/ECS/Tools/Math/
  Curves/
    CurveFrame2D.cs
    CurveLengthTable2D.cs
    Bezier/
      BezierCurve.cs
    EllipseArc2D.cs
    Parabola2D.cs
    CircularArc2D.cs
    WaveMath.cs
  Geometry/
    Distance2D.cs
    ShapeContains2D.cs
    ShapeSampling2D.cs
    GeometryQueryAdapter.cs
  Numerics/
    MyMath.cs
```

### 3.1 曲线层职责

曲线层只负责：

- 输入曲线参数。
- 返回曲线点、切线、长度近似、弧长映射。

曲线层不负责：

- MovementParams。
- 目标跟随。
- 停顿与阶段机。
- 命中判定。
- DataKey.Velocity 写入。

### 3.2 几何层职责

几何层只负责：

- 点与形状关系。
- 距离与投影。
- 形状内部随机采样。

几何层不负责：

- 阵营过滤。
- 实体排序。
- 目标数量裁剪。
- 技能目标协议。

### 3.3 业务层职责

业务层负责：

- 策略生命周期。
- 参数组合与解释。
- 与 Entity / Data / EventBus 的对接。

这意味着：

- `IMovementStrategy` 调用曲线层。
- `TargetSelector` 调用几何层。
- 两个系统共享底层算法，但不共享业务协议。

## 4. 哪些能力值得优先做成通用工具

### 第一优先级：`EllipseArc2D`

原因：

- 当前 `BoomerangStrategy` 已直接需要。
- 语义稳定，参数易懂。
- 能直接复用到回旋刀、回收弹、甩出再回收、绕射技能等。

建议提供能力：

- `CreateFromChord(start, end, arcHeight, sideSign)`
- `Evaluate(progress)`
- `EvaluateTangent(progress)`
- `ApproximateLength()`

### 第二优先级：`Parabola2D`

适用场景：

- 抛射物。
- 起跳 / 落地感更强的技能。
- 指定顶点高度的投射路径。

建议支持两类参数化：

1. `start + end + apexHeight`
2. `start + end + apexPoint`

不要把“抛物线”强行伪装成单纯的业务公式，应该做成显式语义工具。

### 第三优先级：`CircularArc2D`

适用场景：

- 固定转弯半径。
- 刀光短弧。
- 规整、可预测的转向轨迹。

相较 Bézier 的优势：

- 参数更直观。
- 半径与角度更便于设计。
- 对“圆弧感”更稳定。

### 第四优先级：统一弧长映射工具

如果未来多种曲线都要“按实际路程等速前进”，建议抽一个共享的小层，例如：

- `CurveLengthTable2D`
- `ArcLengthRemap`

用途：

- Bézier、椭圆弧、圆弧、抛物线都可以统一按“距离”推进。
- 避免每个曲线类都重复写 LUT / 映射逻辑。

## 5. 不建议的错误方向

### 5.1 不要做成“大杂烩 MathHelper”

错误结果会是：

- 曲线采样、距离计算、几何判定、概率、插值、业务规则全混在一起。
- 文件越来越大，命名越来越模糊。
- 后期调用者根本不知道哪个函数属于哪一层语义。

### 5.2 不要把 `TargetSelectorQuery` 下沉到数学层

`TargetSelectorQuery` 是目标选择领域协议，里面包含：

- Geometry
- TeamFilter
- TypeFilter
- Sorting
- MaxTargets

其中只有几何参数是数学层关心的，过滤和排序明显是领域行为。

### 5.3 不要过早引入“统一接口对象图”

对当前项目来说，不建议先设计一整套高抽象 OOP 曲线层，例如：

- `ICurve2D`
- `ICurveSampler2D`
- `IArcLengthParameterizedCurve2D`

原因：

- 你们的运动策略在高频更新路径上运行。
- 当前更适合“静态纯函数 + readonly struct 参数包”。
- 这样更容易避免 GC 和多态分发成本，也更符合项目当前代码风格。

换句话说，先统一“数据结构和函数风格”，再决定是否需要接口抽象。

## 6. 推荐的实现风格

### 6.1 热路径优先使用静态函数和值类型

建议：

- 曲线参数用 `readonly record struct` 或 `readonly struct`。
- 采样结果用 `readonly record struct CurveSample2D(Vector2 Position, Vector2 Tangent)`。
- 高复用帧结构如 `CurveFrame2D` 用值类型。

原因：

- 符合 `_Process` 禁止不必要分配的项目约束。
- 更适合大量短生命周期计算。

### 6.2 统一外部角度语义

项目已有规则：

- 对外一律使用“度”。
- 内部三角运算时再转弧度。

新的曲线工具必须继续遵守这一点，避免不同工具一个用度、一个用弧度。

### 6.3 统一“等速推进”的输入语义

建议明确区分两种推进方式：

1. 参数推进  
   `progress` 直接从 0 到 1。
2. 距离推进  
   `distance` 或 `speed * delta` 通过弧长映射换算。

否则后续策略很容易出现：

- 同样叫 `progress`，但有的表示参数，有的表示真实距离比例。
- 调用侧误用后出现视觉速度不均匀。

## 7. 对 C# / Godot / 外部库的选型判断

### 7.1 这不是单纯重复造轮子

当前项目手写的数学代码里，真正“重复造轮子”的部分很有限。

可以视为合理自研的部分：

- 面向游戏语义的轨迹工具。
- Boomerang 半椭圆。
- 抛物线顶点控制。
- 面向技能目标选择的几何判定与随机采样。
- 面向匀速曲线运动的轻量 LUT。

这些并不是 .NET BCL 默认会提供的高层能力。

### 7.2 应优先复用已有基础能力

应优先复用：

- Godot 自带 `Vector2`、`Transform2D`、`Mathf`。
- 需要引擎无关时，再考虑 `System.Numerics.Vector2`、`Matrix3x2`。
- Godot `Curve2D` / `Path2D` / `PathFollow2D` 适合编辑器作者化路径。

不建议为了“统一”而把所有向量运算重写一遍。

### 7.3 数值分析库不应直接替代游戏运行时轨迹层

像 Math.NET Numerics 这类库更适合：

- 编辑器工具。
- 数据预处理。
- 曲线拟合。
- 离线分析。

不适合直接替代项目里的核心运动策略热路径。

原因见另一份文档。

## 8. 分阶段落地方案

### Phase 1：先做最小可见收益

目标：

- 把 `BoomerangStrategy` 里的半椭圆数学抽成 `EllipseArc2D`。
- 保持策略行为不变，只减少内联公式。

收益：

- 低风险。
- 立即形成第一块通用曲线工具。

### Phase 2：整理几何算法层

目标：

- 将 `GeometryCalculator` 的纯算法拆成 `Geometry/` 子目录。
- `TargetSelector` 保留 Query / 筛选 / 排序 / 裁剪。

收益：

- 几何算法可以被更多模块复用。
- 目标选择系统的业务边界更清晰。

### Phase 3：补齐第二批曲线

目标：

- 新增 `Parabola2D`。
- 新增 `CircularArc2D`。
- 若确有重复需求，再抽统一弧长重参数化小层。

收益：

- 运动系统的曲线表达能力完整度明显提升。

### Phase 4：视需求决定是否接入 Godot `Curve2D`

适用场景：

- 设计师作者化轨迹。
- 特效路径。
- 非碰撞关键物体。

不适用场景：

- 需要每帧动态改终点。
- 需要与 ECS 速度写入严格对齐。
- 需要复杂阶段机。

---

## 9. 最终建议

一句话总结：

> 应该继续通用化，但要把“曲线数学”“几何数学”“业务调度”三层分开，而不是做一个越来越大的 `MathHelper`。

当前最值得立刻推进的事项：

1. 抽出 `EllipseArc2D`。
2. 逐步把 `GeometryCalculator` 升级为更通用的 `Geometry2D` 算法层。
3. 保持 `TargetSelectorQuery` 与 `MovementParams` 继续留在业务层。
4. 新增曲线时优先补 `Parabola2D`、`CircularArc2D`。

这样做既能避免未来重复写公式，也不会把项目架构拉成混乱的大工具箱。
