## 1. 方案全景图

```
Godot TypeScript 方案生态
├── 1. gode (V8 + Node.js) ── 最成熟，全平台
├── 2. GodotJS (QuickJS/V8) ── 早期，不稳定
├── 3. Wiredot/Cobweb (V8) ── 实验性
├── 4. QuickJS GDExtension ── 轻量级，无 TS 原生
├── 5. godot-ts (类型声明) ── 仅工具链
└── 6. HTTP/API 桥接方案 ── 间接方案
```

## 2. 方案详细分析

### 2.1 gode

| 维度 | 详情 |
|------|------|
| **仓库** | github.com/godothub/gode |
| **版本** | v1.7.0 (2026) |
| **JS 引擎** | V8 (通过 libnode) |
| **Node.js** | 完整支持 |
| **平台** | Windows, Android, macOS, iOS, Linux |
| **TS 支持** | 预编译为 JS 后加载 |
| **npm 生态** | 完整支持 |
| **模块系统** | ESM + CJS |
| **成熟度** | 中高（有官方 showcase） |

**核心特点**：
- 嵌入完整 Node.js 运行时，不只是 V8
- 通过 `res://` 路径适配让 npm 包在 Godot 中工作
- tree-sitter 静态解析元数据（exports/signals/rpc_config）
- GDScript ↔ JS 可互相调用

**关键限制**：
- Web 平台不支持
- TS 运行时只加载 JS，需要 `tsc --watch` 预编译
- V8 + Node.js 运行时内存开销大（~50-80MB）
- 绑定层有已知 crash 历史

### 2.2 GodotJS

| 维度 | 详情 |
|------|------|
| **仓库** | 多个变体，无统一官方 |
| **JS 引擎** | QuickJS 或 V8 |
| **Node.js** | 无或有限 |
| **平台** | 部分平台 |
| **TS 支持** | 有限 |
| **npm 生态** | 有限或无 |
| **成熟度** | 低 |

**核心特点**：
- 多个社区维护的变体，缺乏统一
- QuickJS 变体轻量但功能有限
- V8 变体功能更全但稳定性差

**关键限制**：
- 稳定性问题突出（GodotJS issue #76 等）
- 缺乏 Node.js 运行时，无法使用 npm
- 平台支持不完整
- 社区活跃度低

### 2.3 Wiredot / Cobweb

| 维度 | 详情 |
|------|------|
| **前身** | Cobweb |
| **JS 引擎** | V8 |
| **状态** | 早期实验 |
| **平台** | 部分 |
| **成熟度** | 低 |

**核心特点**：
- TypeScript-first 的设计目标
- 从 Cobweb 重命名而来
- 旨在提供完整的 TS 类型安全

**关键限制**：
- 处于早期开发阶段
- 缺乏生产环境验证
- 社区规模极小

### 2.4 QuickJS GDExtension

| 维度 | 详情 |
|------|------|
| **JS 引擎** | QuickJS (Fabrice Bellard) |
| **Node.js** | 无 |
| **平台** | 部分 |
| **TS 支持** | 需要预编译 |
| **npm 生态** | 无 |
| **成熟度** | 中低 |

**核心特点**：
- QuickJS 极其轻量（~210KB）
- 无 JIT，解释执行
- 内存占用小

**关键限制**：
- 性差极差（纯解释执行，比 V8 慢 10-100 倍）
- 无 Node.js，无法用 npm
- 无 TS 原生支持
- 不适合游戏逻辑

### 2.5 godot-ts（类型声明方案）

| 维度 | 详情 |
|------|------|
| **性质** | 不是运行时，是工具链 |
| **作用** | 为 Godot API 提供 .d.ts 类型声明 |
| **运行时** | 依赖其他方案（gode 等） |
| **成熟度** | 中 |

**核心特点**：
- 纯工具链方案，不提供运行时
- 为 Godot API 提供 TypeScript 类型
- 可与 gode 等运行时配合使用

**关键限制**：
- 本身不解决运行时问题
- 类型声明可能滞后于 Godot 版本

### 2.6 HTTP/API 桥接方案

| 维度 | 详情 |
|------|------|
| **原理** | Godot 通过 HTTP/WebSocket 调用外部 Node.js 服务 |
| **运行时** | 外部 Node.js 进程 |
| **延迟** | 网络延迟（ms 级） |
| **成熟度** | 高（通用方案） |

**核心特点**：
- 不需要 GDExtension
- Node.js 可以完整使用所有 npm 包
- 进程隔离，不会 crash Godot

**关键限制**：
- 网络延迟（不适合实时游戏逻辑）
- 需要管理外部进程
- 部署复杂度增加

## 3. TS 在 Godot 中的核心问题

### 3.1 语言层面

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| **类型擦除** | TS 类型在编译为 JS 后全部消失 | 🔴 致命 |
| **无值类型** | 所有对象都在堆上 | 🔴 致命 |
| **GC 不可控** | V8 GC 暂停不可预测 | 🔴 致命 |
| **无编译期契约** | interface/abstract 运行时不存在 | 🔴 致命 |
| **无 AOT** | 只有 JIT，启动性能差 | 🟡 严重 |
| **无泛型实例化** | 泛型在运行时被擦除 | 🟡 严重 |

### 3.2 集成层面

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| **绑定层稳定性** | V8 ↔ Godot 的 crash 历史 | 🟡 严重 |
| **调试困难** | JS 异常 vs native crash 混合 | 🟡 严重 |
| **热重载** | 不如 C# 的 Edit & Continue | 🟡 中等 |
| **编辑器集成** | Inspector、Debugger 集成有限 | 🟡 中等 |
| **导出复杂度** | 需要包含 node_modules | 🟡 中等 |

### 3.3 生态层面

| 问题 | 说明 | 严重程度 |
|------|------|---------|
| **社区规模** | Godot TS 社区极小 | 🟡 严重 |
| **文档不足** | 缺乏最佳实践 | 🟡 中等 |
| **长期维护** | 依赖第三方维护 | 🟡 严重 |

## 4. TS 相比 C# 的优劣

### 4.1 TS 的优势

| 优势 | 说明 | 对 SlimeAI 的价值 |
|------|------|------------------|
| **npm 生态** | 数十万包可用 | 🔴 高：AI 集成 |
| **async/await** | 原生异步支持 | 🟡 中：比 C# Task 更简洁 |
| **类型推断** | 更少的类型标注 | 🟡 中：开发效率 |
| **动态性** | 运行时可修改对象 | ⚪ 低：对 AI-first 不利 |
| **Web 开发者友好** | 大量 Web 开发者熟悉 | ⚪ 低：不是 SlimeAI 目标 |
| **JSON 原生** | JSON 处理更自然 | 🟡 中：数据处理 |

### 4.2 TS 的劣势

| 劣势 | 说明 | 对 SlimeAI 的影响 |
|------|------|------------------|
| **类型擦除** | 运行时无类型信息 | 🔴 致命：AI 护栏失效 |
| **无值类型** | 全部堆分配 | 🔴 致命：GC 无法优化 |
| **GC 不可控** | V8 GC 暂停不可预测 | 🔴 致命：帧率不稳定 |
| **无编译期契约** | interface 运行时不存在 | 🔴 致命：契约无法验证 |
| **无 AOT** | JIT only | 🟡 严重：启动慢 |
| **Web 不支持** | gode 不支持 Web | 🟡 严重：平台限制 |
| **内存开销大** | V8 对象头开销 | 🟡 严重：实体数量受限 |
| **社区小** | Godot TS 社区极小 | 🟡 严重：缺乏支持 |

### 4.3 对比总结

```
C# 在 Godot 中的优势（TS 无法替代的）：
├── 编译期类型安全 → AI 护栏
├── 值类型 (struct) → 零 GC 路径
├── AOT 编译 → 启动性能
├── 泛型实例化 → 运行时类型安全
├── interface → 编译期契约验证
├── 官方支持 → 长期维护保证
└── 大社区 → 生态和文档

TS 在 Godot 中的优势（C# 无法替代的）：
├── npm 生态 → AI/LLM 集成
└── async/await 原生 → 异步代码更简洁
```

## 5. TS 方案选型建议

### 5.1 如果必须用 TS

**推荐 gode**，理由：
1. 最成熟的方案（v1.7.0，活跃维护）
2. 完整的 Node.js 运行时
3. npm 生态支持
4. 全平台（除 Web）
5. 有官方 showcase（tps-demo-ts）

### 5.2 但更推荐混合架构

```
推荐架构：
┌─────────────────────────────────────┐
│    SlimeAI Core (C#)                │
│    ECS / DataOS / EventBus / System │
│    类型安全 / 性能 / AI 护栏         │
├─────────────────────────────────────┤
│    AI 集成层 (gode/TS 或 HTTP)       │
│    llama-cpp / OpenAI / 行为树       │
│    通过 HTTP/IPC 与 Core 通信        │
└─────────────────────────────────────┘
```

### 5.3 混合架构的优势

1. **保持 C# 核心的类型安全和性能**
2. **利用 gode 的 npm 生态接入 AI 能力**
3. **AI 逻辑（非实时）对 GC 不敏感**
4. **通过 HTTP/IPC 解耦，避免绑定层稳定性问题**
5. **渐进式引入，不需要大规模重写**
