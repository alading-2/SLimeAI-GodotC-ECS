# 深度思考：ECS 架构下的单位分类与阵营设计

> [!NOTE]
> 本文档是对用户提出的 "如何区分敌人与友军" 及 "UnitType 通用分类" 问题的深度架构分析。基于现代 ECS（Entity-Component-System）设计原则，结合 Godot 项目实际情况。

## 1. 核心矛盾：身份 (Identity) vs 关系 (Relation)

在传统的 OOP（面向对象）设计中，我们习惯于使用继承树：
- `GameObject` -> `Unit` -> `Hero`
- `GameObject` -> `Unit` -> `Enemy`

但在 ECS 架构中，这种分类会导致**组合爆炸**和**逻辑僵化**。

### 误区分析：将 Hero/Enemy 放入 UnitType
如果 `UnitType` 包含 `Hero`, `Enemy`, `Ability`, `Item`, `Buff`，这混淆了两个维度的概念：
1.  **物理形态（What I am）**：是否有碰撞？是否有血条？是否可移动？（如 Unit, Item, Projectile）
2.  **阵营关系（Who I am with）**：我是玩家一方，还是敌人一方？（Hero, Enemy）

**问题场景**：
- 如果有一个 "被魅惑的怪物"，它还是 `UnitType.Enemy` 吗？如果是，攻击逻辑会出错；如果改成 `UnitType.Hero`，它本质又是个怪物。
- 如果有一个 "玩家召唤的炮台"，它是 `UnitType.Structure` 还是 `UnitType.Hero`？

## 2. 现代 ECS 的解决方案：正交解耦

现代游戏开发（尤其是 ECS 架构）推崇**组件式思维**。我们将属性解耦为正交（互不干扰）的维度。

### 建议方案

我建议将单一的 "分类" 拆分为以下三个正交维度：

#### 维度 1：实体物理类型 (EntityType)
描述实体的**技术实现**和**物理特性**。决定了它拥有哪些基础组件（Archetype）。

- **Unit (单位)**：有血条、有移动能力、有碰撞。
- **Projectile (投射物)**：有速度、有碰撞、无血条（通常）、有存活时间。
- **Structure (建筑)**：有血条、无移动、有碰撞。
- **Item (物品)**：无血条、可被拾取、静态或简单物理。
- **Ability (技能)**：纯逻辑实体，无物理实体。
- **Buff (效果)**：依附于其他实体的逻辑实体。

#### 维度 2：阵营 (Team)
描述实体的**社会关系**。这是动态的，可通过数据修改（DataKey）。

- **Neutral (0)**：中立（如可破坏箱子、中立生物）。
- **Player (1)**：玩家阵营（包含英雄、召唤物、盟友）。
- **Enemy (2)**：敌人阵营（包含怪物、Boss、敌方炮台）。

#### 维度 3：标签/角色 (Tags/Role)
描述实体的**特殊身份**或**游戏逻辑属性**。

- **Hero**：具有输入组件，由玩家控制。
- **Boss**：具有特殊血条 UI，免疫某些控制。
- **Elite**：精英怪。
- **Summoned**：召唤物（死亡不计入击杀统计）。

## 3. 为什么这样做更好？（资深开发者视角）

### 场景 A：精神控制/魅惑机制
- **旧做法**（UnitType=Enemy）：需要将对象销毁，重新生成一个 `UnitType.Friend` 的对象（因为 Type 通常是不可变的或硬编码的）。
- **新做法**（ECS）：只需修改 `DataKey.Team` 从 2 变为 1。
  - 所有 `SelectTarget` 系统自动将其识别为友军。
  - `EntityType.Unit` 保持不变，动画、AI 行为组件保持不变。

### 场景 B：友伤模式 (Friendly Fire)
- **旧做法**：代码写死 `if (type != myType)`。
- **新做法**：`TargetSelector` 传入 `Filter.All` 或 `Filter.Friendly`。系统只看 Team ID，不关心你是 Hero 还是 Summoned。

### 场景 C：技能的多样性
- 一个 "治疗波" 技能：
  - 目标过滤器：`EntityType = Unit` (只治疗活物，不治疗子弹), `Team = Friendly`。
- 一个 "EMP 冲击" 技能：
  - 目标过滤器：`EntityType = Structure | Unit`, `Team = Enemy`。

## 4. 对本项目的具体重构建议

基于上述分析，对 `DataKeyRegister` 的重构建议如下：

### 1. 修正 `UnitType` 定义
不要包含 `Hero` 和 `Enemy`。
```csharp
// 建议重命名为 EntityType 以示区别，或保留 UnitType 但修改注释
public enum EntityType 
{
    None = 0,
    Unit,       // 标准生物
    Projectile, // 子弹/投射物
    Structure,  // 建筑/静态阻挡
    Item,       // 掉落物
    Ability,    // 技能实体
    Buff,       // Buff实体
    Trap        // 陷阱
}
```

### 2. 强化 `Team` 定义
保持 `int` 类型，但在文档或辅助类中明确枚举。
```csharp
public const string Team = "Team"; // 0=Neutral, 1=Player, 2=Enemy
```

### 3. 数据键迁移
- `Threat` (仇恨值) -> **移动到 `DataKey_AI`**（完全正确，这是 AI 组件的数据，不是 Unit 的固有属性）。
- `Hero`, `Boss` 等身份 -> **建议作为 Tag 或 bool 值**（如 `DataKey.IsBoss`, `DataKey.IsHero`）或单独的 `Role` 字段，而不是混入 `UnitType`。

## 总结
**“通用分类” (Universal Classification) 在 ECS 中通常是反模式。**
最好的做法是：
- **Type** 决定物理/组件结构（Technical Archetype）。
- **Team** 决定敌我关系（Gameplay Relation）。
- **Tag** 决定特殊行为（Gameplay Logic）。

三者组合，千变万化。
