## Evidence

### Godot 官方资料

- Godot FAQ 明确说明 Godot 不使用 ECS，而是依赖继承；同时允许通过带脚本的 child Node 在运行时添加和移除行为。来源：`https://docs.godotengine.org/en/stable/about/faq.html`
- Godot FAQ 对 DOD 的说明是：DOD 主要是 cache coherency 优化，在每帧处理数万对象且修改较少时才显著；少量或中等数量对象不应默认为 DOD 重写架构。来源：`https://docs.godotengine.org/en/stable/about/faq.html`
- Godot Design Philosophy 把 Object-oriented design、scene composition、Node hierarchy 作为核心，并说明 Godot node 不等同于其他引擎的 component。来源：`https://docs.godotengine.org/en/stable/getting_started/introduction/godot_design_philosophy.html`

### ECS 官方/主流资料

- Unity Entities 的 archetype 是相同 component 组合的实体集合；添加或移除 component 会移动实体到对应 archetype；同 archetype 数据存进 16KiB chunk，chunk 内 component 数组紧密排列。来源：`https://docs.unity3d.com/Packages/com.unity.entities@1.0/manual/concepts-archetypes.html`
- Bevy ECS 将程序拆成 Entity、Component、System；System 通过 Query 处理特定 component 集合，ECS 的收益包括数据/逻辑拆分、内存访问优化和更容易并行。来源：`https://bevy.org/learn/quick-start/getting-started/ecs/`
- Flecs 的设计目标强调 system 迭代速度、减少 cache miss、storage engine、模块和 deferred operations；遍历时修改实体需要 defer，避免直接改底层数据结构。来源：`https://www.flecs.dev/flecs/md_docs_2Manual.html`

### QFramework 资料

- Context7 `/liangxiegame/qframework` 显示 QFramework 核心示例围绕 `Architecture<T>` 注册 Model / System / Utility，并用 Command / Query / Event / BindableProperty 组织应用层逻辑。
- 本地源码核对路径：`/home/slime/Code/SlimeAI/Resources/Engine/Engine/QFramework/Source/QFramework.cs` 和 `Source/QFramework.Godot4+/QFramework.cs`。
- 本地源码中核心类型集中在 `IArchitecture`、`Architecture<T>`、`IModel`、`ISystem`、`IUtility`、`ICommand`、`IQuery`、`TypeEventSystem`、`BindableProperty<T>`。这些结构更像应用层分层、类型注册表和事件/命令工具，不是 Godot scene lifecycle、碰撞、对象池或 per-object Data runtime。

## Inference

- 用户对“真正 ECS 核心是 DOD / 数据结构 / cache-friendly storage，而不是 OOP 对象驱动”的判断成立。
- SlimeAI 当前的 Godot Node + 功能 Component + Service/System + DataOS 组合不是传统 ECS runtime。它没有实现 archetype/chunk/sparse-set storage、query cache、schedule dependency 和 command buffer 作为底层核心。
- Godot 已经提供 Node、SceneTree、scene composition、继承、生命周期、编辑器和运行时添加/移除节点能力；这些能力更贴合 SlimeAI 当前“快速 MVP + 功能解耦 + AI 可读验证”的目标。
- Data 系统过去被迫承担 ECS 核心数据结构职责，是方向误判带来的复杂度放大。弃用 ECS 后，Data 名字保留，但应按受控共享状态、表格驱动、DataBinding 和 DataModifier 收窄重新设计。
- QFramework 值得学习少规则分层、强类型 Model/Property、Command/Query/Event 和架构图式注册；但不适合直接作为 SlimeAI 底层框架依赖。

## Adopt Now / Later / Reject

### Adopt Now

- 弃用 ECS 作为 SlimeAI 框架身份。
- 使用 `SlimeAIFramework` 作为当前方向名。
- 以 Godot Node / OOP / scene composition 作为底层事实。
- 保留事件驱动、功能 owner、生命周期 guard、diagnostics 和验证链。
- Data 名字保留并按受控共享状态重新设计：默认状态留在 Component / Feature / System 内，证明需要共享、表格驱动或验证追踪后才进入 Data。

### Adopt Later

- 性能热点出现 profiler 证据后，局部学习 ECS 的数组、池、batch query、command buffer 或 data-oriented storage。
- 如果 Feature 读写混乱，再引入更明确的 Command/Query 命名和轻量协议。
- 如果 UI/debug binding 需求扩大，再借鉴 QFramework `BindableProperty<T>` 的初值通知和静默设置语义。

### Reject

- 不实现完整 ECS storage/query/schedule runtime。
- 不把 Godot Node tree 镜像成 ECS World。
- 不继续把统一 Data dictionary 当 ECS 核心。
- 不直接引入 QFramework 依赖，不复制 `Architecture<T>`、`IController`、全局 `TypeEventSystem` 或 `BindableProperty<T>` 作为 SlimeAI public API。
