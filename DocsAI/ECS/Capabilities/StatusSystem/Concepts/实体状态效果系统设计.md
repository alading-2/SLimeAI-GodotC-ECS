# 实体状态效果系统设计

> 文档类型：正式设计文档  
> 适用范围：实体状态效果、AI 协调、攻击/移动输入限制、无敌叠加  
> 当前状态：第一版已落地

---

## 1. 先说结论

项目级状态不用行为树。  
实体效果状态也不用单一大状态机。

本项目现在采用：

> `StatusDefinition + StatusInstance + StatusCollection + StatusSnapshot + StatusControllerComponent`

这套模型解决的是：

- 多来源无敌如何独立结束
- 多来源沉默、禁攻、禁移动输入如何叠加
- AI、攻击、主动移动如何统一读取“还能不能做”

它不解决：

- 项目级菜单/暂停/测试覆盖层
- 单个敌人的行为树决策

---

## 2. 过去的问题

旧问题集中在两个层面。

### 2.1 单个布尔值无法表示多来源

例如：

- 技能 A 给 2 秒无敌
- 技能 B 给 5 秒无敌

如果只写：

```csharp
Data.Set(GeneratedDataKey.IsInvulnerable, true);
```

那技能 A 结束时一旦写回 `false`，技能 B 的无敌也会被一起清掉。

这就是典型的“**来源丢失**”问题。

### 2.2 `MoveMode` 与行为能力语义混杂

以前一些调用会尝试从这些信号推断状态：

- `MoveMode`
- `AIEnabled`
- `IsMovementLocked`
- `LifecycleState`

但这些字段本来不回答同一个问题：

- `MoveMode` 回答的是“怎么移动”
- `LifecycleState` 回答的是“处于什么生死流程”
- `AIEnabled` 只是局部开关
- `IsMovementLocked` 只是速度/物理层限制

真正缺的是：

> 这个实体此刻还能不能思考、主动移动、攻击、施法？

---

## 3. 设计目标

1. 支持多来源状态独立生效、独立结束。
2. 支持把多个状态聚合成统一快照供系统消费。
3. 状态系统只处理“能力限制 / 免疫 / 保护”，不接管 AI 决策。
4. 明确区分“主动输入限制”和“强制位移”。
5. 与现有项目渐进兼容，不一次性推翻旧 `DataKey`。

---

## 4. 当前模型

### 4.1 `StatusDefinition`

文件：

- `Src/ECS/Capabilities/StatusSystem/System/StatusDefinition.cs`

职责：

- 定义一个状态的逻辑标识
- 定义显示名
- 定义该状态贡献哪些聚合标记

### 4.2 `StatusInstance`

文件：

- `Src/ECS/Capabilities/StatusSystem/System/StatusInstance.cs`

职责：

- 记录一次具体状态实例
- 区分 `SourceId`
- 区分 `StatusId`
- 记录持续时间

关键点：

- `SourceId + StatusId` 共同决定一个实例键
- 这保证不同来源的同名状态不会互相覆盖

### 4.3 `StatusCollection`

文件：

- `Src/ECS/Capabilities/StatusSystem/System/StatusCollection.cs`

职责：

- 存放当前所有状态实例
- 聚合所有 `StatusEffectFlags`
- 生成最终 `StatusSnapshot`

当前聚合策略：

- 同一实例键覆盖刷新
- 不同来源并行保留
- 快照按所有实例的 Flags 做按位并集

### 4.4 `StatusSnapshot`

文件：

- `Src/ECS/Capabilities/StatusSystem/System/StatusSnapshot.cs`

职责：

- 把一堆底层 Flags 收敛成系统可直接消费的判断结果

当前暴露属性：

- `CanThink`
- `CanMoveInput`
- `CanAttack`
- `CanCast`
- `IsInvulnerable`
- `IsControlImmune`
- `IsMovementLocked`
- `AllowsForcedMovement`

### 4.5 `StatusControllerComponent`

文件：

- `Src/ECS/Capabilities/Unit/Component/Common/StatusControllerComponent/StatusControllerComponent.cs`

职责：

- 挂在实体上，持有 `StatusCollection`
- 负责应用/移除状态实例
- 负责定时状态自动结束
- 把快照同步进 `Data`
- 触发统一状态变更事件

当前定时行为通过 `TimerManager` 实现，不直接创建 Godot 原生临时定时器。

## 5. 当前 Flags 语义

文件：

- `Src/ECS/Capabilities/StatusSystem/System/StatusEffectFlags.cs`

当前标记：

| Flag | 语义 |
| --- | --- |
| `SuppressAI` | 禁止 AI/逻辑继续思考 |
| `SuppressMoveInput` | 禁止主动移动输入 |
| `SuppressAttack` | 禁止发起攻击 |
| `SuppressCast` | 禁止技能进入施法流水线 |
| `Invulnerable` | 处于无敌态 |
| `ControlImmune` | 免疫控制类效果 |
| `LockMovement` | 锁定移动 |
| `AllowForcedMovement` | 明确允许强制位移继续生效 |

这里最重要的边界是：

> `SuppressMoveInput` 只禁止主动控制，不等于禁止所有轨迹位移。

所以：

- 击退、拉拽、脚本轨迹可以继续
- 玩家输入和 AI 输入应停止

## 6. 多来源无敌如何工作

这是本次状态系统重构最直接要解决的问题。

例如：

```text
source = skill-a, status = Invulnerable, duration = 2
source = skill-b, status = Invulnerable, duration = 5
```

当前行为：

1. 两个实例都会进入 `StatusCollection`
2. 聚合后 `StatusSnapshot.IsInvulnerable == true`
3. `skill-a` 结束时，只移除 `skill-a::Invulnerable`
4. 由于 `skill-b::Invulnerable` 仍存在，所以无敌继续保持
5. 直到最后一个来源移除，无敌才真正结束

这正是旧 `bool DataKey` 做不到的地方。

## 7. 与 AI、攻击、移动的协作关系

### 7.1 AI

文件：

- `Src/ECS/Capabilities/AI/Component/AIComponent/AIComponent.cs`

当前第一版规则：

- 若 `LifecycleState` 不允许，AI 不运行
- 若 `GeneratedDataKey.StatusCanThink == false`，AI 直接停止 Tick

也就是说：

> AI 行为树继续是决策层，但它现在会尊重状态层给出的“是否还能思考”结论。

### 7.2 攻击

文件：

- `Src/ECS/Capabilities/Unit/Component/Common/AttackComponent/AttackComponent.cs`

当前第一版规则：

- 攻击前验证阶段会读取 `GeneratedDataKey.StatusCanAttack`
- 若状态层判定不可攻击，则攻击请求被拒绝或取消

### 7.3 主动移动输入

文件：

- `Src/ECS/Capabilities/Movement/System/Strategies/Base/PlayerInputStrategy.cs`
- `Src/ECS/Capabilities/Movement/System/Strategies/Base/AIControlledStrategy.cs`

当前第一版规则：

- 若 `GeneratedDataKey.StatusCanMoveInput == false`，主动输入策略返回零速度

这意味着：

- 玩家输入停
- AI 主动走位停
- 但其他非输入型轨迹仍可继续存在

## 8. 与旧链路的兼容

当前 `StatusControllerComponent` 会把聚合结果同时写到：

### 8.1 新状态键

文件：

- `Data/DataKey/Unit/DataKey_Status.cs`

包括：

- `StatusCanThink`
- `StatusCanMoveInput`
- `StatusCanAttack`
- `StatusCanCast`
- `StatusIsInvulnerable`
- `StatusIsControlImmune`

### 8.2 旧兼容键

仍会同步：

- `GeneratedDataKey.IsInvulnerable`
- `GeneratedDataKey.IsMovementLocked`
- `GeneratedDataKey.IsStunned`

这样做的目的不是长期维持双写，而是为了给现有系统留迁移窗口。

---

## 9. 为什么这里不用行为树或项目级状态机

这里处理的是**效果叠加和能力限制**，不是决策流程。

行为树擅长：

- 条件判断
- 决策路径
- 行为选择

状态效果系统需要的是：

- 来源隔离
- 独立计时
- 标记聚合
- 快照同步

这两个问题根本不是一类。

同理，项目级状态机回答的是：

- 菜单
- 局内
- 暂停
- 覆盖层

实体状态系统回答的是：

- 这个单位此刻能不能思考/攻击/施法/主动移动

所以它们必须分开。

## 10. 当前建议用法

### 10.1 施加状态

推荐由业务系统创建 `StatusDefinition` 和 `StatusInstance`，再交给 `StatusControllerComponent`。

可以直接使用：

```csharp
statusController.ApplyStatus(definition, sourceId, durationSeconds);
```

或者：

```csharp
statusController.ApplyStatus(instance);
```

### 10.2 移除状态

按来源和状态 Id 精确移除：

```csharp
statusController.RemoveStatus(sourceId, statusId);
```

不要再直接只写一个总布尔值试图“关掉状态”。
