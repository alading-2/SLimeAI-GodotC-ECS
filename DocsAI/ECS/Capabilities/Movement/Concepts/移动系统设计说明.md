# 移动系统设计说明

## 一、系统定位

移动系统是**通用 Entity 运动执行层**，负责“怎么移动、何时停止、移动碰撞如何解释”，不负责“为什么移动”。

- **业务层**（技能 / AI / EffectTool）：决定使用哪种运动模式，构建 `MovementParams`
- **移动组件**（`EntityMovementComponent`）：持有参数、切换策略、执行位移、消费碰撞候选、统一停止收口
- **运动策略**（`IMovementStrategy`）：只负责轨迹与速度意图计算
- **碰撞策略子模块**（`MovementCollisionPolicy`）：负责过滤、去重、计数和生成碰撞上下文
- **停止协调子模块**（`MovementStopCoordinator`）：统一决定是否发完成事件、是否销毁、是否回退默认模式

这次重构的关键结论是：**不把移动逻辑拆成全局逐帧大系统，继续保持“组件编排 + 子模块”的局部模型。**

## 二、核心架构

### 2.1 参数传递

运动参数通过事件携带的 `MovementParams` 结构体传入：

```csharp
entity.Events.Emit(
    GameEventType.Unit.MovementStarted,
    new GameEventType.Unit.MovementStartedEventData(MoveMode.TargetPoint, new MovementParams
    {
        Mode = MoveMode.TargetPoint,
        TargetPoint = new Vector2(900, 360),
        MaxDistance = 300f, // 可选
        DestroyOnComplete = true, // 可选
    }));
```

`EntityMovementComponent` 将 `MovementParams` 存为私有字段，策略通过 `in MovementParams` 只读访问。

### 2.2 调度器职责

`EntityMovementComponent` 负责：

1. 监听 `MovementStarted` 切换策略
2. 调用策略 `Update` 计算本帧移动意图
3. 通过 `VelocityResolver` 合成最终速度并执行位移
4. 维护 `ElapsedTime / TraveledDistance`
5. 消费原始碰撞候选，交给 `MovementCollisionPolicy`
6. 统一走 `StopMovement(...)` 执行停止、完成事件、销毁、模式回退

### 2.3 策略实例化

每次切换模式时，注册表以工厂函数**新建策略实例**，不复用单例：

- 允许策略持有私有运行态（如 `_currentAngle`、缓存曲线）
- 禁止把外部可观察的业务状态偷偷藏在策略里

## 三、14 种运动模式

| MoveMode | 策略类 | 核心行为 | 关键 MovementParams 字段 |
|---|---|---|---|
| None | — | 不更新位置 | — |
| FixedDirection | FixedDirectionStrategy | 沿当前方向前进 | `ActionSpeed` / `MaxDistance` |
| TargetPoint | TargetPointStrategy | 冲向目标点 | `TargetPoint`, `ReachDistance` |
| TargetEntity | TargetEntityStrategy | 追踪目标实体 | `TargetNode`, `ReachDistance` |
| OrbitPoint | OrbitPointStrategy | 围绕固定点运动 | `OrbitCenter`, `OrbitRadius`, `OrbitAngularSpeed` |
| OrbitEntity | OrbitEntityStrategy | 围绕实体动态环绕 | `TargetNode`, `OrbitRadius`, `OrbitAngularSpeed` |
| Spiral | SpiralStrategy | 螺旋半径变化 | `OrbitCenter`, `OrbitRadius`, `OrbitTargetRadius` |
| SineWave | SineWaveStrategy | 正弦波形前进 | `WaveAmplitude`, `WaveFrequency` |
| BezierCurve | BezierCurveStrategy | 贝塞尔曲线运动 | `BezierPoints`, `MaxDuration`（业务层显式驱动时长） |
| Boomerang | BoomerangStrategy | 去程 -> 停顿 -> 返程 | `TargetPoint`, `TargetNode`, `Boomerang*` |
| AttachToHost | AttachToHostStrategy | 附着跟随宿主 | `TargetNode` |
| PlayerInput | PlayerInputStrategy | 玩家输入驱动 | 读 `DataKey.MoveSpeed` 等 |
| AIControlled | AIControlledStrategy | AI 意图驱动 | 读 `DataKey.AIMoveDirection` 等 |
| Parabola | ParabolaStrategy | 抛物线弹道（推荐固定终点） | `TargetPoint`, `ParabolaApexHeight` |
| CircularArc | CircularArcStrategy | 圆弧弹道（`BowWorldUp` 时自动选更朝上的弧） | `TargetPoint` / `TargetNode`, `CircularArcRadius` |

## 四、Velocity 分层合成

`VelocityResolver.Resolve(data)` 优先级从高到低：

```text
1. IsMovementLocked = true  → Vector2.Zero
2. VelocityOverride ≠ Zero  → VelocityOverride
3. 否则                      → Velocity + VelocityImpulse
```

## 五、停止语义

### 5.1 停止原因

`MovementStopReason` 当前包含：

- `Completed`
- `Collision`
- `Requested`
- `Interrupted`
- `ComponentUnregistered`

### 5.2 统一停止入口

组件不再把“碰撞”“自然完成”“外部打断”分别写死成多条链路，统一走：

```csharp
StopMovement(
    reason,
    emitCompletedEvent,
    requestedNextMode,
    collisionTarget,
    destroyEntity);
```

### 5.3 `MovementStopRequested`

外部系统或内部碰撞策略都通过事件请求停止：

```csharp
entity.Events.Emit(
    GameEventType.Unit.MovementStopRequested,
    new GameEventType.Unit.MovementStopRequestedEventData
    {
        Reason = MovementStopReason.Requested,
        EmitCompletedEvent = true,
        NextMode = MoveMode.None,
        CollisionTarget = null,
        DestroyEntity = false
    });
```

字段说明：

- `EmitCompletedEvent`：是否发 `MovementCompleted`
- `NextMode = MoveMode.None`：表示沿用默认回退逻辑
- `DestroyEntity`：是否在停止后直接销毁实体

### 5.4 `MovementCompleted`

完成事件现在显式携带停止原因：

```csharp
new GameEventType.Unit.MovementCompletedEventData(
    mode,
    elapsedTime,
    traveledDistance,
    reason,
    collisionTarget);
```

这解决了旧版本“外部请求停止后事件看起来像自然完成”的歧义。

## 六、移动碰撞语义重构

### 6.1 旧问题

旧链路把 3 件事写死成一件事：

1. 采样到碰撞
2. 通知业务
3. 直接结束移动并可选销毁

结果是：

- `ArcShot` 碰到任意敌人/玩家就提前结束
- 无法支持“穿透 2 个敌人后再销毁”
- 无法表达“只通知、不停止”
- 无法表达“只对锁定目标计数”

### 6.2 新分层

新链路固定拆为：

```text
RawCollision
  -> CollisionFilter
  -> CollisionNotify
  -> CollisionStop
```

含义分别是：

1. **RawCollision**：原始碰撞候选，由 `CollisionComponent` 或 `MoveAndSlide()` 提供
2. **CollisionFilter**：`MovementCollisionPolicy.TryAccept(...)`
3. **CollisionNotify**：执行 `OnCollision` 回调和可选的 `MovementCollision` 事件
4. **CollisionStop**：只有命中自动停止阈值时才发 `MovementStopRequested`

### 6.3 `MovementCollisionParams`

旧字段 `DestroyOnCollision` 已废弃，改为：

```csharp
Collision = new MovementCollisionParams
{
    TeamFilter = TeamFilter.Enemy,
    EntityTypeFilter = EntityType.Unit,
    TargetMatchMode = MovementCollisionTargetMatchMode.Any,
    StopAfterCollisionCount = 1,
    DestroyOnStop = true,
    EmitCollisionEvent = true,
    OnCollision = ctx => { }
};
```

核心字段语义：

- `StopAfterCollisionCount = -1`：只通知不停止
- `DestroyOnStop`：只有在“已决定停止”后才决定是否销毁
- `TargetMatchMode = TrackedTargetOnly`：只接受当前追踪目标

### 6.4 `MovementCollisionEventData`

有效碰撞事件字段：

- `Mode`
- `Target`
- `TargetEntity`
- `CollisionCount`
- `WillStop`

因此技能层可以直接使用 `evt.TargetEntity`，不必每次自己回溯宿主实体。

### 6.5 `CharacterBody2D` 为什么仍然要手动采样

这是这次设计里最容易误解的点。

结论非常明确：

- `CollisionComponent` 绑定的 4 个 entered/exited 事件，只对 `Area2D` 路径成立
- `CharacterBody2D` 没有这组事件可以直接拿来复用
- `CharacterBody2D` 的标准碰撞读取方式就是 `MoveAndSlide()` 后读取 slide collision

所以“在 `ApplyMovement()` 里手动读取 `GetSlideCollision(i)`”不是架构错误，错误的是旧实现把它和“停止/销毁”直接绑死。

现在正确做法是：

- 保留 slide collision 作为原始候选
- 把后续业务决策交给 `MovementCollisionPolicy`

### 6.6 去重规则

同一目标在同一次运动内只计数一次：

- 优先用宿主实体实例 ID 去重
- 没有宿主实体时回退到碰撞节点实例 ID

这解决了 `CharacterBody2D` 推墙或持续贴脸时每帧重复计数的问题。

## 七、ArcShot 修正

`ArcShot` 的正确语义是“追踪特定目标并自然到达后命中”，不是“碰到任意单位就结束”。

因此现在固定为：

- 使用 `MoveMode.CircularArc`
- `isTrackTarget = true`
- `ReachDistance > 0`
- 不配置 `Collision`
- 在 `MovementParams.OnStop` 中只对 `Reason == Completed` 结算伤害

这意味着：

- 路径中撞到其他敌人/玩家不会提前结束
- 只有真正追到锁定目标时才执行命中逻辑

## 八、技能迁移规则

### 8.1 命中即停投射物

- `SineWaveShot`
- `BezierShot`

统一写法：

- `Collision.TeamFilter = Enemy`
- `Collision.EntityTypeFilter = Unit`
- `Collision.StopAfterCollisionCount = 1`
- `Collision.DestroyOnStop = true`

### 8.1.1 固定落点抛炸弹

- `ParabolaShot`

当前语义：

- 使用 `MoveMode.Parabola`
- 只写固定 `TargetPoint`，不再追踪 `TargetNode`
- `AbilityTriggerMode = Periodic`，由 `AbilityCooldown` 决定投弹间隔
- 在 `MovementParams.OnStop` 中只对 `Reason == Completed` 执行落地范围伤害与特效

说明：

- `ParabolaStrategy` 数学上是固定起点到固定终点的抛物线
- 参数层虽然兼容 `TargetNode / isTrackTarget`，但游戏内标准投射物不推荐这么用
- 追踪目标时终点与曲线会动态重算，视觉上不稳定，更适合调试或特殊位移

### 8.2 持续接触 / 不因碰撞结束

- `BoomerangThrow`
- `OrbitSkill`
- `AuraShield`

统一写法：

- `Collision.TeamFilter = Enemy`
- `Collision.EntityTypeFilter = Unit`
- `Collision.StopAfterCollisionCount = -1`
- `Collision.DestroyOnStop = false`

## 九、为什么不改成全局 Movement System

原因已经很清楚：

- 当前碰撞来源天然是局部物理事件，不需要全局逐帧扫描
- 若拆成全局系统，需要额外同步当前策略、当前参数、当前碰撞计数、当前默认模式
- 复杂度上升，但收益有限

所以当前采用的是折中但正确的方式：

- **组件保留编排权**
- **复杂逻辑拆成子模块**
- **不做全局大系统**

这既保住了局部状态的低成本持有，也把原先堆在组件里的硬编码判断拆散了。
