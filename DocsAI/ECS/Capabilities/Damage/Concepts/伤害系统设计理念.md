# 伤害系统深度设计方案 (Damage System Design)

## 1. 设计愿景与核心理念

作为核心战斗系统的基石，伤害系统不仅仅是“扣血”那么简单。它需要承载：
- **复杂性解耦**：将暴击、闪避、护盾、伤害减免等逻辑拆分为独立模块（Middleware），避免“面条代码”。
- **数据驱动**：深度集成现有的 `Data` 容器，所有数值（暴击率、护甲等）均源自 Data，支持动态修改（Buff/Debuff）。
- **可扩展性**：通过注册机制支持未来随时插入新规则（如“斩杀逻辑”、“元素反应”）。
- **伤害统计追溯**：通过 `IUnit` 接口准确追踪伤害来源，无论伤害是由子弹、陷阱还是 Buff 造成的。

---

## 2. 核心架构：Pipeline 模式（管道模式）

采用 **Pipeline（管道）+ Middleware（中间件）** 的架构。

### 2.1 核心组件

1.  **DamageInfo (伤害上下文)**：承载单次伤害信息的上下文类。
2.  **IDamageProcessor (中间件接口)**：定义处理伤害的标准接口。
3.  **DamageService (入口服务)**：全局单例，管理 Pipeline 并提供对外接口。

### 2.2 新增接口：IUnit (单位接口)

为了解决“子弹造成伤害，但需要统计到玩家身上”的问题，引入 `IUnit` 接口。
所有具有独立战斗归属的主体（玩家、敌人）必须实现此接口。

```csharp
public interface IUnit
{
    // 阵营标识，用于友军伤害判定
    int FactionId { get; } 
}

// 示例：Projectile 并不是 Unit，但它持有 Owner (IUnit)
// 伤害计算时，从 Projectile.Owner 获取真正的施法者
```

## 3. 数据结构定义

### 3.1 DamageInfo (伤害上下文)

```csharp
public class DamageInfo
{
    // === 基础信息 ===
    public Guid Id { get; } = Guid.NewGuid();
    
    // 伤害的直接来源（可能是子弹、陷阱 Area2D、或者近战武器 Area2D）
    public Node Attacker { get; set; }
    
    // 伤害的始作俑者（真正的凶手，用于统计伤害）
    // 如果 Attacker 是 Unit，则 Instigator == Attacker
    // 如果 Attacker 是子弹，则 Instigator == Attacker.Owner
    public IUnit? Instigator { get; set; }
    
    // 受害者实体
    public Node Victim { get; set; }

    // === 数值信息 ===
    public float BaseDamage { get; set; }    // 原始面板伤害
    public float FinalDamage { get; set; }   // 最终结算伤害
    
    // === 标签与类型 ===
    public DamageType Type { get; set; }
    public DamageTags Tags { get; set; }
    
    // === 状态标记 ===
    public bool IsCritical { get; set; }
    public bool IsDodged { get; set; }
    public bool IsBlocked { get; set; }
    
    // === 辅助数据 ===
    public List<DamageLog> Logs { get; } = new();
}
```

## 4. 伤害计算流程 (Pipeline)

修正后的标准结算顺序：**输出增幅 -> 生存判定 -> 护盾抵扣 (Raw) -> 受击减免 -> 生命扣除**。

### 阶段一：输出修正 (Outgoing Modification)
*基于攻击来源的属性增强伤害。*

1.  **BaseDamageProcessor**: 确定基础伤害。
    - 优先读取 `Attacker` 携带的伤害参数。
    - 若 `Instigator` 存在，应用其全局伤害加成。
2.  **DamageAmplificationProcessor**: 伤害增幅。
    - 处理“全局伤害增加%”、“近战/远程伤害增加%”等词条。
3.  **CritProcessor (暴击)**: 
    - 读取 `Instigator` 的 `CritChance`。
    - 若暴击，`FinalDamage *= CritDamage`。

### 阶段二：生存判定 (Survival Check)
*判定伤害是否能够命中。*

4.  **DodgeProcessor (闪避)**:
    - 读取 `Victim.Data["DodgeChance"]`。
    - 若闪避成功，`FinalDamage = 0`，标记 `IsDodged`，**终止流程**。

### 阶段三：护盾抵扣 (Shield Absorption)
*护盾优先于护甲生效，承受原始伤害（不享受护甲减免）。*

5.  **ShieldProcessor (护盾)**:
    - 优先扣除 `Victim` 的临时护盾值。
    - 护盾承受的是 **未经过护甲减免的原始伤害** (`FinalDamage`)。
    - 若护盾足够，`FinalDamage` 归零；若护盾破碎，溢出的伤害进入下一阶段。
    - **设计注意**：必须在减伤前计算护盾。若在减伤后计算，说明护盾也有减伤效果，这不符合设计预期。

### 阶段四：受击减免 (Incoming Mitigation)
*基于受害者的防御属性减少剩余伤害。*

6.  **DefenseProcessor (护甲/魔抗)**:
    - 物理伤害应用 `Armor` 公式。
    - 魔法伤害应用 `MagicResist` 公式。
    - 对 **护盾抵扣后剩余的伤害** 进行减免。
7.  **DamageTakenAmplificationProcessor (增伤/减伤修正)**:
    - 读取 `Victim.Data["DamageTakenMultiplier"]`。
    - 处理“易伤”效果（如受到的伤害增加 20%）。
8.  **FlatReductionProcessor (固定减伤)**:
    - 最后的硬减伤（如“格挡 10 点伤害”）。

### 阶段五：最终结算 (Final Execution)
*实际扣除数值。*

9.  **HealthExecutionProcessor**:
    - 调用 `Victim.HealthComponent.ModifyHealth(-FinalDamage)`。
    - **注意**：`HealthComponent` 不再包含 `TakeDamage` 逻辑，只负责加减血和死亡判定。
10. **LifestealProcessor (吸血)**:
    - 若伤害有效且 > 0。
    - 读取 `Instigator.Data["Lifesteal"]`。
    - 治疗 `Instigator`。
11. **StatisticsProcessor**:
    - 记录 `Instigator` 造成的伤害总量（用于面板统计）。

## 5. 实现细节与 Data 集成建议

### 5.1 注册机制与优先级

使用特性（Attribute）或显式注册来定义优先级。

```csharp
public interface IDamageProcessor
{
    int Priority { get; } // 越小越先执行
    void Process(DamageInfo info);
}

// 示例：护甲计算
public class ArmorProcessor : IDamageProcessor
{
    public int Priority => 300; // 阶段三
    
    public void Process(DamageInfo info)
    {
        if (info.IsDodged || info.Type == DamageType.True) return;

        var armor = info.Victim.GetData().Get<float>("Armor", 0);
        // 土豆兄弟公式：减伤% = Armor / (Armor + Constant)
        // 假设 constant = 15
        if (armor > 0) 
        {
            float reduction = armor / (armor + 15.0f);
            info.FinalDamage *= (1.0f - reduction);
        }
    }
}
```

## 6. 组件职责重构

### 6.1 HealthComponent (生命组件)

**职责**：纯粹的生命值容器与致死判定。

- **职责扩展**：`ApplyDamage()` 方法（从原 DamageComponent 移至此处）。
- **保留**：`CurrentHp`, `MaxHp`。
- **发送事件**：`Damaged`, `HealthChanged`, `Kill`。

## 7. Prompt 提示词更新

*以下是更新后的提示词，用于指导 AI 生成代码：*

> "请基于 Pipeline 模式实现 DamageSystem。
> 1. 定义 `IUnit` 接口（ FactionId）。
> 2. 定义 `DamageInfo` 类，包含 `Attacker` (Node) 和 `Instigator` (IUnit)。
> 3. 实现核心 Processors：Crit（读取 Instigator 暴击率）、Shield（优先扣盾）、Armor（读取 Victim 护甲）。
> 4. 实现 `DamageService` 单例，管理 Pipeline。
> 5. 重构建议：`HealthComponent` 只保留 `ModifyHealth` 接口，移除 `TakeDamage` 及其内部计算逻辑，所有伤害计算逻辑移至 `DamageSystem`。"
