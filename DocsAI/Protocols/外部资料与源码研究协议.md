# 外部资料与源码研究协议

本文定义 AI 在做框架级重构、复杂系统设计、底层 Debug 和玩法系统实现前，如何搜索、clone、阅读和沉淀外部资料。

## 触发条件

必须执行本协议：

- 修改 Runtime、Schedule、Relationship、DataOS、Validation、Observation 等框架核心。
- 处理 Godot 物理、渲染、输入、资源导入等引擎底层问题。
- 实现复杂玩法系统，且项目内没有稳定样板。
- AI 连续 Debug 多轮仍无法定位根因。
- 用户要求“广泛搜索”“clone 源码分析”“参考成熟框架”。

## 研究原则

- 优先 clone 源码到临时目录，再用 `rg / find / tree / git log` 分析。
- 官方文档用于确认概念，源码用于确认真实边界。
- 不把外部框架复杂度直接搬进项目，只抽取对当前问题有验证收益的机制。
- 研究产物必须写入计划或 DocsAI，不依赖聊天上下文。
- 外部仓库只作为参考，不引入依赖，除非任务明确要求并经过人工确认。

## 推荐资料类型

- 成熟 ECS：Bevy、Flecs、Unity Entities、Arch、DefaultEcs 等。
- Godot 源码：本地 `/home/slime/Code/SlimeAI/Engine/godot-4.6.2-stable`。
- 官方 issue / PR：用于确认底层问题、设计权衡和已知限制。
- 成熟游戏项目：用于拆解玩法数据、测试场景和内容组织。

## 标准流程

1. 明确研究问题，不要泛泛搜索。
2. 列出候选资料和选择理由。
3. clone 到 `/tmp/<project>-research/` 或同等临时目录。
4. 用 `find` 查看目录，用 `rg` 定位关键概念。
5. 阅读源码入口、测试、examples、benchmarks。
6. 写 `ResearchBrief.md` 或更新当前计划。
7. 把可落地结论转成项目决策、任务、验证命令。
8. 清楚标记“不照搬”的原因。

## ResearchBrief 模板

```text
研究问题：
候选项目：
阅读入口：
源码证据：
可借鉴点：
不能照搬点：
本项目决策：
需要新增的测试/日志/工具：
参考链接：
```

## 输出要求

最终回复或计划文件必须说明：

- 搜索了什么。
- clone 了什么。
- 读了哪些源码目录。
- 结论如何影响本项目。
- 还有哪些不确定，需要后续验证。

## 禁止事项

- 禁止只贴外部框架概念，不说明本项目怎么落地。
- 禁止把外部项目的目录结构原样搬来。
- 禁止为了“看起来成熟”引入新依赖。
- 禁止把研究结果只留在聊天里。
