# 角色系统详解（Actors）

## 概述

SystemAgent 定义了 15 个角色（Actor），每个角色有：
- **职责定义**：做什么、不做什么
- **输入要求**：需要读什么
- **输出格式**：产出什么
- **评分标准**：PASS/FAIL 的判定条件
- **禁止行为**：绝对不能做的事

## 角色分类

```
角色体系
├── Pipeline（流水线角色）
│   ├── Planner —— 计划生成
│   └── Implementer —— 代码实现
├── Authoring（创作角色）
│   ├── DeepThink —— 深度分析、方向确认
│   ├── TestDesigner —— 测试场景设计
│   └── Documentarian —— 文档维护
├── Review（审查角色）
│   ├── Reviewer —— 门禁审查
│   ├── DesignCritic —— 设计评审
│   ├── SeniorGameDeveloper —— 游戏设计审查（条件触发）
│   └── SeniorProgrammer —— 架构审查（条件触发）
├── Analysis（分析角色）
│   ├── Debugger —— 问题诊断
│   └── ResearchAnalyst —— 外部研究
├── Readiness（就绪角色）
│   └── Verifier —— 完成验证
└── Sprint（冲刺角色）
    └── Retrospective —— 复盘分析
```

## 详细角色说明

### DeepThink（深度分析）

**职责**：在计划冻结前进行方向确认和深度分析
**触发条件**：方向不确定、需要探索多个方案、用户要求深度分析
**输入**：用户问题、相关代码和文档、SDD 设计
**输出**：问题分析、证据搜索、缺口识别、用户确认建议
**关键能力**：
- 用 SystemAgent 思想分析当前状态
- 识别方向性错误
- 搜索多源证据
- 提出需要用户确认的问题

### Planner（计划生成）

**职责**：将需求转化为可执行计划
**输入**：用户需求、SDD 设计、现有代码
**输出**：执行计划（任务列表、影响分析、SDD 推荐）
**评分标准**：
- PASS：任务粒度合理、路径引用真实、能力对齐
- FAIL：任务模糊、路径不存在、超出能力范围

### TestDesigner（测试场景设计）

**职责**：将需求转化为标准答案和验证场景
**输出格式（5 字段标准答案）**：

| 字段 | 含义 | 示例 |
|------|------|------|
| expectedInputs | 输入条件 | `new EntityRegistry()` + 3 个 probe entity |
| expectedObservations | 预期观察 | entity count = 3, spawn log = 3 entries |
| passCriteria | 通过条件 | `count == 3 && all logs are Info severity` |
| failCriteria | 失败条件 | `count != 3 \|\| any log is Error` |
| artifactPath | 产物路径 | `.ai-temp/scene-tests/artifacts/entity-spawn.json` |

**关键约束**：TestDesigner 只设计场景，不写测试代码。测试代码由 Implementer 编写。

### Implementer（代码实现）

**职责**：按计划做最小范围的代码/文档/配置修改
**关键约束**：
- 遵循 TDD 微循环（RED → GREEN → REFACTOR）
- 不超出计划范围
- 不修改同步副本
- 不做额外逻辑（GREEN 阶段只写让测试通过的最少代码）

### Reviewer（门禁审查）

**职责**：基于审查门进行结构化审查
**审查维度**：
- 方向正确性
- 范围边界
- 遗漏检测
- 回归风险
**输出**：每个门的判定（APPROVE/CONCERNS/REJECT）

### Verifier（完成验证）

**职责**：确认完成声明有可复现的证据
**判定**：
- **READY**：有完整可复现证据
- **NEEDS_WORK**：证据不完整或不可复现
- **BLOCKED**：有阻塞项需要解决

### Debugger（问题诊断）

**职责**：基于证据的根因分析和最小修复设计
**输入**：错误日志、复现步骤、相关代码
**输出**：根因分析、最小修复方案

### Retrospective（复盘分析）

**职责**：完成前的缺口检查、事实源更新、效率分析
**关键输出**：
- 结论/判定一致性检查
- 基于证据的发现
- 流程改进建议
- 效率分析（验证循环、文件读取放大、skill 重载、工具失败分类）
- `sessionEvidence`（使用 session-adapter 的结构化证据）

**效率分析维度**：
- 验证循环次数（同一点反复验证 = 信号不清）
- 文件读取放大（同文件反复读 = 入口不明确）
- Skill 重载次数（同 skill 反复加载 = 记忆不足）
- 工具失败分类（path_error / build_failure / command_misuse 等）

### DesignCritic（设计评审）

**职责**：计划冻结前的批判性设计审查
**审查维度**：
- 假设可见性（哪些假设可能不成立）
- 替代方案检查（是否考虑了其他方案）
- 风险识别

### Documentarian（文档维护）

**职责**：SystemAgent/DocsAI/SDD/index 一致性维护

### ResearchAnalyst（外部研究）

**职责**：将外部资料转化为可采纳/可拒绝的证据

### SeniorGameDeveloper / SeniorProgrammer（条件审查角色）

**触发条件**：多系统交互、架构变更等高风险场景
**职责**：分别从游戏设计和程序架构角度审查

## 角色与 Skill 的映射

每个角色（除 Implementer 外）都有对应的 Skill 入口。Skill 是用户触发角色的方式：

```
用户说 "/systemagent-deepthink" → 触发 DeepThink 角色
用户说 "/systemagent-planner" → 触发 Planner 角色
用户说 "/systemagent-new-feature-workflow" → 触发完整 NewFeature 工作流
```

**注意**：Implementer 没有独立 Skill，因为它由 Planner 的计划直接驱动，在工作流中执行。
