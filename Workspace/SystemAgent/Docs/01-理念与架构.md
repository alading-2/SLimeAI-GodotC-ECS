# SystemAgent 理念与架构

## 设计哲学

SystemAgent 的设计基于一个核心观察：**AI 的能力上限不是模型能力，而是流程约束。**

没有流程约束的 AI：
- 会跳过读文档直接写代码
- 会改了代码不验证
- 会验证通过不复盘
- 会复盘完不更新文档
- 会改了同步源不跑同步脚本

SystemAgent 通过五层约束解决这些问题。

## 五层洋葱模型

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 5: Retrospective（复盘层）                              │
│  每个工作流结束前强制复盘：效率分析、缺口检查、事实源更新        │
│                                                              │
│  ┌──────────────────────────────────────────────────────────┐ │
│  │  Layer 4: Review Gates（审查门层）                         │ │
│  │  8 个结构化审查门，每个有触发条件、上下文、提示词、判定词     │ │
│  │                                                          │ │
│  │  ┌──────────────────────────────────────────────────────┐│ │
│  │  │  Layer 3: Actors（角色层）                             ││ │
│  │  │  15 个角色定义：职责、输入、输出、评分标准、禁止行为      ││ │
│  │  │                                                      ││ │
│  │  │  ┌──────────────────────────────────────────────────┐││ │
│  │  │  │  Layer 2: Routes（流程层）                         │││ │
│  │  │  │  6 个执行工作流：阶段定义、角色顺序、门顺序          │││ │
│  │  │  │                                                  │││ │
│  │  │  │  ┌──────────────────────────────────────────────┐│││ │
│  │  │  │  │  Layer 1: Rules（约束层）                      ││││ │
│  │  │  │  │  10 个行为约束：哲学、边界、Git、子代理、TDD...  ││││ │
│  │  │  │  └──────────────────────────────────────────────┘│││ │
│  │  │  └──────────────────────────────────────────────────┘││ │
│  │  └──────────────────────────────────────────────────────┘│ │
│  └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

每一层的作用：
- **Layer 1 (Rules)**：定义"什么不能做"——底线约束
- **Layer 2 (Routes)**：定义"按什么顺序做"——流程编排
- **Layer 3 (Actors)**：定义"谁做什么"——职责分离
- **Layer 4 (Review Gates)**：定义"做的质量如何"——质量保障
- **Layer 5 (Retrospective)**：定义"怎么做得更好"——持续改进

## 信息架构

SystemAgent 的信息架构遵循一个原则：**一个事实源，多个同步副本**。

### 事实源层级

```
事实源（只在这里改）
├── Workspace/SystemAgent/     ← Route/Actor/Rule/Gate 定义
├── .ai-config/skills/         ← Skill 定义（唯一可维护源）
├── .ai-config/rules/          ← Rule 定义
└── Workspace/SDD/             ← SDD CLI/模板/校验规则

同步副本（由脚本生成，禁止直接修改；目标见 .ai-config/sync-targets.json）
├── .claude/skills/            ← skill 副本
├── .codex/skills/             ← skill 副本
├── .trae/skills/              ← skill 副本
├── AGENTS.md                  ← rule 副本
├── CLAUDE.md                  ← rule 副本
└── .trae/rules/               ← rule 副本

运行配置（直接维护，不走同步）
├── .claude/settings.json      ← Claude hook
├── .claude/agents/            ← Claude subagent
├── .codex/hooks.json          ← Codex hook

```

## Skill 体系

Skill 是 SystemAgent 的用户入口。分为两类：

### Capability Skills（12 个，独立角色入口）

| Skill | 角色 | 用途 |
|-------|------|------|
| systemagent-deepthink | DeepThink | 方向确认、深度分析 |
| systemagent-design-critic | DesignCritic | 设计评审 |
| systemagent-design-document | DeepThink | 设计文档生成 |
| systemagent-planner | Planner | 需求分析、计划生成 |
| systemagent-test-designer | TestDesigner | 测试场景设计 |
| systemagent-reviewer | Reviewer | 门禁审查 |
| systemagent-verifier | Verifier | 完成验证 |
| systemagent-debugger | Debugger | 问题诊断 |
| systemagent-research-analyst | ResearchAnalyst | 外部研究 |
| systemagent-retrospective | Retrospective | 复盘分析 |
| systemagent-skill-test | — | Skill 静态检查 |
| systemagent-code-review | — | 代码审查（需求对比） |

### Workflow Skills（6 个，完整工作流入口）

| Skill | Route | 用途 |
|-------|-------|------|
| systemagent-new-feature-workflow | NewFeature | 新功能/重构/迁移 |
| systemagent-debug-fix-workflow | DebugFix | Bug 修复 |
| systemagent-workflow-iteration | WorkflowIteration | 改进 SystemAgent 自身 |
| systemagent-config-maintenance-workflow | ConfigMaintenance | 配置维护 |
| systemagent-research-adoption-workflow | ResearchAdoption | 外部研究采纳 |
| systemagent-validation-release-workflow | ValidationRelease | 验证发布 |

### Skill 设计原则

1. **Workflow skill 是薄入口**：不复制 Route/Actor/Rule/Gate 正文，只引用事实源
2. **Capability skill 是角色入口**：定义触发条件、必读文件、输出要求、禁止行为
3. **同步副本由脚本生成**：`bash Workspace/Tools/ai-config-sync/sync-ai-config.sh`
4. **禁止修改同步副本**：改 `.ai-config/skills/` 中的源文件

## Subagent 体系

Subagent 是 SystemAgent 的可选执行层。

### 当前注册的 Subagent

| 名称 | 角色 | 配置 | 状态 |
|------|------|------|------|
| systemagent-planner | Planner | `.claude/agents/` + `.codex/agents/` | 活跃 |
| systemagent-test-designer | TestDesigner | `.claude/agents/` + `.codex/agents/` | 活跃 |
| systemagent-reviewer | Reviewer | `.claude/agents/` + `.codex/agents/` | 活跃 |
| systemagent-retrospective | Retrospective | `.claude/agents/` + `.codex/agents/` | 活跃 |

### Subagent 约束

- **默认只读**：Subagent 禁止写文件，所有写操作由主对话执行
- **结构化输出**：必须包含 Scope/Evidence/Inference/Unknown/Risks/Recommended Main-Thread Action/Files Touched
- **没有 Dispatcher**：并行写操作未实现，需要 SDD 设计 work package 格式、文件所有权、worktree 策略等

## Hook 体系

Hook 是 SystemAgent 的主动提醒机制。

### 当前状态：**暂停**

原因：`systemagent-hook.py` 被暂停，等待 SystemAgent hook 策略重新设计。

### 可用节点

| Hook 触发 | 作用 |
|-----------|------|
| SessionStart | 提醒当前 SDD 状态、State、明显待办 |
| FirstEdit | 首次编辑时提醒 must-read 和事实源边界 |
| PostValidation | build/test/validate 后提醒是否需要 Code Review / Verifier |
| PreStop | 会话结束前提醒 progress、retrospective 和 git status |

### 为什么暂停

旧 Hook 设计把提醒和检查混在一起，问题是：
1. **触发频率太高**：PostToolUse 在每次工具调用后触发，干扰正常工作流
2. **判定逻辑不够智能**：无法区分"探索性读取"和"实质性修改"
3. **与 lean review mode 冲突**：lean 模式想减少开销，但 hook 强制全量检查

当前裁决：Hook 暂不启用。未来只作为低噪音提醒器，不替代 Reviewer、Verifier、Code Review 或 SDD validate。

## AI-First 的本质

"AI-first"不是让 AI 更聪明，而是让 AI **更容易做对、更难做错**：

| 传统方式 | AI-first 方式 |
|----------|---------------|
| AI 自己判断该读什么文档 | must-read 入口链强制指定 |
| AI 自己决定什么时候验证 | Review Gate 强制验证 |
| AI 自己记住改了什么 | SDD progress.md 维护 State / Decisions / Validation |
| AI 自己判断测试是否通过 | ValidationSession + artifact 强制证据 |
| AI 自己决定什么时候复盘 | Retrospective 强制复盘 |

核心思想：**不信任 AI 的记忆和判断，只信任结构化的事实源和可检查的 artifact。**

## Control Plane vs Evidence Plane

为了避免把所有 AI-first 组件混成一团，当前推荐把工程层拆成两个平面：

```text
AI-first 工程层
├── Control Plane
│   ├── Workspace/SystemAgent/
│   ├── SDD
│   ├── Skill / Review Gate / Retrospective
│   └── 负责：任务入口、角色、流程、门禁、复盘
└── Evidence Plane
    ├── DocsAI
    ├── Log / Validation / Test
    ├── artifact / logctl
    └── 负责：运行事实、验证结果、离线分析、证据恢复
```

- **SystemAgent** 属于 control plane：定义 AI 怎么做事、怎么验证、怎么复盘。
- **Log / Validation / Test** 不属于 SystemAgent 本体，但属于它依赖的 evidence plane：负责让 DebugFix、Reviewer、Verifier、Retrospective 有可复查证据，而不是依赖 AI 猜测。
- **DocsAI** 横跨两边：既是框架事实源入口，也是证据与设计的索引层。

这个分层尤其重要，因为当前 Log 只能说“记录层 + 离线整理层第一版已落地”；源码调用点语义化仍未完成，不能再把 analyzer 完成误说成整个 Log 已完成。
