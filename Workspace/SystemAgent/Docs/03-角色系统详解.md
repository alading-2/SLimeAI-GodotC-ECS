# 角色系统详解（Actors）

> 本文档是 02-工作流详解.md 中角色部分的扩展说明。每个角色的完整定义在 `Workspace/SystemAgent/Actors/*.md` 中，此处只提供概览和关键设计决策。

## 角色设计原则

### 1. 职责单一

每个角色只做一件事。违反示例：
- ❌ Planner 同时写代码 → 应由 Implementer 做
- ❌ Reviewer 同时修复问题 → 应由 Implementer 修复后再 Review
- ❌ TestDesigner 同时运行测试 → 应由 Implementer 运行

### 2. 输出结构化

每个角色的输出必须有明确的结构，不能是自由文本。

**Planner 输出**：
```
- 任务列表（编号、描述、依赖、验证命令）
- 影响分析（受影响模块、风险评估）
- SDD 推荐（是否需要 SDD、SDD 范围）
```

**TestDesigner 输出**：
```
- 标准答案（5 字段：expectedInputs, expectedObservations,
  passCriteria, failCriteria, artifactPath）
- 最小验证项（哪些必须验证、哪些可以跳过）
```

**Reviewer 输出**：
```
- 每个 Gate 的判定（APPROVE / CONCERNS / REJECT）
- 判定理由（证据引用）
- CONCERNS/REJECT 的改进建议
```

**Verifier 输出**：
```
- READY / NEEDS_WORK / BLOCKED
- 证据清单（路径、行号、命令、输出）
- 阻塞项列表（如果是 BLOCKED）
```

**Retrospective 输出**：
```
- 结论/判定一致性发现
- 效率分析（验证循环、文件放大、工具失败）
- 流程改进建议
- sessionEvidence（结构化会话证据）
```

### 3. 禁止行为明确

每个角色都有明确的"不能做什么"：

| 角色 | 关键禁止行为 |
|------|-------------|
| Planner | 不能直接写代码 |
| TestDesigner | 不能直接写测试代码 |
| Implementer | 不能超出计划范围、不能改同步副本 |
| Reviewer | 不能修复问题（只审查不修） |
| Verifier | 不能接受"应该能通过"作为证据 |
| Debugger | 不能不做复现就修复 |
| Retrospective | 不能复制 Route/Actor/Rule/Gate 正文 |
| ResearchAnalyst | 不能直接复制外部代码 |
| DesignCritic | 不能阻止决策（只提供视角） |
| Documentarian | 不能改变事实源边界 |

## 当前角色体系的问题

### 1. 角色过多导致的认知负担

15 个角色对 AI 来说是很大的认知负担。在 lean 模式下，很多角色实际上不会被单独触发，而是被合并到工作流中的某个阶段。

**建议**：区分"独立触发"角色和"工作流内嵌"角色。大部分角色应该是工作流内嵌的，只有 DeepThink、CodeReview 等少数角色需要独立触发。

### 2. 别名角色的必要性存疑

`DesignDiscovery` 和 `DirectionDiscovery` 是 DeepThink 的历史别名。保留它们增加了索引复杂度但没有额外价值。

**建议**：迁移到只保留 DeepThink，在别名文件中添加重定向说明。

### 3. Implementer 没有独立 Skill

Implementer 是唯一没有独立 Skill 的角色，因为它只在工作流中使用。但这意味着用户无法单独触发 Implementer（比如"只帮我实现这个小功能"）。

**建议**：考虑在某些场景下允许独立触发 Implementer（需要 Planner 的最小计划）。
