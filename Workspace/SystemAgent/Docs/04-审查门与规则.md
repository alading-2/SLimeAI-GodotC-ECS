# 审查门与规则详解

## Review Gates（审查门）

### 设计原则

审查门是 SystemAgent 的质量保障机制。每个门有 5 个字段：

| 字段 | 含义 |
|------|------|
| Trigger | 什么时候触发这个门 |
| Context | 需要读什么上下文 |
| Prompt | 给 Reviewer 的具体审查提示词 |
| Verdicts | 判定词汇（APPROVE/CONCERNS/REJECT） |
| Special | 特殊处理（如 Gate 之间的聚合规则） |

### 8 个审查门详解

#### RV-PLAN-FEASIBILITY（计划可行性）

**触发**：Plan 阶段完成后，Execute 开始前
**审查内容**：
- 任务粒度是否合理（不能太大也不能太小）
- 路径引用是否真实存在
- 能力是否对齐（是否超出 AI 能力范围）
- SDD 推荐是否合理

#### RV-TEST-COVERAGE（测试覆盖）

**触发**：Execute 阶段，每次 TDD 微循环后
**审查内容**：
- 每个新增/修改的公共方法是否有对应测试
- RED 证据是否来自"缺少行为"（不是编译错误）
- GREEN 证据是否引用了同一个测试/check 输出
- 是否有"太简单不测"的方法被跳过

**这是 TDD 的关键执行点。** 如果这个门被跳过，TDD 就形同虚设。

#### RV-IMPL-BOUNDARY（实现边界）

**触发**：Execute 阶段完成后
**审查内容**：
- 修改范围是否在计划内
- 是否修改了同步副本
- 是否引入了不必要的依赖
- 是否违反了 ECS 架构红线

#### RV-INTEGRATION-BEHAVIOR（集成行为）

**触发**：多系统交互的实现完成后
**审查内容**：
- 跨系统状态冲突
- 游戏生命周期 FeatureSpec / BDD
- GodotPitfalls（已知的 Godot 陷阱）

#### RV-DOC-SYNC（文档同步）

**触发**：Validate 阶段
**审查内容**：
- README/INDEX/DocsAI/SDD 路径一致性
- 改了代码是否更新了对应文档
- 入口链是否仍然有效

#### RV-CONFIG-SYNC（配置同步）

**触发**：修改了 `.ai-config/` 后
**审查内容**：
- 是否运行了同步脚本
- Lint 是否通过
- Smoke 测试是否通过

#### RV-BEHAVIOR-COMPLIANCE（行为合规）

**触发**：WorkflowIteration 和 Retrospective 中
**审查内容**：
- AI 是否遵守了 must-read 入口
- 角色激活顺序是否正确
- 是否跳过了门

#### RV-RETROSPECTIVE（复盘）

**触发**：所有工作流的最后阶段
**审查内容**：
- 复盘是否完整
- SDD progress 是否对齐
- sessionEvidence 是否充分

### 判定词汇（VerdictVocabulary）

使用严格的 3 级判定：

| 判定 | 含义 | 聚合规则 |
|------|------|----------|
| APPROVE | 通过 | — |
| CONCERNS | 有顾虑但不阻塞 | 可被 REJECT 覆盖 |
| REJECT | 不通过 | 最高优先级 |

**聚合规则**：REJECT > CONCERNS > APPROVE。任何 REJECT 都阻塞流程。

## Code Review 与 TDD 的边界

Code Review 不是 TDD 的重复，也不是只看测试文件。它的第一问题是：**代码是否按用户需求、FeatureSpec、SDD 设计或 BDD 场景实现了功能。**

最小审查维度：

1. 功能/设计实现度：需求、完成定义、FeatureSpec / BDD 关键行为是否真的落到生产代码。
2. ECS 架构红线：是否违反 Entity 生命周期、DataKey、EventBus、资源加载、Timer、TargetSelector、DamageService、对象池等硬约束。
3. 修改范围和依赖边界：是否越界、引入不必要依赖或跨 git 边界混改。
4. 可维护性底线：只抓明显影响维护的命名、重复、复杂度问题。
5. 热路径性能和日志噪音：高频路径中是否新增分配、LINQ、过量日志。
6. 失败语义和错误处理：是否吞错、伪造成功或用默认值掩盖失败。

测试结果、ValidationSession artifact 和构建输出可以作为 evidenceRef，但不能替代读代码。缺测试通常交给 `RV-TEST-COVERAGE` 或记录为证据缺口；只有测试失败暴露实现错误，或没有证据导致无法判断功能是否实现时，才影响 Code Review 的 `DONE/PARTIAL/MISSING/WRONG` 判定。

Hook 未来可以在验证后或收尾前提醒运行 Code Review，但 Hook 不直接审查代码。

## Rules（规则）

### 10 个规则概览

| 规则 | 核心要求 | 违反后果 |
|------|----------|----------|
| **Philosophy** | AI-first、小内核、无遗留兼容 | 方向偏离 |
| **Boundary** | 框架/游戏边界、归属树 | 跨界污染 |
| **ReviewGates** | 8 个门的完整执行 | 质量遗漏 |
| **VerdictVocabulary** | 3 级判定词汇 | 判定混乱 |
| **Git** | Git 边界、worktree、允许/禁止操作 | 数据丢失 |
| **Subagent** | 只读 subagent、结构化输出 | 写冲突 |
| **AIConfig** | 事实源边界、同步机制 | 配置漂移 |
| **TDD** | RED-GREEN-REFACTOR 微循环 | 测试缺失 |
| **Documentation** | 旧路径分类、事实源层级 | 信息误导 |
| **DesignDocument** | 保留用户原始问题、问题优先 | 遗忘初衷 |

### 关键规则深入

#### TDD 规则的核心

```
铁律：没有失败测试，就没有产品代码。

RED：写一个测试，确认它失败（正确原因：缺少功能）
GREEN：写最少代码，确认测试通过
REFACTOR：消除重复，改善命名

禁止行为：
- 先写代码再补测试
- 测试从一开始就通过
- "太简单不用测"
- 只有手动测试
```

#### Git 规则的核心

```
每个仓库是独立的 Git 边界。
操作前必须 git status --short。
worktree 策略已写但未激活。
禁止：force push, reset --hard, clean -fd, 历史改写
```

#### Subagent 规则的核心

```
Subagent 默认只读。
输出必须结构化（7 字段）。
写操作由主对话执行。
没有 Dispatcher（并行写未实现）。
```
