# systemagent-code-review

## 触发条件

- SDD task 实现完成后、声称完成前。
- 用户要求"审查代码"、"检查实现是否符合需求"、"总结为什么功能没实现"。
- Retrospective 发现实现偏差需要独立审查。

## 必读

- SDD design 文档（需求 / 完成定义 / 数据契约 / 目标架构）
- 当前实现代码（git diff 或指定文件）
- 相关 owner skill / DocsAI（如涉及特定模块）

## 流程

### Phase 1: 提取需求清单

从 SDD design 文档中提取所有可审查的需求项。每项必须是具体、可验证的陈述，不能是模糊的"能"字句。

提取规则：

- 完成定义（Definition of Done）的每条 → 独立需求项
- 目标架构中的关键契约（字段、接口、行为）→ 独立需求项
- "不采用"列表中的每条 → 独立需求项（应验证确实没做）
- 明确的裁决（如"默认关闭 GodotEditorSink"）→ 独立需求项

输出格式：

```markdown
| # | 来源 | 需求项 | 验证方式 |
|---|------|--------|----------|
| 1 | 完成定义.1 | 业务日志可通过结构化 envelope 输出 | 检查 LogEntry 字段完整性 |
| 2 | 完成定义.2 | CLI 能临时打开、关闭、查询和回写规则 | 检查 logctl 命令 |
```

### Phase 2: 逐条审查

对每条需求项，执行：

1. **定位实现代码**：找到对应的代码文件和函数。
2. **判断状态**：
   - `DONE` — 实现完整，符合需求。
   - `PARTIAL` — 实现存在但不完整或质量不足。
   - `MISSING` — 完全没有实现。
   - `WRONG` — 实现了但方向错误或违反需求。
3. **收集证据**：引用具体文件:行号、命令输出、artifact 内容。
4. **分析根因**（仅对 PARTIAL/MISSING/WRONG）：
   - 需求理解偏差：AI 误解了需求的含义。
   - 验收口径过低：验证命令只检查存在性，不检查质量。
   - 概念边界模糊：如 flow 和 operation 没有明确区分。
   - 技术约束阻塞：如 Godot API 限制。
   - 遗漏：需求存在但执行时被跳过。

### Phase 3: 输出报告

```markdown
# 代码审查报告

## 概览
- 需求总数：N
- DONE：X | PARTIAL：Y | MISSING：Z | WRONG：W
- 根因分布：需求理解偏差=A, 验收口径过低=B, 概念边界=C, 技术阻塞=D, 遗漏=E

## 逐条审查

### ✅ DONE
| # | 需求项 | 证据 |
|---|--------|------|
| 1 | ... | file:line, 命令输出 |

### ⚠️ PARTIAL
| # | 需求项 | 当前状态 | 差距 | 根因 | 修复建议 |
|---|--------|----------|------|------|----------|
| 3 | ... | 已有 X，缺 Y | ... | 验收口径过低 | 补 Z |

### ❌ MISSING
| # | 需求项 | 根因 | 修复建议 |
|---|--------|------|----------|
| 5 | ... | 遗漏 | ... |

### 🚫 WRONG
| # | 需求项 | 实际实现 | 期望行为 | 根因 | 修复建议 |
|---|--------|----------|----------|------|----------|
| 7 | ... | ... | ... | 概念边界模糊 | ... |

## 根因分析

### 根因 1: [名称]
- 影响的需求项：#3, #5, #7
- 具体表现：...
- 为什么发生：...
- 建议修复方式：...

## 修复优先级

1. [最高影响] ...
2. ...
```

## 禁止

- 不写实现代码；只审查和报告。
- 不接受"应该会过"作为 DONE 证据；必须引用具体代码或命令输出。
- 不把编译通过（`dotnet build`）当作需求完成的证据。
- 不把目录/文件存在当作内容合格的证据。
- 不自由发挥 verdict 文本；状态只允许 DONE / PARTIAL / MISSING / WRONG。
- 不直接修改 `.ai-config/sync-targets.json` 定义的 skill 同步副本作为源。

## 与现有 skill 的关系

- 本 skill 在 `test-designer`（设计验收标准）和 `verifier`（确认证据完整性）之间，负责**需求 vs 实现的逐条对比**。
- `reviewer` 检查流程合规（gate、边界、禁改副本）；本 skill 检查需求合规（实现是否符合设计）。
- `retrospective` 检查流程缺口；本 skill 检查实现缺口。
- 审查报告的 MISSING/WRONG 项可直接转为 SDD `tasks.md` 的 follow-up 任务。
